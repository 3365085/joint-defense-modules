# 运行资产数据库与 B 模块输出闭环记录

日期：2026-06-01

## 背景

用户提出净化模型、加速导出模型、扫描/净化报告、检测证据等“会产生输出的地方”需要按业务分类统一管理，避免散落在 `model/runtime` 下难以复现或清理。此前项目已有 `runtime_catalog.sqlite3` 产物目录雏形，但 B 模块和 A 模块证据的展示、筛选、落点仍存在不完整处。

## 当前代码链路依据

- 统一产物数据库由 `model/src/defense/runtime/catalog.py` 维护，核心函数为 `register_artifact()` 和 `list_artifacts()`，数据库路径为 `model/runtime/db/runtime_catalog.sqlite3`。
- B 模块通过 `model/src/defense/model_security/service.py` 的 `_register_output()` 写入 `business_domain="model_security"`，类别包含 `scan_report`、`purification_report`、`purified_model`、`accelerated_model`。
- A 模块证据由 `model/src/defense/runtime/evidence.py` 的 `_index_evidence_event()` 写入 SQLite 事件索引，并把 `event_clip`、`representative_frame` 登记为 `business_domain="module_a"`、`category="evidence"`。
- Web API `model/src/defense/web/fastapi_app.py` 的 `/api/runtime/artifacts` 现在可按 `business_domain` 和 `category` 查询统一 catalog。
- 安全中心页面 `model/src/defense/web/static/model_security.html` 读取 `/api/runtime/artifacts`，用于集中展示扫描报告、净化报告、可信模型、加速模型和检测证据。

## 本次结论

1. 不需要另建一套数据库；当前更稳妥的方案是增强既有 `runtime_catalog.sqlite3`，让它成为运行期输出资产目录。
2. 白名单仍然是安全准入根凭据，继续由 `trusted_registry.json` + `trusted_registry.seal.json` 管理；runtime catalog 只管理产物索引，不替代白名单。
3. 净化成功后的 PT 模型继续按用户规则放在源模型同目录，并带“净化完毕”后缀；加速导出的 engine/onnx 统一放在 `model/runtime/model_security/exports`，并登记 catalog、写入白名单。
4. 证据预览的 `clip.mp4` 和代表帧属于 A 模块证据，必须进入同一个 `model/runtime/db/runtime_catalog.sqlite3`，不能因测试或自定义 evidence root 污染项目 catalog。

## 影响范围

- B 模块安全中心会看到更多业务分类产物，包括 `model_security` 和 `module_a`。
- `/api/runtime/artifacts?business_domain=module_a` 可独立查询检测证据；`business_domain=model_security` 可独立查询 B 模块报告和模型产物。
- 日志清空现在真正清空日志文件，不再立即写入一条“清空日志”记录，避免用户看到“清空后仍有日志”的误导。
- 证据页导出按钮通过 `/api/evidence/file?inline=false` 下载视频片段，预览仍使用 inline 媒体 URL。

## 待后续完善

- runtime catalog 当前只做索引和分类，还没有提供删除/归档/锁定策略；若要实现“安全、可控、被修改后失效”的产物管理，需要补产物哈希复核和批量失效接口。
- 三类 PPE 净化实验区的 triplet ASR 结果尚未成为生产 full scan 的正式放行依据，不能把实验 smoke 直接当作白名单证据。
- 加速导出依赖 Ultralytics/TensorRT 环境，导出耗时和失败原因需要在 UI 中继续细化展示。
