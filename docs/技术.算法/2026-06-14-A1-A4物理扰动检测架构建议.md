# A1-A4 物理扰动检测架构建议

日期：2026-06-14

## 背景

当前 A1-A4 模块的核心矛盾不是单个阈值偏高或偏低，而是“底层特征证据”和“对外报警结论”耦合过重。正常视频中常见的相机移动、自动曝光、人体自然运动，也会持续产生 temporal、motion、flow 等底层 reason；如果直接把这些 reason 映射为高报警分数，就会出现正常视频全程报警。反过来，如果只靠强抑制压住误报，攻击视频虽然偶尔确认报警，但 A1-A4 的分数归因不够清楚，难以达到“每个攻击视频尽量一个模块报警值高”的展示和验收要求。

## 当前判断

当前主项目已经具备 A1 过曝、A2 纹理/时序、A3 运动/模糊/目标一致性、A3b 静态介质、A4 融合与 AlertState，但合理架构应调整为四层：

1. 证据层：只产出归一化物理证据，不直接报警。
2. 场景门控层：区分目标锚定、全局相机行为、无目标消失、自然曝光。
3. 事件层：把连续帧证据聚合成攻击事件，输出主导模块和置信度。
4. 显示层：只展示事件层认可的模块分数，避免把诊断噪声显示成攻击。

该判断来自当前代码链路：`process_pipeline.py` 中 A1-A4 顺序执行并由 `target_anchored.py` 产生 suspicious，`alert_state.py` 做滚动确认，`details` 暴露 A1-A4 特征。当前可改进点是 A4 之前缺少统一的“候选事件对象”，导致 reason_codes 同时承担诊断、融合、展示三种职责。

## 建议架构

### 1. A1：光照攻击通道

职责：只识别强光、闪光、激光、遮挡导致的极端亮暗变化。

输出：

- `a1_score`：0 到 1 的光照攻击强度。
- `a1_kind`：`flash`、`static_glare`、`blackout`、`natural_exposure`。
- `a1_evidence`：过曝比例、欠曝比例、帧间亮度跳变比例、亮暗极性、局部/全局分布。
- `a1_gate`：是否被自然曝光门控抑制。

报警原则：

- `flash` 可以快速进入事件候选，因为帧间亮度突变更接近攻击。
- `static_glare` 必须结合目标区域或强全局异常。
- `natural_exposure` 永远只作为诊断，不进入报警。

### 2. A2：纹理与时序扰动通道

职责：提供“画面发生变化”的证据，但默认不独立报警。

输出：

- `a2_score`：局部纹理/时序异常强度。
- `a2_locality`：异常是否集中在目标附近。
- `a2_persistence`：异常是否连续存在。
- `a2_context`：是全局运动、背景运动，还是目标相关变化。

报警原则：

- A2 作为辅助证据，不单独确认攻击。
- A2 与 A1/A3/A4 同时命中时，提高事件置信度。
- A2 在正常人体活动和相机运动中很容易常驻，因此不能直接映射成红色报警。

### 3. A3：目标质量退化通道

职责：识别模糊、遮挡、目标消失、检测置信度突然下降等对目标本身造成的质量退化。

输出：

- `a3_blur_score`：目标区域模糊退化。
- `a3_occlusion_score`：遮挡或目标突然缺失。
- `a3_track_score`：目标轨迹/置信度一致性下降。
- `a3_motion_score`：局部运动扰动强度。

报警原则：

- A3 必须目标锚定：目标框内、目标框附近，或刚刚可见的目标突然消失。
- 模糊类攻击以 `blur + temporal/locality` 为主导。
- 遮挡类攻击以 `target_disappear + occlusion_motion/locality` 为主导。
- 全局相机晃动只进入诊断，不直接作为攻击。

### 4. A3b：静态介质/翻拍/贴片通道

职责：识别屏幕、纸张、贴片、重放介质等“物理介质”攻击。

输出：

- `a3b_score`：静态介质置信度。
- `a3b_kind`：`screen_replay`、`paper_patch`、`static_spoof`、`unknown_media`。
- `a3b_target_related`：候选介质是否与目标绑定。
- `a3b_suppression`：是否被边界、相机运动、自然曝光抑制。

报警原则：

- 只有 target-related 的 A3b 可以直接进入事件候选。
- 背景大面积边框、字幕、画面边缘、高速相机运动必须抑制。
- 高分 bypass 可以保留，但必须在 suppression 之后，且输出主导模块为 A3b。

### 5. A4：事件融合与归因层

职责：不是简单加权求 `p_adv`，而是从 A1/A2/A3/A3b 候选中选出“主导事件”。

建议输出统一事件对象：

```text
AttackCandidate
  family: glare | blur | occlusion | visibility | patch | replay | unknown
  primary_module: A1 | A3 | A3b | A4
  score: 0..1
  display_score: 0..1
  evidence_count: int
  target_anchored: bool
  suppressed: bool
  suppression_reason: str
  reason_codes: list[str]
```

融合规则：

- A1 主导：`flash` 或强过曝/欠曝命中，显示 A1 高，A2/A3 只做辅助。
- A3 主导：模糊、遮挡、目标消失、质量下降命中，显示 A3 高。
- A3b 主导：贴片、纸张、屏幕、翻拍命中，显示 A3b 高。
- A4 主导：单模块不足但跨模块证据一致，例如弱遮挡加目标置信度下降加局部运动集中。

关键原则：

- `p_adv` 应由主导事件 score 决定，而不是由所有底层 reason 的噪声累积决定。
- `reason_codes` 分为三类：`debug_reasons`、`candidate_reasons`、`public_reasons`。
- 对外 UI 只展示 `public_reasons` 和主导模块分数。

## 分数建议

对外 A1-A4 分数应服务于验收和解释：

- 正常视频：所有显示分数应长期低于 0.45，确认报警为 0。
- 攻击视频：至少一个主导模块 display_score 应升到 0.75 以上。
- 非主导模块可以有辅助分数，但不应全部同时拉高。

建议映射：

- `glare`：A1 0.80-0.95，A4 0.65-0.85。
- `motion_blur`：A3 0.75-0.90，A2 可辅助 0.40-0.65。
- `occlusion`：A3 0.75-0.90，A4 0.65-0.85。
- `visibility_degradation`：A3 0.70-0.85，必要时 A4 0.70-0.85。
- `adv_patch`：A3b 或 A4 0.75-0.90，A2/A3 只辅助。

以上分数区间是架构目标，具体阈值需用当前攻击视频和正常视频重新标定，待实验确认。

## 状态机建议

每个 attack family 拥有独立短窗状态：

- `instant`: flash、极端 blackout 可单帧强触发。
- `short_window`: 3/5 或 2/4 用于 blur、occlusion、patch。
- `hold`: 只保持已经确认的 family，不允许新 family 被 hold 伪造。

最终 `alert_confirmed` 应来自事件状态机，而不是来自原始 suspicious 布尔值。这样可以防止正常视频中的单帧 `single_frame_suspicious` 被 UI 放大成高报警。

## 实施建议

1. 在 `module_a/types.py` 增加内部候选事件结构，或先用 dict 过渡。
2. 在 `process_pipeline.py` 中把 A1/A2/A3/A3b 原始输出整理成候选证据。
3. 新建或重构 `fusion/event_fusion.py`，负责 family 选择、主导模块归因、display_score 映射。
4. `target_anchored.py` 保留为场景门控层，但不再直接混合所有 reason_codes。
5. `detail_builders.py` 分离 `debug_reasons` 和 `public_reasons`。
6. `rebuilt_demo` 删除 `single_frame_suspicious -> p_adv=0.85` 的展示放大逻辑，改为读取主项目的事件输出。

## 验收口径

必须使用项目实际视频重新跑：

- 正常视频：`室内到室外视频.mp4`、`固定镜头室外视频.mp4`，确认报警应为 0。
- 攻击视频：`glare`、`motion_blur`、`occlusion`、`visibility_degradation`、`adv_patch`，应各有明确主导模块。
- 每个视频输出逐帧 CSV：包含 `primary_module`、`family`、`display_score`、`debug_reasons`、`public_reasons`。
- 不能只看单元测试，必须生成结果视频并抽取连续 3 秒逐帧复核。

## 结论

合理的 A1-A4 架构应从“特征直接报警”改为“证据候选到事件归因”。A1-A3b 负责各自物理证据，A4 负责事件选择和主导模块归因，AlertState 负责事件确认，UI 只展示确认后的主导模块分数。这样才能同时避免正常视频全报警，并让每个攻击视频呈现一个清晰的高分模块。
