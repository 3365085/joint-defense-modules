# 2026-06-16 实时界面误报与规则堆叠复盘

## 背景

用户反馈实时界面在正常视频上显示大量报警倾向，并且处理卡顿。截图中可见：

- 当前视频为正常固定镜头室外视频。
- `Active Alarm State` 显示 `SECURE`。
- `Candidate Source` 为 `NONE`，`Primary Channel` 为 `none`。
- 但右侧 `A4 Fusion p_adv` 显示约 `0.567`，超过当前阈值附近，界面视觉上呈现为高风险。
- 实时帧率约 `3.0 FPS`，安全模块延迟约 `217 ms`。

这说明 CSV 短窗验收与实时系统体验之间存在偏差：联合决策可能没有确认告警，但界面仍把 raw/融合分数按“攻击概率”突出展示，造成正常视频也像在报警。

## 当前问题判断

1. 规则堆叠过重。
   - `rebuilt_demo/src/module_a/detector.py` 中 `_apply_media_policy` 约 386 行，包含 29 个布尔策略分支和 27 次 `score_cap` 压分。
   - `_joint_decision` 约 441 行，也包含 29 个布尔判断。
   - 这已经偏离“清晰 A1/A2/A3 -> A4 + A3B 后处理”的设计初衷，维护性和性能均受影响。

2. p_adv 与最终告警语义混淆。
   - `p_adv` 是 A4 输出/融合分数，不等价于最终告警。
   - 最终告警应以 `alert_confirmed=True`、`candidate_source`、`primary_channel`、短窗确认共同决定。
   - 当前界面若将 `p_adv` 超阈值作为主要视觉报警，会与 `SECURE/NONE` 状态冲突。

3. 实时性能未被纳入上一轮验收。
   - 上一轮主要看 CSV、短窗视频和合同校验。
   - 没有以 Web 实时监控的 FPS、延迟、UI 告警语义作为验收指标。
   - 因此即使短窗 CSV 看起来正常，实际体验仍可能是负优化。

## 结论

本轮应认定为需要回滚/瘦身的版本，而不是继续叠加规则的版本。

建议暂停继续调参，优先做三件事：

1. 明确 UI 展示语义：
   - `p_adv` 只显示为“融合原始分/候选分”。
   - “报警”只由 `alert_confirmed` 或 `attack_state_active` 驱动。
   - 正常视频中若 `candidate_source=NONE`，界面不应呈现攻击报警视觉。

2. 减少 A3B policy 规则：
   - 将 A3B policy 从大量命名分支收敛为少数可解释条件：强屏幕证据、强纸张证据、背景平面抑制、目标相关性抑制。
   - 保留 `p_media_raw / p_media_policy`，但避免继续增加特例。

3. 做实时性能验收：
   - 用同一正常视频在 Web 端跑实时监控。
   - 记录 FPS、Module A timing、Candidate Source、Active Alarm State。
   - 只有正常视频稳定 `SECURE/NONE` 且 FPS 达到可接受范围，才继续做攻击视频调参。
