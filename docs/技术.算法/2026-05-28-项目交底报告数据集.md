## 1. 项目目标和业务场景

- 项目目标：面向智能安防监控系统，在连续视频流中同时处理物理对抗扰动、视频源异常、PPE 安全帽业务告警，以及运行模型的后门/投毒安全准入。依据：`doc/前置文档/基础文档.txt` 的“三个步骤”；代码入口为 `model/src/defense/web/fastapi_app.py::create_app()`。
- 业务场景：施工现场、仓库巡检、工地/园区监控、RTSP/HTTP 视频流、本机摄像头和 MP4 离线检测。输入源由 `model/src/defense/runtime/runner.py::sample_sources()`、`scan_camera_devices()`、`open_capture()` 一类运行时能力支撑。
- 业务输出：实时预览、检测框、模块 A 风险、A3b 翻拍/假目标状态、PPE 未戴安全帽告警、事件证据帧/视频、B 模块扫描/净化/白名单状态。状态汇总由 `model/src/defense/web/helpers.py::enrich_status()` 统一补齐。
- 边界：PPE 是业务安全提示，不等于模块 A 的物理对抗扰动；B 模块是模型生命周期安全，不进入每帧检测热路径。

## 3. A/B 模块完整检测链路

### A 模块实时链路

1. 前端调用 `/api/start`，请求体包含 `source_type`、`source`、`profile`、`feature_options`、`custom_model`。
2. `fastapi_app.py::start()` 先执行 B 模块准入；准入通过后调用 `MonitorEngine.start()`。
3. `MonitorEngine._backend_capture_loop()` 采集最新帧；预览和检测解耦。
4. `MonitorEngine._backend_process_loop()` 创建 `FrameProcessor` 和 `EvidenceSession`，按 latest-only 策略处理检测帧。
5. `FrameProcessor.process()` 调用 `VideoDefensePipeline.process_frame()`，取得 YOLO 检测、Module A 结果、PPE 状态和 JPEG 预览。
6. `MonitorEngine._after_processed()` 写入状态、overlay、最近事件和预览帧。
7. 前端通过 MJPEG、WebSocket 和 `/api/status` 更新画面与状态。

### B 模块准入链路

1. `ModelSecurityService._config_and_fingerprint()` 读取 runtime config 并构建模型 fingerprint。
2. `ModelSecurityService.status()` 查询白名单和完整性状态，输出 `admission_status`、`can_scan`、`can_purify`、`whitelist_hit`。
3. 未知模型进入 `blocked_scan_required`；`/api/start` 会阻断 A 模块并触发 background full scan。
4. `ModelSecurityService.scan()` 只允许 full scan 的 `clean/trusted` 成为白名单依据；quick scan 只作诊断。
5. full scan 为 `suspicious` 时进入 `ModelSecurityService.purify()`，由 `model_security/purifier.py::run_new_purification()` 生成候选净化模型。
6. 净化候选复扫为 `clean/trusted` 后，`ModelTrustRegistry.mark_trusted()` 写入白名单。
7. 后续命中白名单时，A 模块可直接启动；白名单损坏、签名不匹配或主机指纹不匹配时，`trust_store_ok=false`，A 模块不可正常启动。

## 5. 模型加载、推理、后处理、告警、证据记录流程

- 模型格式到后端：`model/src/defense/runtime/config.py::infer_backend_from_model_path()` 规定 `.engine -> tensorrt`、`.onnx -> onnx`、`.pt/.pth -> pytorch`；`apply_custom_model()` 将结果写入 `inference.backend` 和 `inference.artifacts`。
- 模型族：`infer_model_family_from_model_path()` 通过路径、PT/ONNX/TensorRT 输出形态推断 `ultralytics/yolov5`，不能确认时默认 `ultralytics`。
- 推理：`UltralyticsDetectorBackend.predict()` 使用 `confidence` 或 `candidate_confidence` 调 YOLO；`YoloV5DetectorBackend` 支持官方 YOLOv5 路径。
- PPE 后处理：`ppe_postprocess.py::summarize_ppe_from_detections()` 以 head/helmet 为主导，person 只作上下文；`ppe_tracking.py::PPEDisplayTracker.update()` 做时序持框、弱证据晋升和显示稳定。
- PPE 业务告警：`ppe_business.py::evaluate_ppe_business()` 组合单帧 PPE、时序 tracker 和 `SafetyHelmetState`，输出 `ppe_warning`、`ppe_confirmed`、`ppe_event_active`。
- A3b：`runtime/a3b_soft_trigger.py::A3BSoftTriggerState.update()` 做 observing/suspect/confirmed 软状态，A3b 灵敏度通过 `runtime/config.py::apply_a3b_sensitivity()` 调整阈值和检测间隔。
- 证据：`EvidenceSession.update()` 在 `module_a/ppe/a3b` 通道活跃时保存关键帧；`_finalize()` 写 `event.json`、`events.jsonl`、代表帧、采样视频 `clip.mp4` 和 Web 预览 URL。

## 7. 核心算法、规则、状态机、调度、缓存和安全准入

### Module A

- A1：过曝/光照异常，来自 `module_a/features/overexposure.py`。
- A2：LBP 纹理与时序纹理，来自 `module_a/features/temporal_texture.py`。
- A3：运动伪影、模糊、轨迹一致性、轻光流，来自 `module_a/features/*`。
- A3b：静态媒介/翻拍/假目标，核心为 `module_a/features/a3/static_media/detector.py` 和 `runtime/a3b_soft_trigger.py`。
- A4：规则/分类器融合，当前在 `module_a/process_pipeline.py`、`module_a/fusion/*`、`module_a/detail_builders.py` 中组织。

### Runtime 调度

- 预览与检测解耦：`doc/docs/architecture.md` 的 Performance contract；实现为 `MonitorEngine._backend_capture_loop()`、`_backend_process_loop()`、`_preview_render_loop()`。
- latest-only backpressure：`model/configs/module_a_runtime.yaml` 的 `capture_queue_size: 1`、`result_queue_size: 1`、`detector_queue_policy: latest_only`。
- Pipeline 缓存：`PipelineCache.get()` 将 profile、模型、安全帽候选阈值、A3b 灵敏度纳入 key，避免错复用。

### PPE

- 主判定已从 `person_count - helmet_count` 改为 head/helmet 主导；字段兼容保留 `ppe_person_count`，但其语义变为 person 上下文数量。
- 二类 head/helmet-only 模型使用双阈值：`business_min_confidence=0.25`、`temporal_candidate_min_confidence=0.18`。
- 快速报警：`fast_alert_window=3`、`fast_alert_trigger_count=2`、`fast_alert_min_head_confidence=0.65`，仅正式高置信裸头走快响。

### B 模块

- 白名单只允许系统写入，用户只能删除或清空；实现为 `ModelSecurityService.delete_trust()`、`clear_trust()`。
- 白名单完整性由 `model_security/integrity.py::verify_trust_store()` 校验；状态字段包括 `trust_store_ok`、`trust_store_reason`、`host_fingerprint_status`。
- `.engine` 不作为神经元级扫描/净化对象，安全依据是同源或净化后的 PT；`ModelSecurityService._source_pt_candidates()` 负责解析源 PT。

## 9. 已做过的检测算法改进和性能优化

- 预览/检测解耦：避免检测慢时画面卡住，依据 `doc/docs/architecture.md` 和 `runner.py` 三线程结构。
- 文件视频结束态修复：`wait_latest_jpeg()`、`mjpeg_generator()` 和前端生命周期避免最后一帧滞留，记录见 `docs/技术.算法/2026-05-27-文件视频结束态与预览残留问题.md`。
- source_epoch 保护：seek/stop 后丢弃旧 epoch 结果，避免迟到检测污染状态，记录见 `2026-05-27-检测链路体检与运行边界.md`。
- PPE head/helmet 主导：兼容无 person 输出模型，记录见 `docs/技术.算法/2026-05-27-head-helmet-only时序候选探索.md`。
- PPE 双阈值候选：二类模型 raw 空窗可恢复样本上最长稳定断窗降到约 4 帧，速度基本不变。
- ROI 复检禁用在实时路径：避免首次 crop 推理耗时约 1.4s 导致 overlay 空窗，记录见 `2026-05-27-PPE-ROI复检导致实时检测中断.md`。
- A3b 灵敏度：四档 preset 通过 `apply_a3b_sensitivity()` 同时调阈值和采样间隔；UI 需明确它降低触发门槛，不提高模型原始置信度。
- B 闭环：full scan -> suspicious -> 自动净化 -> 复扫 -> 白名单写入，记录见 `2026-05-27-B模块闭环与攻击视频验证问题.md`。

## 13. 后续交底材料建议结构

1. 项目缘起：智能安防监控下的物理对抗和模型后门双风险。
2. 实验方法重置：攻击有效性、检测有效性、净化有效性分开评估。
3. A 模块：A1/A2/A3/A3b/A4，从实验到实时工程化。
4. Web/runtime：FastAPI、三线程、latest-only、overlay 和证据。
5. PPE：head/helmet 主导、双阈值、时序告警和证据记录。
6. B 模块：攻击矩阵、full scan、净化、复扫、白名单完整性。
