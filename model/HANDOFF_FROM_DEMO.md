## 0. 一句话结论

主项目模块A（A1→A4）检测质量差，**根因不是阈值/特征还要再手调，而是前期分析没到位**：
标签噪声、按帧划分的数据泄漏、记忆而非泛化、特征天花板。demo 已把这些**逐条查清并验证修复**：
- 按场景分组CV：**AUC 0.82、留出误报 8%**（起点 0.65 / 40%）
- 系统级（12干净+10跨场景攻击视频）：**跨场景命中 10/10、干净误报 1/12**（起点 7/10、4/12）

**建议：整块采用 demo 的模块A 与方法论，而不是继续在主项目里对坏掉的 A 增量打补丁。**

## 1. 根因分析（为什么"越迭代越糟"）

A 模块从 A1 调到 A4 越调越乱的本质：**在错的指标（泄漏的高分）上、对着错的根因反复手工打补丁**，每次迭代都在错误前提上叠加。

| # | 根因 | 证据 | 影响 |
|---|---|---|---|
| 1 | **标签噪声 ~36%** | 整段视频一个标签，攻击前缀/爬升/未生效帧被误标为攻击 | **头号根因**。逐帧重标后系统级误报 4/12→0/12 |
| 2 | **数据泄漏评估** | 按帧随机划分得 AUC 0.998/0.99（相邻帧高度相关=考前看答案）| 被假象误导，越调越偏 |
| 3 | **记忆 vs 泛化** | 翻拍视频训练内 p_adv 0.93、**留出 0.14** | 训练内"看着很好"，换文件就崩 |
| 4 | **特征天花板** | A1/A2 测"被加上的纹理/结构"，对"去信号/翻拍"天然无力；干净高能场景(焊接/剧烈运动)与攻击在特征上重叠(贝叶斯地板~27%) | 调参/换模型都越不过，得靠数据+方法 |

## 2. demo 已验证的修复与方法（可直接借鉴/移植）

1. **逐帧标签**：用攻击清单(`attack_manifest.jsonl`)/合成 meta 的**逐帧时序**重标（攻击前缀→干净、爬升期丢弃）。→ 误报 4/12→0/12。**这是最高杠杆、最便宜的一步。**
2. **两支路检测**（按攻击物理本质分流）：
   - 支路A（XGBoost）：加结构型 glare/occlusion/adv_patch（A1/A2/A3 适用）
   - 支路B（rule-based 效果检测）：致盲/去信号型 motion_blur/visibility/glare致盲——测"YOLO 相对场景基线的清晰度/对比度/置信度骤降"，干净段 p_blind≈0.006、致盲段≈0.7，**良定义、几乎零误报**
3. **场景自适应基线抑制**：当前最强特征分在本场景滚动分布正常带内即抑制 → 压制干净高能场景误报
4. **数据多样性**：合成攻击生成器（5类型×多场景）+ stock 干净片下载器 → 场景组 38→134
5. **诚实评测**：StratifiedGroupKFold **按场景分组** + 留出 20% 场景；系统级以 `eval_detection_broad` 视频级召回/误报为准
6. **否决记录**（避免重走弯路）：P3(相对特征)/P4(ROI判别特征)/小CNN 均经数据验证**不值**——CNN 在 14→127 场景组下仍输 XGBoost(AUC 0.66 vs 0.82)，瓶颈是**真实攻击场景多样性**(仅~7个)，非模型。**边端约束下 XGBoost 反而比 CNN 更合适**（文档 A4 本就点名 XGBoost）。

## 3. 两套模块A 的差异与不兼容点（移植前必看）

| 维度 | 主项目模块A | demo 模块A |
|---|---|---|
| 分类器 | TorchLogistic/MLP **JSON** (`configs/module_a_classifier_universal_*.json`) | **XGBoost pkl** (`rebuilt_demo/data/a4_classifier.pkl`) |
| 特征 | `build_classifier_features` **~40维**(含 track/rule_*_triggered 等) | **20维**(A1×5/A2×5/A3×6/A3b×4) |
| 标签纪律 | 整段(疑似) | 逐帧时序(manifest) |
| 评测 | 待确认是否按场景分组 | 按场景分组CV + 留出 |
| 轻量化 | ✅ **已有**：keyframe(3)+检测复用(静止跳YOLO)+各模块节流(config profile) | ❌ 逐帧(实验需要光流连续) |
| 模块B 后门 | ✅ 已接入 | ❌ |

**重要**：`configs/module_a_runtime.yaml` 里 `a4_classifier_path` 指向 demo 的 pkl 是**死配置**——
`detector_setup.py` 实际读的是 `classifier_artifact`(JSON)，不读 `a4_classifier_path`。两套**特征维度+分类器格式都不兼容**，
不能简单互指。

## 4. 建议路线：整块替换，不要增量打补丁

坏掉的 A 继续增量手调 = 重复"越修越乱"。推荐：

1. **把 demo 的 A 检测逻辑整块移入主项目**：`detector.py` 的 A1/A2/A3/A4 + 支路B 致盲 + 场景自适应基线。
2. **保留主项目已有的**：轻量化层(keyframe/检测复用/节流)、模块B、边端路径、安全准入。
   即"用主项目的轻量化外壳，包 demo 的检测内核"。
3. **统一分类器口径**：二选一——
   - (a) 主项目改用 demo 的 20维特征 + XGBoost（轻、边端友好、文档点名）；或
   - (b) 用主项目训练管线重训，但**喂逐帧标签 + 按场景留出**（把 demo 的 manifest/relabel/分组CV 纪律带过去）。
   **推荐 (a)**：XGBoost <1ms 纯CPU，最适合 Jetson 类弱边端。
4. **移植后用留出场景评测当回归门禁**，杜绝再退化。

> 移植是真工程（特征/格式要打通），但这是**对的那种工程**，比继续打补丁强。

## 5. 工程纪律（主项目必须采纳，防再退化）

1. **任何"提升"都要有可复现的验收数字**，写进 commit 与文档；没验证不算完成。
2. **严防数据泄漏**：模型评估一律**按视频/场景分组**(StratifiedGroupKFold)，**绝不按帧随机划分**。
3. **小实验先于重型集成**：上 CNN/新特征/重训前，先用现有数据做分组CV小实验验证前提。
4. **系统级 ≠ 分类器级**：分类器 CV 好不代表系统好；以**视频级留出召回/误报**为准。
5. **训练维度 = 运行期维度**：detector 产出特征维度必须等于分类器 `n_features_in_`，改特征集同步改三处(采集/训练/detector)，否则运行期填零→预测垃圾。
6. **诚实记负结果**：方案被数据否决就写进路线图(含根因与代价)，不假装没发生。
7. **检测数值不变量**：任何提速/重构不得改变 A1/A2/A3/A4 数值结果（须对拍验证等价）。

## 6. Git 规范（主项目当前有 ~61 个未提交改动，亟需规范）

- **小步提交、一次一逻辑**：每个独立改动单独 commit，不攒大杂烩。
- **提交信息讲清"做了什么 + 为什么 + 验收数字"**，禁用 "update"/"fix bug" 这种空话。
  例：`模块A标签修复: 逐帧重标剔除36%错标, 分组CV误报0.40->0.18`
- **改动前先 `git status`/`git log` 确认基线**；每个可验证里程碑即提交，便于回退。
- **拿不准的方案先提交可用状态再试**，失败用 `git checkout HEAD -- <file>` 回退，不留半成品。
- **只提交该提交的**：源码/文档/小模型提交；大数据(特征csv/视频)、密钥**不提交**（.gitignore + 提交前 `git status --short` 核对）。
- **临时探索脚本** `_` 前缀(`_exp_*`/`_diag_*`)，验证完即删；长期工具留 `verify_*`/`eval_*`。
- 破坏性 git 操作(force push / reset --hard / 改 main 历史)一律先确认。

## 7. 数据规格与管理（搬 demo `doc/data-spec.md` 要点）

- **场景多样性 > 帧数**：分组CV 瓶颈是独立场景组数，不是帧数。100 个不同工地 ≫ 5 个工地各 2000 帧。
- **逐帧攻击时序标签**：每段记 attack_start/ramp，按帧解析 0/1/丢弃。
- **按场景留出 20%** 做诚实评测，留出场景绝不进训练。
- **manifest 单一真相源**：`dataset_manifest.csv`，加数据=加一行，下游(采集/重标/训练/评测)自动纳入。
- **当前最大缺口 = 真实攻击多样性**：攻击场景仅 ~7 个。补"真实物理补丁/强光/遮挡/屏幕翻拍"跨 50-100+ 场景（须实拍/自录，stock 站只有干净工地无攻击），是解锁更高质量(乃至 CNN)的关键。

## 10. 实用方法/技巧（demo 验证有效，建议主项目复用）

这些是 demo 这轮沉淀的"轻量好用"做法，工具都现成（在 `rebuilt_demo/tools`），可直接复用或照搬思路。

### 10.1 本地 HTML 预览素材（强烈推荐，零依赖看数据）
**场景**：要肉眼核验几十上百段视频，又不想一个个用播放器打开。
**做法**：`tools/preview_server.py` —— FastAPI + StaticFiles 起一个**缩略图墙**网页：
- 缩略图栅格（每张左上角带 **#编号**），点开弹层播放**完整视频**；
- 视频走 `StaticFiles`，**原生支持 HTTP Range** → 可拖动进度、边下边播；
- 缩略图上**画了 YOLO 检测框**（见 10.2），一眼看出内容对不对；
- 配 `tools/prune_by_index.py`：你报编号 → **dry-run 先打印"编号→文件名"确认 → `--delete` 再删**（删文件+旁车+缩略图+清单移除）。两段式，安全。
```powershell
pixi run python tools/preview_server.py --port 7880      # 浏览器开 http://<机器IP>:7880/
pixi run python tools/prune_by_index.py --ids "81,95"          # 看映射
pixi run python tools/prune_by_index.py --ids "81,95" --delete # 确认后删
```
要点：`uvicorn.run(host="0.0.0.0", ...)` 才能被同网/远程访问；无鉴权只读服务**勿做公网端口转发**。

### 10.2 用 YOLO 抽帧核验素材质量（不靠人眼也能先过滤）
**场景**：批量下载/收来的素材，先自动判定"有没有真目标、是不是无关画面"。
**做法**：`tools/inspect_clips.py` —— 每段等距抽 ~4 帧跑**与系统同一份 YOLO engine**，统计 person/helmet/head 命中，
**把检测最多那帧画框存成缩略图**，并标出"零目标"片建议剔除，出 `clean_supplement_qc.csv` 报告。
（demo 这批 96% 含目标，自动揪出 6 段无目标/超低清剔除。）

### 10.3 远程访问家里机器的数据/服务（Tailscale 子网路由）
**场景**：人在外面，要看/操作家里这台跑着服务和数据的机器。
**做法**：路由器装 Tailscale 当**子网路由器**，advertise 家里 LAN(如 `192.168.1.0/24`)；远程设备启用该路由后，
直接用**家里机器的局域网 IP** 访问其上任意服务（如上面的 `http://192.168.1.167:7880/`）。
注意：Windows 防火墙要放行对应端口入站（管理员 `New-NetFirewallRule -Direction Inbound -Action Allow -Protocol TCP -LocalPort <port>`）。

### 10.4 Web 端实时展示检测效果（演示/调试用）
**场景**：给人演示"检测在做什么"，或调试时看每帧分数。
**做法**：`run_demo.py`（FastAPI+WebSocket）渲染 HUD：
- **决断式威胁条**：`max(p_adv, p_blind)` 三段着色(绿CLEAR/琥珀SUSPECT/红ATTACK)+ θ 阈值刻度，"该高高该低低"一眼可见；
- 同屏显示 p_adv / p_blind / p_media / 各支路分 / 原因；
- **离线两阶段回放**：扫描中预览跟随进度、扫完按源 FPS 从头流畅循环（前端体验好）。

### 10.5 数据下载器（合规拉 stock 干净片）
**场景**：补干净场景多样性。
**做法**：`tools/fetch_stock_clips.py` —— 走 Pexels/Pixabay **官方 API**，多关键词×两源交替拉不同场景，
每段记来源+许可旁车，可选直接登记进 manifest。**API key 放 `data/.stock_keys.env`（gitignored，绝不进 git）**。
注意：stock 站只有干净工地，**没有"对抗攻击工地"**——攻击素材须实拍/自录。

### 10.6 长任务/后台进程管理（边跑边监控）
**场景**：采集/训练/评测动辄几十分钟。
**做法**：把长任务放后台进程跑，轮询其输出看进度/结果，而不是阻塞等待；临时实验脚本 `_` 前缀、跑完即删。
（采集 70k 帧、分组CV 调参、系统级评测都是这么跑的。）

> 共同心法：**能用一个小 HTML / 一个抽帧脚本快速"看见"数据和结果，就不要靠想象**——
> demo 这轮的多次纠偏（发现记忆陷阱、剔除坏素材、确认检测框可信）都是靠"先看见"。
