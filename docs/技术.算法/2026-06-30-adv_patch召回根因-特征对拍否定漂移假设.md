# 2026-06-30 adv_patch 召回根因：特征对拍否定"特征漂移"假设，瓶颈在联合决策抑制

## 背景

留出评测显示主项目 rebuilt 内核 adv_patch 召回 1/4（最差项）。路线图阶段1.1 的首要假设是
**特征漂移**（pkl 用 demo detector 特征训练，主项目 rebuilt 经不同 YOLO 后端/ROI provider/隔帧复用
提取的 20 维 A4 特征可能不一致，HANDOFF §5.5）。本次做定量对拍验证该假设。

## 方法（只读，未改生产代码）

同一段 `素材/物理扰动攻击视频/adv_patch/raw_attack_adversarial_patch.mp4`（前 400 帧，含全部 186 个
`attack_active` 帧，onset=30、末=283）：
- **demo 路径**：`rebuilt_demo/tools/collect_a4_features.py`（ultralytics `.pt` + demo detector，逐帧）→ demo_adv.csv
- **主项目路径**：真实 `VideoDefensePipeline`（TRT `.engine` + 主 ROI provider + 检测复用）导出

- 用**同一个** `a4_classifier.pkl` 分别给两套特征打分，并逐维 diff；用 manifest 的 `attack_active` 分层。
- 临时脚本：`_diag_a4_features_main.py` / `_diag_diff_a4.py` / `_diag_supp_check.py`（验证后删）。

## 结果

### 1. 特征基本无漂移（否定假设）
逐维 mean|Δ|（demo vs 主项目），最大者：
- a3b_flow_gap 0.107、a3b_p_media_raw 0.106、delta_h_roi_max 0.046、delta_h_target_contrast 0.046；
- 其余 <0.035，多个 ROI/光流维度（flow_target_relation 0.015、delta_h_local_max 0、change_t_local_max 0）几乎逐位一致。
→ **不同 YOLO 后端 + ROI provider + 隔帧复用并未造成有意义的 20 维特征漂移。阶段1.1 的漂移假设被数据否定。**

### 2. 分类器在主项目特征上照常触发（否定"分类器失灵"）
用 pkl 打分（θ=0.65）：
| 分层 | n | p_adv demo | p_adv 主项目 | ≥θ demo | ≥θ 主项目 |
|---|---|---|---|---|---|
| 攻击帧 | 186 | 0.872 | 0.843 | 167 | **158** |
| 干净帧(非active) | 214 | 0.835 | 0.775 | 174 | 160 |
→ 主项目路径下,adv_patch 攻击帧 **158/186 的 p_adv≥0.65**,分类器本身在触发。

### 3. 真正瓶颈：联合决策 `adv_candidate_allowed` 抑制（决定性证据）
主项目路径逐帧统计（186 攻击帧）：
- p_adv≥0.65：**158**
- `adv_candidate_allowed==1`：**仅 11**
- p_adv≥0.65 且 allowed：11
- 单帧可疑：11 → 达不到 3/5 时序确认 → **alert_confirmed：0（该视频漏报）**
- 干净帧(214)：p_adv≥0.65 达 160,但 allowed==0（全被抑制）,confirmed 0（无误报）。

→ 分类器给出高 p_adv,但 `_joint_decision` 的抑制门(场景基线/背景平面/nonlocal spike 等)把 175/186 个
高分攻击帧的 `adv_candidate_allowed` 置 False,只剩 11 帧候选,永远到不了 3/5 确认 → 漏报。
**adv_patch 1/4 的直接原因是联合决策抑制,不是特征漂移、也不是分类器。**

## 判断与影响范围

- **分类器无法区分本视频的攻击帧与干净帧**（两者 p_adv 都 ~0.8）。联合决策抑制门为控制误报(干净帧
  confirmed 0)必须否决,但同一门也否决了攻击帧 → 召回被抑制门吃掉。这正是 demo 自己文档记录的
  **"adv_patch 弱攻击与干净突发在 20 维特征上无法分开"的特征天花板**（a4-xgboost-classifier.md），
  根因是真实攻击场景多样性仅 ~7 个,非模型/非集成。
- 结论：主项目在 adv_patch 上的表现**与 demo 内核同源同码、特征一致**,瓶颈是**特征可分性天花板 +
  抑制门取舍**,不是移植引入的漂移或回归。**路线图阶段1.2a（对齐特征提取）无必要**——特征已对齐。

## 待确认（诚实标注）

## 【更正】决策对拍推翻"天花板/parity"推测：确为主项目侧回归（隔帧复用打散候选）

上一节末尾把 adv_patch 漏报归为"特征天花板 / 与 demo parity（demo 大概率也漏）"，并标注为待确认。
按用户要求跑了 **demo 自身 pipeline 的决策对拍**（`_diag_demo_decision.py`：ultralytics `.pt` + demo ROI
+ demo detector 逐帧，输出告警决策），**结论被推翻**：

### 决策对拍（同段 adv_patch 视频，186 攻击帧）
| 路径 | p_adv≥0.65 | adv_allowed | 单帧susp | **alert_confirmed** | 视频级命中 |
|---|---|---|---|---|---|
| demo | 167 | 13 | 13 | **12** | ✅命中 |
| 主项目 | 158 | 11 | 11 | **0** | ❌漏报 |

→ **判定：主项目侧回归（demo 命中、主项目漏），不是 parity、不是数据天花板。**

### 根因（候选帧时序分布，决定性）
- demo 候选帧：`[158,159,…,170]` —— **13 帧连续**；5帧窗内最多 5 命中 → 达 N-of-M 确认 → 12 帧 confirmed。
- 主项目候选帧：`[150-153] [158-161] [168-170]` —— **被打散成 3 段(4/4/3)**；5帧窗内最多仅 4 命中 → 达不到
  所需 ~5 → **0 confirmed**。
- 主项目在 **44 个攻击帧命中检测复用(reuse_hit)**（demo 每帧跑新 YOLO）。复用导致这些帧用的是**上一帧的
  陈旧 ROI**，逐帧 `adv_candidate_allowed` 因此抖动，把本应连续的候选流**打断成短段** → N-of-M 时序确认失效。

### 结论订正
- adv_patch 1/4 的直接原因 = **隔帧/检测复用外壳打散了候选帧时序,使 demo 内核赖以确认的"连续候选"变碎**,
  N-of-M(约 5/8)确认不到。这正是本项目最初标注的 **"隔帧 vs 时序连续性" 头号风险**,现由数据坐实。
- **不是**特征漂移(§1 已否定)、**不是**分类器(§2,158/186 触发)、**不是**数据天花板(demo 命中本视频)。
- 主项目距确认阈值**只差 1**(碎片最长 4 vs 需 ~5),说明**可恢复**,非硬伤。

### 修复方向（候选,均需双口径 heldout 验证后再定,尚未实施）
1. **候选状态穿透复用**:复用帧沿用上一处理帧的候选判定(carry adv candidate through reuse),不制造"候选空洞"。
2. **攻击疑似时抑制复用**:当 p_adv 抬升时强制跑新 YOLO(不复用),保候选连续——代价是这些帧多跑一次 YOLO。
3. **确认窗放宽**:N-of-M 阈值/窗对"被复用打散"更鲁棒(风险:抬高误报,需双口径)。
- 我加的 `rebuilt_alert_hold_frames`(报警保持)对本例无效:它需先有一次确认才 arm,而主项目连第一次确认都到不了。

### 仍待确认
- 上述仅单段 adv_patch;需在留出集 4 段 adv_patch + 其它类型上复核复用打散的普遍性。
- 具体修复(1/2/3)对干净误报的影响未测,**必须双口径(基准视频零误报 + heldout)验证后才改**。

## 【再更正】"隔帧复用打散候选"假设也被实验否定——真凶是抑制门在边界帧翻转

上一节把候选碎片化归因于"隔帧复用(reuse)用陈旧 ROI"。**关掉复用重跑,该假设被否定**：

- `_diag_a4_features_main.py --no-reuse`(每帧跑 YOLO,reuse_hit=0)：候选帧仍是
  `[150-153][158-161][168-170]`（与开复用时**完全一样**）、alert_confirmed 仍 **0**。→ 复用不是原因。

### 逐帧对拍(demo vs 主项目 no-reuse, 帧 148-172)
- 两侧 `n_rois` 全为 1、p_adv 都高。**差异只在 `adv_candidate_allowed` 布尔门**：
  - 帧 **162-167**：demo allow=1(p 0.97-0.99)、主项目 allow=0(p 0.68-0.89) → 主项目**抑制了本该放行的中段**。
  - 帧 **150-153**：反向,主项目 allow=1、demo allow=0。
- demo 放行区 `158-170` 连成 13 帧 → N-of-M 达标(12 confirmed)；主项目放行区被抑制门在 162-167 掐断
  → 碎成 4/4/3 → 达不到所需 ~5 → **0 confirmed**。

### 订正后的根因
adv_patch 漏报 = **联合决策 `_joint_decision` 的抑制门在边界帧上,主项目与 demo 判定不同**,把连续候选流掐断。
差异来源(**待进一步验证,未逐一坐实**)：主项目输入路径与 demo 不同——
- YOLO 后端 **TensorRT `.engine`** vs demo **ultralytics `.pt`**：同为 1 个框,但**框坐标/置信度有细微差**;
- ROI 供给 **`DetectionROIProvider`(稳定化/margin/min_conf)** vs demo **原始框 + `_map_label`**;
- 上述细微差 → 抑制门依赖的子信号(target_related / flow_target_relation / a3b_reason / 场景基线 z-score /
  曝光/光流)在边界帧翻转 → `adv_candidate_allowed` 结果不同。这些子信号**不在 20 维特征里**,故 §1 特征对拍
  看起来"基本一致"仍可与本结论并存。

### 三次假设更迭的诚实记录
1. 特征漂移 → **否定**(§1,20 维 |Δ|≤0.11)。
2. 隔帧复用打散候选 → **否定**(本节,no-reuse 仍碎、仍 0 confirmed)。
3. **当前结论**：抑制门(adv_candidate_allowed)在边界帧因输入路径细微差而翻转,掐断候选连续性 → N-of-M 差~1。

### "能不能直接用 demo" 的准确回答
- rebuilt 内核**已经是** demo 算法(默认启用)。但"效果=demo"还依赖**同样的输入喂法**(demo 的 `.pt` YOLO +
  原始 ROI + demo 自身的逐帧状态积累)。主项目为省算力/工程一致用了 `.engine` + `DetectionROIProvider`,
  在边界帧喂出的子信号不同 → 抑制门翻转 → 少数关键帧被掐 → N-of-M 差一口气。
- 因此"直接用 demo"要么(a)**连输入路径一起用 demo 的**(换 YOLO 后端为 .pt + 绕开 ROI provider,
  部分放弃主项目外壳),要么(b)**让确认/抑制对这种边界抖动更鲁棒**(主项目现在只差~1,门槛降一点或
  抑制门对 adv 类放宽即可能救回,须双口径验证误报)。
- **未确认**：上述子信号具体哪一个翻转、是否 4 段 adv_patch 普遍如此——需再抽这些边界帧的 joint_decision
  原因码逐帧核对(下一步可做)。

## 【解决】候选连续性桥接：留出集召回 71.4%→90.5%（双口径验证）

修复实现:`rebuilt/detector.py` 新增**候选连续性桥接**(`rebuilt_adv_candidate_bridge_frames`,默认0=关):
adv 候选被放行后,近 K 帧内若本帧 p_adv 仍触发,即便 `adv_candidate_allowed` 偶发翻假也桥接为候选,
修复抑制门抖动打散连续候选、N-of-M 确认失败的根因。K=6 全量留出集(27 干净+21 攻击)实测:

| 指标 | 修复前 | bridge=6 |
|---|---|---|
| 攻击召回 | 15/21 = 71.4% | **19/21 = 90.5%** |
| 干净误报 | 2/27 = 7.4% | 3/27 = 11.1% |
| adv_patch | 1/4 | **3/4** |
| glare | 3/4 | **4/4** |
| visibility | 3/4 | **4/4** |
| motion_blur / occlusion | 5/5 / 3/4 | 5/5 / 3/4 |

## 【2026-07-09 重构】持续对抗升级改为「场景自适应 + fps 归一化」：去掉固定 100/0.68 过拟合阈值

### 动机
上一节的固定阈值（长窗 window=100、req=68 / ratio=0.68）是从**当前 heldout 反推**的：干净片段长窗原始触发
峰值 60 vs 需求 68，仅 8 帧安全裕度，且 100/68 都是**帧数硬编码 → 随 fps 变化、易过拟合**。虽修好了 adv_patch
（28→234），但换 25/30/60fps 或新场景就不可靠。本次把升级判据改为**物理可解释 + 场景自适应**，不再硬编码帧数。

### 自适应设计（`_joint_decision`，最小改动、config-gated、默认开）
用两个判据取代固定长窗占比：
- **(a) 时间持续下限（fps 归一化）**：`floor = round(process_fps * sustained_seconds)`，
  `rebuilt_sustained_adv_seconds` 默认 **2.0s**。物理先验——真实物理补丁攻击持续 >~2s，干净的运动/纹理突刺更短；
  用 `process_fps` 归一化 → 25/30/60fps 通用。
- **(b) 场景自身良性突发尺度**：跟踪 `_adv_run`（当前**连续**原始 `p_adv>=theta_adv` 帧数，容忍段内 1 帧掉落，
  掉 2 帧判段结束）。当一段**未升级即结束**时，把其长度学入 `_benign_run_ref`（本场景良性突发参考，带慢衰减
  `ref=max(run_len, ref*decay)`，decay 默认 0.9）；**已升级的段冻结不污染**参考。
- **升级条件**：`_adv_run >= floor` **且** `_adv_run >= run_mult * _benign_run_ref`
  （`rebuilt_sustained_adv_run_mult` 默认 **1.6**）→ 直接升级为 adv 确认，绕过逐帧 `adv_candidate_allowed`，
  **不整体削弱该门**。天然长突发的场景自抬门槛（更少误报），突发短的场景由时间下限兜底。
- `rebuilt_sustained_adv_require_target` 默认 **False**（保留为出现干净误报时的收紧旋钮）。
- 旧 `rebuilt_sustained_adv_window` / `_ratio` 保留读取以兼容旧 config，**逻辑不再依赖**（已弃用）。

- 结论基于当前 heldout（4 段 adv_patch）+ 该单段；`run_mult`/`benign_decay` 的跨大规模真实高能干净场景鲁棒性待补验。
