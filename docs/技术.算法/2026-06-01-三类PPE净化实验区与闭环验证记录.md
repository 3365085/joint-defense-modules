# 三类 PPE 净化实验区与闭环验证记录

## 背景

用户要求在不污染主项目 `model/src` 和现有 A/B 运行链路的前提下，于 `D:\联合防御模块` 下建立独立净化实验工作区，面向 `helmet/head/person` 三类输出完成投毒训练、投毒视频合成、净化迭代、结果评估和边界判断。

## 当前实现

独立实验区位于 `D:\联合防御模块\purification_lab`。生产链路仍保持 `model/src` 为唯一生产包根，实验产物按目录隔离：

- `purification_lab/configs/attack_specs.yaml`：七类攻击族的三类 PPE 版攻击规格。
- `purification_lab/scripts/attack_specs.py`：统一 trigger、source filter、label mode、OGA/ODA/semantic ASR 判定。
- `purification_lab/scripts/build_poison_dataset.py`：按 attack spec 生成投毒训练集和 canonical image attack eval manifest。
- `purification_lab/scripts/make_poison_video.py`：按 attack spec 生成视觉 smoke 视频。
- `purification_lab/scripts/sample_video_frames.py`：抽连续 3 秒完整帧，并生成 contact sheet；已修复 OpenCV 中文路径读写失败问题。
- `purification_lab/scripts/eval_triplet.py`：输出 clean/poisoned/purified 三组报告，包含 ASR、Wilson95 上界、person 上下文率、latency、逐类 mAP 字段。
- `purification_lab/scripts/run_full_lab_cycle.py`：分阶段执行 dataset/train/purify/eval，可 `--resume`。

## 当前已确认资产

`D:\联合防御模块\素材\训练图片` 是当前三类训练/验证数据来源：

- train：4500 图，helmet 17183，head 5258，person 15796。
- val：500 图，helmet 1783，head 527，person 1664。
- 类别顺序固定为 `0=helmet, 1=head, 2=person`。

`D:\联合防御模块\model\baseline_training` 只包含权重快照，不是完整训练目录：

- YOLOv8 baseline 可由 Ultralytics 加载，作为当前三类 clean 基准。
- YOLOv5 PT 不能被当前 Ultralytics YOLOv8 loader 直接加载，只能作为运行兼容基线或另走 YOLOv5 官方链路。

## 已跑通的闭环

已跑通 `oga_visible_patch` 小闭环：

1. 生成投毒数据集：`purification_lab/datasets/poisoned/oga_visible_patch_helmet_v1`。
2. 1 epoch 投毒训练：`purification_lab/models/poisoned/oga_visible_patch_helmet_v1/weights/best.pt`。
3. weight-soup 净化候选：`purification_lab/models/purified/oga_visible_patch_helmet_v1_weight_soup/weight_soup/`。
4. 三组 smoke 评估：`purification_lab/reports/triplet_oga_visible_patch_helmet_v1_weight_soup_smoke.json`。
5. 视觉视频 smoke：`purification_lab/videos/synthetic_poisoned/smoke_oga_visible_patch_video.mp4`。
6. 连续 3 秒抽帧检查：`purification_lab/reports/frame_reviews/smoke_oga_visible_patch_0s_3s`，结论为 pass。

关键结果：

- clean baseline 在可见 patch eval 上 ASR 约 2.35%，Wilson95 上界约 8.18%。
- 1 epoch poisoned 模型 ASR 为 100%。
- 当前 alpha=0.01 weight-soup purified 模型 ASR 仍为 100%。

该结果说明“投毒训练与评估链路已经有效”，但“当前简单 weight-soup 净化不足以解除该 OGA 可见贴片攻击”。

## 与旧七类净化方案的边界

旧 B 模块七类方案可复用为：

- trigger 家族定义参考；
- ASR/Wilson/gate 报告范式；
- full scan 后再复扫候选的准入思路；
- fingerprint/hash-bound 产物管理思路。

旧方案不能直接作为三类 PPE 方案证明：

- 旧七类 strict audit 只证明特定旧模型字节和旧候选；
- 旧默认目标多为 helmet 单目标，不覆盖 person 上下文保真；
- 旧 OGA/ODA 评估口径不等于当前三类模型业务后处理口径；
- 三类 PPE 必须重新产出 clean/poisoned/purified triplet 指标。

## 已发现问题

1. Windows + CUDA + Ultralytics 在训练/评估进程退出时会出现 native access violation。实验脚本已提供 `--hard-exit`，流水线默认用于 train/eval，避免报告已写出却被错误返回码标记为失败。
2. 三模型连续 GPU 评估偶发提前崩溃；当前可用 `--skip-val` 生成 ASR smoke 报告，完整 mAP 评估建议分模型/分进程批跑。
3. 单纯 weight-soup 对当前 `oga_visible_patch` smoke 投毒模型无效，下一步要尝试 clean_finetune、冻结层组合、RNP-lite、head/helmet ROI 对抗复训和 trigger replay。

## 下一步建议

1. 对七类 attack spec 全量生成 canonical image eval，并先只跑 clean baseline ASR，确认攻击载体和评估口径。
2. 对 `oga_visible_patch` 优先尝试 clean_finetune 与 trigger replay，因为 smoke 结果已经证明该攻击能稳定把 ASR 拉到 100%。
3. 对 ODA 家族单独验证“helmet 消失”的目标类召回，不能用 OGA 的目标出现口径替代。
4. 完整三类报告必须同时保留 mAP、per-class AP/precision/recall、person 上下文率、视频断框/延迟和人工帧审记录。
