1. rebuilt 内核是否产生了足够高的 A3b 原始证据；
2. 证据在哪一层被阻断；
3. “难触发”是算法策略本身，还是字段接线、异步生命周期或展示契约造成。

审计使用 CodeGraph 追踪生产默认调用链，并通过 Pixi 在真实素材上运行逐门探针。探针直接读取：

- `details.a3b`
- `details.joint_decision`
- rebuilt detector 的只读运行态

未通过 legacy `details.module_a_features.static_media` 诊断路径，避免字段丢失污染结论。

## 二、生产调用链

生产默认链路为：

1. `runtime.pipeline_factory.PipelineCache` 加载 `desktop_rtx`；
2. `pipelines.video_defense_pipeline.VideoDefensePipeline` 默认创建 rebuilt detector；
3. rebuilt 主路径每帧计算 A1/A2/A3/A4，A3b 按间隔在后台线程计算；
4. `_compute_a3b` 提取媒体平面候选并计算 raw 分数；
5. `_apply_media_policy` 执行背景、运动、目标关系和攻击归因抑制；

7. `ModuleAResult.to_info_dict()` 输出运行时结果；
8. `FrameProcessor` 把 A3b 结果交给 `A3BSoftTriggerState`；
9. status、overlay、FastAPI 和前端展示最终状态。

## 六、已确认的非阈值问题

### 6.1 rebuilt 到 runtime 的质量字段丢失

rebuilt 将 A3b 写在：

```text
details.a3b
```

legacy 消费者期望：

```text
details.module_a_features.static_media
```

`FrameProcessor._static_media_details()` 的 fallback 只映射了分数、bbox 和少量状态，未映射：

- `p_media_scores`
- `p_media_candidate_count`
- `p_media_strong_evidence`
- Policy/suppression 诊断

而 `A3BSoftTriggerState._debug()` 的 quality gate 正依赖这些字段。

结果是 rebuilt 明明存在候选和 track，runtime 却可能看到：

```text
candidate_count = 0
track_score = 0
quality_gate_passed = false
```

这是确认的接线缺陷，不是算法阈值问题。

### 6.2 pipeline A3b bbox suppression 仍只读 legacy 字段

`VideoDefensePipeline._apply_a3b_suppression()` 只读取 legacy static media 路径和 `static_image_triggered`。

生产默认 rebuilt 确认后，这条框抑制链可能不生效。该问题会造成：

- A3b 已确认；
- 假视频区域内 person/helmet/head 框仍显示；
- 用户误以为 A3b 没有触发或没有接管。

### 6.3 诊断工具仍按 legacy 路径读取

`diagnostics.video_rows.frame_row()` 仍从 legacy 路径提取 A3b。

因此 `a3b_material_audit.py`、`video_diagnostic.py` 等工具在 rebuilt 下可能把真实分数统计为 0，产生误导。

### 6.4 后台 reset 代际未隔离

当前 `reset()` 不等待旧 daemon worker，也没有 generation token。

旧 source 或 warmup worker 可以在 reset 后把旧结果写回新状态。该问题影响：

- seek；
- source switch；
- pipeline cache warmup/reset；
- restart。

### 6.5 同一后台结果被重复计票

当前主路径每帧消费最近一次 `_a3b_bg_result`，但 A3b 只按间隔产生新结果。

同一结果会被重复用于：

- `_media_run`
- `_media_run_gap`
- `media_hits`
- media N-of-M

影响是双向的：

- 同一正结果重复加票；
- 同一负结果重复扣票，并可能在下一独立结果到来前耗尽 gap tolerance。

当前 `15` 和 `3` 的单位实际是前台处理帧，不是独立 A3b 观测。该语义问题已经确认，但修复会改变时序策略，本轮只增加可观测性，不直接改变计票方式。

### 6.6 公共结果字段可能互相矛盾

media-only confirmed 可以形成：

```text
alert_confirmed = true
attack_state_active = true
attack_detected = false
is_attack = false
layer_triggered = NORMAL
```

这会让不同 API/UI 区域对同一 A3b 状态给出不同表达。

### 6.7 Web 活动事件对象被错误嵌套

前端把 active A3b event 先构造成数组，再作为单个元素插入事件数组，形成嵌套数组。后续读取 `event_id`、`peak_a3b_score` 等字段时会得到空值。

## 七、修复分类与行为边界

本轮改动必须区分“契约/生命周期修复”和“行为调优”，不能把所有提前触发都描述为纯接线恢复。

### 7.1 契约、接线和生命周期修复

- 新增统一 rebuilt/legacy A3b result adapter，保留 candidate、scores、track、Policy、suppression、bbox、worker health 等字段；
- `FrameProcessor`、`VideoDefensePipeline` 和 diagnostics 使用同一 A3b 字段契约；
- legacy authoritative trigger 保持原有权威性，rebuilt authoritative trigger 仍只认严格内核 `media_confirmed`；
- pipeline 在 rebuilt A3b 活动期使用同一 effective bbox 刷新 PPE suppression；
- rebuilt 后台 worker 增加 generation 隔离、结果序号、源帧、时间、错误与 retired-worker 诊断；
- 新一轮后台计算失败时立即失效上一轮成功缓存，保留 last-success/source/error 诊断；下游已有 hold 只提供有界宽限，不再允许旧 confirmed bbox 在持续异常期间无限续期；
- A3b worker 增加独立 `3s` monotonic timeout；超时 generation 失效，晚到 worker 只能释放 token，不能发布；
- 成功结果增加独立 `5s` monotonic lease 与 `a3b_result_fresh`；lease 到期清除权威 payload，rebuilt pipeline 只有 fresh 结果才能在 hold 到期时续租；
- 进程级 A3b live-worker token 硬上限为 `2`，跨 detector/PipelineCache 重建仍生效；同一 detector 的 retired worker 上限同为 `2`；
- copy/deepcopy、compute、结果 normalization 和发布准备统一进入原子失败路径；发布失败回滚 seq/last-success/media-track，并失效旧 payload，不会把旧结果伪装成 fresh/new-seq；
- `reset()` 不允许旧 source worker 写回新状态，`close()` 在有界时间内先关闭 detector 再关闭 backend；
- 公共 attack 字段在 authoritative attack 活动时保持一致；
- Web active A3b event 修复为扁平事件对象，不再生成嵌套数组；
- status 暴露实际 effective runtime config，避免只看 YAML 文本误判；
- 公共 status/a3b_debug 补齐 active/retired worker、last-attempt frame/time，effective config 补齐 `static_image_enabled` 与 `rebuilt_a3b_independent_trigger`。

### 7.2 明确属于 behavior-tuning 的变化

以下变化虽然没有修改 rebuilt 严格内核阈值，但会改变 runtime A3b 的触发时间、持续时长或显示框选择，因此应按 behavior-tuning 验证：

- rebuilt 质量字段恢复后，source-keyword 允许范围内的 soft trigger 可以重新使用 quality/window/track 证据，而不再只靠稀疏单帧强分支；
- soft state 每个处理帧只更新一次，并在 PPE business 逻辑前完成，使告警与 suppression 使用同一状态快照；
- active/hold 期间保留可信 bbox；低质量或几何突变候选不能立即接管；
- 新位置需要 3 个独立高质量 `a3b_result_seq` 结果才能接管；
- 几何突变除 IoU、面积和中心变化外，增加宽高比剧变检查；
- 若新框几乎包含旧可信框、但面积瞬时扩张到至少 `1.45` 倍，则按 outward expansion 进入 3 个独立结果 pending；后续收缩回更精确内框仍可立即接受；
- contained contraction 的快速接受要求新内框至少 `85%` 被旧外框覆盖，且宽、高分别不小于旧框的 `30%`；合理内框可立即恢复，细条内框仍走 3-result pending；
- 长静默、source switch、reset 或可信结果超龄后清理旧框。

因此本轮不能只以“字段接线修复”概括。严格 rebuilt 内核阈值未放宽，但 runtime soft-trigger 与框生命周期确实发生了有意、受门禁约束的行为变化。

### 7.3 本轮明确未改

- candidate threshold、edge 区间、border contrast、media run floor/gap；
- rebuilt Policy、tighten、N-of-M 和同一缓存结果重复参与计票的现有语义；
- source keyword 允许范围；
- 模型权重、类别语义和 PPE 语义。

- 后续若调整严格内核门控或扩大 source gate，必须先补独立 A3b 正样本 heldout。

## 九、最终跨改动一致性审查

截至本记录更新，主线程和独立子 agent 均已通过 CodeGraph 复核当前磁盘源码与影响面。当前确认：

1. legacy authoritative trigger 与 rebuilt strict `media_confirmed` 分流保留；
2. runtime、pipeline suppression 和 diagnostics 共享统一 A3b adapter；
3. backend worker 错误、active/retired worker、last attempt、result seq、source frame/time、generation 和 effective config 在公共 status 中可见；
4. source switch/reset/close 不允许旧 worker 或旧可信框污染新 source；
5. confirmed 活动期新独立结果可刷新 suppression，同一结果不会每个预览帧重复重置；
6. 新后台尝试失败会使上一成功缓存失效，旧 confirmed 不能再借 pipeline expiry refresh 无限续期；正常成功结果间仍由现有 hold 跨越短暂发布间隔；
7. target preview、全量 heldout 和自动化测试使用 production-default 路径；
8. 当前仍保留“同一缓存 A3b 结果重复参与严格内核 run/N-of-M”的既有语义，该项若修复会改变时序策略，必须另立 behavior-tuning 任务；
9. 永久挂起 worker 由 `3s` monotonic timeout 失效 generation；晚到结果不能发布，并由 process-global token hard limit `2` 防止跨 PipelineCache 重建无限积累；
10. 成功结果使用独立 `5s` monotonic lease；rebuilt suppression 的 hold-expiring 续租必须显式 `result_fresh=true`，缺失 freshness 的同一缓存不能无限续租；
11. worker 发布业务完成与 active 清理在同一锁内完成，关闭 success-vs-timeout 竞态；normalization/发布准备异常走原子失败与回滚路径；
12. `MonitorEngine._empty_status()` 已预置完整 health/effective-config schema，初始化、预热、首帧前、seek/reset 与首帧后字段集合一致；
13. timeout/rejected/result-expired 计数定义为当前 source/run 口径，`reset()`/`close()` 清零；schedule-blocked 独立按状态转换记录 warning。

## 十、当前结论

1. 用户反馈成立：原 rebuilt 严格内核在目标视频第 257 帧才首次确认，用户感知上明显过晚。
2. 问题不能简化为单一阈值；目标视频和 heldout 干净硬负样本在 candidate/edge 上存在重叠，直接放宽会增加误报风险。
3. 已确认的主要工程根因包括 rebuilt 字段契约丢失、suppression/diagnostics legacy-only、后台代际污染风险、错误不可见、公共字段与 Web 事件结构不一致。
4. runtime soft trigger 的恢复与可信 bbox 收口属于明确 behavior-tuning；虽然严格内核阈值未改，仍必须以目标视频和 heldout 门禁验收。
5. 最终 v9 在实际 MonitorEngine latest-only 预览中约 0.53 秒进入 A3b，原问题区间未再观察到细条、横向大框、包含墙面的外扩框接管或内层 PPE 框重现。
6. 48 个 heldout clip 中 clean A3b FP、attack wrong-channel、backend error 均为 0，但因缺少 A3b heldout 正样本，不能据此宣称泛化召回已解决。
7. 当前提前触发依赖现有 source keyword：普通摄像头、RTSP 或路径不含允许关键词的文件，不会自动获得同等 soft-trigger 救援。是否扩大来源范围必须另行决策。
8. 后台异常、发布异常、永久挂起、陈旧 result lease、同 detector 与跨 detector worker 上限、首帧前 status schema 等已知 lifecycle P1/P2 均已修复并经独立 reviewer 关闭。
9. 当前未发现新的 P0/P1/P2 阻塞项；仍保留的严格内核重复计票语义属于需独立正样本门禁的 behavior-tuning，不在本轮生命周期修复中擅改。
10. 当前新改动不能仅凭自动化、agent 审查或工程抽检提交，仍需用户明确最终授权。
