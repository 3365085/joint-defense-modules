# 2026-06-30 主项目 rebuilt 内核留出集诚实评测

## 背景

HANDOFF_FROM_DEMO.md §9.2 要求："在同一批留出场景上量化主项目 A 现状 vs demo A，用数字确认替换值得"。
本次按 `rebuilt_demo/data/dataset_manifest.csv` 的 **heldout 场景分组留出集**，跑【主项目 rebuilt 内核】
（经真实 `VideoDefensePipeline` / default profile），输出视频级召回与误报，作为移植后的首份诚实基线。

判据：
- 攻击段：`attack_start+ramp`（onset=10）之后任一帧 `alert_confirmed`（3/5 时序确认）→ 命中。
- 干净段：整段任一帧 `alert_confirmed` → 误报。
- 留出集：27 干净 + 21 攻击 = 48 段，26 个独立场景，每段帧上限 240。
- 评测脚本：`model/_eval_heldout_rebuilt.py`（临时脚本，验证后删）；结果 `_eval_heldout_result.json`。

## 关键发现（代码链路依据）

### 1. 干净误报口径与 demo 持平（7.4% vs ~8%）
说明移植后的负样本抑制（场景自适应基线 + target-anchored）整体生效，未发生大面积退化。

### 2. adv_patch 召回严重偏低（1/4）——最大回归点
adv_patch 是唯一**不破坏 YOLO**的攻击，按 demo 设计应由**支路A（XGBoost）**捕获。
实测 3 个漏报段 onset 后 `single_frame_suspicious` 仅 2~15 帧（180 帧中），从未达到 3/5 确认；
唯一命中的 016 段反而有 74 susp→112 confirmed。

- 代码链路：rebuilt `detector.py` 加载 `rebuilt_demo/data/a4_classifier.pkl`（实测 `XGBClassifier`，`a4_classifier_used=True`），
  支路A 走该分类器的 20 维特征。
- **首要怀疑（待实验确认）**：该 pkl 是用 **demo 自己的 detector** 产出的特征训练的；
  主项目 rebuilt 内核若特征提取与 demo 有任何漂移（HANDOFF §5.5「训练维度=运行期维度」），
  XGBoost 即在 adv_patch 上系统性失灵。**尚未直接对拍两侧 20 维特征向量，不能下定论。**

### 3. 016_multi_person 场景病态：干净狂报 + 攻击漏报
同一"多人交叉"场景：干净版误报 204/240，而其 glare/visibility 攻击版反而漏报，adv_patch 版的"命中"
（112 confirmed）与干净版的狂报高度相似。

- 判断（待实验确认）：该场景的告警**由多人交叉运动驱动的虚假触发**主导，并非目标锚定的真实攻击识别；
  即"命中"与"误报"可能是同一虚假信号，而非有效检测。需抽帧核验该场景 overlay 的触发原因码。

### 4. 支路B（致盲类）整体可用
glare/motion_blur/occlusion/visibility 合计 14/17 ≈ 82%，motion_blur 5/5。
与 HANDOFF §2.2「支路B 良定义、几乎零误报」一致。短板集中在支路A(adv_patch) 与个别病态场景。

## 影响范围

- 移植**功能链路已通**（XGBoost/RAFT-TRT/支路B/Rust 全部在线，~69fps 满足 <15ms/帧）。
- 但"达到 demo 效果"在**攻击召回维度尚未达标**：71.4% vs 自报 10/10，缺口集中在 adv_patch 与多人交叉场景。
- 干净误报维度已达标。

## 重要口径说明（避免误读）

- 本次留出集（21 攻击）与 demo 自报"10/10"所用的具体 10 段**不是同一批**，数值不可直接等同；
  这是基于 manifest heldout 的**全新独立测量**，更贴近系统真实表现。
- 本次走**真实 VideoDefensePipeline**（含跳帧/检测复用/ROI provider/A3b 抑制链），
  与 demo eval 脚本直接喂 detector 的简化路径不同；差异本身也可能贡献召回差。

## 结论与后续建议

1. **结论**：主项目 rebuilt 内核 = 干净误报达标（7.4%）、攻击召回未达标（71.4%，adv_patch 1/4 为主因）。
   "整块移植"已完成且可运行，但**尚未复现 demo 的检测效果**。
2. **下一步（按杠杆排序）**：
   - (P1) **对拍 20 维特征**：同段视频，分别用 demo detector 与主项目 rebuilt 内核导出 A4 特征，
     逐维 diff，定位 adv_patch 失灵是否源于特征漂移（HANDOFF §5.5）。这是 adv_patch 1/4 的首要假设。
   - (P2) **抽帧核验 016 场景**：看 overlay 触发原因码，确认是否多人运动驱动的虚假触发。
   - (P3) 若确认特征漂移：要么修正 rebuilt 内核特征提取与 demo 对齐，要么按 HANDOFF §4.3(b)
     用主项目内核重新采集特征 + 逐帧标签 + 分组CV 重训 pkl。
   - (P4) 将本评测固化为留出回归门禁（HANDOFF §9.4），防再退化。
3. **未确认项**：特征漂移假设、016 虚假触发假设均**待实验确认**，本文不作为定论。
