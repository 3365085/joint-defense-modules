# rebuilt_demo 实时模式与实训卡点功能核查

日期：2026-07-11

## 问题背景

用户询问当前“作业监测”是否已经是真正实时监测，以及“实训学生卡住点记录”如何实现，并要求说明近期实际完成内容。

## 代码链路依据

### 实时检测能力

- `rebuilt_demo/src/runtime/runner.py` 已实现采集、检测、异步落盘三线程。
- `offline_mode=False` 时使用 `_LatestFrameSlot`；新帧覆盖未处理旧帧，检测线程始终消费最新帧，并统计 `detect_fps` 与 `dropped_frames`。
- 检测线程按帧执行 YOLO、Module A、结果渲染和 WebSocket 广播。
- 为避免丢帧后光流跨越过大，实时检测会把连续视频帧单独传给 A2/A3 的时序比较。

### 当前页面实际启动方式

- `DemoRunner.start()` 的默认参数是 `offline_mode=True`。
- Web API 的 `StartRequest.offline_mode` 默认也是 `True`。
- 当前 `rebuilt_demo/src/web/static/index.html` 调用 `/api/start` 时只发送模型路径和视频路径，没有显式发送 `offline_mode=false`。
- 因此当前页面默认执行“离线全速扫描后按源 FPS 回放”，不是实时 latest-only 模式。
- `/api/start` 当前要求 `video_path` 是本地文件，尚未从代码确认 RTSP、摄像头或实时采集设备接入。

## 实训学生卡住点记录核查

在当前仓库中未检索到以下业务对象或链路：

- 作业、实训任务、学生、知识点、步骤进度的数据模型；
- 学生长时间无进展、重复失败、反复求助等卡点判定规则；
- 卡点事件表、相关 API、教师端查询或学生端提示；
- 卡点开始时间、持续时间、解除方式和人工标注字段。

当前系统已有的是检测证据记录：每帧写入 CSV/JSONL，包括模块分数、候选来源、确认窗口、主通道、抑制原因和解释文本。该记录属于视频攻击检测审计，不能等同于学生学习卡点记录。

## 当前判断

1. 后端具备实时检测执行模式，但当前页面默认未启用；“当前页面已经是真正实时监测”这一说法不成立。
2. 当前仅支持本地视频文件；是否已经具备真实摄像头/RTSP 在线监测能力，未能从代码中确认。
3. “实训学生卡住点记录”尚未实现，不能用 Module A 的帧级告警日志替代。

## 影响范围

- 若演示或验收口径要求实时监测，应在前端明确提供实时/离线模式选择，并在实时模式请求中发送 `offline_mode=false`。
- 若产品要求现场实时摄像头，应补视频源适配器，而不是仅修改前端参数。
- 若产品要求学生卡点分析，需要独立的学习事件模型、规则引擎、持久化和查询界面；它与 A1-A4 检测算法属于不同业务域。

## 后续建议

1. 先确认“作业监测”指视频安全作业监控，还是教学作业/实训管理，避免继续跨业务域误解。
2. 对视频实时监测增加显式模式开关，并分别验收实时延迟、检测 FPS、丢帧数和告警时延。
3. 对学生卡点功能先定义卡点事件契约：`student_id`、`training_id`、`step_id`、`knowledge_point_id`、`stuck_reason`、`started_at`、`resolved_at`、`evidence`。
4. 卡点判定规则和阈值需要结合真实教学流程制定，当前属于待需求确认，不能从现有 A 模块算法推导。
