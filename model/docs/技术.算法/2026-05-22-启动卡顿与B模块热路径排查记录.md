# 启动卡顿与 B 模块热路径排查记录

## 背景

用户反馈：服务启动后直接开始检测文件视频时，画面开头 1 到 2 秒容易卡顿或抽帧；但在页面上重新选择一次相同自定义模型后，检测流畅度明显恢复。该现象与输入像素并非稳定相关，更像模型/管线初始化路径导致的偶发冷启动抖动。

## 代码链路依据

- `src/defense/web/fastapi_app.py:create_app()` 的 `/api/start` 入口负责启动模块 A 检测。
- `src/defense/runtime/runner.py:MonitorEngine.start()` 在启动新检测前会停止旧运行并获取检测管线。
- `src/defense/runtime/pipeline_factory.py:PipelineCache.get()` 负责按 profile、backend、model artifact、custom model 等 key 复用或创建 `PipelineBundle`。
- `src/defense/model_security/service.py:ModelSecurityService.status()` 会调用 `current_fingerprint()`，而 `current_fingerprint()` 会读取运行配置并对模型 artifact 构建指纹。

## 当前判断

1. B 模块不是每帧检测热路径。
   `ModelSecurityService` 只在 Web API 状态、扫描和信任登记相关接口运行，不参与 `MonitorEngine` 的帧处理循环。

2. B 模块曾经处在开始检测热路径。
   `/api/start` 曾在启动模块 A 前调用 B 模块状态判断，并在 unknown 状态下触发后台扫描。这不会直接降低稳态 FPS，但会和模型加载、CUDA/TensorRT 初始化叠加，放大冷启动首秒卡顿。

3. 重新选择同一模型后变流畅，主要指向管线缓存命中。
   复用已加载模型后，`PipelineCache.get()` 的耗时从数百毫秒降为亚毫秒级，首帧检测耗时也显著下降。因此问题核心不是视频像素本身，而是冷启动模型加载、后端初始化、预览先行播放之间的同步问题。

4. 文件视频预览不能早于首个检测结果进入显示。
   若文件源在首个检测结果前已经开始预览，页面会先播放无检测或旧状态画面，然后等检测结果追上，肉眼表现为开头卡顿、抽帧或画面/状态不同步。

## 已实施调整

- `src/defense/web/fastapi_app.py`
  - `/api/start` 不再同步执行 B 模块状态查询或自动扫描。
  - `/api/stop` 保留当前检测管线缓存，避免停止后再次开始时无意义卸载/重载模型。
  - 新增 `POST /api/model-security/status`，让页面按当前 profile 和 custom model 查询 B 模块状态。

- `src/defense/web/static/index.html`
  - 模型安全状态刷新改为携带当前自定义模型配置。
  - 页面加载和开始检测后的 B 模块状态刷新延迟执行，避免抢占启动首帧路径。

- `src/defense/runtime/runner.py`
  - `MonitorEngine.start()` 停旧运行时默认保留管线缓存。
  - `MonitorEngine.stop()` 新增 `release_pipeline_cache` 参数，显式区分“停止当前检测”和“释放模型资源”。
  - 文件源实时预览在 `first_detection_ready` 之前进入 `waiting_first_detection`，避免检测前裸预览抢跑。

- `src/defense/runtime/pipeline_factory.py`
  - `PipelineCache.get()` 在模型 key 变化时先关闭旧 bundle，避免旧后端对象和 GPU/TensorRT 资源残留。

## 性能观测

- 冷启动首轮：`/api/start` 约 1995.6ms，`pipeline_cache_hit=false`，首帧检测约 1245.88ms。
- 缓存命中第二轮：`/api/start` 约 319.4ms，`pipeline_cache_hit=true`，首帧检测约 9.71ms。
- 缓存命中第三轮：`/api/start` 约 309.5ms，`pipeline_cache_hit=true`，首帧检测约 9.55ms。

## 影响范围

- 不改变模型权重、类别语义、阈值、PPE 主判定语义或 A3b 策略。
- 不增加主检测路径的额外 GPU 推理。
- B 模块仍可通过页面状态查询和手动扫描运行，只是不再阻塞模块 A 启动。
- 冷启动仍需要模型/CUDA/TensorRT 初始化成本；本次优化目标是避免该成本表现为视频播放抽帧和同步错乱。

## 后续建议

- 若仍希望进一步降低冷启动等待，可在服务启动后做默认模型后台预加载，但需要评估 GPU 显存占用和用户切换自定义模型时的资源策略。
- 若用户使用多个 TensorRT/PyTorch 模型来回切换，需要继续观察 key 变化时的资源释放是否足够彻底。
- 对偶发卡顿仍应记录 `pipeline_cache_hit`、`first_detection_processing_ms`、`preview_fps`、`dropped_detection_frames`、`source_fps` 和 `backend`，用于区分冷启动、视频解码和检测推理瓶颈。
