# 2026-07-11 Module A 最终验收路径判断

## 背景

项目目标从开放调试转为交付收束：按基础设计文档，模块 A 需要覆盖 P1 目标相关物理对抗纹理/穿戴补丁、P2 闪烁/强光时序扰动、P3 动态遮挡/异常流，同时在冻结负例上不瞎报警。

## 当前判断

- v1 冻结矩阵在 B 分支运动模糊收敛后，负例达到 0/16 误报。
- v1 正例仍有 4/12 漏报，但旧合成器存在合同缺陷：P1 噪声逐帧重采、干净前缀过短、部分样本目标锚定弱。
- v2 正例生成器改为长干净前缀、稳定攻击窗口、P1 固定纹理并加入穿戴抖动/形变/亮度脉动，负例继续复用 v1 冻结负例。
- v2 诊断显示，多数漏报帧 A1/A2/A3 证据强且目标相关成立，但因 A4 单帧分数或场景抑制未进入 3/5 确认窗口。

## 影响范围

本轮修改不改变模型权重、类别语义、PPE 语义或 A3b 策略；只在模块 A 联合决策中增加一个窄口径结构证据救援：目标相关、A2 时序证据强，并且 A1 或 A3 有结构佐证、且非普通运动/背景平面时，允许进入原有 3/5 确认窗口。

## 结论

结束路径应以“冻结 v2 场景矩阵 + 冻结负例 + 视觉抽帧验收”为准，而不是继续追逐旧 demo 的所有合成瑕疵。若 v2 达到 0 漏报、0 误报，再进入目标视频结果生成、随机 3 秒逐帧视觉验收和最终 review agent。

## 后续建议

- 保留 v1 报告作为旧合成器诊断，不作为最终 0/0 交付证明。
- v2 若仍漏报，应优先看结构救援命中数和负例回归，而不是继续加大合成强度。
- 最终验收 CLI 必须作为门禁使用：任一 clip 评估失败、正例漏报、负例误报、攻击前误报、攻击后滞留或严格时序前帧不完整，都应返回非零退出码。
- 若负例回归重新出现误报，应收紧结构救援到更高 A1/A2/A3 组合阈值，并记录失败类别。
