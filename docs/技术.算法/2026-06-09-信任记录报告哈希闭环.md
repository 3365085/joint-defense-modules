# 信任记录报告哈希闭环

记录时间：2026-06-09

## 问题背景

模型安全准入链路已经把 runtime artifact、source PT、class names、PPE mapping、scanner version 和报告路径写入 trust registry，但旧实现中 `report_path` 与 `purification_report_path` 只是路径记录。只要路径指向的文件被替换或篡改，白名单记录仍可能被误判为可信。

这会削弱“扫描报告、净化报告、运行时模型”之间的证据绑定，不符合当前净化与准入流程的审计要求。

## 当前判断

报告路径本身不是稳定身份。新的信任记录需要保存报告文件哈希，并在 admission/status 时复算文件哈希进行比对。

为了兼容旧记录，本轮只在记录中存在哈希字段时强制校验；新写入的 clean full scan、purified runtime 和 accelerated export 记录都会写入或继承报告哈希。

## 代码链路依据

1. `model/src/defense/model_security/registry.py`
   - `TrustRecord.report_hash`
   - `TrustRecord.purification_report_hash`
   - `ModelTrustRegistry.mark_trusted()`
2. `model/src/defense/model_security/service.py`
   - `_report_hash_for_trust()`
   - `_trusted_record_report_identity()`
   - `_trusted_record_mismatch_reason()`
   - `_trusted_record_matches()`
   - `_mark_clean_full_scan_trusted()`
   - `purify()`
   - `export_accelerated_model()`
   - `admission_status()`
3. `model/tests/test_model_security_bypass_and_metrics.py`
   - `test_trusted_record_report_hash_blocks_tampered_full_scan_report`
   - `test_trusted_purified_record_binds_scan_and_purification_report_hashes`

## 影响范围

1. clean full scan 写入白名单后，scan report 被改动会导致准入失败。
2. purified runtime 写入白名单后，净化后复扫报告或净化流程报告任一被改动都会导致准入失败。
3. accelerated export 的白名单记录会继承源可信模型的报告身份，避免导出产物脱离原始准入证据。
4. status payload 能暴露报告哈希及匹配状态，方便前端、安全页和日志定位白名单记录为何不再可信。

## 结论

报告哈希绑定已完成第一轮闭环。当前准入判断不再只依赖 `report_path`，而是把报告文件内容纳入白名单匹配。

旧白名单记录如果没有哈希字段，当前实现暂不强制阻断，避免破坏历史数据；后续如要更严格，可以增加迁移或重扫策略，把无哈希旧记录标记为需要 full scan 重新准入。

## 后续建议

1. 在模型安全页面显示 `report_hash_matches` 和 `purification_report_hash_matches`，让用户能看懂 blocked 原因。
2. 对旧 trust registry 增加迁移提示：无 report hash 的记录建议重新 full scan。
3. 在 runtime replacement 状态中继续补充 blocked source 与 selected purified runtime 的报告身份展示。
4. 保持 `/api/start` 只读准入，不在生产启动路径静默触发 scan 或 purification。
