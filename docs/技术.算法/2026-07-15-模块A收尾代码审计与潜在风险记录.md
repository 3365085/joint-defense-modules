# 模块 A 收尾代码审计与潜在风险记录

日期：2026-07-15  
范围：`model/src/defense` 全链路，重点为生产默认 `rebuilt` 模块 A、运行时 latest-only 管线、A3b、Web/overlay 契约及测试门禁。  
方式：以 CodeGraph 调用链和影响分析为主，辅以 Pixi 下的聚焦测试与只读复现脚本。未修改检测权重、类别语义、阈值、PPE 语义或 A3b 策略。

## 一、审计结论

当前不能得出“模块 A 功能设计没有问题”的结论。

架构主方向是正确的：

- 生产包根保持在 `src/defense`；
- FastAPI 是唯一 Web API；
- preview 与 detection 使用独立总线/线程；
- detection backpressure 为 latest-only；
- 运行时提供源帧直接前驱，避免简单地把上一次检测帧当作时序前驱；
- GPU 不可用时存在 CPU 回退路径。

但生产默认内核已经从 legacy 切换为 rebuilt，而若干下游仍按 legacy 字段和状态语义消费结果。审计确认存在多项可直接复现的功能/契约缺陷，且现有绿色测试主要覆盖 legacy 或 happy path，不能证明 rebuilt 生产链路完整正确。

## 二、端到端代码链路

主链路：

1. `runtime.runner.MonitorEngine` 采集源帧；
2. `runtime.backend_pipeline.DetectionBus` 只保留最新待检测帧；
3. `runtime.frame_processor.FrameProcessor.process` 调用 `VideoDefensePipeline.process_runtime_frame`；
4. `VideoDefensePipeline` 默认构造 `module_a.rebuilt.RebuiltModuleADetector`；
5. rebuilt 内核计算 A1、A2、A3、后台 A3b、A4、blind branch 和 joint decision；
6. `ModuleAResult.to_info_dict()` 转成运行时 `info`；
7. `FrameProcessor` 生成 Module A/A3b/PPE 状态；
8. `MonitorEngine` 发布 status、overlay 和 evidence；
9. FastAPI/前端读取 status、overlay 和 MJPEG preview。

关键生产默认：

- `VideoDefensePipeline.__init__` 未配置 `detector_impl` 时默认使用 `rebuilt`；
- 当前 `configs/module_a_runtime.yaml` 未显式写 `detector_impl`，因此生产实际走 rebuilt；
- 配置中的 legacy 键并不等于 rebuilt 的实际生效参数。

## 三、已确认问题

### P1-1：rebuilt 只满足 ModuleAResult 结构，不满足下游依赖的 legacy 语义契约

代码依据：

- `module_a/rebuilt/detector.py::process` 构造了 `features`，但 `details` 仅包含 `a1/a2/a3/a4/a3b/blinding/joint_decision/...`；
- `module_a/types.py::ModuleAResult.to_info_dict` 不输出 `features` 和 `roi_results`；
- `pipelines/video_defense_pipeline.py::_run_detection` 仍从 `details.module_a.p_adv_display` 和 `details.module_a_breakdown` 读取兼容字段；
- `pipelines/video_defense_pipeline.py::_apply_a3b_suppression` 仅从 `details.module_a_features.static_media` 读取 A3b；
- `runtime/frame_processor.py::_static_media_details` 为 rebuilt 增加了局部映射，但未覆盖 `p_media_scores`、candidate count、strong evidence、各抑制状态等 A3BSoftTrigger 质量门字段。

已复现：

- rebuilt `details.a3b.media_confirmed=True` 且 bbox 有效时，`_apply_a3b_suppression` 不启动，框数保持 1；
- 同等 legacy `details.module_a_features.static_media` 输入会启动抑制并把框数降为 0；
- rebuilt A3b 原始结果即使带 candidate count、track、edge、strong evidence，经过 `_static_media_details` 后这些字段丢失，A3BSoftTrigger 看到 `candidate_count=0`、`track_score=0`、`quality_gate_passed=False`。

影响：

- 生产默认 rebuilt 下，A3b 假视频区域 YOLO/PPE 框抑制链路失效；
- rebuilt 计算出的 `p_adv` 平滑显示分和 A1/A2/A3/A4 兼容特征没有进入运行时；
- Module A latency breakdown/A3b timing 可能为空或为 0；
- 诊断工具继续按 legacy 详情读取时，会把 rebuilt 的部分信号统计成 0；
- Web、证据、诊断与真实检测内核可能对同一帧给出不同语义。

结论：这是已确认的字段契约断裂，不是待实验项。

### P1-2：A3b 已确认时，公共攻击字段仍可能表达 NORMAL/未检测

代码依据：

- rebuilt 的 `media_single_frame_candidate` 单独计算；
- `single_frame_candidate` 只包含 adv 和 blind，不包含 media；
- `media_confirmed` 可以独立令 `alert_confirmed=True`；
- `ModuleAResult.to_info_dict()` 的 `layer_triggered/is_attack/attack_detected` 只取 `single_frame_suspicious`。

因此 A3b-only confirmed 可形成：

```text
media_confirmed = true
alert_confirmed = true
attack_state_active = true
single_frame_suspicious = false
is_attack = false
attack_detected = false
layer_triggered = NORMAL
```

影响：

- API、页面、证据和外部消费者如果分别读取 `attack_detected` 或 `alert_confirmed`，会得到相反结论；
- A3b 的已确认攻击可能在顶部通用 Module A 状态中仍显示正常。

结论：这是可由控制流直接推出的公共字段语义冲突。

### P1-3：TargetAnchoredAnalyzer 在 rebuilt 中被执行，但判定结果没有参与 joint decision

代码依据：

- `module_a/rebuilt/detector.py::process` 调用 `self._ta.evaluate(...)`；
- `_joint_decision` 读取 `ta_suspicious` 和 `ta_classifier_bonus`；
- 全文件仅有这两次赋值，没有任何后续消费者。

影响：

- Target-anchored 的自然曝光抑制、目标锚定异常和 classifier bonus 对 rebuilt 最终候选/确认没有直接作用；
- 相关测试即使通过，也不能证明生产默认 rebuilt 使用了该策略；
- 当前存在两个被 skip 的自然曝光/A3b 算法契约测试，且 skip 原因明确写为“待算法裁定”。

结论：这是已确认的死接线。是否删除该分析器或重新接入 joint decision 属于行为调优工作，必须用留出集重新验证。

### P1-4：A3b 后台线程在 reset/warmup/切源时可把旧结果写回新状态

代码依据：

- rebuilt `process` 每隔若干帧启动 daemon thread；
- `_run_a3b_bg` 直接把结果写入共享 `_a3b_bg_result`；
- `reset` 只把 `_a3b_bg_thread` 和 `_a3b_bg_result` 置空，不等待旧线程退出，也没有 generation/token 校验；
- `_run_a3b_bg` 对所有异常执行空 `except`。

已复现：

1. 启动一个延迟返回的 A3b 线程；
2. 在线程完成前调用 `reset()`；
3. 线程完成后 `_a3b_bg_result` 又变成 reset 前的结果。

影响：

- warmup 后第一批真实帧可能读取 warmup A3b 结果；
- seek/切源/reset 后可能出现跨源 A3b 状态污染；
- reset 后允许新 A3b 线程启动，旧、新线程可能并发修改 `media_track` 和结果；
- A3b 内部异常会被静默转换为“继续使用旧结果/空结果”，状态中无错误信息。

结论：这是已确认的生命周期竞态和错误可见性问题。

### P1-5：A3b 时序确认会把同一个后台缓存结果重复当成多次独立观测

代码依据：

- A3b 只按 `static_image_interval` 在后台产生新结果；
- 主线程每帧读取同一个 `_a3b_bg_result`；
- `_joint_decision` 每个主处理帧都按当前缓存结果增加 `_media_run`；
- 每个主处理帧都会向 `media_hits` 追加一次；
- 结果中没有 produced frame、generation、fresh/not-fresh 标记。

影响：

- 一次 A3b 正观测可在下一次后台结果到达前被重复投票多次；
- “持续 15 帧”和 N-of-M 的实际语义依赖主循环 FPS、A3b interval 和后台耗时，而不是独立 A3b 观测数；
- 后台越慢，同一缓存结果贡献的重复票数越多，可能提前确认；
- 该问题与 reset 竞态叠加时，旧结果还可能被重复计入新流。

结论：重复计数机制由代码确定；是否已造成目标视频误确认属于待实验确认。

### P1-6：生产配置展示的阈值/窗口与 rebuilt 实际生效值不一致

当前配置：

- `module_a.p_adv_threshold: 0.55`
- `module_a.alert_window: 5`
- `module_a.alert_trigger_count: 3`

rebuilt 实际读取：

- `rebuilt_theta_adv`，缺省为 `0.65`；
- `max_history=8`；
- 确认窗口按 `process_fps * 0.5` 动态计算并限制为 3～8 帧；
- 确认数按场景比例计算。

Pixi 只读探针确认：

- configured `p_adv_threshold=0.55`
- configured `rebuilt_theta_adv=None`
- effective `theta_adv=0.65`
- configured alert window/count 为 `5/3`
- effective history maxlen 为 `8`

影响：

- 运维或调试人员修改 `p_adv_threshold`，对生产默认 rebuilt 不生效；
- 配置文件无法作为实际策略的可信说明；
- 阈值审计、复现实验和版本交接容易使用错误口径。

结论：这是已确认的配置契约问题。修复时不得直接改变阈值，应先明确 canonical 配置键并保持现有实际行为。

### P1-7：rebuilt 未接入关键 enable/device/profile 配置

配置声明了：

- `module_a.device`
- `require_gpu`
- `light_flow_enabled`
- `static_image_enabled`

但 rebuilt 构造和主流程未读取这些开关：

- A3b 始终按 `static_image_interval` 调度；
- 光流始终每帧计算；
- FlowNet 在 CUDA 可用时硬编码 `cuda:0`；
- `cpu_smoke` 中的 `static_image_enabled=false`、`light_flow_enabled=false` 不等于 rebuilt 真正关闭这些路径；
- 指定 `cuda:1` 或在有 CUDA 的机器上指定 CPU，也不能保证 rebuilt 按该 device 执行 FlowNet。

影响：

- Web feature option“关闭静态媒体检测”可能只对 legacy 生效；
- CPU/profile 性能预算与实际执行路径不一致；
- 多 GPU 设备选择可能失效；
- GPU/CPU 回退测试无法证明生产默认 rebuilt 遵守配置。

结论：这是已确认的配置接线缺失。修复时必须保持默认实际行为，并单独评估关闭支路后的检测影响。

### P1-8：严格源前驱只更新 gray/LBP，没有同步 brightness；latest-only 源帧间隔也未进入检测复用决策

代码依据：

- `_inject_temporal_previous_frame` 只写 `detector.prev_gray` 和 `detector.prev_lbp`；
- 即使 `source_gap_frames > 1`，当前实现仍先注入 previous frame，只把 `strict_source_predecessor` 标成 false；
- rebuilt `_compute_scene_context` 的 `frame_diff_global` 使用严格前驱 gray，但 `exposure_delta` 使用旧的 `prev_brightness`；
- `_maybe_reuse_detections` 使用 pipeline 内部连续 `self.frame_idx` 计算 gap；
- `ModuleAInput.frame_idx` 也使用内部处理帧号；
- 真实 `current_source_frame_idx` 只在检测完成后写入诊断字段。

已复现：

- 注入均值 100 的严格前驱帧，但保留旧 `prev_brightness=10`；
- 当前帧均值 110；
- `frame_diff_global` 基于 100→110，而 `exposure_delta` 实际按 10→110 得到 `0.392`，严格前驱口径应约为 `0.039`。

影响：

- 同一帧的不同特征使用不同时间基准；
- exposure 相关抑制/救援门可能在 latest-only 跳帧时出现错误；
- 两次检测之间即使跨越大量源帧，只要最终画面相似，仍可能复用很久以前的检测框/ROI；
- 中间发生目标进出后画面恢复相似时，YOLO 变化可能被漏掉。

结论：严格前驱设计方向正确，但接入不完整。

### P1-9：stop 超时后可立即复用同一个 pipeline，存在旧/新检测线程并发操作模块 A 的风险

代码依据：

- `MonitorEngine.stop` 线程 join 超时后仍清空线程/bus 引用并返回；
- `start` 随即清除共享 stop event；
- `PipelineCache.get` 缓存命中时对共享 pipeline 直接 `reset()`；
- profile/key 变化时可能直接 close 旧 pipeline。

影响：

- 阻塞中的旧 GPU 推理线程可能与新线程并发使用同一个模块 A/后端；
- 新运行 reset 可能清空旧线程正在使用的时序状态；
- 可能出现跨源污染、CUDA 对象并发访问、偶发漏检或启动失败。

结论：从代码链路可确认竞态窗口存在；触发概率依赖 GPU/后端阻塞时长，属于“待压力实验确认”的运行时风险。

## 四、其他重要风险

### P2-1：A4 classifier 是本机可用、交付不可自包含的绝对路径资产

当前配置：

`D:/联合防御模块/rebuilt_demo/data/a4_classifier.pkl`

当前机器上该文件存在、可加载、维度为 20，运行时可用；但：

- `rebuilt_demo/` 被 `.gitignore` 忽略；
- classifier 不在 Git 跟踪文件中；
- bundled `model/src/defense/module_a/rebuilt/data/a4_classifier.pkl` 不存在；
- 显式配置路径缺失时，rebuilt `_load_classifier` 静默返回 `None`，不会回退到 bundled classifier，A4 改走规则融合；
- `a4_classifier_used` 仅表示 classifier 对象非空，单帧 `predict_proba` 异常后回退规则时仍可能报告已使用 classifier。

影响：

- 当前工作机正常不代表 clean clone/交付机行为一致；
- classifier 缺失或损坏会静默改变模块 A 决策模型。

结论：已确认发布可复现性风险。

### P2-2：A3BSoftTrigger 的高分绕过分支被后置 source guard 抵消

代码注释声明 `observed_score >= 0.85` 可绕过 keyword 和 quality gate，但后续：

- `trigger_source_allowed=False` 时无条件清除任何非 none source；
- 非 quality gate 且不是 observed-only source 时也会清除。

只读探针输入 `observed_score=0.90`、quality gate 通过、未知 source，最终仍为：

- `triggered=False`
- `triggered_source=none`

结论：代码行为与注释声明冲突。是否允许未知源高分绕过属于 A3b 策略，必须由行为调优确认。

### P2-4：当前缺少与 2026-07 代码同版本的可追溯量化报告

结论：算法效果是否满足最终门禁属于“待实验确认”，不能仅凭注释当作已验收证据。

结论：若该文件仍被当作当前交付完整性依据，会产生误导；若仅为历史产物，应明确归档或停止引用。

## 五、已确认正确的设计

- `DetectionBus` 的 latest-only 覆盖和 dropped 计数逻辑正确；
- preview 与 detection 使用独立 bus 和线程，preview 不等待检测；
- seek 使用 source epoch，检测线程遇到 epoch 变化会 reset processor；
- 旧 epoch 的 status/overlay/evidence 发布有拒收保护；
- 后端创建/推理主路径没有把异常普遍转换为空 detection；
- Pixi 全量 smoke 当前可通过；
- 当前本机 A4 classifier 和 YOLO TensorRT artifact 均存在且可读取。

## 六、建议处理顺序

### 第一阶段：不调算法，只修契约和生命周期

1. 定义 rebuilt/legacy 共用的规范化 Module A result schema；
2. 让 `_apply_a3b_suppression` 和 runtime A3b 只消费规范化字段；
3. 为 A3b background 引入 generation token 或可取消 worker，reset/close 时拒绝旧结果写回；
4. 将 A3b/background/classifier 失败写入 status/diagnostics；
5. 明确 canonical rebuilt 配置键，保持当前实际阈值不变；
6. strict predecessor 同步 brightness 等同源时序状态，并用 source frame gap 禁止过期 detection reuse；
7. stop 超时后隔离 pipeline 或拒绝复用。

### 第二阶段：补生产默认内核回归测试

最少新增：

- rebuilt 第一帧/静态序列/攻击序列/reset；
- A3b background reset race；
- rebuilt A3b bbox suppression；
- rebuilt result schema；
- classifier 缺失/损坏/单帧异常的可见性；
- source gap 禁止 reuse；
- strict predecessor brightness 一致性；
- stop timeout 后不得共享 pipeline；
- legacy/rebuilt 公共 API 字段同构测试。

### 第三阶段：行为调优与最终验收

在不混入契约修复的独立工作中裁定：

- TargetAnchoredAnalyzer 是否应重新参与 rebuilt joint decision；
- 自然曝光与 strong static glare 的冲突语义；
- A3BSoftTrigger 的 source keyword 和高分 bypass 语义。

完成后必须：

- 跑与当前提交绑定的留出集召回/误报；
- 实跑目标视频并抽检关键帧；
- 检查拖框、断框、旧框滞留、页面/画面同步；
- 记录结果视频路径和抽检结论。

## 七、最终判断

模块 A 的总体架构方向可继续使用，但当前生产默认 rebuilt 链路存在明确的字段契约断裂、A3b reset 竞态、配置生效口径不透明、时序前驱接入不完整和测试覆盖偏离生产内核等问题。

因此当前状态应判定为：

**“主算法框架可用，但尚不能证明功能设计完全正确；至少需要先完成契约/生命周期修复和同版本量化复验。”**
