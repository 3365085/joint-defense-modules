# 默认视频首启卡顿与模型配置入口收敛记录

## 问题背景

用户在 Web 端反馈：每次重新运行服务后，直接检测默认路径视频时，画面和检测框会出现一帧一帧的卡顿；但点击“浏览/选择视频”后再启动，画面马上变正常。同时，主监控页右侧的“算力档位 / 检测模型 / 自定义模型”区域与 B 模块安全中心职责混在一起，`TensorRT` 文案容易被理解成可随意切换的检测后端开关。

## 当前判断

卡顿根因不是 `pickFile()` 或 `/api/pick-file` 本身。`model/src/defense/web/static/index.html` 的 `pickFile()` 只调用 `/api/pick-file`、回填 `sourceValue`、保存 `moduleA.lastSourcePath`，没有调用 `MonitorEngine.start()`、`MonitorEngine.stop()` 或任何后端队列重置。

真正影响默认视频首启体验的是 `model/src/defense/runtime/runner.py` 中的文件源首帧门控：`MonitorEngine._backend_capture_loop()` 原先对 `source_type == "file"` 设置 `preview_can_start = False`，随后 `_preview_render_loop()` 在 `first_detection_ready == False` 时跳过渲染。这会让 MP4 预览被首帧检测结果牵制。服务刚启动时还叠加 `PipelineCache.get()` 冷启动、TensorRT backend 构建、`VideoDefensePipeline.warmup()` 和 `_backend_process_loop()` 线程预热，所以用户肉眼看到“直接检测默认视频时画面/框一帧帧卡”。

“选择视频后正常”更可能来自后续启动变成模型 cache hit 或 seek/start 重新同步后的热路径，而不是文件选择器有特殊预热逻辑。

## 代码链路依据

- 主页面启动入口：`model/src/defense/web/static/index.html` 的 `startBtn.onclick` 组装 `/api/start` 请求。
- 文件选择入口：`model/src/defense/web/static/index.html` 的 `pickFile()` 只回填输入源路径。
- Web 启动路由：`model/src/defense/web/fastapi_app.py` 的 `/api/start` 先执行 `ModelSecurityService.ensure_admitted()`，通过后调用 `MonitorEngine.start()`。
- 运行时启动：`model/src/defense/runtime/runner.py` 的 `MonitorEngine.start()` 创建 `PreviewBus`、`DetectionBus`，启动 `_backend_process_loop()`、`_backend_capture_loop()`、`_preview_render_loop()`。
- 文件源预览门控：`model/src/defense/runtime/runner.py` 的 `_backend_capture_loop()` 和 `_preview_render_loop()`。
- 检测队列背压：`model/src/defense/runtime/backend_pipeline.py` 的 `DetectionBus.pop_latest()` / `DetectionBus.push()` 保持 latest-only。

## 本次处理

1. 主监控页删除“算力档位 / 检测模型 / 自定义模型”整块 UI，只保留 B 模块安全摘要和“打开安全中心 / 刷新状态”。
2. 主监控页启动请求固定使用 `DEFAULT_RUNTIME_PROFILE = "desktop_rtx"`，`customModelOptions()` 固定为未启用，避免旧 localStorage 中的自定义模型残留悄悄影响默认检测。
3. 安全中心删除运行档位下拉、导出预留按钮和手工切换净化模型按钮，改为展示“运行档位由系统固定选择；TensorRT 只代表默认加速模型链路，不是可手工切换的空壳开关”。
4. `MonitorEngine._backend_capture_loop()` 对文件源也立即设置 `ready_for_preview = True`、`preview_started = True`，预览按源时钟先显示。
5. `_preview_render_loop()` 只有在显式关闭 `preview_never_wait_for_detection` 时才等待首帧检测；默认继续保持“预览不等检测”，检测结果到达后通过 overlay 叠加。

## 影响范围

- A 模块检测算法、模型权重、类别语义、阈值未修改。
- B 模块安全准入策略未放宽，未知模型仍会阻断 A 模块启动并触发 full scan。
- 文件源预览体验恢复为预览与检测解耦：允许先显示无框源画面，之后叠加检测结果。这能降低首启卡顿感，但首帧检测框仍取决于 detector warmup 和第一帧推理耗时。
- 主监控页不再允许选择自定义模型；模型准入、扫描、白名单管理集中到安全中心。

## 验证记录

- 运行测试：`pixi run python -X utf8 -m pytest -q tests/test_web_prebuffer_contract.py tests/test_runtime_file_preview_contract.py tests/test_web_detection_readiness_contract.py tests/test_model_security_runtime.py tests/test_web_cache_headers.py`，结果 `38 passed`。
- 浏览器验证：`http://127.0.0.1:7860/` 主监控页不再存在 `profile`、`enableCustomModel`、`customModelPath` 控件；`http://127.0.0.1:7860/model-security` 不再存在 `profileSelect`、`usePurifiedBtn`、`exportBtn`。
- 接口验证：`/api/start` 在 B 模块未放行时返回 `409 model_security_blocked`；B full scan clean 后默认模型进入 `trusted`；随后 `/api/start` 返回 `ready_for_preview=true` 且初始 `first_detection_ready=false`，符合“源预览先行、检测后叠加”的目标。

## 后续建议

如果仍偶发首启卡顿，应继续记录 `pipeline_cache_hit`、`pipeline_cache_get_ms`、`detector_thread_warmup_ms`、`first_detection_processing_ms`、`preview_fps`、`dropped_detection_frames`，区分冷启动、B 准入、视频解码和 overlay 同步问题。若用户要求检测框必须首帧同步显示，则需要引入短 overlay 预缓冲，但这会牺牲启动流畅性。
