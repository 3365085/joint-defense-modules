## 0. 结论摘要

1. 主项目当前默认运行的已经是从 demo 移植的 `rebuilt` 内核，并加载
   `rebuilt_demo/data/a4_classifier.pkl` 的 variant C XGBoost 分类器。
2. “让主项目达到 demo 程度”已经基本实现，但 demo 本身不是可靠泛化基准。2026-07-11 重新实跑：
   - demo `eval_detection_broad.py`：前 240 帧内合成攻击命中 `7/10`，干净片误报 `1/12`；
   - 主项目开发/模型选择集：前 240 帧内合成代理攻击命中 `17/21 = 81.0%`，干净片误报
     `2/27 = 7.4%`。
   两次运行使用的 YOLO 资产和 profile 不同，只能分别说明各自当前状态，不是严格 A/B 对拍。
3. 这两个攻击召回数字都不能解释为“真实物理攻击召回率”。21 段 heldout 攻击全部是程序化
   合成代理，真实攻击 heldout 为 0。
4. A1、A2、A3、A3b 不是都需要大数据训练。它们主要是手工物理/时序传感器；真正直接进行
   监督学习的是 A4。问题在于所有模块的阈值、基线、融合、抑制规则和适用边界仍需要大量
   **独立场景**验证。
5. 在不能实拍的条件下，可以继续把系统做成低误报、可解释、经过代理攻击压力测试的
   “视频完整性异常预警器”；不能诚实宣称已经解决真实打印补丁、真实强光、真实遮挡或
   跨设备屏幕翻拍检测。

## 1. 当前运行事实

### 1.1 主项目实际运行组合

当前新浏览器默认组合为：

`desktop_rtx -> rebuilt Module A -> variant C XGBoost`

代码依据：

- `model/src/defense/pipelines/video_defense_pipeline.py` 未配置 `detector_impl` 时默认选择
  `rebuilt`，只有显式写成 `legacy` 才使用旧内核。
- `model/configs/module_a_runtime.yaml` 的分类器路径指向
  `rebuilt_demo/data/a4_classifier.pkl`。
- 当前分类器 SHA256：
  `ED1BB6626C2CD101F2404AC959A3C2778C654770A854F57B38ECD2F06C985DCD`。
- 当前分类器为 20 维 `XGBClassifier`。
- rebuilt 实际攻击阈值来自 `rebuilt_theta_adv`，默认 `0.65`。配置中的
  `p_adv_threshold: 0.55` 属 legacy 路径，不控制 rebuilt。

2026-07-11 的主项目 heldout 脚本使用 `profile="default"`，并不等于浏览器默认的
`desktop_rtx`。demo broad 还使用了另一份 `baseline_yolov8_three_put_cx` engine，而主项目默认配置
使用 `baseline_yolov8_three_put` engine。因此本文记录的两组结果不可直接归因于 detector 差异。

### 1.2 主项目与 demo 已经分叉

主项目生产 rebuilt 最初来自 demo 整块移植，但当前已不是逐行等价副本：

- 主项目增加了确认告警保持。
- 主项目工作树还有未提交的候选桥接、持续攻击升级、A1-only 场景基线 carveout。
- 主项目与 demo detector 当前比较约有 `+203/-6` 行差异。
- demo 默认离线逐帧处理；主项目是实时 latest-only、可能跳源帧，并允许 YOLO 检测复用。
- demo 离线使用源视频时间戳；主项目 Web 主链仍通过 `process_frame()` 向 Module A 传
  `timestamp=0.0`，内核退回处理墙钟估算 FPS。
- demo realtime 会补入严格相邻源帧作为 A2/A3 前帧；主项目没有同等修正，丢帧后 A2 帧差和
  A3 光流的实际时间跨度会放大。

因此，不能把 demo 离线结果直接等同于主项目 Web 实时结果。

## 3. A1-A4 是否天然需要大数据

### A1

A1 是 LBP 纹理直方图、网格/ROI 差异和场景 EMA 基线构成的手工传感器，本身没有训练过程。

它不需要大数据“训练”，但需要大量正常场景验证以下边界：

- 工服、格栅、碎石、树叶、焊接、反光和压缩伪影是否造成高分；
- 冷启动前约 8 帧是否把攻击吸收入基线；
- 攻击从视频第一帧存在时是否仍可区分；
- 局部高纹理到底是攻击补丁还是正常衣物/背景。

结论：A1 的核心问题是信息可辨识性和正常分布覆盖，不是简单缺少训练轮次。

### A2

A2 是相邻帧 LBP 变化和运动解释后的时序纹理异常，同样不是监督模型。

它需要多场景验证相机抖动、人员运动、灯光闪烁、编码抖动和真实攻击之间的重叠。主项目 latest-only
和跳源帧还会改变它的时间尺度。

### A3

A3 的局部残差、ROI 关系和规则聚合不是监督学习；光流骨干使用预训练 RAFT，demo 没有自行训练
光流网络。

需要验证的是光流在真实相机运动、快速人员运动、遮挡和丢帧条件下是否保持可解释，而不是继续用
少量本地攻击视频训练 RAFT。但“本项目没有本地训练 RAFT”不等于 A3 与数据无关：它仍继承
预训练光流网络的上游数据分布和域外泛化风险。

### A3b

A3b 是矩形、边缘、平面、跟踪和静态媒体规则。当前真实 replay/interference 样本各约一段，
并且因墙面、门框等背景假阳，独立单帧媒体候选已被关闭。

它不需要海量模型数据，但恢复独立翻拍告警前，需要较多不同屏幕、手机、角度、距离、光照和困难
负样本。当前 A3b 仍会：

- 向 A4 提供媒体特征；
- 参与物理补丁救援、背景抑制和 `adv_candidate_allowed` 联合规则；
- 提供诊断与媒体候选信息。

因此它不是“只用于显示”，但不能宣称已经完成独立真实翻拍告警或泛化验证。

### A4

A4 是 20 维 XGBoost，直接依赖监督数据。但需要的不是同一视频的更多帧，而是更多独立的：

`场景 x 攻击类型 x 介质/设备 x 相机 x 光照`

当前分类器虽有 65,739 个有效帧，真实独立攻击设置仍极少。相邻帧高度相关，不能把 6 万帧理解为
6 万个独立样本。

当前报告：

- CV AUC：0.841；
- 帧级代理攻击召回：0.542；
- 帧级假阳比例：0.071；
- 阈值：0.65。

其中 `cv_fp` 不是视频级误报率，`p_adv` 也没有完成概率校准，应理解为攻击分数而非严格概率。

## 4. demo 当前数据与评测问题

### 4.1 正样本几乎全部是合成代理

manifest 当前共 232 段：

- 干净：130 段；
- 正标签：102 段；
- demo 合成：95 段；
- 旧生成链路的 `raw_*`：5 段，虽标为 `source=real`，元数据确认也是从 015 正常视频生成；
- replay/interference：各 1 段。

`素材/真实物理攻击/` 当前为空。

### 4.2 合成生成器提供了容易学习的捷径

`rebuilt_demo/tools/generate_attack_videos.py` 中：

- adv patch 是高对比彩色块，可逐帧重采噪声并大幅抖动；
- occlusion 是高不透明快速移动矩形；
- visibility 会主动加入闪烁；
- 所有合成片统一经 `mp4v` 重编码。

实际验收帧显示，补丁可大面积遮挡人脸并逐帧改变纹理。这会主动放大 A1/A2/A3 响应，与真实打印
补丁的材质、透视、光照一致性和运动连续性差异较大。

因此当前合成数据能证明“传感器对该生成器响应”，不能证明真实物理攻击召回。

### 4.3 训练/留出谱系不干净

已确认：

1. `stock_pexels_8964728` 存在 train/heldout 完全重复视频。
2. `_scene_id()` 注释声称会将原始 015 系列归并，但代码只剥离 `__attack` 后缀。
3. 5 个 `raw_*` 攻击来自同一 015 正常视频，却被登记为 5 个独立 train scene；
   对应 015 正常视频在 heldout。
4. manifest 当前还有 1 个 scene_id 同时出现在 train 和 heldout。

这会让 CV 和开发集看到同源画面风格，形成乐观偏差风险。具体影响幅度需要按 capture/source
重新分组后做消融，当前属于待实验确认。

### 4.4 variant C 存在人为选择与不可复现问题

variant C 是先根据已知真人误报结果选出 7 个 FP-prone 场景，再删除这些场景的攻击正样本后训练。

问题：

- 场景选择使用了已经看过的误报结果，后续在同一场景集合上评测会产生选择偏差；
- `tools/_exp_variantC.py` 当前未跟踪；
- 实际训练使用的 `a4_features_disjointC.csv` 已删除；
- 仓库跟踪的 `a4_features.csv` 不是当前 pkl 的直接训练输入。

当前分类器可运行，但不能从已提交内容严格复现。

### 4.5 有效特征与文档存在偏差

A4 读取 `p_media_scores["display"]`，A3b 实际写入 `display_frame`。当前训练 CSV 的
`a3b_display_frame` 恒为 0，名义 20 维中有一维目前没有有效信息。

此问题应先修正并重新采集/评测，不能只修改键名后继续沿用旧分类器。

## 5. 无法实拍时的可行路线

### 5.1 先改变可交付口径

在没有真实攻击正样本时，建议将 Module A 定位为：

> 低误报优先的视频完整性异常与可见性退化预警器，经过物理化代理攻击压力测试。

可以声明：

- 真实正常视频上的误报回归结果；
- 场景自适应、状态机和时序稳定性；
- 对合成代理攻击的响应、延迟和强度单调性；
- A3b 在现有两段 replay/interference 上的开发集回归结果；两段均在 train，不是独立泛化验证。

不能声明：

- 真实打印补丁召回率；
- 真实强光/激光攻击召回率；
- 真实遮挡攻击分类准确率；
- 跨屏幕、跨手机、跨相机的翻拍泛化；
- 面对自适应攻击者的可靠性。

### 5.2 阶段 0：先修证据口径

为数据清单增加或固化：

- `capture_id`
- `source_clip_id`
- `generator_id`
- `generator_version`
- `provenance`
- `artifact_hash`

provenance 至少区分：

- `real_clean`
- `real_replay_proxy`
- `synthetic_physical_proxy`
- `digital_backdoor_proxy`

同一原始视频的裁剪、转码、分辨率版本和派生攻击必须归到同一 capture/source 组，训练、校准和
测试之间禁止跨组。

同时应：

- 删除 train/heldout 重复；
- 修正 015 谱系；
- 固定评测使用的代码、配置、YOLO 权重、A4 pkl 和生成器哈希；
- 将 A4 artifact 固定到主项目管理的版本化资产位置，避免生产运行依赖独立 demo 仓的可变文件；
- 将 variant C 数据配方变成可跟踪、可复现的正式流程；
- 将 broad 明确降级为 smoke，另建严格 heldout 事件评测。

### 5.3 阶段 1：用正常视频解决真正可解决的问题

不能拍攻击视频，不代表不能获得有价值数据。真实正常视频最适合解决：

- 误报；
- 场景自适应；
- 正常运动、灯光、焊接、多人交叉和转场边界；
- 状态机长时间稳定性；
- 目标出现、消失、重现的正常模式。

建议：

- 31 段项目真实正常视频作为高权重挑战集；
- 99 段 stock 视频用于正常分布扩展；
- 阈值只用正常数据确定，不用合成召回反向追阈值；
- 对 A1/A2/A3、清晰度、对比度、目标置信度建立按场景的分位数/EMA 正常带；
- 以 source-group out-of-fold 方式产生正常分数，避免同源画面泄漏；
- 主指标改为每小时误报事件、每小时告警秒数和事件持续时间，不只报帧比例。

当前干净素材总时长约 0.517 小时，其中项目真实视频约 0.148 小时。即使零告警，按零事件
Poisson 近似，95% 误报率上界仍约为 `3 / 0.517 = 5.8 次/小时`。若要声称误报率低于
`0.1 次/小时`，约需 30 个独立正常小时且零事件。

循环播放可以做 8 到 24 小时状态机 soak，但不能当成 8 到 24 小时独立真实场景数据。

### 5.4 阶段 2：把合成数据降级为代理压力矩阵

保留现有生成器，但不要再把“同生成器 heldout 召回”解释成真实召回。

建议形成正交代理矩阵：

- 源场景；
- 攻击类型；
- 强度；
- 随机种子；
- 分辨率；
- 编码器/码率；
- 帧率；
- 相机运动；
- 目标尺度；
- YOLOv5/YOLOv8 离线响应。

生成器应补充：

- 有效目标锚点门禁：没有可信 person/PPE 目标时拒绝生成，不能回退到任意最高置信框或画面中心；
- 真实补丁图像，而不是只用红黄 X 或随机彩格；
- 目标跟踪后的透视、尺度和姿态连续变化；
- 纸张/布料形变近似；
- 颜色、阴影和局部光照匹配；
- 镜头模糊、噪声、压缩和 rolling-shutter 近似；
- 屏幕像素网格、gamma、刷新拍频和摩尔纹近似；
- 至少两个彼此独立的生成器域，训练一个、留出另一个做跨生成器测试。

这些改进仍是代理，不替代真实验证。

### 5.5 阶段 3：按可辨识证据拆分告警语义

不建议继续把所有情况压成一个 `ATTACK / p_adv`。

更可信的输出应区分：

- `video_integrity_degraded`：模糊、雾化、强过曝、视野严重下降；
- `media_replay_suspected`：矩形介质、显示特征、平面跟踪等证据；
- `adversarial_texture_suspected`：目标附近持续异常纹理；
- `unknown_visual_anomaly`：异常成立但无法归因；
- `suspected_attack`：系统已完成时序确认，但攻击真实性仍需外部证据或人工复核。

现有 `alert_confirmed` 仅表示内部状态机的时序确认，不等于真实攻击地真。A1-A4 大多来自同一 RGB/
YOLO 链路，不能把它们简单视为统计独立证据；在没有外部传感器、可信设备状态或人工复核时，不建议
对外使用“真实攻击已确认”的语义。

物理边界：

- motion blur、visibility 当前可由相对场景退化支路稳定承接；
- variant F 已证明，直接把 glare/occlusion 从 A4 移除会使 glare 降到 `0/4`、occlusion 降到
  `1/4`。是否能通过重新设计退化支路减少它们对攻击标签的依赖，属于待实验确认；
- adv patch 若在目标出现后才贴上，利用同一目标前后纹理变化和 recent target history 是合理研究
  方向，但当前没有真实数据验证，属于待实验确认；
- adv patch 从第一帧就存在时，仅凭单摄像头 RGB，往往无法区分“攻击补丁”和正常高纹理服装；
- 没有真实屏幕/介质数据时，A3b 应输出 suspect，不应输出真实攻击已确认。

### 5.6 阶段 4：shadow 部署积累正常运行小时

若无法组织实拍，最有价值的现实数据来源是主项目实际运行：

- Module A 先 shadow，不触发强处置；
- 保存高分候选、场景摘要和少量事件证据；
- 人工只审高分和代表性低分样本；
- 按误报类型形成 hard-negative 场景簇；
- 逐步积累真实正常运行小时和真实误报分布。

这不能补真实正样本，但能让系统首先做到“少骚扰、可解释、稳定”。

## 6. 推荐验收指标

### 真实正常视频

- 误报事件数/小时；
- 告警秒数/小时；
- 片级误报率；
- 按场景 bootstrap 置信区间；
- 冷启动误报；
- 转场、seek、丢帧和重连后的误报。

### 合成代理攻击

- 攻击前误报事件；
- 首次确认延迟；
- 攻击段覆盖率；
- 攻击结束后的告警滞留；
- 响应随强度的总体单调性；
- 跨分辨率、帧率、编码和生成器一致性。

所有攻击结果必须明确标注“合成代理”，不得简称为真实召回。

## 7. 建议优先级

P0：

1. 修正数据谱系、重复视频和 train/heldout 泄漏。
2. 固化 variant C 可复现流程及模型/配置哈希。
3. 将 A4 artifact 收归主项目版本化管理，消除生产对 demo 可变资产的隐式依赖。
4. 将 broad 评测降级为 smoke，建立严格事件级 heldout。

P1：

1. 接通主项目真实源时间戳和相邻帧语义，量化与 demo 离线的差异。
2. 修复 A3b display 特征键不一致后重新采集和训练，不沿用旧 pkl。
3. 在状态中明确暴露分类器是否成功加载，禁止静默回退后仍显示同样能力。

P2：

1. 用正常视频建立 shadow 误报小时门禁。
2. 改造代理生成器，增加物理一致性和跨生成器留出。
3. 按完整性退化、媒体疑似、纹理疑似和未知异常拆分告警语义。

## 8. 最终判断

主项目已经接近当前 demo 的工程检测水平，当前差距主要不在“有没有把 demo 代码搬过来”，而在：

- demo 自身的真实攻击数据为空；
- 数据谱系和评测仍有泄漏；
- 合成生成器提供了明显捷径；
- variant C 是低误报换召回的人工折中；
- 主项目实时语义与 demo 离线语义不完全一致；
- `p_adv` 承担了超过现有证据能力的“攻击概率”含义。

在无法实拍的前提下，最优路线不是继续围绕同一批合成片微调 A1-A4，而是先修证据口径、用真实
正常视频把误报压稳、将合成攻击降级为代理压力测试，并把无法确认的真实攻击能力明确标记为
“待实验确认”或“未能从现有数据确认”。
