# PPE 断框/托框第二轮 A2 诊断记录

日期：2026-06-05

## 背景

用户反馈 Web 端目标视频仍存在 PPE 框断续、短时托框、多人重叠时共框，以及远距离人员 head/helmet 证据不足的问题。第一轮 A1 仅放宽短时显示桥接与 render miss 上限，未降低 head abrupt drop，且增加 held track 记录，因此第二轮不继续盲调参数，而是先给跟踪链路增加 text-only 分阶段诊断。

本轮未查看、导出或展示任何图片/视频画面；证据来自 Web API 的 status/overlay JSON 与 CSV 汇总。

## 代码链路

诊断入口位于 `PPEDisplayTracker.update()`，按以下阶段采样：

`incoming -> incoming_merged -> after_assignment_and_age -> after_prune -> after_shadow_removal -> after_low_context_filter -> rendered`

诊断字段通过 `evaluate_ppe_business()` 写入 `ppe["display_tracking_diagnostics"]`，再由 `FrameProcessor` 暴露到 status，并由 `build_overlay_record()` 写入 overlay record。该改动用于观测，不改模型权重、类别语义、A3b 策略、阈值、hold 参数或实际过滤条件。

## 当前判断

1. 断框主因不是单一参数过紧。A2b 显示 `shadow_track_removed` 与 render 阶段合计占主要 drop，说明多人重叠、同目标/影子轨迹去重、短时 held 渲染上限分别在不同片段触发。
2. head 的主要损失发生在 tracker 内部，尤其是 shadow removal 与 render cap。放宽 render cap 可以减少一部分短断，但会增加旧框滞留和托框风险，A1 已经给出负面迹象。
3. helmet 的可见性除 tracker 内部损失外，还受到 `_filter_tracks_for_ppe_counts()` 业务出口过滤影响。业务字段当前不允许某类标签时，已存在的 held helmet/head track 仍可能被最终隐藏，导致“检测/跟踪层还有框，但 overlay 结果没有框”。
4. `person` 在本次目标视频的 overlay 与 tracker 诊断中仍为 0，但历史状态里可能出现 `person_context_available` 语义。此处不能直接断言模型完全没有 person 能力，只能确认本次 API 复跑的 PPE display tracker 输入/输出未形成 person track。需要下一轮用原始 detector 类别直方图进一步确认 person 是未检出、被 PPE 抽取过滤，还是被业务层只作为 context 不入 display track。
5. 当前 A2b 仍属于诊断实验，未产生可提交的行为调优方案；也未进入提交前视觉验收流程。

## 下一轮实验设计

A3 overlap-safe：

- 目标：解决多人重叠、两人共框、head/helmet 误互斥。
- 只观察或隔离调整同目标合并、shadow removal、head/helmet mutex 相关条件。
- 指标：`shadow_track_removed` 数量、head/helmet abrupt drop、重复框数量、business filter 前后差异。
- 风险：直接放宽 shadow removal 可能把重复框、错类框留下，降低画面可信度。

A4 render/business gate：

- 目标：区分短时断框、托框、业务出口隐藏三类问题。
- 分别实验 `max_render_misses`、`held_track_not_eligible`、`business_label_not_allowed`，不能混合改。
- 指标：`misses_exceed_render_cap`、held track records、max track misses、head/helmet abrupt drop、旧框滞留 proxy。
- 风险：提高 render cap 可能把断框换成托框；绕过业务过滤可能导致画面框与 PPE 状态不同步。

A2 补充诊断：person-source histogram：

- 目标：确认远距离 person、head、helmet 证据在 detector 输出、PPE 抽取、business summary、display tracker 中分别是否存在。
- 方法：继续 text-only 输出每帧 raw detector class counts、PPE raw counts、context counts、display track counts。
- 风险：若直接把 person 画出来，会干扰当前“person 仅辅助”的显示意图；应保持 `show_person_boxes=false`，只用于诊断与上下文辅助。

本次 A2 补充诊断汇总中的标签集合只有 `head` 与 `helmet`，没有 `person`。因此对这段目标视频而言，`ppe_has_person_class=True` 只能说明模型类别表支持 person；实际 raw detector 输出层没有形成 person 检测，后续 PPE raw 与 display 阶段自然也没有 person track。该结论仅针对本次目标视频和当前运行配置，其他视频仍需同样用直方图复核。

工程含义：

1. `show_person_boxes=false` 不是本次 person 缺失的原因；它只影响绘制层。
2. 当前视频不能依赖 person 作为稳定辅助上下文来修正 head/helmet，A3/A4 应继续以 head/helmet 的跟踪、互斥、业务出口过滤为主。
3. 若后续需要 person 参与抑制 head/helmet 误报，应先确认目标模型在相同场景下能稳定输出 person；否则只能把 person 作为可选证据，而不是硬前置条件。
4. 远距离 person 有框但 head/helmet 缺失的场景仍需另选有 raw person 输出的视频或样本做专项复核；不能由本次视频推出相反结论。
