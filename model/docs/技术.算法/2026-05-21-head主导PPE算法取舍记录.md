# Head 主导 PPE 算法取舍记录

## 问题背景

用户提出是否应从当前检测算法中移除 `person` 类，改为以 `head` 为主导的 PPE 安全帽判定。当前阶段明确不修模型权重，先修算法，并以 `best.engine` 的实际检测输出为主要适配目标。

## 当前判断

不建议彻底移除 `person` 类。建议将 `person` 从 PPE 告警的主决策条件降级为辅助上下文，让 `head` 成为“疑似未戴帽”的主证据，`helmet` 成为“已戴帽/抑制未戴帽”的反证，`person` 只参与上下文约束、ROI 候选和显示稳定。

## 代码链路依据

- `src/defense/module_a/ppe_postprocess.py`
  - `PERSON_HINTS`、`is_person_label()` 定义 `person` 识别。
  - `suppress_helmet_false_positives()` 中 `persons` 参与头部孤立目标过滤、安全帽上下文过滤和安全帽相对人体面积过滤。
  - `summarize_ppe_from_detections()` 中 `person_count`、`helmet_count`、`head_count` 共同计算 `missing_helmet_count` 与 `candidate`。
- `src/defense/pipelines/video_defense_pipeline.py`
  - `DetectionROIProvider` 使用 `roi_target_labels` 生成 Module A ROI，默认包含 `person`、`helmet`、`head`。
- `configs/module_a_runtime.yaml`
  - `module_a.roi_target_labels: [person, helmet, head]` 明确将 `person` 纳入 ROI。
- `src/defense/module_a/postprocess/ppe_tracking.py`
  - `PPEDisplayTracker.recommend_redetect_rois()` 会把 `person` 纳入小目标复检 ROI。
  - `_filter_low_context_display_tracks()` 用 `head` 或 `person` 给低置信 `helmet` 提供上下文。
  - `_merge_same_target_items()` 明确避免把 `person` 与 `head/helmet` 合并为同一目标。
- `src/defense/runtime/frame_processor.py`
  - `_build_status()` 暴露 `ppe_person_count`、`ppe_helmet_count`、`ppe_head_count`、`ppe_missing_helmet_count` 等状态字段。

## 影响范围

当前 `person` 会影响以下行为：

1. PPE 告警计数：`person_count - helmet_count` 会形成 `missing_helmet_count`。
2. 裸头保留：有 `person` 上下文时，小的 `head` 更不容易被当成边缘孤立误检过滤。
3. 安全帽误检抑制：没有 `head` 或 `person` 上下文的 `helmet` 更容易被抑制。
4. ROI 生成：`person` 框会参与 Module A 的目标区域计算。
5. 显示稳定：`person` 可作为低置信 `helmet` 的上下文，但不会和 `head/helmet` 合并。
6. Web 状态字段：前端/状态快照会继续暴露 `ppe_person_count`。

## 推荐修改方向

推荐做“head 主导、person 辅助”的改法：

- PPE 候选条件以有效 `head_count > 0` 为主。
- 当存在 `head` 与有效 `helmet` 高重叠时，按置信度和重叠关系做互斥，只保留一个语义。
- `person_count` 不再直接决定 `missing_helmet_count`，只作为 `context_person_count` 或 `person_context_count` 输出。
- `person` 仍保留在 ROI 和显示跟踪中，用于辅助稳定和减少孤立误检。
- 状态字段保持兼容：保留 `ppe_person_count`，新增或明确 `ppe_head_driven`、`ppe_current_evidence`、`ppe_context_person_count`。

## 预期好处

- 更适配 `best.engine` 实际输出中 `head/helmet` 比 `person` 更关键的场景。
- 避免因为 `person` 空检导致“画面有人但算法说未检测到人员”的语义错误。
- 降低 `person` 类漏检对安全帽告警的影响。
- 保留 `person` 的上下文价值，避免完全删除后增加边缘小头部误检。
- 保持 Web API 字段兼容，降低前端和状态消费方改动成本。

## 风险

- 如果完全不使用 `person` 上下文，远处小头部、边缘物体、人体外类似头部纹理的误检可能增加。
- 如果 `helmet` 检测也稀疏，单纯 head 主导会更偏向“未戴帽”告警，需要靠时间窗口和互斥后处理抑制抖动。
- 如果前端或外部系统依赖 `ppe_person_count` 解释告警，必须同步说明该字段变为上下文指标而非主判定指标。

## 结论

不建议从算法中彻底移除 `person`。建议把主判定改为 `head` 驱动，并保留 `person` 作为辅助上下文、ROI 和显示稳定信息。这样对当前 `best.engine` 更稳，也更符合“未戴帽检测”本身的业务语义。

## 未能从代码中确认

- 未能仅从代码确认所有现场视频中 `person` 类的真实召回率，需要用标注或原项目输出做实验确认。
- 未能仅从代码确认前端或外部接口是否将 `ppe_person_count` 当成强业务指标，需要再查使用方或实际页面行为。
