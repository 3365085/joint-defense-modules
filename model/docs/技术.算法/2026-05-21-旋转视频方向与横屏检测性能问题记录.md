# 旋转视频方向与横屏检测性能问题记录

## 问题背景

用户反馈两个竖屏视频中有一个实际旋转了 90 度，容易出现各类误报；同时横屏视频直接检测很卡。经抽样读取素材首帧确认，`d6415677a016dbf211e665f648e75607.mp4` 的解码像素为 `1280x720`，但画面内容是侧向竖屏手机视频；`5e145bf778577e75118502e263d00c41.mp4` 为正常 `720x1280` 竖屏像素；`VID20260512200916.mp4` 为 `3840x2160` 横屏视频。

## 当前判断

旋转 90 度视频的问题属于输入素材方向异常，不计划修复。当前 OpenCV 链路读取到的 `CAP_PROP_ORIENTATION_META` 为 `0`，不会自动按手机显示方向旋转，因此模型会看到侧着的人、侧着的手机 UI 和横向强边缘，PPE 与 A3B 都可能被误导。该情况在正常素材规范下不应出现，暂作为已知输入质量问题记录。

横屏 4K/60fps 视频卡顿需要单独处理。其主要风险不在模型权重，而在运行链路：视频读取、预览编码、检测帧抽样、检测 backpressure、以及 `FrameProcessor.prepare_frame_640()` / `VideoDefensePipeline._run_detection()` 中的缩放路径都会影响实时性。

## 代码链路依据

- `src/defense/runtime/frame_processor.py`
  - `prepare_frame_640()` 会将输入最终缩放到 `640x640`，当前不做方向修正。
  - `FrameProcessor.process()` 负责预处理、推理、PPE/A3B 状态与渲染。
- `src/defense/pipelines/video_defense_pipeline.py`
  - `VideoDefensePipeline._run_detection()` 在非 `640x640` 输入上再次缩放到 `640x640`，并运行检测与 Module A。
- `src/defense/runtime/runner.py`
  - `MonitorEngine._backend_capture_loop()`、检测线程和预览线程共同决定视频文件模式下是否卡顿。
  - 已在 capture 层增加 `capture_max_side` 与 `file_source_fps_cap`，避免高分辨率/高帧率文件逐帧传递原始大图。
- `src/defense/runtime/a3b_soft_trigger.py`
  - `A3BSoftTriggerState.update()` 会基于静态媒体/屏幕线索做窗口累积，侧向手机界面会增加误触发风险。
- `src/defense/module_a/ppe_postprocess.py`
  - `summarize_ppe_from_detections()` 中小面积孤立 head 已降级为 `uncertain`，但方向异常输入仍可能产生弱 head 证据。

## 结论

旋转 90 度素材属于输入方向异常，暂不修复。横屏卡顿需要优先从运行性能链路优化，避免高分辨率/高帧率视频让预览和检测互相阻塞。

## 性能修复记录

- 文件输入在 capture 层先按 `capture_max_side=1280` 限制长边，4K 横屏视频会从 `3840x2160` 降采样为 `1280x720` 后再进入预览/检测总线。
- 文件输入按 `file_source_fps_cap` 抽样读取，默认取预览帧率和检测帧率的较大值；对 `60fps` 视频，当前步长约为 `2.4`，避免逐帧取回。
- 验证 `VID20260512200916.mp4` 前段运行状态：
  - `source_fps=60.0`
  - `source_frame_step=2.4`
  - `capture_resized=True`
  - `capture_frame_width=1280`
  - `capture_frame_height=720`
  - `preview_fps` 约 `23fps`
  - `dropped_detection_frames=0`

## 性能修正补充记录

经与原项目链路对比，自动把文件源帧率限制到预览帧率/检测帧率上限会影响预览播放节奏。原项目虽然检测链路会按需处理帧，但文件源读取和预览没有被这个默认抽帧策略限制，因此 4K/60fps 横屏视频在视觉上更流畅。

本次修正为：`file_source_fps_cap` 默认关闭，`MonitorEngine._file_frame_step()` 在未显式配置 cap 时返回 `1.0`，文件源按原始帧节奏进入预览；检测仍由 `detector_process_fps_cap` 和 latest-only `DetectionBus` 控制，不额外增加 GPU 推理压力。`capture_max_side=1280` 继续保留，用于降低 4K 帧进入预览编码和检测总线前的 CPU/内存开销。

修正后用 `VID20260512200916.mp4` 和 `baseline_yolov8/weights/best.engine` 短跑验证：

- `source_fps=60.0`
- `source_frame_step=1.0`
- `file_source_fps_cap=0.0`
- `capture_resized=True`
- `capture_frame_width=1280`
- `capture_frame_height=720`
- `preview_fps` 约 `24.8fps`

## 进一步性能判断与修正

用户确认原项目在同一 4K 横屏素材上也存在明显卡顿，因此该问题不是迁移回归，而是 4K/60fps 文件源、检测模型、A3b 静态媒介检测和 Web 预览共同构成的实时预算问题。当前 `FRAME ... | ... | xx.xms` 展示的是检测分支耗时，若单帧达到 `53ms`，理论检测吞吐只有约 `18fps`，再叠加预览编码和浏览器显示会表现为卡顿。

压测发现 `desktop_rtx` 档位的主要尖峰来自 `A3b static_media` 的 multi-scale fallback：典型尖峰中 `a3b_static_media_ms` 可达 `10-12ms`。本次修正将 `desktop_rtx`、`edge_fast`、`low_power` 这类实时档位的 `static_image_multiscale_fallback_enabled` 默认值改为 `false`；未显式配置时只在高质量/默认算法链路保留 multi-scale fallback。这样不改模型、类别和阈值，但会降低小屏/小补丁补扫覆盖率，属于实时性优先的策略取舍。

同时移除检测线程中未被使用的预览渲染，避免 backend-source 预览模式下检测线程重复画框；实际预览仍由独立 preview 线程根据 overlay record 渲染。

## 输入源影响性能的原因与追帧修正

输入视频源会影响性能，原因是文件源链路不仅包含模型推理，还包含视频解码、色彩转换、源帧缩放、预览缩放、HUD/框渲染和 JPEG 编码。`VID20260512200916.mp4` 是 `3840x2160/60fps`，如果后台逐帧 `cap.read()`，就要求 Python/OpenCV 每秒完成 60 次 4K 解码和缩放；即使检测分支 latest-only，源视频时间仍会因为采集线程跟不上而变成约 `0.6x` 播放。

本次修正将文件源实时播放改为播放器式追帧：文件源默认按 `preview_render_fps` 采样，采集线程每轮按墙钟计算应该到达的源帧；如果当前帧落后，就用 `cap.grab()` 丢掉过期帧，保证视频时间接近 1.0x。`capture_max_side` 默认跟随 `preview_max_side=960`，避免 4K 帧先降到 1280 后预览再二次降到 960 的额外开销。

预览线程也同步调整为固定节拍读取 latest frame：不再先等待新帧再渲染，而是在每个预览 tick 直接取当前最新帧，必要时重复上一帧。这样文件源采样、检测 latest-only 和 MJPEG 预览三者解耦，避免“等帧 + 编码 + 睡眠”叠加导致预览 FPS 掉到 18fps 左右。

## 为什么会同时费性能又抽帧

当前 Web 预览链路把后端当作“视频播放器”和“检测器”同时使用：`MonitorEngine._backend_capture_loop()` 用 OpenCV 解码文件帧，`_render_backend_preview()` 缩放并绘制 HUD/检测框，`_preview_render_loop()` 再把画面编码成 JPEG，经 `preview.mjpg` 推给浏览器。这意味着 4K/60fps 文件不能直接走浏览器/显卡的 MP4 硬件播放能力，而是变成后端逐帧解码、缩放、绘制、JPEG 编码、HTTP 传输。

因此会出现两个矛盾：

- 如果不抽帧，后端需要尽量处理 60 张 4K 源帧/秒，CPU/内存带宽/JPEG 编码都会很重，检测线程即使 latest-only 也会被源帧链路拖累。
- 如果按实时墙钟追帧，系统会丢掉过期帧以保持 1.0x 播放速度，画面就会变成 25fps 左右的 MJPEG 预览，不再是原始 60fps 的丝滑播放。

从架构角度看，浏览器原生 `<video>` 播放 MP4 确实能降低后端解码/编码压力，但它不是当前项目的无代价最优解。复核原项目文档后，原项目明确采用后端 MJPEG 预览、latest-only 检测和 overlay 时间匹配作为实时监控取舍，而不是逐帧完整检测或前端原生播放器主导：

- `D:\security_project_d\Model_A\docs\技术.算法\倍速播放与检测采样密度.md` 记录：当前运行链路不是“每帧必检”，视频源按播放时钟推送预览帧，检测链路通过 latest-only 队列消费最新帧，并受 `detector_process_fps_cap` 限制。
- `D:\security_project_d\Model_A\docs\技术.算法\项目检测链路模块功能技术报告.md` 记录：`_backend_capture_loop()` 从 `cv2.VideoCapture` 读取帧，`_preview_render_loop()` 渲染 MJPEG，`PreviewBus`/`DetectionBus` 解耦，重点是“预览连续、检测最新帧”。
- `D:\security_project_d\doc\项目问题诊断报告.md` 记录：latest-only 会导致检测不是逐帧完整处理，但很可能是合理的业务/性能策略；如需离线全量诊断，应使用独立工具而非实时 monitor。
- `D:\security_project_d\doc\项目问题诊断报告.md` 同时记录 `/media/current` 可读取当前源文件存在数据暴露风险，这说明把原生 `<video>` 作为主预览链路还会放大本地视频文件暴露、鉴权和访问边界问题。

因此当前结论修正为：对实时 Web monitor，后端 MJPEG 预览 + latest-only 检测 + overlay hold/interpolate + 文件源墙钟追帧，是现阶段更符合原项目设计的折中方案。原生 `<video>` 只能作为后续架构实验方向，适合“浏览器流畅复核 + 后端低频检测叠加”的产品模式；若未同步解决 overlay 时间对齐、seek/倍速语义、文件访问安全、RTSP/摄像头兼容和检测采样密度解释，直接切换反而会引入新的问题。

## 间歇性卡顿与初始化判断

用户反馈 4K 横屏卡顿并非每次稳定复现，现象更像与管线初始化、缓存命中和内容触发分支有关，而不是单纯由像素决定。当前代码链路可确认以下因素：

- `src/defense/runtime/pipeline_factory.py` 的 `PipelineCache.get()` 只在 `profile`、`static_image_enabled`、自定义模型开关、模型路径、backend、model_family 或 config path 组成的 key 完全一致时复用 pipeline；否则会重新 `create_detector_backend()`、构造 `VideoDefensePipeline` 并执行 `pipeline.warmup()`。
- `src/defense/runtime/runner.py` 的 `MonitorEngine.start()` 在每次启动源之前会 `self.stop()` 当前 session，但不清空 `PipelineCache`。因此同模型/同 profile 再启动通常是热缓存；切换 profile、切换模型或切换静态图像开关会变成冷初始化。
- `src/defense/runtime/frame_processor.py` 的 `FrameProcessor.process()` 在 `show_boxes=true` 且历史平均处理时间仍在预算内时，可能根据 `PPEDisplayTracker.recommend_redetect_rois()` 对 1 个 ROI 再执行一次 `detector_backend.predict(crop)`。该分支依赖检测结果和画面内容，可能造成非稳定的单帧耗时尖峰。
- `src/defense/module_a/features/a3/static_media` 分支耗时也与画面内容和候选区域有关。虽然实时档位已默认关闭 multi-scale fallback，但静态媒体基础计算仍可能在不同画面中波动。

一次当前状态抽样显示：`init_ms` 约 `218ms`，末帧 `detector_inference_ms` 约 `4.7ms`，`a3b_static_media_ms` 约 `8.7ms`，`ppe_roi_redetect_ms` 约 `6.7ms`，`processing_ms` 约 `25.7ms`。这说明稳定帧不一定慢，但当冷初始化、PPE ROI 复检、A3b 内容分支或 GPU/CPU 调度叠加时，仍可能出现用户看到的短时卡顿。

后续若要继续优化，优先方向不是再次降低输入分辨率，而是增加可观测指标：记录 pipeline cache hit/miss、warmup_ms、首帧检测耗时、ROI 复检触发次数、A3b 分支耗时分位数和 detection queue dropped 变化。只有先区分“冷初始化慢”和“运行中偶发分支慢”，才能避免误把所有卡顿都归因于视频像素。

## 连续测试启动卡顿与抽帧修正记录

用户在 Web 控制台连续跑三次后反馈：检测开始前 1-2 秒会比较卡，并且仍能看到抽帧感。新增观测字段显示最近一次运行 `pipeline_cache_hit=false`，`init_ms` 约 `101ms`，`first_detection_processing_ms` 约 `12ms`；这说明模型首帧推理并不慢，但连续测试仍没有命中热 pipeline。

代码原因可确认：

- `src/defense/runtime/runner.py` 的 `MonitorEngine.start()` 在启动新 run 前调用 `stop()`。
- `src/defense/runtime/runner.py` 的 `MonitorEngine.stop()` 原先会无条件 `_release_pipeline_cache()`。
- 文件自然播放结束后，检测线程 finally 分支也会释放 pipeline cache。
- 因此连续点击开始时，即便 profile 和模型没有变化，也容易重新创建 TensorRT backend 并重新 warmup，造成浏览器端感知的启动不稳定。

本次修正：

- `MonitorEngine.start()` 调用 `stop(release_pipeline_cache=False)`，连续重新开始时保留同模型 pipeline。
- `PipelineCache.get()` 在 key 命中时复用 pipeline 并记录 `pipeline_cache_hit=true`；key 不命中时先关闭旧 pipeline，再创建新 backend，避免换模型/profile 后泄漏旧资源。
- 文件源正常播放结束后不再立即释放 pipeline，便于本地反复测试同一模型/同一 profile。
- 文件源预览最低提升到 `30fps`，并新增 `source_skip_frames_last` 和 `source_skipped_frames_total`，用于解释为了追实时播放而发生的源帧跳过。

当前结论：启动前 1-2 秒卡顿更接近“重复测试仍冷初始化 + 浏览器 MJPEG 流重新建立”的体验问题；抽帧感来自文件源为保持实时播放而主动跳过过期帧。修正后连续同配置测试应优先命中热 pipeline，启动抖动应下降；抽帧仍不会完全消失，因为实时 monitor 仍以“30fps 预览 + latest-only 检测”为主，而不是 60fps 原生播放器。

## 检测框拖框问题复现与修正记录

用户反馈优化启动和追帧后“拖框”再次出现。复查原项目文档可确认该类现象与 overlay 显示稳定策略有关：

- `D:\security_project_d\Model_A\docs\技术.算法\项目检测链路模块功能技术报告.md` 记录 `overlay_match_window_ms=180`、`overlay_hold_ms=550`、`overlay_interpolate_ms=400`、`overlay_max_age_ms=950`。
- 同文档说明 overlay 具备 hold/interpolate/max age 策略，因此检测框在倍速播放下可能仍然看起来连续。
- `D:\security_project_d\Model_A\docs\技术.算法\倍速播放与检测采样密度.md` 记录检测链路不是每帧必检，而是 latest-only 消费最新帧；检测框稳定来自 overlay 时间匹配、保持和插值机制。

当前问题判断：在 4K/60fps 文件源上，预览已提升到约 30fps，而检测仍按 `detector_process_fps_cap` 约 15fps 输出 overlay。若继续沿用 `overlay_hold_ms=550ms`，旧 overlay 会在多帧新画面上继续显示，视觉上就像框拖着目标走。该现象不是模型输出本身延迟，而是显示稳定策略在高帧率文件预览里的副作用。

本次修正：在 `src/defense/runtime/runner.py` 的 `_select_preview_overlay()` 中，仅对 `source_type=file` 且 `realtime=true` 的文件源预览收窄显示窗口。根据 `detector_process_fps_cap` 计算短 bridge，默认约 `80-140ms`；`match_s`、`hold_s`、`max_age_s` 被限制到该范围，`interpolate_s` 只保留检测帧之间的短插值。这样旧框不会再按 550ms 长时间挂在新画面上，代价是检测线程跟不上时框可能更早消失，显示会比原策略少一点“稳定感”。

## 回退记录

用户实测“连续检测启动与预览追帧”修正后出现约一秒一帧的严重卡顿。该现象说明提高文件源预览最低帧率和调整连续启动缓存策略可能引入新的调度/资源竞争问题，暂不能作为稳定优化保留。已回退提交 `33baf5a 优化连续检测启动与预览追帧`，恢复到上一版 `c3e75fe 增加运行时卡顿观测指标` 的运行策略；后续若继续处理，应先通过现有 `pipeline_cache_hit`、`init_ms`、`preview_fps`、`source_frame_step`、`source_skipped_frames_total` 等指标定位，再做更小粒度实验。
