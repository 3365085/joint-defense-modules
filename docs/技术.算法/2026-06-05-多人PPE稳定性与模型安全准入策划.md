# 多人PPE稳定性与模型安全准入策划

日期：2026-06-05

## 背景

用户在 Web 监控台复核多人场景时指出三个问题：

1. 多人、多人重叠时绘制框不稳定，可能出现两个人共用一个 `person` 框。
2. `head` 被误检或误显示为 `helmet`，这更像检测后处理和展示策略问题，不应简单归因于模型。
3. 远距离人员能检出 `person`，但 `head/helmet` 缺失，需要优化。

用户同时要求重新设计一套工程性的模型扫描、净化、白名单准入流程，解决干净模型、投毒模型在实际开始检测流程中的扫描处理盲区。

## 当前判断

多人重叠框问题存在明确代码风险。`PPEDisplayTracker` 的 `_same_person_target()` 原先同时使用 IoU、中心距离和 containment 判断同一 person；在重叠行人、前后遮挡、人体框高度相近但位置错开的场景中，中心距离与 containment 条件可能过度合并，导致 Web 可见框少于真实人员。

`head/helmet` 问题也存在后处理偏置。当前 Web 默认启用 `prefer_helmet_on_head_overlap`，并且展示层 `_select_cluster_representative()`、`_remove_shadow_tracks()`、业务层 `_apply_temporal_helmet_mutex()` 都偏向让稳定 `helmet` 覆盖同区 `head`。这对语义背心投毒视频有利，但在真实裸头、多人重叠、远距离小目标场景下，若仅靠中心距离就覆盖，会产生“head 被当作 helmet”的误导。

远距离人员不能直接从 `person` 推断 `head` 或 `helmet`。正确工程语义应是“PPE 未确认”或“弱证据待时序确认”，不能把只有 person 的情况直接变成未戴帽告警或安全帽证据。本轮采用低置信候选 + person 上半身上下文 + 时序提升的方向，不增加额外主路径推理。

## 已落地修正

1. 收紧 `person/person` 同目标合并。
   - 文件：`model/src/defense/module_a/postprocess/ppe_tracking.py`
   - `_same_person_target()` 从宽松 IoU/中心/containment 改为更强重复框条件：更高 IoU、更高小框覆盖率、相近面积比例、上下边缘一致性。
   - 目标：合并同一人的大小重复框，但不合并互相遮挡的两个人。

2. 收紧 `head/helmet` 互斥。
   - 文件：`model/src/defense/module_a/ppe_postprocess.py`
   - 新增 `bbox_min_overlap_ratio()` 与 `head_helmet_mutex_min_overlap`。
   - 中心距离命中不再单独足够，必须同时有最小几何重叠。
   - 文件：`model/src/defense/runtime/ppe_business.py`
   - 业务层 temporal helmet mutex 与后处理层使用相同 overlap 约束，避免显示层和业务计数分裂。

3. 调整展示层 head/helmet 代表选择。
   - 文件：`model/src/defense/module_a/postprocess/ppe_tracking.py`
   - `helmet` 只有在置信度明确领先时才覆盖 `head`；否则保留 `head`，防止低置信 helmet 抢占真实裸头。

4. 远距离弱证据补偿。
   - 文件：`model/src/defense/module_a/backends/detector_backend.py`
   - `candidate_confidence` 对三类模型也生效，使低置信 head/helmet 候选能进入后处理。
   - 文件：`model/src/defense/module_a/ppe_postprocess.py`
   - 有 person 类时，低置信 head/helmet 只有落在 person 上半身区域内才作为 weak temporal candidate；仍不直接触发告警。

5. 模型安全报告复用防护。
   - 文件：`model/src/defense/model_security/service.py`
   - `admission_status()` 加载净化报告时校验当前 `source_pt_hash`；旧 source PT 的净化报告不能让当前模型进入 `purified_alternative_available`。
   - 后续补充：当当前 source PT 可计算时，旧 scan/purification report 若缺失 `source_model_hash` 也不得复用；`_trusted_pt_for_export()`、`trusted_purified_runtime_model()`、`purify()` 与状态查询使用同一套 source hash 过滤。

## 当前模型安全链路

实际 Web 启动路径如下：

1. 前端提交 `custom_model`，包含 `enabled/path/backend/model_family/source_pt_path/class_names`。
2. `/api/start` 先调用 `ModelSecurityService.prepare_runtime_for_start(auto_remediate=True)`。
3. `normalize_custom_model_options()` 和 `apply_custom_model()` 把自定义模型写入 runtime config。
4. `build_model_fingerprint()` 绑定 runtime artifact hash、backend、model family、image size、confidence、class names hash、PPE mapping hash、scanner version。
5. `admission_status()` 查询白名单；白名单记录必须匹配 runtime model hash、source PT hash、class names hash、PPE mapping hash、scanner version。
6. 未命中白名单时，启动被阻断并触发 full scan。
7. full scan 为 `clean/trusted` 才写入白名单。
8. full scan 为 `suspicious` 时才能净化；净化候选必须复扫 `clean/trusted` 后，才写入净化模型白名单。
9. 准入通过后，`runner.py` 和 `pipeline_factory.py` 才加载实际检测 backend。

## 模型安全盲区

1. 加速模型源 PT 推断仍是策略风险。
   - 当前 `.engine/.onnx` 若未传 `source_pt_path`，服务会尝试同名 `.pt` 或 sibling `best.pt`。
   - 风险：实际运行的 engine 可能对应毒模型，扫描却误扫旁边干净 PT。
   - 结论：这是准入设计问题，不应只靠 UI 提示解决。

2. 测试绕过入口过宽。
   - `/api/start` 仍接受多个 bypass alias。
   - 风险：测试入口误传到非测试流程，会直接绕过 B 模块准入。

3. class names 仍可能只告警不阻断。
   - fingerprint 绑定 class names hash，但未知自定义模型的内嵌类名不一定可读。
   - 风险：扫描目标类与实际检测类语义错位，尤其是 `person/head/helmet` 顺序错配。

4. runtime catalog 不是白名单。
   - catalog 存 scan/purification/evidence 资产记录，但准入不直接信任 catalog。
   - 这是安全正确的，但页面必须避免把 catalog 的 `trusted` 文案误解为“当前模型已白名单放行”。

## 完整工程策划

### 目标原则

1. 运行前必须有唯一 `ModelAdmissionContext`。
2. 白名单只信任 context，不信任路径名、目录邻居、catalog 文案。
3. source PT、runtime artifact、class mapping、PPE mapping、scanner version 任一变化，都必须重新进入准入判断。
4. 净化产物必须绑定 original source hash 和 purified runtime hash。
5. 未知模型可以扫描、可以净化，但不能静默运行。

### 目标流程

1. Intake：接收自定义模型。
   - PT/PTH：runtime artifact 与 source PT 默认同一文件。
   - ENGINE/ONNX：必须显式提供 source PT；没有 source PT 时返回 `unverifiable_requires_source_pt`。
   - class names 必须显式确认 `person/head/helmet` 映射；缺失时返回 `class_names_unverified`。

2. Context：生成 `ModelAdmissionContext`。
   - 字段包括 `runtime_model_hash`、`runtime_model_path`、`source_pt_hash`、`source_pt_path`、`backend`、`model_family`、`class_names_hash`、`ppe_mapping_hash`、`profile`、`scanner_version`。
   - scan report、purification report、registry record 都以该 context 作为校验对象。

3. Admission：运行前准入。
   - 命中白名单且 context 全字段匹配：允许启动。
   - 已知七实验毒模型 hash：阻断原模型，给出 paired purified candidate。
   - 已知七实验净化模型 hash：若证书与 class mapping 匹配，可直接 clean/trusted。
   - 未知模型：阻断并 full scan。

4. Scan：完整扫描。
   - 默认 target classes 为 `helmet/head`。
   - `person` 作为 context/preserve/suppressor，不作为默认攻击目标。
   - full scan clean/trusted 写白名单；review 不写白名单；suspicious 可进入净化。

5. Purify：净化与复扫。
   - 净化输入必须是当前 context 的 source PT。
   - 每个 candidate 复扫时生成新的 purified context。
   - 只有 purified full scan clean/trusted 才写入净化模型白名单。
   - registry 记录必须保存 `original_source_model_hash` 与 `purification_report_path`。

6. Runtime Replacement：启动替换。
   - 原模型 suspicious 且存在通过复扫的 purified PT 时，`prepare_runtime_for_start()` 自动替换为 purified runtime custom model。
   - 替换前再次断言 purified registry record 与 original source hash 匹配。

7. UI 状态。
   - 拆分 `admission_status` 与 `admission_reason_code`。
   - 增加 `source_pt_resolution=explicit/self/missing/inferred_legacy`。
   - 增加 `purification_reuse_checked=true/false`。
   - catalog 显示为“资产记录”，白名单显示为“准入记录”。

## 后续建议

1. 下一轮先把 `.engine/.onnx` silent sibling source PT 推断降级为 legacy 警告，再逐步改为强制显式 source PT。
2. 收紧 `/api/start` 的测试绕过入口，只保留一个测试字段，并绑定测试 profile 或环境开关。
3. 增加 Web 页面上的 `source_pt_resolution` 与 class mapping 确认状态。
4. 对多人重叠和远距离 PPE，必须用真实视频抽帧验证。当前代码层修正已经有单元测试覆盖，但“远距离小目标召回率提升”仍需实际视频确认，不能只从单元测试下结论。
