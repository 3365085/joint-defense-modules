## 背景

用户提供 `D:\联合防御模块\b模块新算法\backbone_soup_full_pipeline_v2_2026-05-24`，要求将新的净化方法完整接入当前 AB 模块融合分支，并跑通净化测试。

## 当前接入结论

- 新净化包源码已接入 `model/src/model_security_gate/autodetox/`、`model/src/model_security_gate/detox/ccsync/`、`model/src/model_security_gate/detox/ccsync_yolo/`、`model/src/model_security_gate/detox/hybrid_purify/`、`model/src/model_security_gate/detox/oc3/`。
- 当前业务闭环优先接入可自动执行、可复现的 Backbone-Soup 路径，服务入口位于 `model/src/defense/model_security/service.py::ModelSecurityService.purify`。
- 净化执行封装位于 `model/src/defense/model_security/purifier.py::run_new_purification`，会解析源 PT、查找 clean anchor、生成 AutoDetox plan，并调用 `model/src/model_security_gate/detox/weight_soup.py::build_weight_soup_candidates` 输出净化 PT。
- Web/API 接入口为 `model/src/defense/web/fastapi_app.py::model_security_purify`，路径是 `POST /api/model-security/purify`。
- 前端按钮位于 `model/src/defense/web/static/index.html` 的 `Run Detox`，状态轮询函数为 `pollModelSecurityPurification`。

## 新旧方法差异

- 旧路径以 `asr_closed_loop_train.py`、`strong_pipeline.py`、`hybrid_purify_train.py`、`weight_soup.py` 等独立训练/剪枝/蒸馏/回归修复脚本为主，业务层没有统一净化任务状态。
- 新包新增 AutoDetox 控制器、Backbone-Soup、CCSync-YOLO、OC3、Hybrid-PURIFY-OD 等路径。当前业务闭环先接 Backbone-Soup，因为它只依赖 `poisoned/source PT + clean baseline PT`，最适合在 Web 准入层自动执行。
- CCSync-YOLO、OC3、Hybrid-PURIFY 已作为源码能力接入，但还没有在业务层做自动路由执行。原因是它们需要触发图像池、witness 数据或训练/评估数据集，不能在缺少数据的情况下伪造成功。

## 后续建议

- 默认模型如缺少 clean anchor，只能生成 AutoDetox plan，不能声称已净化。
- 若要把 `.engine` 放行，应在净化 PT 通过 full scan 后，再接导出 engine 和 PT/engine 哈希绑定。
- CCSync/OC3/Hybrid 自动路由需要先明确 witness/trigger/baseline 数据来源，否则只能保留为可调用算法能力。
