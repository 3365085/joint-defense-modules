# adv_patch 空场景"帧帧 ATTACK"根因与持续升级静止背景排除修正

日期: 2026-07-09
分支: codex/demo-module-a-port (未提交)
涉及文件: `model/src/defense/module_a/rebuilt/detector.py`

## 1. 问题背景

用户在验收 adv_patch 攻击视频 (`素材/物理扰动攻击视频/adv_patch/raw_attack_adversarial_patch.mp4`)
的抽帧时提出质疑: **画面全程是空旷地面, 看不到人, 但 HUD 却帧帧显示 ATTACK (p_adv 0.94-0.99), 报警持续不断。**

## 2. 事实核查 (来自 attack_manifest.jsonl, 权威真值)

对该 300 帧采样清单逐帧统计:

- `person` 目标: **0 帧** (全程无人)。
- `helmet` 目标: 300 帧 (每帧 2-3 顶安全帽, clean_conf≈0.82)。
- `attack_active`: 帧 30-283; `effective_attack_frame` / `adv_patch_success`: 帧 44-283。
- 帧 0-29 是**攻击前缀 (未布设补丁)**, confidence_drop=0。

对照五类物理攻击视频清单 (adv_patch/glare/motion_blur/occlusion/visibility): **全部 person=0、helmet=300**。
即这批合成攻击素材的受保护目标是**安全帽 (PPE)**, 而非人; "空场景无人"是素材本身特性, 不是检测 bug。

结论: 用户的观察属实——这些是**无人、仅安全帽的合成攻击片**, 补丁贴在"person 锚点"位置但源视频从未渲染出人。
它可以测"补丁纹理是否被 A1-A4 感知", 但**不能**验证"遮挡真实作业人员"。

## 3. 报警链路拆解 (frames.csv, 前 400 帧)

- **原始 p_adv 从帧 6 起持续 0.77-0.97**: XGBoost 传感器对合成补丁/地面高频纹理的响应。
  帧 0-29 (补丁尚未布设) 也高 → 属传感器对该合成场景纹理的**伪高分**。此为已训练模型的固有输出,
  不在本次行为调优范围 (改动分类器需专门重训, 违反 AGENTS 纪律)。
- **逐帧抑制门判定正确**: 帧 6-149 `suppressed_reason=pure_static_background`
  (无 target_related + 无运动 flow_gap<0.25 + warp_residual<0.10), `adv_candidate_allowed=False`,
  `adv_count=0`。即系统**已正确判定这些帧为"纯静止背景、非攻击候选"**。
- **持续对抗升级 (sustained escalation) 越权**: 该逻辑逐帧累计"原始 p_adv≥θ 的连续段" `_adv_run`,
  且冷启动时 `_benign_run_ref=0` → 场景相对门槛 `1.6×ref=0` → 升级退化为"仅时间下限 floor"。
  离线批处理 `process_fps` 逼近 60fps 上限 → floor≈round(60×2.0)=120 → 约帧 120 把一段
  **全是 pure_static_background、无目标**的连续段凭空升级为确认, 叠加 alert_hold 持续到 ~189。

**根因**: 持续升级本是为救援"被场景自适应基线 (scene_baseline_normal) 打散的**真实候选**",
却把"无目标纯静止背景的 XGBoost 伪高分帧"也累计进连续段, 制造出空场景的确认报警。

## 4. 修正 (config-gated, 默认开)

`detector.py` 新增 `rebuilt_sustained_adv_exclude_static_bg` (默认 True):
持续段 `sustained_hit` 增加条件——当本帧 `无 target_related` 且被判为
`background_static_suppressed` 家族 (含 pure_static_background) 时, **不计入 `_adv_run`**。
即持续升级只在"真正有攻击证据 (目标相关/运动/结构) 的连续段"上生效, 不再从静止背景伪高分凭空升级。

纯 CPU 标量判断, 不增加任何 GPU 推理, 不改动分类器/阈值/PPE 语义。

## 5. 双口径验证

| 指标 | 修正前 (adaptive) | 修正后 | 结论 |

| 干净误报 | 2/27 = 7.4% (015,016) | **2/27 = 7.4% (015,016)** | 不变, 无新增 |
| 分类型命中 | adv3/glare4/mb5/occ3/vis4 | **完全一致** | 不变 |
| adv_patch 片确认帧 (前400) | ~70, 首确认~120 | **28, 首确认162** | 空场景/静止段伪报警清除 |
| 该片持续升级触发帧 | 有 | **0** | 升级路径在该片彻底不再越权 |

留出召回**不变**说明: 持续升级此前对留出集真实召回**无贡献**, 90.5% 来自另两项修正
(候选连续性桥接 + C1 场景基线 A1-only 豁免); 它在 adv_patch 合成片上的额外报警**全部是静止背景伪报**。
修正后 adv_patch 片确认帧回落到 28 (即加持续升级之前的真实检出水平), 且仅落在真实逐帧候选段 (帧162-189,
此时有安全帽目标 + 补丁激活), 符合"对带攻击画面报警"的诉求, 同时消除空场景误报。

## 6. 结论与后续建议

1. **修正有效且安全**: 消除空场景/静止背景伪报警, 留出召回与误报零回退。(已验证)
2. **该合成 adv_patch 片是弱验收目标**: 全程无人、仅安全帽、合成贴片。真实验收应以留出场景集
   (48 片 / 26 场景, 召回 90.5% / 误报 7.4%) 为准。
3. **待实验确认**: `process_fps` 在离线批处理下逼近 60fps 上限, 使"2 秒持续先验"的 floor 变成
   固定 ~120 帧, 与视频真实时序脱钩; 在线流式下才等于真实帧率。是否给持续升级改用"视频源 fps
   / 帧序"而非"墙钟处理速率", 需另做实验确认, 本次未改。
4. 用户诉求 (#7 "即使模型被污染, 检测器仍应对带攻击画面报警") 仍满足: 补丁激活且有目标的真实候选段
   (162-189) 依旧确认报警; 仅去掉了无证据的静止背景伪报。
