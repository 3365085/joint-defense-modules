## 2. 当前准入闭环总览

### 已从代码确认

当前项目的自定义模型安全准入链路可以概括为：

```text
前端/调用方 custom_model
  -> FastAPI /api/start 或 /api/model-security/*
  -> ModelSecurityService 加载 runtime config 并构造指纹
  -> trust registry seal 校验
  -> admission_status 状态机
  -> 启动准入只读判定 / 显式 full scan / 显式 purify / purified runtime replacement
  -> MonitorEngine.start 只接收已准入或测试绕过后的 runtime custom_model
```

准入的主状态来自 `ModelSecurityService.admission_status()`：

- `trusted`：白名单命中，允许启动 A 模块。
- `blocked_scan_required`：未进入白名单，默认阻止启动并要求完整扫描。
- `scanning`：目标模型正在扫描，阻止启动。
- `suspicious`：最近 full scan 发现可疑或已知投毒，阻止原模型启动，可进入净化。
- `purifying`：目标模型正在净化，阻止启动。
- `purified_alternative_available`：原模型不可信，但已有复扫通过的净化 PT，可作为 runtime replacement。
- `review` / `unverifiable`：扫描结果不可直接信任，阻止启动。
- `trust_store_compromised` / `error`：白名单或服务状态异常，阻止启动。

`/api/start` 当前调用 `prepare_runtime_for_start(profile, custom_model, auto_remediate=False)`，且底层 `prepare_runtime_for_start()` 默认值也已收紧为 `False`。启动 API 只做准入解析，不在启动请求内同步扫描或净化：

1. 如果当前模型已 `trusted`，直接把请求中的 `custom_model` 交给 `MonitorEngine.start()`。
2. 如果当前模型为 `purified_alternative_available`，尝试解析并选择已信任的净化 PT，作为 `runtime_custom_model` 启动。
3. 如果当前模型为 `blocked_scan_required`，返回 409 `model_security_blocked`，并给出 `next_action=start_full_scan`。
4. 如果当前模型为 `suspicious`，返回 409，并给出 `next_action=start_purification`。
5. 如果当前模型为 `review/unverifiable/trust_store_compromised/error`，返回 409 或错误状态，并给出可执行的 `next_action/operator_message`。

这意味着“干净模型”和“毒模型”在实际检测开始前的处理分叉仍然存在，但重任务已经从 `/api/start` 同步路径移到 `/api/model-security/scan` 与 `/api/model-security/purify` 等显式后台流程。

## 4. 指纹、source PT 与白名单匹配

### 已从代码确认

`build_model_fingerprint()` 将下列内容纳入模型运行身份：

- 运行 artifact 的 SHA256。
- 选中的 artifact 路径。
- backend。
- model_family。
- image_size。
- confidence。
- nms_iou。
- class names hash。
- PPE mapping hash。
- scanner_version。

白名单记录不是只按文件 hash 命中。`_trusted_record_matches()` 要求：

- `approved_for_runtime=True`。
- `scanner_version == model_security_runtime_v2`。
- `runtime_model_hash == 当前运行 artifact hash`。
- `class_names_hash == 当前 class names hash`。
- `ppe_mapping_hash == 当前 PPE mapping hash`。
- `source_model_hash == 当前解析出的 source PT hash`。

因此，只替换模型文件、改类别映射、改 PPE 后处理映射、换 source PT，都会导致旧白名单失效。

`_source_pt_path()` 的解析顺序包括：

1. `runtime.custom_model.path` 若是 `.pt/.pth`，直接作为 source PT。
2. `runtime.custom_model.source_pt_path/source_model_path/source_path`。
3. 对非 PT 自定义 runtime artifact，不再猜测同名 `.pt` 或同目录 `best.pt`，必须显式提供 source PT。
4. 非自定义模型时从 `model_security.source_pt*`、`inference.artifacts.pytorch/pt`、当前 artifact 同名 `.pt`、同目录 `best.pt` 中寻找。

这对 TensorRT/ONNX 等加速模型很关键：它们要想通过 full scan 与白名单，必须能回溯到原始 PT，否则 full scan 会进入 `unverifiable`。

## 5. quick scan 与 full scan

### 已从代码确认

`quick_scan()` 是结构性快速探测：

- 可读取 activation cache。
- 对 artifact 做 entropy sample。
- 尝试调用 ABS-style activation probe。
- 输出状态可为 `trusted`、`review`、`suspicious`。
- 写入 quick report，但当前 `ModelSecurityService.scan()` 只有 `scan_type == "full"` 且状态为 `trusted/clean` 时才会写白名单。
- `trust_if_low_risk` 参数在 `scan()` 中被显式丢弃，不会因为 quick scan 低风险而写白名单。

`full_scan()` 是准入级扫描：

- 必须有原始 `.pt/.pth` source model。
- 如果没有 source PT、source PT 不存在、source PT 不是 PT/PTH，返回 `unverifiable`，风险分 1.0。
- 先进行已知投毒模型哈希匹配。
- 再进行已知净化模型严格证书/归档哈希匹配。
- 若不是已知模型，则要求验证集资产可用，并运行 external hard suite。
- external hard suite 使用 PyTorch source model 作为验证模型。
- 默认目标类为 `helmet/head`，`person` 作为 context/preserve。
- `max_asr <= external_eval_allowed_max_asr` 时返回 `clean`。
- `max_asr >= external_eval_suspicious_asr` 时返回 `suspicious`。
- 二者之间返回 `review`。
- 无可评分 rows 时返回 `unverifiable` 或 `review`，不会写白名单。

`scan_type == "full"` 且结果为 `trusted/clean` 时，`_mark_clean_full_scan_trusted()` 会自动写入白名单，approval_source 为 `full_scan`。

### 待实验确认

本轮未运行 `pixi run` 测试或 full scan 实验，因此以下内容只做代码路径判断，尚未确认运行时表现：

- quick scan cache 对不同 custom_model 参数变更的实际命中/失效表现。

## 6. 已知毒模型与已知净化模型

### 已从代码确认

已知毒模型检测分两类：

1. 七实验归档：`purification_lab/seven_experiment_archive/manifest.json` 与各实验 `summary.json` 记录 poisoned/purified checkpoint 哈希。`packaged_poisoned_evidence_for_model()` 对 source PT 做哈希匹配，命中后返回 `seven_experiment_known_poisoned_archive`。
2. 新 B 模块算法包：在 `b模块新算法/.../models/poisoned` 中按哈希匹配 packaged poisoned model，命中后返回 `new_algorithm_known_poisoned_catalog`。

命中已知毒模型时，`full_scan()` 直接返回：

- `status: suspicious`
- `risk_score: 1.0`
- reason 表示模型 hash 命中已知投毒归档或新算法包
- diagnostics 中包含 poisoned evidence、family tag、候选净化模型等证据

已知净化模型检测也分两类：

1. 七实验 purified archive：`packaged_strict_certification_for_model()` 对 purified checkpoint 做哈希匹配，命中后返回 `seven_experiment_purified_archive`，full scan 直接给 `clean`，风险为 0.0。
2. 新算法 strict audit：文件名含 `purified`，哈希匹配 packaged purified model，且 strict audit 记录满足 `strict_pass && wilson_upper <= 0.05 && mAP_drop_pp <= 5.0`，full scan 返回 `clean`。

这些判断均为 hash-bound。单纯把随机文件改名为 purified 不会继承证书。

## 7. 净化与 runtime replacement

### 已从代码确认

净化入口为 `ModelSecurityService.purify()`，只允许在最近 full scan 状态为 `suspicious` 时执行。若没有 suspicious full scan，会抛出 `purification_requires_suspicious_full_scan`；若原模型已经 clean，会抛出 `source_model_already_clean`。

净化实现 `run_new_purification()` 的主要路径：

- 要求 source PT 存在且为 `.pt/.pth`。
- 查找 clean anchor。
- 写 AutoDetox plan。
- 优先 staging packaged strict purified candidates，尤其是七实验 poisoned 模型对应的 paired purified checkpoint。
- 若没有 clean anchor 但有 packaged candidates，则选择 packaged candidate，状态为 `candidate_generated`，并要求后续 full scan。
- 若有 clean anchor，则构造 weight soup candidates，并复制到源模型目录，命名策略为 `净化完毕` 后缀。
- 返回 purification report，包括 `purified_model_path`、`purified_model_hash`、candidate manifest 与诊断信息。

`purify(scan_after=True)` 会对净化候选逐个执行 full scan 或严格候选报告：

- 第一个 `trusted/clean` 候选被接受。
- 写入 purification report，状态改为 `scan_clean_trusted`。
- 将净化模型注册到 runtime catalog，category 为 `purified_model`，artifact_type 为 `trusted_purified_pt`。
- 将净化模型指纹写入白名单，approval_source 为 `purified_full_scan`。
- 记录原始毒模型 source hash 与净化模型 hash 的关联。

`trusted_purified_runtime_model()` 只在以下条件全部满足时返回 runtime replacement：

- 原模型对应的 purification report 存在。
- report 状态为 `scan_clean_trusted`。
- `purified_model_path` 存在且是 PT/PTH。
- 净化 PT 自身的 admission_status 为 allowed。
- 净化 PT 的 fingerprint 在 registry 中有记录。

返回的 runtime model 固定使用：

```text
enabled=True
path=<purified_pt>
backend=pytorch
model_family=<原模型 family 或 auto>
source_pt_path=<purified_pt>
```

因此，当前 runtime replacement 的安全口径是：不让原 suspicious 模型进入检测，而是在检测开始前切换到已经复扫并写白名单的净化 PT。

## 8. Web API 准入分叉

### 已从代码确认

`/api/model-security/status`

- GET 按 profile 查询默认模型安全状态。
- POST 可带 `custom_model` 查询自定义模型状态。

`/api/model-security/scan`

- `scan_type` 仅允许 `quick/full`，非法值回退为 `quick`。
- 默认 `background=True`。
- full scan 默认 `auto_purify=True`。
- background 时调用 `start_background_scan()`，返回 job 状态。
- 非 background 时同步调用 `scan()`。

`/api/model-security/purify`

- 默认 `background=True`，`scan_after=True`。
- 先调用 `status()`，仅当 `can_purify=True` 才允许。
- 否则返回 409 `purification_requires_suspicious_full_scan`。

`/api/model-security/trust`

- POST 手工新增信任被禁用，返回 403 `manual_trust_disabled`。
- GET 可列白名单。
- delete/clear 只允许删除或清空白名单。

`/api/runtime/artifacts`

- 查询 runtime catalog，可按 business_domain/category 过滤。

`/api/start` / `/api/runs/start`

- 默认走模型安全准入。
- 只保留 `test_bypass_model_security` 作为本地/显式测试绕过字段；`bypass_model_security_for_test` 与 `bypass_model_security` 已拒绝。
- 非测试绕过时，会调用 `prepare_runtime_for_start(auto_remediate=False)`。
- 阻止启动时返回 409，不自动排队 full scan 或 purification，只返回 `next_action/operator_message`。

### 风险提示

测试绕过入口已收窄为单一 canonical key，且需要本地安全策略或显式环境开关。生产 UI 与自动化脚本仍应避免发送该字段。

## 9. 干净模型/毒模型在实际检测开始前的处理规则

### 已从代码确认的当前行为

干净模型：

1. 若 registry 已有匹配记录，直接允许启动。
2. 若未知，`/api/start` 返回 409 与 `next_action=start_full_scan`，不会同步扫描。
3. 调用 `/api/model-security/scan` background full scan，通过后会写白名单；后续 `/api/start` 允许启动。
4. quick scan 低风险不会写白名单，不能作为生产准入依据。

已知毒模型：

1. full scan 命中 poisoned archive/package hash 后返回 `suspicious`。
2. 原模型不会被写入白名单，也不会以原始 custom_model 启动。
3. 需要显式 background purify 或 background full scan `auto_purify=True` 才进入净化。
4. 若净化候选复扫为 `clean/trusted`，净化 PT 写入白名单。
5. 启动前选择净化 PT 作为 `runtime_custom_model`，并在响应中携带 `model_security_runtime_replacement`。

未知模型：

1. 未命中白名单时初始状态为 `blocked_scan_required`。
2. `/api/start` 只返回 409 与 full scan 操作建议，不会同步 full scan。
3. full scan 无 source PT、无验证集、target policy 错误或 external validation 运行失败时返回 `unverifiable`，阻止启动。
4. full scan `review` 阻止启动，不自动净化。
5. full scan `suspicious` 可净化。

加速模型：

1. 若 registry 中 runtime artifact hash、class/PPE hash、source PT hash 均匹配，则允许启动。
2. 若缺少可解析 source PT，full scan 返回 `unverifiable`。
3. 已信任 PT 可导出 ONNX/TensorRT，导出结果会以 `trusted_accelerated_export` 逻辑写白名单和 runtime catalog。

### 建议形成的目标准入规则

检测开始前应遵循以下硬分叉：

- `trusted`：允许启动当前 artifact。
- `clean full scan but not yet trusted`：应立即写 registry 后再启动，不直接绕过 registry。
- `known poisoned / suspicious`：禁止启动原模型；只允许净化后 PT 复扫通过并写白名单，再进行 runtime replacement。
- `purified_alternative_available`：允许启动净化 PT，不允许启动原模型。
- `review`：禁止启动，要求人工/算法复核或补充验证集，不应自动净化为生产可用。
- `unverifiable`：禁止启动，必须补 source PT、验证集或修正目标类映射。
- `trust_store_compromised`：禁止启动，并要求恢复 registry/seal。
- `bypassed_for_test`：只允许显式测试环境，响应与日志必须明显标记，不得进入生产 UI 默认路径。

## 10. 错误流程风险与改造建议

### 风险 1：启动同步执行 full scan/purify 的风险已收紧

已从代码确认：`/api/start` 显式调用 `prepare_runtime_for_start(auto_remediate=False)`，底层默认值也是 `False`，不会在启动请求内同步调用 `scan(scan_type="full")` 或 `purify(scan_after=True)`。

当前效果：

- 首次启动未知模型直接 409，并提示显式后台 full scan。
- full scan/purify 的耗时与错误归属到 B 模块后台任务，不再混入 A 模块启动。
- 避免启动检测时额外占用 GPU 执行扫描或净化。

仍需保持：

- 由前端或调用方显式触发 `/api/model-security/scan(background=true, scan_type=full, auto_purify=true)`。
- full scan/purify 完成后再重试 start。
- 不新增生产默认 `auto_remediate_on_start`；如未来需要，只能作为本地调试开关并有明显日志。

### 风险 2：测试绕过入口命名过宽

已从代码确认：`/api/start` 只接受 `test_bypass_model_security`；旧 key `bypass_model_security_for_test` 与 `bypass_model_security` 会返回 403。

当前效果：

- 宽命名 key 不能再启动未准入模型。
- canonical key 也受本地策略或环境变量限制。

仍需保持：

- 测试绕过时在 runtime status 中持续暴露 `model_security_bypassed=True`，避免启动后状态页失去风险上下文。

### 风险 3：配置中的 unknown policy 与实际启动逻辑存在语义落差

已从代码确认：配置有 `unknown_model_policy: warn`，但 `/api/start` 实际未按 warn 放行，而是通过 admission 状态阻止未知模型并返回 full scan 建议。

影响：

- 运维或前端人员可能误读配置，以为未知模型只是警告。
- 不同入口若未来直接读取 config policy，可能产生不一致行为。

建议：

- 将配置字段收敛为明确的 `startup_unknown_model_action: block`。
- 如果保留 `warn`，必须限定为 UI 提示，不得影响 `/api/start` 准入。
- 在 `status()` payload 中增加 `effective_startup_policy`，展示实际生效策略。

### 风险 4：source PT 推断可能误绑定同目录 `best.pt`

已从代码确认：非自定义模型仍可按配置和默认 artifact 规则推断 source PT；自定义非 PT runtime artifact 已不再猜测同名 `.pt` 或同目录 `best.pt`，必须显式提供 `source_pt_path/source_model_path/source_path`。

当前效果：

- 自定义加速模型不会误把同目录历史 PT 当作 full scan 来源。
- 缺失 source PT 时进入 `unverifiable`，并通过 `source_pt_resolution` 指出显式 source PT 必需。

仍需保持：

- status 中持续暴露 `source_pt_resolution`。
- full scan report 后续可写入更详细的 `source_pt_resolution_reason`，便于审计。

### 风险 5：quick scan 状态名 `trusted` 容易被误解

已从代码确认：quick scan 可返回 `trusted`，但不会写白名单。

影响：

- 前端或用户可能误以为 quick trusted 等于生产准入。

建议：

- 将 quick scan 的低风险状态展示为 `quick_low_risk` 或在 API payload 增加 `admission_effective=false`。
- 前端文案明确：quick scan 只用于预筛，不能启动生产检测。

### 风险 6：`review` 状态没有后续闭环

已从代码确认：`review` 会阻止启动，但 `can_purify` 只对 `suspicious` 为 true。

影响：

- ASR 介于 allowed 与 suspicious 之间的模型会停留在 review，缺少可执行下一步。

建议：

- 增加 `review_next_actions` 字段：补充验证集、扩大 hard suite、人工批准禁止、重新训练/净化建议。
- 不建议让 review 自动写白名单或自动净化后放行；可以允许生成净化计划，但必须复扫 clean 后才能准入。

### 风险 7：runtime replacement 的响应语义应更强

已从代码确认：成功 runtime replacement 时响应包含 `model_security_runtime_replacement`，但 engine status 中仅保存最终 `custom_model`。

影响：

- 用户启动后可能只看到“正在用某个 PT”，不清楚它是替代原毒模型的净化 PT。

建议：

- 在 `MonitorEngine.start()` status 中增加 `model_security_runtime_replacement` 或 `original_custom_model`。
- 日志中保留 source poisoned fingerprint、purified fingerprint、两个 report path。
- 前端状态页显示“原模型已阻断，当前运行净化模型”。

### 风险 8：白名单 seal 绑定本机，迁移场景需要操作指引

已从代码确认：trust registry seal 使用当前 host fingerprint 校验。registry hash、host fingerprint 或 signature 不匹配都会进入 `trust_store_compromised`。

影响：

- 复制 runtime/model_security 到另一台机器会导致白名单不可用。
- 这是安全设计，但需要用户可理解的恢复流程。

建议：

- 在 UI/status 中把 `trust_store_compromised` 的 reason 转译为可执行说明。
- 提供“清空本机白名单并重新扫描”入口，不提供手工新增信任入口。

## 11. 建议目标闭环

### P0：收紧启动准入

- `/api/start` 默认只做准入判定，不做 full scan/purify 重任务。
- `blocked_scan_required` 返回 409 + background scan 建议。
- `suspicious` 返回 409 + purify 建议。
- `purified_alternative_available` 可自动切换净化 PT。
- 只允许 `trusted` 或可信净化 replacement 进入 `MonitorEngine.start()`。

### P1：明确未知模型处理

- 未知模型必须先 full scan。
- full scan 只能基于 source PT。
- 加速模型必须显式携带 source PT 才能进入生产扫描。
- quick scan 只做预筛，不能写白名单。

### P1：完善毒模型处理

- 已知毒模型 full scan 命中后立即阻断原模型。
- 若有 paired packaged purified candidate，自动 staging。
- 净化候选必须 full scan clean 后写白名单。
- 启动时只运行净化 PT，不运行原毒 PT。

### P1：完善净化模型处理

- 已知净化归档 hash 命中可直接 full scan clean，并写白名单。
- 新算法 strict audit 命中也可 clean，但必须保留 strict audit 证据。
- 所有净化准入记录都应关联 original_source_model_hash，便于审计。

### P2：增强 API 可观测性

- status 增加 `effective_startup_policy`。
- status 增加 `source_pt_resolution`。
- start 成功响应和 runtime status 增加 replacement 溯源。
- report 增加 `next_action` 与 `operator_message`。

### P2：前端/操作流建议

推荐用户流：

```text
选择 custom_model
  -> 查询 /api/model-security/status
  -> 未知：启动 background full scan
  -> clean：提示可启动
  -> suspicious：启动 background purify + scan_after
  -> scan_clean_trusted：提示将使用净化模型启动
  -> review/unverifiable：展示修复项，不允许启动
```

## 13. 本轮结论

### 已从代码确认

当前项目已经具备“扫描 -> 净化 -> 复扫 -> 白名单 -> 启动替换”的主闭环。核心安全原则是：

- 原始未知模型不能凭 quick scan 进入生产检测。
- full scan clean 才会自动写白名单。
- 已知毒模型 full scan 直接 suspicious，不允许原模型启动。
- 净化模型必须复扫 clean 并写白名单后，才能作为 runtime replacement。
- 白名单校验绑定 runtime artifact、source PT、class names、PPE mapping、scanner version 与本机 seal。

### 后续保持重点

当前最大工程风险已从启动链路中收紧：`/api/start` 不再同步执行 full scan/purify，旧测试绕过 key 已拒绝，自定义加速模型 source PT 也不再猜测同目录 `best.pt`。后续应继续强化 runtime replacement 的前端可观测性、trust store 恢复指引，以及显式 scan/purify 的批量验证材料。

## 14. 本轮 P0 收紧补充记录

时间：2026-06-06

### 背景

上轮代码审查后确认还存在一个 P0 边界缺口：默认/生产加速模型在没有显式 `source_pt_path` 时，仍可能通过同名 `.pt` 或同目录 `best.pt` 猜测源 PT。这个行为会把 full scan 绑定到错误训练检查点，导致未知 engine/onnx runtime 被错误地借用其他 PT 的扫描与白名单结论。

### 当前调整

1. `ModelSecurityService._source_pt_candidates()` 已移除默认/生产加速 artifact 的 `runtime_path.with_suffix(".pt")` 和 `runtime_path.parent / "best.pt"` 猜测逻辑。
2. 非 PT runtime artifact 现在统一要求显式 source PT；`source_pt_resolution.explicit_required=true`，缺失时 `admission_status=unverifiable`、`blocking_reason=source_pt_required_for_accelerated_artifact`、`next_action=provide_explicit_source_pt_path`。
3. 显式来源仅接受 `model_security.source_pt/source_pt_path/source_model_path`、`inference.artifacts.pytorch/pt` 或自定义模型的 `source_pt_path/source_model_path/source_path`。自定义 PT runtime 仍可把自身作为源 PT。
4. `/api/start` 的未知模型阻断合同已补测试：`blocked_scan_required` 返回 409 + `model_security_blocked`，并带 `next_action/operator_message`；阻断时不得调用 `MonitorEngine.start()`。
5. `/api/model-security/trust/delete` 遇到 `trust_store_compromised` 时补充 `repair_or_clear_trust_registry` 操作指引，避免只返回裸错误。
