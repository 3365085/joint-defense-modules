# 文件视频结束态与预览残留问题

## 背景

Web 端检测文件视频时，用户观察到两个现象：某些场景像是“停止检测”，手动停止后过一会恢复；视频检测完后页面停在最后一帧，像资源没有释放。

## 当前判断

问题不是单一算法误判。当前链路里至少有两类状态问题：

1. `MonitorEngine._backend_capture_loop` 在文件源 EOF 后会设置 `running=false`、`source_ended=true`，但旧状态里 `ready_for_preview` 可能残留为 `true`。这会让前端把已经结束的旧运行仍当成可恢复预览。
2. `model/src/defense/web/static/index.html` 的 `showModelSecurityBlock` 会展示 B 模块阻断提示，但之前没有先清理旧 MJPEG 预览流；`stopEndedPreview` 也没有完整清空 overlay 画布和时间线。因此页面可能同时显示阻断/结束提示和上一轮最后一帧。
3. `refreshModelSecurity` 属于右侧安全面板刷新，但旧逻辑会在发现当前模型安全状态阻断时直接调用 `showModelSecurityBlock` 覆盖主画面。若 A 模块已经通过测试绕过、净化替代或上一轮运行启动，安全面板刷新会把正在运行的预览清掉，用户会看到“某些场景突然停止检测”。这不是检测线程必然停止，而是前端状态层错误覆盖。
4. `mergeOverlayStatusForPanel` 会把最新 overlay 记录合并到 `/api/status`，旧逻辑用 `activePreviewRunId` 把已经 `running=false` 的后端状态重新推断为 `running=true`。因此手动 `/api/stop` 后，页面仍可能显示“检测中/首帧检测”并保留旧 MJPEG。
5. `ensurePreview` 对 `running=false` 但 `frame_idx>0` 的历史状态没有主动关闭旧 `<img>`，导致按钮和状态已显示“已停止”，但主画面仍停留在最后一帧。
6. `PreviewBus.wait_for_frame` 和 `MonitorEngine._preview_render_loop` 之前允许在没有新 `seq` 时继续使用缓存帧；`MonitorEngine.wait_latest_jpeg` 在超时后也可能返回同一 `latest_jpeg`。这会把“没有新帧”伪装成“MJPEG 仍在持续输出”，直接放大页面最后一帧残留和“检测中”错觉。
7. `model/src/defense/web/static/index.html` 的 `stateText` 原先把 `source_ended` 放在 `alert_confirmed` 和 `running` 后面。若结束瞬间仍有告警或混合态字段，页面会优先显示“确认告警/检测中”，而不是最高优先级的“播放结束”。

## 代码链路依据

- 后端结束态：`model/src/defense/runtime/runner.py` 的 `MonitorEngine._backend_capture_loop`。
- MJPEG 输出：`model/src/defense/runtime/runner.py` 的 `MonitorEngine.wait_latest_jpeg` 和 `model/src/defense/web/fastapi_app.py` 的 `mjpeg_generator`。
- 前端预览清理：`model/src/defense/web/static/index.html` 的 `hideAllPreview`、`stopEndedPreview`、`showModelSecurityBlock`、`ensurePreview`。
- 安全面板刷新：`model/src/defense/web/static/index.html` 的 `refreshModelSecurity`，该函数应只更新安全面板；只有 `/api/start` 返回 `model_security_blocked` 或显式阻断状态时才覆盖主画面。
- 状态合并：`model/src/defense/web/static/index.html` 的 `mergeOverlayStatusForPanel`，overlay 只能补充展示字段，不能把后端 `running=false` 改回运行态。
- 停止态显示：`model/src/defense/web/static/index.html` 的 `ensurePreview`，非运行态必须关闭旧 MJPEG 流。

## 修改方向

- 文件源结束后明确清除 `ready_for_preview`、`preview_started`，并保留 `source_ended=true` 作为唯一的结束语义。
- `wait_latest_jpeg` 在运行已停止时不再返回旧 JPEG，避免 MJPEG 连接重新吐出最后一帧。
- `wait_latest_jpeg` 在运行中但 JPEG `seq` 未增长时也只返回空数据，避免 MJPEG 把同一帧重复 yield。
- `PreviewBus.wait_for_frame` 只有在 `latest_seq > last_seq` 时才返回 packet；`_preview_render_loop` 改为等待新帧，不再每轮直接拿缓存帧重复编码。
- `source_ended` 被前端归一化为最高优先级生命周期状态：强制 `running=false`、`ready_for_preview=false`、`preview_started=false`、`preview_mode=source_ended`、`detector_pipeline_mode=ended`。
- `/api/start` 成功后不再无条件打开 MJPEG，而是把返回状态交给 `ensurePreview` 判断。短视频在启动等待窗口内已经结束时，页面直接进入“播放结束”，不会再打开一个马上失效的流。
- B 模块阻断和播放结束都统一调用 `hideAllPreview`，清空旧流、overlay timeline、canvas 和画面比例。
- `refreshModelSecurity` 不再直接覆盖主画面；正在运行的 A 模块状态会清除过期的 `modelSecurityBlockState`。
- `mergeOverlayStatusForPanel` 以后以 `/api/status` 的 `running` 为权威；当后端已停止时，overlay 不再恢复 `ready_for_preview`、`preview_started` 或 `first_detection_ready`。
- `ensurePreview` 在停止态统一调用 `hideAllPreview`，使页面不再保留最后一帧。
- 手动停止后后端明确写入 `preview_mode=stopped`、`detector_pipeline_mode=idle`、`preview_fps=0`，避免前端拿到半旧状态。
- 文件解码中途读帧失败时，在非 EOF 位置尝试跳过少量坏帧，避免个别坏帧被误判成视频结束。该恢复次数通过 `source_decode_recoveries` 暴露，后续可用于诊断素材质量或 OpenCV 解码抖动。

## 影响范围

- 只影响 Web 文件源播放/结束/阻断状态管理，不改变模型权重、类别语义、PPE 判定阈值或 A3b 策略。
- 对实时流仍保留原有重连逻辑；文件源只增加轻量坏帧恢复。

## 后续建议

- 若仍出现“某些场景无框”，应区分是模型 raw detection 空窗、overlay 时间窗过短，还是视频解码失败。优先记录 `source_decode_recoveries`、`dropped_detection_frames`、`detector_inference_ms`、`overlay_seq` 和 `preview_seq` 的时间线。
- 涉及页面状态和检测效果的提交前，仍需按 `AGENTS.md` 要求跑真实检测视频并抽连续 3 秒逐帧检查。
