# 三类模型类序与归档准入口径复核

日期：2026-06-05

## 背景

用户指出 Web 端检测视频中，橙色马甲人的 `head` 与 `helmet` 输出稳定，但 `person` 输出不稳定。该问题属于检测效果、类别语义解释和系统准入口径问题，需要记录到技术/算法档案。

用户随后明确允许直接查看本地抽帧图像，并要求删除 `AGENTS.md` 中禁止展示/上传抽帧图片的限制规则。当前复核确认，该限制规则已不在 `AGENTS.md` 中；视觉验收段落仍保留“记录检测视频路径、抽帧区间、抽帧图片路径和肉眼检查结论后才允许提交”的门槛。

## 当前判断

1. 旧验收视频中的 `person` 不稳定，主要原因是 class id 到 label 的映射错误。旧记录使用 `0=person, 1=head, 2=helmet`，但三类实验 checkpoint 内嵌 names 为 `0=helmet, 1=head, 2=person`。
2. 修正版验收记录使用 checkpoint 内嵌顺序 `{0: "helmet", 1: "head", 2: "person"}`，`visual_acceptance_record.json` 中 `person_missing_frames=[]`。
3. 继续审计发现，若同一自定义模型路径切换 `class_names`，旧 `PipelineCache` key 未包含类别映射，可能复用旧 backend names，导致用户改了类序但画面仍按旧类序解释。该问题已修复。
4. seven-experiment archive 的 purified checkpoint 属于 hash-bound archive verification，不等同于严格 Wilson/mAP 审计。当前已将 `wilson_upper/mAP_drop_pp` 标为 `None`，并用 `metric_source=archive_hash_verification_only` 说明口径。

## 代码链路依据

- `model/src/defense/runtime/pipeline_factory.py`：`PipelineCache.get()` 的 key 现在纳入归一化后的 `custom_model.class_names`，同一路径不同类序会重建 detector backend。
- `model/src/defense/module_a/backends/detector_backend.py`：backend 构造时优先使用配置传入的 `class_names/names/labels` 解释 class id。
- `model/src/defense/module_a/ppe_postprocess.py`：`person` 进入 PPE 计数、状态字段和 `head/helmet` 离体误报抑制，但不直接作为未戴帽违规触发条件。
- `model/src/defense/model_security/purifier.py`：seven archive purified 命中保留 `validation_scope=seven_experiment_purified_archive`，archive 配对候选不会因 packaged 子目录缺失而丢失。
- `model/src/defense/model_security/scanner.py` 与 `model/src/defense/model_security/service.py`：对 archive purified 使用 archive 验证口径生成 clean 报告，同时保留真实 validation scope。
