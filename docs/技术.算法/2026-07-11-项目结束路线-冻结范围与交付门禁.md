# 项目结束路线：冻结范围与交付门禁

日期：2026-07-11

目标：在不继续扩大算法研究范围的前提下，将当前项目收束为可运行、可验收、可解释、可交付的版本。

状态：结项规划。本轮未修改生产代码、模型权重、阈值、类别语义或信任库。

## 0. 一句话结论

项目当前缺少的不是“最后一轮调参”，而是明确的结束定义。

主项目已经接入 demo 的 rebuilt Module A，并达到当前 variant C 的工程检测水平。继续围绕同一批
合成视频调整 A1-A4，无法证明真实攻击能力，只会继续在召回和误报之间移动。

建议冻结开放式算法研究，将项目按以下定位交付：

> PPE 检测、Web 运行时、证据链和模型安全准入为主要交付能力；Module A 必须完成基础设计文档
> 明确规定的对抗纹理、闪烁光照和动态遮挡场景验收，同时将真实物理攻击泛化标为尚未验证。

最终 Module A 验收矩阵见：

`docs/技术.算法/2026-07-11-基础设计文档对齐与最终场景验收矩阵.md`

## 1. 项目真正的问题

### 1.1 目标与可用证据不匹配

目标是检测真实物理补丁、强光、遮挡和翻拍攻击，但当前没有独立真实物理攻击测试集：

- heldout 攻击全部是程序化合成代理；
- 5 段 `raw_*` 也是从同一正常视频生成；
- replay/interference 各约一段，且已进入训练和反复调试；
- `素材/真实物理攻击/` 为空。

因此真实攻击召回率在现有条件下不可验证。继续优化只能提高对现有生成器和开发集的适应度。

### 1.2 demo 被当成目标，但 demo 自身不是完成态

当前重新验证：

- demo broad：前 240 帧内合成攻击 `7/10`，干净片误报 `1/12`；
- 主项目开发/模型选择集：前 240 帧内合成代理攻击 `17/21`，干净片误报 `2/27`；
- A4 CV：AUC `0.841`，帧级代理召回 `0.542`，帧级假阳 `0.071`。

demo 的价值是工程方法、特征链路和评测教训，不是已经证明的真实攻击检测效果。

### 1.3 A1-A4 存在信息上限

- A1 只能测纹理异常，分不清攻击补丁与正常高纹理衣物/背景。
- A2 只能测时序纹理变化，正常运动、灯光和丢帧也会产生响应。
- A3 只能测光流残差，真实运动、遮挡和帧间跨度变化会影响结果。
- A3b 的独立媒体告警因背景假阳关闭，目前只参与联合决策和疑似判断。
- A4 只能学习现有特征与标签中的统计关系，不能创造输入里不存在的攻击证据。

这不是继续增加规则就一定能解决的问题。

### 1.4 评测长期承担了训练反馈

当前所谓 heldout 已被 A/B/C/D/F 多个变体反复比较并用于选择 variant C，已经成为开发/模型选择集。

此外还存在：

- train/heldout 字节级重复视频；
- 015 正常视频与其派生攻击没有统一 capture/source 谱系；
- broad 的大部分样本来自 train；
- variant C 训练配方没有完整入库，当前 pkl 不能从已提交内容严格复现。

所以当前数字可以作为版本回归，不能作为无偏泛化证明。

### 1.5 工程状态仍有几个真正的交付阻塞

截至 2026-07-11：

- 默认模型安全状态为 `trust_store_compromised`；
- `blocking_reason=host_fingerprint_mismatch`；
- `allowed=false`，默认 Web 检测启动会被 B 模块阻断；
- 主项目运行时依赖 demo 仓中的 A4 pkl、RAFT ONNX/engine 和 Rust native 安装状态；
- 分类器加载失败会静默退回规则，页面没有明确暴露实际运行能力；
- 主项目 Module A 使用 `timestamp=0.0`，与 demo 离线源时间语义不同；
- 主项目 rebuilt detector 仍有未提交行为调优。

这些比继续训练一个新分类器更接近“项目能否结束”的核心。

### 1.6 项目范围持续扩张，没有停止规则

四个月中项目同时承担：

- PPE 检测与跟踪；
- 物理攻击检测；
- 静态媒体/翻拍检测；
- 模型后门扫描与净化；
- Web 实时预览和 overlay；
- 证据链；
- GPU/TensorRT/RAFT/Rust 性能优化；
- 多平台和 Jetson 方向；
- 数据生成、训练和评测。

任何一项都可以继续做数月。没有明确“哪些不做”，项目就不会自然结束。

## 2. 结项版本的产品定义

### 2.1 保留为正式能力

- FastAPI Web 运行时；
- 文件视频和摄像头输入；
- PPE person/head/helmet 检测、跟踪和告警；
- Module B 模型安全准入、扫描和状态展示；
- 运行状态、错误信息和配置快照；
- 证据 JPEG/MP4/JSON/JSONL/SQLite；
- Module A 的完整性退化与攻击疑似信号；
- Pixi 环境、一键启动和停止。

### 2.2 降级为实验/疑似能力

- 真实对抗补丁检测；
- 真实强光/激光攻击检测；
- 真实遮挡攻击归因；
- 跨设备屏幕翻拍检测；
- `p_adv` 作为严格攻击概率；
- `alert_confirmed` 作为真实攻击地真。

对外交付语义建议为：

- `video_integrity_degraded`
- `media_replay_suspected`
- `adversarial_texture_suspected`
- `unknown_visual_anomaly`
- `suspected_attack`

现有公共字段为兼容可以保留，但文档必须说明 `alert_confirmed` 只表示系统时序确认。

## 3. 最快结束路径

预估：3 到 5 个有效工作日，不包含用户个人验收等待时间。

### 阶段 1：冻结，半天

目标：停止算法漂移。

执行：

1. 将当前 variant C、rebuilt detector 和配置设为基础设计对齐起点。
2. 记录代码状态、dirty diff、YOLO/A4/RAFT 哈希和 Pixi 环境。
3. 宣布停止开放式新增 A1-A4 特征、分类器变体、攻击类别和专项阈值。
4. 将 36 个 skip 分类为“已知未实现能力”，不为结项全部补齐。
5. 将真实攻击召回标记为“未能从现有数据确认”。
6. 只保留一轮有边界的基础设计对齐：
   - P1 目标相关对抗纹理；
   - P2 闪烁光照；
   - P3 动态遮挡；
   - N1-N4 对应困难负场景。

退出门禁：

- 形成唯一冻结基线；
- 后续只允许修交付阻塞、明显运行 bug，以及与基础设计契约直接冲突的根因。

### 阶段 2：修真正阻塞，1 到 2 天

目标：保证目标机器可启动、运行能力明确、资产可复现。

必须完成：

1. 通过受支持流程修复目标主机的 trust-store host seal，使默认准入可用；不得测试绕过。
2. 将生产 A4 artifact 归入主项目管理的版本化运行资产，记录哈希，解除对 demo 可变文件的隐式依赖。
3. 启动状态明确显示：
   - A4 分类器是否成功加载；
   - RAFT backend；
   - Rust native 是否可用；
   - 是否发生规则回退；
   - 当前 YOLO/A4 artifact 哈希。
4. 固化一份 release config snapshot。
5. 修正主项目源时间戳和严格相邻帧输入语义。基础文档明确要求关键帧处理时仍保留相邻帧光流，
   因此该项属于最终检测验收硬门禁。
6. 对当前未提交 detector 调优只做二选一：
   - 通过最终视觉验收后冻结；
   - 未通过则不继续调参，回到最近已验证状态。

退出门禁：

- `start_web.bat` 在目标机器可正常启动；
- 页面能明确说明为什么允许或阻止检测；
- 运行时使用的模型和分类器可以从交付资产唯一确定。

### 阶段 3：做一次最终诚实验收，1 到 2 天

目标：不再训练，只回答“当前冻结版本能否交付”。

建立固定验收包：

- P1 对抗纹理、P2 闪烁光照、P3 动态遮挡各至少 4 段；
- N1 正常高纹理、N2 正常光照、N3 正常遮挡、N4 时序压力各至少 4 段；
- PPE 目标视频；
- replay/interference 开发集回归；
- Web 实际运行视频。

执行：

1. 按 capture/source 修正验收清单，避免同源跨集合。
2. 全视频运行，不只截前 240 帧。
3. 报告：
   - 正常视频误报事件数；
   - 告警秒数；
   - PPE 断框、旧框和标签稳定性；
   - 代理攻击首次响应延迟和覆盖率；
   - 停止、seek、重连和视频结束状态；
   - B 模块准入结果。
4. 按 AGENTS.md 随机抽连续 3 秒逐帧查看，并记录视频、区间、抽帧路径和肉眼结论。
5. 运行 `pixi run smoke`。
6. 固定矩阵最多运行两轮：
   - 第一轮定位基础设计根因；
   - 第二轮作为最终冻结验收。

退出门禁：

- 无阻塞启动故障；
- 无明显拖框、旧框滞留、状态错位或误导性显示；
- PPE 主流程达到用户可接受程度；
- Module A 正场景事件级漏报 `0/12`；
- Module A 困难负场景事件级误报 `0/16`；
- 攻击前误报、旧告警滞留和逐帧视觉失败均为 0；
- 所有代理攻击结果明确标注为代理。

### 阶段 4：交付与停止，半天

产出：

1. 最终使用说明：启动、停止、输入源、状态和证据位置。
2. 最终能力声明：已验证、待实验确认、未能确认。
3. release manifest：
   - Git commit；
   - 配置哈希；
   - YOLO 模型哈希；
   - A4 pkl 哈希；
   - RAFT/native 状态；
   - Pixi/Torch/CUDA/GPU；
   - 测试结果；
   - 最终验收资产路径。
4. 已知问题清单和不做清单。

只有用户亲自完成测试并明确允许后，才进行分类中文提交和版本标记。每次提交后刷新 CodeGraph。

完成后停止 Module A 行为调优，将后续真实攻击数据工作定义为下一项目，而不是当前项目尾项。

## 4. 明确不做

为了让项目结束，以下内容不进入当前结项范围：

- 不继续做 A/B/C/D/F/G 分类器变体；
- 不再针对同一 48 片开发集开放式调阈值；
- 不新增 CNN、Transformer、序列模型或新框架；
- 不继续生成更多攻击类别；只允许按基础设计修正 P1/P2/P3 代理样本的目标锚点、时序和物理一致性；
- 不恢复 A3b 独立确认告警；
- 不承诺真实攻击召回率；
- 不为结项补完全部 36 个超前契约 skip；
- 不做新的 Jetson/NVDEC/INT8 性能路线；
- 不重构整个 runtime；
- 不继续追求所有场景零误报；
- 不在没有新证据时继续增加 detector 特判。

## 5. Definition of Done

以下全部满足，即宣布项目完成：

1. 目标机器双击启动和停止可用。
2. B 模块准入可用，阻断原因和下一步清晰。
3. PPE 主流程在固定目标视频上通过用户视觉验收。
4. Module A 在基础设计固定矩阵上达到正场景 `0/12` 漏报、负场景 `0/16` 误报。
5. Web 画面、框、状态、seek、结束态和证据链没有阻塞问题。
6. 所有运行资产和配置可通过 manifest 唯一追溯。
7. `pixi run smoke` 无失败；保留 skip 已解释。
8. 最终 3 秒逐帧视觉验收记录完整。
9. 已知限制文档完成。
10. 用户亲自确认可交付并允许提交。

这一定义不要求：

- 真实攻击召回得到证明；
- 所有研究假设被解决；
- 所有 skip 被实现；
- 所有误报归零；
- demo 和主项目逐位等价。

## 6. 最终判断

项目已经具备交付骨架，剩余工作主要是基础设计对齐和收口，不是继续开放式研究。

最快结束方式是接受一个事实：

> 在没有真实攻击数据的条件下，Module A 无法被证明为覆盖全部真实世界的攻击检测器；但它必须
> 在基础设计明确的对抗纹理、闪烁光照、动态遮挡及对应困难负场景上完成固定 0/0 验收，之后与
> PPE、Web、证据链和模型安全能力一起交付。

继续追求“让合成召回再高一点”不会让项目真正完成。冻结范围、修默认启动、固定资产、跑一次完整
验收、写清限制并交付，才是当前项目的结束路径。
