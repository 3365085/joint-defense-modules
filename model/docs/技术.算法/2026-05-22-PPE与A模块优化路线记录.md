# PPE 与 A 模块优化路线记录

## 背景

用户询问当前 PPE 检测、A1-A4 每帧/隔帧逻辑之后应如何优化。当前问题同时涉及远距离小目标召回、首帧/启动卡顿、预览流畅度、overlay 拖框和 A 模块性能预算，必须分层处理。

## 当前判断

1. 最优先优化方向不是继续提高主检测频率，而是在 PPE 后处理层做弱证据时序升级。
2. 远处小目标检测差主要受 640 输入尺度、小目标像素占比、模型类别输出和后处理压制共同影响；仅提高检测频率不能解决召回。
3. 启动前 1-2 秒卡顿主要像是 TensorRT/CUDA/模型缓存的懒初始化问题，应通过预热和缓存生命周期处理，而不是降低检测算法质量。
4. A1-A4 与 PPE 共享同一处理帧和检测结果；只做 PPE 后处理状态机不会明显拖慢 A1-A4，但新增额外推理会竞争性能预算。

## 推荐优化顺序

### 第一阶段：PPE 弱证据时序升级

在 `model/src/defense/module_a/postprocess/ppe_tracking.py` 或 PPE 后处理层增加弱 head/helmet 的短窗口稳定判断：

- `weakHead` 连续命中、位置连续、平均置信度达标后，只在当前帧升级为有效 head 证据。
- `weakHelmet` 使用同类逻辑，稳定后升级为 helmet 安全证据。
- helmet 证据优先级高于 head 告警；稳定 helmet 应阻止同区域 weakHead 升级为 missing helmet。
- 不允许 weak 目标长时间 hold 成有效告警，避免旧框滞留和拖框。
- 输出状态字段区分 raw/effective/weak/promoted，便于 UI 和测试解释。

该阶段不增加模型推理次数，对 A1-A4 性能影响应非常小。

### 第二阶段：启动/切模型预热优化

针对“启动检测前 1-2 秒卡顿、重新选择同模型后正常”的现象，优化模型生命周期：

- 服务启动或选择模型后后台创建/复用 `PipelineCache`。
- 对 TensorRT backend 执行固定次数 warmup，确保 CUDA context、engine、buffer 和首轮 kernel 编译/分配提前完成。
- 检测开始时直接使用已预热 pipeline，不在首个视频帧上承担初始化成本。
- 页面状态明确显示 `initializing/prewarming/ready`，避免把初始化误认为检测空帧。

该阶段优化卡顿，不应改变 PPE/A 模块算法语义。

### 第三阶段：调度与预算保护

保持 preview 与 detection 解耦、latest-only backpressure 和检测帧率上限。新增任何重逻辑必须遵守：

- 主路径不新增额外 GPU 推理。
- 如确需 ROI 重检，必须受 `budget_ok` 控制，并限制 ROI 数量。
- 处理耗时超过目标帧预算时，优先降级可选重检/显示增强，不降级核心检测状态机。

### 第四阶段：高分辨率/多尺度方案仅作为后续实验

当前 TensorRT engine 固定 640 输入。若要生产化 960/1280 或动态输入，需要重新导出 TensorRT engine 并做完整回归；否则只能作为 PyTorch 实验趋势，不能直接塞进实时主路径。

## 影响范围

- PPE 弱证据升级影响 `ppe_postprocess.py`、`ppe_tracking.py`、`ppe_business.py`、`frame_processor.py` 和 overlay 显示字段。
- 模型预热优化影响 `pipeline_factory.py`、`runner.py`、启动脚本或服务初始化状态。
- 高分辨率方案影响模型 artifact、TensorRT engine、推理后端和性能预算，风险最高。

## 结论

当前最稳妥路线是：先做 PPE `weakHead + weakHelmet` 的短窗口稳定升级，再单独处理 TensorRT/CUDA 预热卡顿。不要通过提高每帧检测频率或默认开启额外 ROI 推理来解决远处小目标问题，否则容易把预览流畅度再次打坏。
