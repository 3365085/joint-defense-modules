# B模块安全准入白名单完整性优化技术交底

日期：2026-05-25

## 1. 背景与定位

本版优化将 B 模块明确定位为 A 模块启动前的模型安全准入层。A 模块负责运行期视频检测与物理扰动识别，B 模块负责判断“当前即将被 A 模块加载的模型产物”是否允许进入生产检测链路。

准入策略不是“启动时一律全量扫描”，而是：

- 已通过 clean full scan 且白名单完整性校验通过的模型，快速放行 A 模块。
- 未知模型、语义配置变化、源 PT 变化、运行产物 hash 变化、白名单损坏或系统指纹不匹配时，阻断 A 模块启动。
- 只有原始 PT 完整扫描 clean/trusted，或净化产物完整复扫 clean/trusted，才能自动写入白名单。
- quick scan 只作为诊断信息，不作为生产白名单写入依据。

本版重点是补齐“白名单可信度”本身：白名单记录不再只依赖 `trusted_registry.json`，还通过 seal 文件和系统唯一指纹约束使用环境，降低白名单被拷贝、篡改或脱离本机环境复用的风险。

## 2. 启动准入主流程

Web 启动入口在 `model/src/defense/web/fastapi_app.py::start`。当用户从前端启动检测时，后端先读取 `profile` 和 `custom_model`，调用 `ModelSecurityService.ensure_admitted()` 获取 B 模块准入状态。

执行顺序为：

1. 生成当前运行模型 fingerprint。
2. 校验白名单 registry 与 seal 的完整性。
3. 检查当前 fingerprint、运行模型 hash、源 PT hash、类别语义 hash、PPE 语义 hash 是否命中白名单。
4. 若 `allowed=True`，才继续调用 `MonitorEngine.start()` 启动 A 模块。
5. 若 `allowed=False`，返回 `409` 和 `error="model_security_blocked"`，A 模块不启动。
6. 若状态为 `blocked_scan_required`，后端自动启动后台 full scan。

关键代码：

- `model/src/defense/web/fastapi_app.py::start`
- `model/src/defense/model_security/service.py::ModelSecurityService.ensure_admitted`
- `model/src/defense/model_security/service.py::ModelSecurityService.admission_status`
- `model/src/defense/runtime/runner.py::MonitorEngine.start`

## 3. 状态机与处置语义

### 3.1 未知模型

未知模型通常表现为 `blocked_scan_required`，原因是当前 fingerprint 未命中白名单，或当前模型/语义配置与历史记录不一致。

策略：

- 阻断 A 模块启动。
- 后台触发 `start_background_scan(scan_type="full")`。
- full scan 通过后自动写入白名单。
- full scan 未通过时保持阻断，由前端安全中心展示原因。

### 3.2 full scan

full scan 是当前唯一自动写入生产白名单的扫描证据。它要求可验证的源 PT，并基于验证集/攻击集执行 external hard suite，形成 ASR 风险判定。

结果语义：

- `clean` 或 `trusted`：允许自动写入白名单，后续相同模型和语义配置可快速放行。
- `suspicious`：疑似风险，阻断 A 模块，可进入净化流程。
- `review`：需要人工复核，阻断 A 模块，不自动净化。
- `unverifiable`：缺少源 PT、缺少验证资产、报告不可评分等，阻断 A 模块。

关键代码：

- `model/src/defense/model_security/service.py::ModelSecurityService.scan`
- `model/src/defense/model_security/scanner.py::full_scan`
- `model/src/model_security_gate/detox/external_hard_suite.py::run_external_hard_suite`
- `model/src/model_security_gate/detox/external_hard_suite.py::summarize_external_rows`

### 3.3 clean

`clean` 是“当前扫描证据满足准入阈值”的状态，不代表对原模型做了净化，也不代表模型被物理改写。

本版需要特别避免混淆：

- clean 不等于 purified。
- clean 不会生成新模型。
- clean 只会作为白名单写入条件。
- clean full scan 后的运行仍使用当前选中的运行模型产物。

### 3.4 suspicious

`suspicious` 表示 full scan 发现风险高于可接受阈值。当前策略不会放行生产检测，也不会把 suspicious 模型写入白名单。

策略：

- 阻断 A 模块启动。
- 允许用户在安全中心触发“净化并复扫”。
- 净化完成后必须对净化 PT 再执行 full scan。
- 只有复扫 clean/trusted 才能进入白名单。

### 3.5 review

`review` 表示风险结果处于 clean 与 suspicious 阈值之间，需要人工复核。当前实现不会自动净化，也不会自动放行。

策略：

- 阻断 A 模块启动。
- 保留扫描报告用于人工判断。
- 后续可预留人工审批、复测矩阵扩大或阈值策略调整，但不能把 review 直接视为 clean。

### 3.6 unverifiable

`unverifiable` 表示当前无法形成可审计的安全结论，典型原因包括：

- 当前运行产物是 `.engine` 或 `.onnx`，但找不到同源 `.pt/.pth`。
- 缺少可用验证集/攻击集根目录。
- 扫描输出没有可评分样本。
- 运行模型 artifact 或配置不足以生成完整 fingerprint。

策略：

- 阻断 A 模块启动。
- 前端展示具体阻断原因。
- 运维侧补齐源 PT、验证资产或模型配置后重新 full scan。

## 4. 白名单 seal 与系统唯一指纹

白名单文件位于运行目录：

- `model/runtime/model_security/trusted_registry.json`
- `model/runtime/model_security/trusted_registry.seal.json`

本版通过 `trusted_registry.seal.json` 对白名单完整性做约束。seal 包含 registry hash、系统指纹 hash、更新时间和签名。每次白名单保存时，会重新写入 seal；每次准入检查时，会先验证白名单完整性。

系统唯一指纹由 `current_host_fingerprint()` 生成，当前使用以下来源：

- Windows MachineGuid
- 系统盘卷序列号
- 机器名

若可用来源不足，指纹状态会降级为 `degraded`，但仍会在状态中暴露警告。若 registry hash、seal 签名或 host fingerprint 不匹配，则状态进入 `trust_store_compromised`，白名单不再被信任，A 模块禁止启动。

关键代码：

- `model/src/defense/model_security/integrity.py::current_host_fingerprint`
- `model/src/defense/model_security/integrity.py::build_trust_store_seal`
- `model/src/defense/model_security/integrity.py::write_trust_store_seal`
- `model/src/defense/model_security/integrity.py::verify_trust_store`
- `model/src/defense/model_security/service.py::ModelSecurityService._trust_store_integrity`
- `model/src/defense/model_security/service.py::ModelSecurityService._write_registry_seal`

## 5. 白名单命中条件

白名单命中由 `ModelSecurityService._trusted_record_matches()` 判定，不是简单按路径放行。必须同时满足：

- `approved_for_runtime=True`
- `scanner_version` 与当前扫描器版本一致
- `runtime_model_hash == fp.model_hash`
- `class_names_hash == fp.class_names_hash`
- `ppe_mapping_hash == fp.ppe_mapping_hash`
- `source_model_hash == source_pt_hash`

因此，以下变化都会导致重新准入：

- 运行模型文件内容变化
- 同源 PT 内容变化
- 类别名称或类别顺序变化
- PPE 语义映射变化
- 扫描器版本变化
- 白名单文件或 seal 被篡改
- 白名单从其他机器拷贝导致系统指纹不匹配

关键代码：

- `model/src/defense/model_security/fingerprint.py::build_model_fingerprint`
- `model/src/defense/model_security/service.py::ModelSecurityService._trusted_record_matches`
- `model/src/defense/model_security/registry.py::ModelTrustRegistry`
- `model/src/defense/model_security/registry.py::TrustRecord`

## 6. 净化产物受控目录

净化不是修改原模型，而是在运行目录下生成受控产物。当前净化入口为 `ModelSecurityService.start_background_purification()` 和 `ModelSecurityService.purify()`，底层调用 `run_new_purification()`。

净化输出目录规则：

- 根目录：`model/runtime/model_security/purified/`
- 单模型目录：`model/runtime/model_security/purified/<fingerprint>/`
- weight soup 候选目录：`model/runtime/model_security/purified/<fingerprint>/weight_soup/`

净化报告会记录源模型路径、源模型 hash、clean anchor、候选模型、选中候选、净化模型路径和净化模型 hash。净化成功只表示“生成了候选产物”，不等于准入通过。

受控策略：

- 原模型不被原地改写。
- 净化产物只写入 `runtime/model_security/purified/`。
- 净化产物必须 `scan_after=True` 复扫。
- 复扫 clean/trusted 后才可写入白名单。
- 前端“使用净化模型”只是把净化模型路径带回启动配置，当前运行模型切换后仍需满足 B 模块准入。

关键代码：

- `model/src/defense/model_security/service.py::ModelSecurityService.start_background_purification`
- `model/src/defense/model_security/service.py::ModelSecurityService.purify`
- `model/src/defense/model_security/purifier.py::run_new_purification`
- `model/src/defense/model_security/storage.py::ModelSecurityStorage.purified_dir`

## 7. 前端安全中心

本版前端提供独立的 B 模块模型安全中心：

- 页面：`/model-security`
- 日志页：`/model-security/logs`

安全中心展示：

- 当前准入状态
- 风险分数
- 最近扫描时间
- 白名单命中状态
- 运行模型、源 PT、净化模型、模型指纹
- 扫描报告路径
- 白名单文件、seal 文件、系统指纹、registry hash
- 白名单记录列表
- 最近 B 模块事件日志

主要操作：

- 完整扫描
- 净化并复扫
- 使用净化模型
- 查看扫描报告
- 查看净化报告
- 打开日志页
- 删除当前白名单记录
- 清空全部白名单记录

安全边界：

- 前端不能手工添加白名单。
- 后端 `/api/model-security/trust` 的 POST 信任入口返回禁用信息。
- 用户只能删除或清空白名单，不能绕过 full scan 自动写入规则。

## 8. 触发与恢复策略

### 8.1 自动触发

以下场景会触发准入阻断或后台任务：

- 未命中白名单：阻断并启动后台 full scan。
- full scan 正在运行：保持 `scanning`，A 模块暂不启动。
- 白名单完整性失败：进入 `trust_store_compromised`，A 模块禁止启动。
- suspicious：允许安全中心触发净化并复扫。
- purified alternative available：净化 PT 已复扫通过，但当前运行模型尚未切换到该净化 PT。

### 8.2 人工恢复

建议恢复路径：

- `blocked_scan_required`：等待 full scan 完成；若缺少源 PT 或验证资产，补齐后重新扫描。
- `suspicious`：执行净化并复扫；若复扫通过，切换到净化模型。
- `review`：人工检查扫描报告，必要时扩大验证集或重新评估阈值，不直接放行。
- `unverifiable`：补齐同源 PT、heldout roots、external eval roots 或 runtime custom model 配置。
- `trust_store_compromised`：确认是否为误拷贝或篡改；清空白名单后用本机源 PT 重新 full scan。
- `purified_alternative_available`：在安全中心选择净化模型作为运行模型，再走一次准入检查。

## 9. 测试与验证关注点

本次交底覆盖的测试重点应包括：

- 未知模型启动时返回 `409 model_security_blocked`，且 A 模块不启动。
- 未知模型触发后台 full scan。
- clean full scan 自动写入白名单与 seal。
- 白名单命中后 A 模块可以启动。
- 修改运行模型、源 PT、类别语义或 PPE 语义后不再命中旧白名单。
- 删除 seal、修改 registry、跨机器拷贝白名单时进入 `trust_store_compromised`。
- suspicious 状态下允许净化，review/unverifiable 状态下不应自动净化或放行。
- 净化产物落在 `runtime/model_security/purified/`，原模型不被覆盖。
- 净化后必须复扫 clean/trusted 才能进入白名单。
- 前端安全中心能展示状态、报告、日志和白名单完整性信息。

## 10. 残留预留

当前仍保留以下后续能力空间：

- review 状态的人工审批流程尚未设计为可写白名单的正式闭环。
- 加速模型导出能力在前端已有预留提示，本轮重点仍是 PT 扫描、净化、白名单和 A 模块准入闭环。
- host fingerprint 在极端环境下可能降级，需要在部署规范中明确机器更换、系统盘更换后的白名单重建流程。
- external hard suite 的验证集覆盖度仍决定 full scan 证据强度，后续应持续扩展 heldout/attack validation assets。
- 净化策略当前以 AutoDetox Backbone-Soup 路径为主，其他算法路径仍需独立实验和证据记录后再纳入自动闭环。

## 11. 结论

本版优化的核心结论是：B 模块已经从“模型扫描工具”收敛为 A 模块启动前的安全准入层。生产启动只信任经过 clean full scan 或净化后 clean full scan 的白名单记录，并且白名单必须通过 seal 与系统唯一指纹完整性校验。clean 只代表准入证据通过，不会净化或改写原模型；净化产物必须进入受控目录，复扫通过后才能作为可切换、可准入的运行模型。
