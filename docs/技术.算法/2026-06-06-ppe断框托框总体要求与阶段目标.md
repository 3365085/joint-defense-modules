# PPE 断框/托框总体要求与阶段目标

日期：2026-06-06

## 总体要求

当前 PPE 稳定性工作不应被拆成用户每轮催促的小修小补，而应按工程闭环推进。总体要求如下：

1. 保持 `person / head / helmet` 三类语义稳定。`person` 可作为上下文和误报抑制辅助，但不能未经证据就作为强前置条件；`head` 与 `helmet` 是当前 PPE 显示和业务判断主路径。
2. 不改模型权重、不改类别语义、不改 B 模块准入口径、不改 A3b 策略，除非进入单独的行为调优任务。
3. 所有项目命令必须通过 Pixi 运行，不使用全局 Python、pip 或可执行文件。
4. Preview 与 detection 继续解耦；检测保持 latest-only backpressure，不为显示稳定性增加额外主路径 GPU 推理。
5. 先做 text-only 证据，再做 dry-run 候选，再做行为候选。不得用“感觉像参数问题”直接调阈值。
6. 每个候选实验只能改一个维度：shadow removal、render cap、business filter、person context、low-context filter 不混在一次实验里。
7. 所有技术/算法判断必须记录到 `docs/技术.算法/`，不确定项明确写“待实验确认”。
8. 行为改动如果涉及检测效果、框显示、overlay、拖框/断框/流畅度，提交前必须跑目标视频生成结果视频，并抽连续 3 秒逐帧肉眼验收；未完成视觉验收不得提交。
9. 不提交未经用户亲自测试确认的改动；runtime evidence 与本地缓存不进入提交。

## 当前阶段目标

本阶段目标是完成 PPE 断框/托框工程闭环第一阶段：

1. 固化 A2/A3 诊断口径，避免把 “A2 补充诊断”误写成新增算法模块。
2. 完成 A3.1 stricter-shadow dry-run：不改变当前 shadow removal 行为，只模拟更严格 same-head/same-helmet 删除规则会改变哪些决策。
3. 完成 A4 render/business gate dry-run：不改变 render 或 business filter 行为，只记录候选 gate 对断框/托框风险的影响。
4. 通过 Web API 跑同一目标视频，生成 JSON/CSV 文本证据。
5. 运行 Pixi 重点回归，证明新增字段不破坏现有 Web/PPE/模型安全合同。
6. 输出下一步行为候选：只有 dry-run 指标显示收益和风险边界清楚，才进入真实行为调参。

## 当前已知事实

- A1 仅放宽短时 hold/bridge，未改善 head abrupt drop，并增加 held track 记录，不能作为可采纳方案。
- A2b 显示主要断点分布在 shadow removal、render cap、业务出口过滤三层。
- A2 补充诊断显示本目标视频 raw detector 输出没有 `person`，因此本视频不能依赖 person 作为强上下文。
- A3 显示 shadow removal 主要是 same-head overlap 清理，真正 head/helmet overlap 只有少量；最大 abrupt drop 仍更多指向 render cap 和 held eligibility。
- A4-render-plus1 行为复验确认配置生效，但 abrupt event 从 43 增到 45/46，held misses>=3 出现 20+ 次；该候选暂不具备提交条件。

## 完成标准

本阶段完成时应具备：

- 代码层：新增 dry-run 字段，但实际检测、过滤、绘制结果不变。
- 证据层：A3.1/A4 的 overlay、summary、CSV 均在 runtime evidence 下可追溯。
- 文档层：`docs/技术.算法/` 有清晰中文记录，说明背景、链路、数字、判断、风险和下一步。
- 验证层：Pixi 重点回归通过，`git diff --check` 无错误，缓存已清理。
- 决策层：给出明确候选，不让用户继续一轮轮追问“下一步做什么”。
