# 2026-05-26 AB安全页、A3b与PPE显示优化交底

## 背景

本轮在 `codex/ab-model-security-gate` 分支继续完善 AB 模块融合后的 Web 交互、A3b 灵敏度、B 模块净化输出位置和 PPE 显示稳定性。用户要求主页面不要堆叠 B 模块控件，安全相关能力统一进入安全中心；A3b 希望可调；针对 `model/run_model` 下 2 类 head/helmet 模型出现检测框闪烁的问题，需要用真实视频生成检测视频并抽帧确认。

## 关键代码链路

- 首页入口与启动参数：`model/src/defense/web/static/index.html`
  - `featureOptions()` 将 `a3b_sensitivity` 送入 `/api/start`。
  - 首页 `modelSecurityPanel` 仅保留安全中心入口、简短状态和 `openModelSecurityCenterBtn`。
  - 测试绕过入口仍由 `localStorage.moduleA.testBypassModelSecurity` 控制，但可视控件迁移到安全页。
- 安全中心：`model/src/defense/web/static/model_security.html`
  - 自定义模型配置变更后立即 `saveModelConfig({silent: true})` 并刷新状态。
  - 运行后端提示改为由模型格式决定：`.engine` 走 TensorRT，`.onnx` 走 ONNX，`.pt/.pth` 走 PyTorch。
- B 模块日志页：`model/src/defense/web/static/model_security_logs.html`
  - 移除自动刷新和清空日志能力。
  - 增加事件、状态、字段和值的中文映射。
- A3b 配置：`model/src/defense/runtime/config.py`
  - `normalize_a3b_sensitivity()` 只接受 `conservative/balanced/sensitive/high`。
  - `apply_a3b_sensitivity()` 调整 `a3b.observed_threshold`、`trigger_threshold`、`strong_single_frame_threshold`、observed-only 阈值和 `module_a.static_image_interval`。
- A3b 调试：`model/src/defense/runtime/a3b_soft_trigger.py`
  - `debug` 输出当前阈值，便于验证 UI 档位是否真正生效。
- Pipeline 缓存：`model/src/defense/runtime/pipeline_factory.py`
  - cache key 纳入 `a3b_sensitivity`，避免切换档位后复用旧阈值管线。
- B 净化输出：`model/src/defense/model_security/purifier.py`
  - 净化候选复制回源 PT 同目录，命名为 `*_净化完毕.pt`，必要时追加序号。
- B 准入与白名单绑定：`model/src/defense/model_security/service.py`
  - 只接受严格净化候选和复扫报告绑定，白名单命中仍依赖模型 hash/指纹。
- PPE 生产判定：`model/src/defense/runtime/ppe_business.py`
  - head/helmet-only 模型的文件预览显示保持窗口从主流程传入的 2 帧提升到 8 帧。
  - `_filter_tracks_for_ppe_counts()` 允许 head/helmet-only 模式在短暂无当前 PPE count 时继续渲染 held head/helmet track。
- PPE 显示跟踪：`model/src/defense/module_a/postprocess/ppe_tracking.py`
  - temporal weak promotion 后的 track 可进入短时 hold。
  - 稳定 track 后续遇到弱证据时不立即失去 hold 资格。
  - 在 `max_render_misses` 允许窗口内，held head/helmet 不再因置信度衰减到显示阈值以下而提前隐藏。
- PPE 调试视频：`model/tools/render_ppe_debug_video.py`
  - 改为调用生产函数 `evaluate_ppe_business()`，避免调试工具绕过生产 PPE 逻辑。

## 实验结论

- A3b：
  - `5e145bf778577e75118502e263d00c41.mp4` 在四个档位均稳定告警，`alert_frames=257`。
  - `d6415677a016dbf211e665f648e75607.mp4` 和 `VID20260512200916.mp4` 只有少量 A3b trigger，最终 `alert_frames=0`。这说明单纯调阈值不足以把弱异常样本推入最终告警，需要进一步分析 quality gate、source gate、时序确认或最终 fusion。
  - 正常仓库巡检视频四档均 `a3b_trigger_frames=0`、`alert_frames=0`，本组样本未观察到误报。
- PPE：
  - 2 类模型原始 raw detection 存在连续空窗：最长 23 帧和 18 帧。
  - 生产业务路径修正后，稳定显示空窗从 40 帧降到 31 帧，最长空窗从 23/15 帧降到 15/10 帧。
  - 抽帧观察显示短时续框没有明显旧框乱飘，但 90-108 帧附近仍有约半秒无框。这是模型 raw detection 连续为空导致，不适合继续通过长时间旧框硬拖来掩盖。
  - 降低推理置信度到 0.18 的实验未改善稳定显示，反而让 stable zero frames 增加到 43，暂不进入生产配置。

## 残留边界

- 2 类 head/helmet 模型在部分视频段落 raw detection 连续为空，后处理只能短时桥接，不能替代模型召回。
- A3b 灵敏度实验样本仍少，且当前 preset 同时修改阈值和 `static_image_interval`，后续若要精调，应拆分“只改阈值”和“只改采样间隔”两组实验。
- 安全日志页已中文化常见事件和值，但报告文件名、模型哈希、真实路径片段和底层异常文本会保留原始技术字符串。
- 本轮尚未提交。按项目规则，涉及页面状态和检测显示的修改必须等待用户亲自测试确认后才能提交。

## 建议

- 默认保留当前 head/helmet-only 短时显示保持策略，不再继续拉长到 16 帧以上。
- 若必须进一步减少 2 类模型断框，应优先处理模型召回、训练数据或按模型族配置更合理的推理阈值，而不是无限延长旧框显示。
- 对 A3b 的下一步优化应导出 per-frame `a3b_reason` 和 `a3b_debug.failed_gates`，定位弱异常样本未进入最终告警的具体门控。
