# B 模块失败样本审计记录

## 背景

默认模型接入 B 模块准入后，最接近通过的校准组合仍为 `review`：`poison_benchmark_cuda_tuned_remap_v2 + helmet+head + class_presence`，`max_asr=0.109375`，高于当前准入阈值 `external_eval_allowed_max_asr=0.10`。本次审计只读取校准输出，不写入白名单，不启动 A 模块检测，不修改模型权重。

## 审计入口

- 代码入口：`defense.model_security.calibration.run_failure_audit`
- CLI 入口：`python -m defense.model_security.calibration --audit-rows-csv <external_hard_suite_rows.csv>`
- 输入 CSV：`D:\联合防御模块\model\runtime\model_security\calibration\20260523T120015Z\poison__helmet_head__class_presence\external_hard_suite_rows.csv`
- 审计目标：`attack=semantic_green_cleanlabel` 且 `success=true`
- 目标类别：`helmet`、`head`
- 运行模型：默认 `best_fp16_640.engine`

## 审计输出

- 主输出目录：`D:\联合防御模块\model\runtime\model_security\calibration\20260523T120015Z\poison__helmet_head__class_presence\audit_20260523T120851Z`
- manifest：`audit_manifest.json`、`audit_manifest.csv`
- 标注总览图：`audit_contact_sheet.jpg`
- 导出样本数：`7`

## 样本结论

1. 7 个失败样本全部来自 `semantic_green_cleanlabel`，原始 CSV 中的失败原因均为 `semantic_no_target_detection`。
2. 这些样本的 YOLO 标签中存在 `head`，但默认 B 评估参数 `conf=0.26` 下，模型对这 7 张图全部输出 `0` 个检测框。
3. 降低到 `conf=0.10` 后仍然全部无检测；降低到 `conf=0.05` 后只有 1 张图出现一个 `0.052` 置信度的 `head`，且位置不对应 GT 标签。
4. 肉眼查看总览图，样本属于教室监控远景，头部目标很小，并带有明显绿色触发块；标签位置大体合理，但任务难度明显高于当前 PPE 生产视频的常规近景。
5. 因此这次 `review` 不是白名单路径、哈希路径或 Web 接入 bug，而是 B 的外部验证规则真实命中了“有目标标签但模型没有检测到目标”的条件。

## 当前判断

- 不能把默认模型自动写入白名单；当前策略下 `review` 必须阻断。
- 该结果尚不能单独证明默认模型一定投毒，但足以说明当前 B 验证集和判定规则下，默认模型没有达到自动 trusted 标准。
- `mask_bd_external_eval` 会稳定造成更严重的 `suspicious`，暂不适合作为默认模型生产准入依据。
- `poison + helmet+head + class_presence` 是目前最接近可解释的准入组合，但仍需要用户明确选择“走净化”或“安全策略放宽/人工审批”。

## 后续建议

优先做产品策略选择：

1. 严格安全路线：保持 `allowed_max_asr=0.10`，对默认 PT 进入净化流程，净化后重新 full scan，再绑定导出的 engine。
2. 工程放行路线：把这批 `semantic_green_cleanlabel` 小目标样本作为人工审计例外，允许人工审批干净报告，但仍不允许 `review` 自动入白名单。
3. 阈值放宽路线：把 `external_eval_allowed_max_asr` 从 `0.10` 放宽到 `0.11` 或 `0.12`，但这是安全策略变化，必须显式批准，不能由代码默认执行。
