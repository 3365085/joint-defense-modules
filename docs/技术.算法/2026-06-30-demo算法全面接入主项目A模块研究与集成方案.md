## 0. 一句话结论

demo 与主项目**共享同一套契约**（demo 的 `detector.py` 直接 `from defense.module_a import
ModuleAInput, ModuleAResult, ROI`，两者都暴露 `process(item) -> ModuleAResult`）。因此正确路线
是**把 demo 检测内核作为主项目的一个可切换 detector 后端接入**（backend-swap），而不是高风险的
逐行重写或整包覆盖。主项目保留自身轻量化外壳与现有 detector，用 config 开关在两套内核间切换、
A/B 对拍、按场景留出评测做门禁。

> 最大风险点（必须先解）：demo 是**逐帧 + 光流连续**设计，主项目热路径是 **latest-only 丢帧 +
> keyframe 间隔 + 检测复用**。两者直接叠加会破坏 demo 赖以工作的"相邻帧光流/LBP 时序连续性"。
> 这是"轻量化外壳包 demo 内核"能否成立的核心，**待实验确认**。

## 1. demo 算法全貌（已读代码确认）

### 1.1 分层：传感器 A1-A3 → 决策 A4，外加 A3b / 支路B / 场景基线 / 目标锚定

| 层 | 含义 | 关键产出 |
|---|---|---|
| A1 单帧 LBP 纹理异常 ΔH | 当前帧 LBP 直方图与**场景自适应 EMA 基线**的距离 | `delta_h / delta_h_roi_max / delta_h_local_max / delta_h_target_contrast / a1_feature_score` |
| A2 时序纹理不一致 Change_t | 相邻帧 LBP 变化速率（带 without_motion 修正） | `change_t / change_t_roi_max / change_t_local_max / change_t_without_motion / a2_feature_score` |
| A3 光流伪影 f_flow | 去除全局运动后的局部残差（RAFT-TRT/GPU-LK @256² ） | `f_flow / flow_local_anomaly_ratio / flow_residual / flow_shape_score / flow_target_relation / a3_feature_score` |
| A3b 静态媒体 p_media | Canny+轮廓找"规则矩形边界+内部纹理"（翻拍/回放/纸张） | 后台线程算，主路径用上一次结果；4 维进 A4 |
| A4 决策 p_adv | **20 维特征 → XGBoost** 输出攻击概率 | `_compute_a4`；缺分类器回退 `_rule_p_adv` sigmoid |
| 支路B 致盲 p_blind | **rule-based**，测"相对本场景基线的清晰度/对比度/置信度骤降" | `_compute_blinding` |
| 目标锚定 TA | 复用主项目 `TargetAnchoredAnalyzer` 思路，做目标锚定抑制 | `target_anchored.py`（demo 自带一份）|
| 联合决策 | 大量场景化抑制/救援规则 + 三支路 N-of-M 时序确认 | `_joint_decision`（约 670 行） |

### 1.2 A4 的 20 维特征向量（`_compute_a4`，已对代码逐项确认）

```
A1×5 : delta_h, delta_h_roi_max, delta_h_local_max, delta_h_target_contrast, a1_feature_score
A2×5 : change_t, change_t_roi_max, change_t_local_max, change_t_without_motion_target, a2_feature_score
A3×6 : f_flow, flow_local_anomaly_ratio, flow_residual, flow_shape_score, flow_target_relation, a3_feature_score
A3b×4: p_media_raw, p_media_scores.flow_gap, p_media_scores.warp_residual, p_media_scores.display
```
- 训练侧 `tools/train_a4_auto.py` 的 `FEATURE_NAMES` 与之**一一对齐**（20 维，P4 5 维已否决、运行期不计算）。
- 分类器按 `n_features_in_` 自适应裁/补零（兼容旧 20 维 / 实验 25 维）。
- 训练纪律：**StratifiedGroupKFold 按场景分组**，`scale_pos_weight` 处理类不平衡。

### 1.3 支路B `_compute_blinding`（主项目完全缺失，需新增）

- 物理本质：致盲/去信号型攻击（motion_blur / visibility / glare 致盲）会**抹掉纹理**→ A1/A2 反而
  **低于**干净帧，A4(支路A) 检不出。改测"相对本场景近 N 帧自身基线的退化"。
- 度量：清晰度（拉普拉斯方差，优先 Rust `blinding_laplacian_var`）、对比度（亮度 std）、
  YOLO 目标置信度强度；取本场景滚动分位数(p70)为参考，算相对骤降。
- 冷启动绝对兜底（P1）：基线未就绪时靠"曾有目标却骤然漏检 + 帧间清晰度骤降/强过曝"判定。
- 实测：干净段 p_blind≈0.006，致盲段≈0.7，几乎零误报。

### 1.4 场景自适应基线抑制（主项目缺失，需新增）

- `_sb_sharp/_sb_contrast/_sb_detstr/_sb_maxfeat` 四个滚动窗口（默认 30 帧，攻击候选时冻结更新）。
- 对**支路A 也**做"在本场景近况正常带内即抑制"，并由 P0 改为对运动场景也生效的 **z-score 判定**
  （当前 max_feature 在本场景滚动分布正常带内即抑制；真实攻击 z≫2 不被误抑）。
- 目的：压制焊接/剧烈运动等"干净高能场景"的误报（特征天花板的工程缓解）。

### 1.5 联合决策 + N-of-M 时序确认

- `window_frames = clamp(ceil(process_fps*0.5), 3, 8)`；adv/media/blind 三支路各有命中阈值
  （`adv_hit_required` 随高误报场景上调到 0.67）。窗口内 N-of-M 命中才确认告警，过滤瞬时突发。
- `_joint_decision` 含大量按 `a3b.suppressed_reason` 分类的抑制/救援规则（physical_patch_rescue /
  a3_residual_fallback / no_target_fallback / cold_start_low_motion_adv 等），是 demo 调优沉淀的核心。

## 4. demo 已被数据否决的方案（主项目不要重走）

来源 `optimization-roadmap.md` / `a4-xgboost-classifier.md`，均经分组CV/系统级验证：
- **P3 相对场景偏离喂分类器**：FP 帧本身就是相对场景离群点，z-score 门控分不开，无效。
- **P4 ROI 判别特征（hf/lap/edge/sat/color）**：分组CV 不增益(0.58)、系统级误报恶化 4/12→5/12，
  运行期禁用（`_p4_enabled=False`，零开销，实现保留备 P4v2）。
- **小 CNN**：14→127 场景组下 AUC 0.66 仍全面输 XGBoost 0.82，根因是**真实攻击场景多样性(~7)**，非模型。
- **NVDEC 硬解**：4K 快 6 倍但改变检测输入（p_media 翻转 8%），违背数值红线，不采用。
- 结论：边端约束下 **XGBoost 是当前最优**；真正瓶颈是真实攻击素材多样性，需实拍/自录。

### 阶段 2｜内核依赖落地
- Rust native（`native/module_a_native`，pyo3）：作为可选加速，导入失败自动回退纯 Python。

- **三处维度对齐铁律**：采集(`collect_all`)/训练(`train_a4_auto`)/detector(`_compute_a4`) 必须同 20 维。

### 阶段 3｜套主项目轻量化外壳（**关键风险，需实验**）
- 目标："主项目轻量化外壳 + demo 检测内核"。
- 冲突：demo 依赖逐帧光流/LBP 连续性；主项目 latest-only 丢帧 + keyframe 间隔会打断连续性。
- 待实验确认的选项（按风险）：
  1. demo 内核走"检测线程逐帧、预览解耦"，不强加 keyframe 跳帧（最贴近 demo，FPS 靠 Rust/RAFT/三线程）。
  2. 给 demo 内核加"prev_gray 缺帧守卫 + 时序状态在丢帧时降级"，容忍 latest-only（需对拍量化漂移）。
  3. keyframe 仅作用于 YOLO，光流/LBP 仍逐帧（折中，需测算力）。
- 不变量：保留模块B、边端 profile、安全准入、Web API/状态字段兼容。

### 阶段 4｜门禁与验收
- 接入 demo 的按场景留出评测当 CI 门禁（误报 ≤1/12 且跨场景命中 ≥9/10）。
- 涉及检测框/预览/overlay 的改动，按 AGENTS.md「提交前视觉验收」跑目标视频、抽连续 3 秒逐帧核验。

## 6. 取舍与待确认项

- **数值不变量的边界**：这是"行为调优"而非等价重构——接入新内核会改变 A1-A4 数值与告警结果，
  须以**新基线 + 留出评测**衡量，而非要求与旧 A 逐位等价。AGENTS.md 的数值不变量约束应理解为
  "同一内核内的提速/重构不得改数值"，跨内核替换属显式调优工作。
- **轻量化 vs 光流连续性**（阶段3）：最大未决项，**待实验确认**。
- **分类器口径统一**：推荐 (a) 20 维 + XGBoost（轻、CPU <1ms、边端友好、文档点名）。
- **真实攻击素材多样性**：当前 ~7 个攻击场景是质量天花板，长期需实拍/自录扩充（demo 已记此为瓶颈）。
- **未能从代码中确认**：demo 双进程运行时层与主项目 runtime 的合并代价（本期不评估，仅接检测内核）。
