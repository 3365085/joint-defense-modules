# 检测输入分辨率与远处小目标问题记录

## 背景

用户提出：远处人员检测差可能与检测输入像素过少有关，考虑把检测像素调整为原视频像素的 1/4，并保持原始宽高比例。

## 代码链路依据

- `src/defense/runtime/frame_processor.py:prepare_frame_640()` 当前先把超大帧按最长边限制到 1280，再强制 resize 到 640x640。
- `src/defense/runtime/frame_processor.py:FrameProcessor.process()` 使用 `prepare_frame_640()` 的输出进入 `VideoDefensePipeline.process_frame()`。
- `src/defense/module_a/backends/detector_backend.py:UltralyticsDetectorBackend.predict()` 使用 `imgsz=self.image_size`，默认配置为 `640`。
- `configs/module_a_runtime.yaml` 中 `inference.image_size: 640`，默认 TensorRT artifact 为 640 engine。

## 当前判断

远处目标效果差确实与有效检测像素过少有关。以 4K 横屏为例，原始 3840x2160 被处理到 640x640 后，横向有效缩放约为 1/6，远处头部可能只剩十几像素，模型很容易只输出弱 `head` 或完全漏检。

“原像素 1/4、比例不变”在几何上等价于宽高各缩小到 1/2：

- 3840x2160 -> 1920x1080
- 1920x1080 -> 960x540
- 1280x720 -> 640x360

该方案能显著增加远处头部像素，但不能直接套到当前 640 TensorRT engine 上。固定 640 engine 通常要求固定输入尺寸；即使前处理保留比例，后端仍会按 `imgsz=640` 做推理，收益会被再次压回 640 尺度。

## 风险和取舍

- 直接把主推理输入提高到 960/1280/1920，需要对应动态 shape 或重新导出更大输入尺寸的 TensorRT engine。
- 输入面积提升会近似线性增加推理、NMS 和后处理成本，可能重新引入视频卡顿。
- 保持比例有助于避免 640x640 强制拉伸带来的几何失真，但 overlay、ROI、A3b 特征和状态记录都需要处理坐标尺度映射。
- 对远处小目标更稳的工程路线可能是“双尺度/ROI 复检”：主路保持 640 保障实时性，只对疑似远处 weak head 或高风险区域做局部高分辨率复检。

## 后续建议

1. 先确认默认 TensorRT engine 是否固定 640；若固定，需要导出 960 或 1280 engine 后再评估。
2. 优先实验 `960` 或 `1280` 长边保持比例输入，而不是直接 1/4 4K 像素到 1920x1080。
3. 若不想牺牲实时性，应优先做 ROI 高分辨率复检：远处小目标区域扩大裁剪后重新检测，再映射回 640/预览坐标。
4. 任何分辨率调整都必须重新跑检测视频和连续 3 秒抽帧验收，重点观察流畅度、拖框、误检和远处人员召回。
