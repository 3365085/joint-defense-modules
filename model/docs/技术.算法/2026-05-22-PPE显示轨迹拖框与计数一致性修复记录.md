# PPE显示轨迹拖框与业务计数一致性修复记录

日期：2026-05-22

## 问题背景

用户在固定镜头室外视频回放中观察到页面画面明显抽帧、检测框拖框、旧框滞留，以及画面底部 PPE 计数与可见检测框不一致。例如状态显示 `head=0` 时，画面上仍可见历史 `head` 框。这类问题会误导人工判断，并使“检测是否真的识别到目标”变得不可信。

## 当前判断

本次问题不只来自 `runtime/runner.py` 的 overlay 选择窗口。旧框来源有三层：

1. `src/defense/module_a/postprocess/ppe_tracking.py` 中 `PPEDisplayTracker` 会把短时丢失的目标作为 `held` 轨迹继续渲染。
2. `src/defense/web/overlay_timeline.py` 的 `interpolate_overlay()` 在两条 overlay 记录之间插值时，原逻辑会保留上一条记录中未被下一条记录匹配的轨迹。
3. `src/defense/runtime/ppe_business.py` 原先直接输出 tracker 轨迹，没有再次按业务有效计数过滤，导致 `ppe_head_count=0` 但 `ppe_tracks` 仍含 `head` 框。

因此，只缩短 `MonitorEngine._select_preview_overlay()` 的 held 时间不能彻底解决拖框与计数不一致。

## 代码链路依据

- `src/defense/runtime/frame_processor.py::FrameProcessor.process()`：每帧调用模型、ROI 复检和 `evaluate_ppe_business()`，并把 `ppe_tracks` 写入状态与渲染链路。
- `src/defense/runtime/ppe_business.py::evaluate_ppe_business()`：把 `summarize_ppe_from_detections()` 的 PPE 语义结果、`SafetyHelmetState` 时序状态和 `PPEDisplayTracker` 显示轨迹合成业务输出。
- `src/defense/module_a/postprocess/ppe_tracking.py::PPEDisplayTracker.update()` / `_render_tracks()`：负责 PPE 框的显示稳定、短时 hold 和最终渲染轨迹。
- `src/defense/runtime/runner.py::MonitorEngine._select_preview_overlay()`：按视频时间选择后端 overlay 记录，决定最近帧、插值帧或 held 帧。
- `src/defense/web/overlay_timeline.py::interpolate_overlay()` 与 `src/defense/web/static/overlay_timeline.js::interpolateOverlay()`：页面/后端 overlay timeline 的插值语义。

## 修复结论

1. `PPEDisplayTracker` 增加 `hold_eligible`，弱 head 或不应延续的框不再进入 held 渲染和 ROI held 复检候选。
2. 文件实时预览通过 `FrameProcessor._ppe_max_render_misses()` 对 `PPEDisplayTracker.update(max_render_misses=0)` 施加限制，避免文件回放画面继续显示丢失帧目标。
3. `evaluate_ppe_business()` 新增 `_filter_tracks_for_ppe_counts()`，最终 `ppe_tracks` 只能包含当前业务计数仍有效的 `person`、`helmet`、`head` 类型。
4. `MonitorEngine._select_preview_overlay()` 对文件实时预览缩短 display-only smoothing，并在插值时传入 `keep_unmatched_tracks=False`，不再把上一条 overlay 中未匹配的 PPE 轨迹带到当前视频时间。
5. Python 与浏览器静态 `overlay_timeline` 均过滤 `hold_eligible=False` 的轨迹，避免前后端语义分叉。

## 影响范围

- 本次修改只影响 PPE 显示轨迹、overlay 选择和文件实时预览的旧框延续策略。
- 不修改模型权重、类别表、置信度阈值、PPE 主判定语义或 A3b/Module A 策略。
- 实时流/摄像头仍允许按既有策略进行短时 held 显示，以降低真实实时采集下的闪烁。

## 未能从代码中确认

- 该修复不保证模型本身不会产生误检；如果当前帧业务计数有效，例如 `head_count=1`，画面仍会显示该模型输出对应框。
- 偶发启动初期卡顿是否完全来自管线初始化，仍需结合 GPU/TensorRT 首帧 warmup、浏览器 MJPEG 解码和系统负载日志进一步实验确认。

## 补充判断：检测效果与显示修复的边界

用户追问“那岂不是检测效果很差”。基于本次固定镜头室外视频的 overlay 统计，PPE 有效轨迹只出现在少量记录中，说明当前模型/阈值/场景组合下的 PPE 召回确实偏弱。该结论仅针对此视频和当前模型配置，不能直接外推到所有工业安全帽场景。

本次修复没有降低模型输出能力，而是把弱 head、uncertain、小误检和丢失后的 held 旧框从显示层清理掉，使画面框与 `ppe_head_count`、`ppe_helmet_count` 等业务计数字段一致。因此，修复后的画面看起来“框更少”，本质是减少误导性显示；若业务要求“尽可能框出所有人头/人员”，后续需要单独做模型召回、类别策略、阈值和复检策略评估。
