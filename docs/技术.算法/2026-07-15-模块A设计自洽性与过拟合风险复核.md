## 一、执行结论

当前不能给出“Module A 各模块逻辑完全自洽、没有过拟合风险”的结论。

更准确的判断是：

1. **总体架构方向可运行且部分关键生命周期已经硬化。**
   - FastAPI 是生产 Web API；
   - preview 与 detection 解耦；
   - detection 使用 latest-only backpressure；
   - 默认生产链路实际使用 rebuilt；
   - A3b 后台 worker 已具备 generation、timeout、lease、原子发布、进程级 worker 上限和错误状态；
   - rebuilt/legacy A3b 公共结果契约已得到统一。

2. **没有确认会立即令默认服务完全不可用的运行时 P0。**
   但存在多项已复现的 P1/P2，涉及 A3b 证据独立性、路径标签泄漏、PPE suppression、restart/source switch、配置失效、A4 资产可复现性和证据保存真实性。

3. **过拟合与评测泄漏风险为高。**
   - 默认 runtime 的 A3b soft-trigger/补充确认读取 source path 关键词，属于确认的标签泄漏；rebuilt authoritative `media_confirmed` 内核本身仍是内容驱动；
   - 当前 heldout 已反复用于策略选择，应降级为 development/calibration set；
   - train/heldout 存在字节级重复；
   - heldout 攻击全部是 synthetic，且没有 A3b replay/interference 正例；
   - 正负样本与编码器/生成器高度混杂；
   - 当前生产模型没有与当前代码、配置和资产哈希绑定的独立最终验收报告。

4. **当前自动化测试主要证明代码契约，而不是生产默认 rebuilt 的真实泛化效果。**
   429 项全量 smoke 通过是重要工程证据，但不能替代独立真实攻击、完整 clean 长流、跨设备、跨编码、跨路径和跨链路验证。

### 风险分级口径

- `P0`：默认生产能力立即不可用、不可恢复，或存在已确认的灾难性安全破坏。
- `P1`：默认生产链路中的确定性高风险缺陷，或足以阻断可信验收/泛化结论的问题。
- `P2`：确定的设计脆弱性、非默认路径缺陷或需要实验量化影响的问题。
- `P3`：可观测性、文档、测试卫生或维护性问题。

数据泄漏和独立测试集失效在本文中列为“**验收结论阻断项**”。它们不等同于服务运行 P0，但会阻断“已证明泛化”的交付声明。

## 二、生产默认链路

CodeGraph 和配置有效值复核确认当前主链路为：

```text
FastAPI /api/start
  -> MonitorEngine.start
  -> PipelineCache.get
  -> capture thread
       -> PreviewBus.publish
       -> DetectionBus.push            # 单槽 latest-only
  -> process thread
       -> DetectionBus.pop_latest
       -> FrameProcessor.process
       -> VideoDefensePipeline.process_runtime_frame
       -> detector backend
       -> RebuiltModuleADetector.process
       -> A1/A2/A3/A3b/A4/blind/joint decision
       -> ModuleAResult.to_info_dict
       -> PPE/status/evidence
  -> preview thread
       -> preview render/MJPEG
```

关键事实：

- `model/configs/module_a_runtime.yaml` 当前 `detector_impl: rebuilt`；
- 所有已加载 profile 的 detector implementation 均为 rebuilt；
- `VideoDefensePipeline` 只有精确值 `legacy` 才选择 legacy，其他值均进入 rebuilt；
- 生产 PPE postprocess/tracking 仍由 `FrameProcessor` 接线；
- `DetectionBus` 是 latest-only 单槽，不形成无界检测积压；
- preview thread 不调用检测，慢检测不会直接阻塞抓帧和预览。

## 三、已确认正确或方向合理的设计

### 3.1 Runtime 与公共契约

- preview/detection 分离符合实时监控设计；
- latest-only backpressure 符合“宁可跳帧，不积压旧检测”的约束；
- 检测结果发布前存在 run ID 和 source epoch 检查；
- seek 后检测侧会清 DetectionBus，并在 epoch 变化时 reset processor；
- A3b worker 的旧 generation 结果不能发布到新 generation；
- A3b 后台错误、timeout、rejection、worker 数量和 result freshness 已进入 status；
- `ModuleAResult.to_info_dict()` 已修复 confirmed/active 攻击被公共字段表达为 NORMAL 的旧问题；
- rebuilt/legacy A3b 通过统一 adapter 向 runtime 和 suppression 提供公共字段；
- 公共 A3b status schema 在首帧前、初始化、预热、seek 和首帧后保持稳定。

### 3.2 A1/A2/A3 基础方向

- A1 灰度输入和过曝/欠曝阈值方向一致，分数越高代表异常越强；
- temporal flash 同时检查变化面积、变化极性和亮度幅度，基础物理解释合理；
- motion、flow、track、blur 的分数方向整体一致；
- track missing 只在首次丢失时计票，避免同一 track 的每个 miss 帧重复增加 missing count；
- PPE 当前帧 summary 在 `head/helmet` 互斥类别假设下，使用未被 helmet 覆盖的 head 作为缺帽候选，基本语义自洽；
- 未发现 A1/A2/A3a/PPE 生产代码读取 GT、manifest 标签或未来帧作为业务判定输入。

### 3.3 A3b 本轮已关闭的旧问题

相较于此前审计，以下问题在当前工作树中已有实现和测试证据：

| 历史问题 | 当前状态 |
|---|---|
| rebuilt/legacy A3b 字段不兼容 | 已通过统一 result adapter 处理 |
| confirmed A3b 导出为 NORMAL | 已修复公共攻击字段 |
| reset/close 后旧 worker 写回 | 已有 generation 隔离 |
| worker 挂起后污染结果或无限创建替代 worker | 已有 generation 失效、monotonic timeout 和 worker hard limit；底层 Python 线程本身仍不能被安全强杀 |
| 成功结果永久有效 | 已有 result lease/freshness |
| 发布一半成功、一半失败 | 已改为锁内原子发布 |
| 首帧前 status 字段缺失 | 已预置稳定 schema |
| 后台错误不可见 | 已进入 status/warning |

这些修复说明 A3b 的工程生命周期已有明显改进，但不能消除下文的策略和泛化问题。

## 四、逐模块自洽性复核

## 4.1 A1：曝光/强光/致盲

### 当前判断

A1 的基础物理方向合理，但独立泛化证据不足，且其结果会受到尺寸预处理、场景基线、固定阈值和 joint carveout 的共同影响。

### 自洽点

- 灰度值域和阈值方向一致；
- 过曝、欠曝、flash/亮度跃迁是不同输出；
- A1 不依赖 source path 或数据标签。

### 风险

1. 生产多处直接执行：

   ```python
   cv2.resize(frame, (640, 640))
   ```

   未使用 letterbox。宽高比变化会改变：

   - 饱和区域比例；
   - 局部边缘和纹理；
   - bbox 几何；
   - border contrast；
   - A1/A3b 的目标相对位置。

   已有历史实测指出竖屏输入会出现能力损失。该项是跨方向、跨设备泛化风险。

2. 当前真实攻击 heldout 为 0，不能从合成 glare 代理样本外推出真实强光、焊接、自动曝光或镜头炫光召回。

### 结论

A1 的基础特征方向自洽，但真实强光、自然曝光和镜头炫光的业务边界尚未由当前独立数据证明。现有“自然曝光/strong static glare”冲突测试属于 legacy target-anchored 逻辑，不能直接当作默认 rebuilt A1 的缺陷证据。

## 4.2 A2：纹理与时序纹理

### 默认 rebuilt 链路风险

1. latest-only 下，runtime 会提供严格源前驱，但 `_inject_temporal_previous_frame()` 目前只同步：

   ```text
   prev_gray
   prev_lbp
   ```

   未同步：

   ```text
   prev_brightness
   ```

   因而同一帧的 scene context 可能同时使用：

   - `frame_diff_global`：当前帧与严格源前驱；
   - `exposure_delta`：当前帧与更旧的上一次处理帧。

   这是确定的前驱状态混用。`prev_timestamp` 当前用于估算处理 cadence，是否应改为 source cadence 仍属于语义待裁定项，不能与 brightness 缺陷等同。

2. detection reuse 使用 pipeline 内部处理帧序号，不使用真实 source frame gap。latest-only 从源帧 10 跳到 20 时，processed gap 可能仍只有 1，静态相似画面仍可能复用旧 detection。

### legacy A2 的确定性设计问题

legacy 并非当前默认 detector，但仍是公开导出和可配置实现：

1. LBP 被编码为 0—255 整数后，直接使用：

   ```python
   abs(curr_lbp - prev_lbp) / 255
   ```

   这把 bit pattern 当作有序连续值。同样一次 bit 翻转发生在低位或高位时，距离可相差 128 倍，不具备 Hamming/histogram 距离的语义。

2. noise suppression 将输出固定压到约 `0.029/0.044`，但下游 fusion 门可配置。当门值改低后，可能同时出现：

   ```text
   noise_suppressed = true
   temporal_triggered = true
   ```

3. legacy 的固定归一化常数会使 frame size 改变时量纲变化。

### 结论

默认 rebuilt 的主要问题是时序输入和 source gap 不完整；legacy A2 还存在明确的度量与配置不变量问题。A2 对旋转、镜像、编码噪声和分辨率的稳健性需要配对实验。

## 4.3 A3：运动、光流、模糊、track 与 blind

### P1：flow/device 配置契约失效

rebuilt 当前：

- 构造时无条件加载 flow backend；
- 每帧无条件调用 `_compute_flow()`；
- 不读取 `light_flow_enabled`；
- CUDA 设备硬编码为 `cuda:0`；
- 不尊重 profile 中的 `device: cpu` 或其他 CUDA device；
- RAFT 失败后可能静默回退到 GPU LK/DIS。

因此：

- `cpu_smoke.light_flow_enabled=false` 不等于关闭 flow；
- 有 CUDA 时 CPU profile 仍可能使用 `cuda:0`；
- 无 CUDA 时即使配置声称关闭，也会执行 CPU DIS；
- status 没有完整暴露 requested/effective device、flow backend 和 fallback reason。

这会使性能 profile、部署契约和算法能力状态不可信。

### P2：blind 长记忆和固定帧阈值

`_blind_target_established` 一旦为真，直到 reset 才清除。长实时流中，目标合法离场后仍可能保留“曾有目标”的长期记忆，后续无人场景中的虚焦、曝光退化或偶发异常可能继续进入 blind sustained。

当前常用 heldout 每段最多只评测 240 帧，无法覆盖：

- 长时间目标离场；
- 长流场景基线漂移；
- 多次断流重连；
- 数分钟后的偶发 blind 累积。

实际误触发属于待实验确认。

### 时间语义不一致

rebuilt 中部分 sustained 逻辑按 `process_fps * seconds` 归一化，但下列逻辑仍按固定处理帧：

- blind sustained 12；
- alert hold 12；
- scene baseline 30；
- A1 hold 5；
- A3 hold 4；
- recent target deque 8。

同一规则在 60/30/6 detector FPS 下代表完全不同的物理时间。该问题在 profile effective FPS 失效时会进一步放大。

### 结论

A3 的多证据方向合理，但 flow 设备契约、时序状态寿命和固定帧/秒混用不自洽；长流风险未被当前短片评测覆盖。

## 4.4 A3b：replay/static media/interference

A3b 是当前最明显的设计和过拟合风险集中区。

### P1：同一缓存结果被重复视为独立证据

rebuilt authoritative kernel 每个主处理帧重复读取同一个 `_a3b_bg_result`，并在未检查 `a3b_result_seq` 变化的情况下：

- 增加 `_media_run`；
- 追加 `media_hits`；
- 推进 N-of-M/持续确认。

最小探针确认同一 `seq/source_frame` 重复输入 6 次后：

```text
media_run = 6
confirmed = true
```

soft trigger 同样每帧向 window 追加；`result_seq` 当前主要用于 bbox 跃迁，不用于投票去重。

因此：

```text
media_run >= 15
```

当前表示“主处理循环重复消费了 15 次”，不等于“完成了 15 次独立 A3b 推理”。

这是确定的证据独立性问题。修复会改变触发时序，必须作为独立 behavior-tuning 任务并使用新 A3b 正负样本验证。

### P1：runtime soft-trigger 的 source path 关键词构成标签泄漏

当前配置包含：

```yaml
observed_only_source_keywords:
  - 视频中出现干扰视频
trigger_source_keywords:
  - 视频中出现干扰视频
```

source 字符串是在 rebuilt pipeline 返回结果后由 `FrameProcessor` 注入 `A3BSoftTriggerState`。soft-trigger 通过：

```python
keyword in source_text
```

决定 soft rescue 是否允许。同一视频字节只因移动目录、改名或换成 camera/RTSP source 字符串，结果就可能改变。

更具体地：

- 目标视频路径本身包含允许关键词；
- 目标视频已用于 A3b 行为调整和验收；
- camera/RTSP 通常不包含该关键词；
- `high_score_bypass` 虽在前段逻辑声称绕过 source keyword，后置 source guard 又会把结果清掉。

这不是一般意义上的“可能过拟合”，而是确认的 runtime source-label leakage。需要收窄作用范围：

- rebuilt authoritative `media_confirmed` 内核不读取 source path，仍是内容驱动；
- 受影响的是默认 runtime 的 soft-trigger、补充确认以及由其继续驱动的 status/PPE suppression 路径；
- 当前目标视频的“约 0.5 秒提前触发”不能被外推为不依赖路径的 rebuilt 内核能力。

### P1：policy suppression 与 PPE suppression 语义冲突

`FrameProcessor._source_auth_media_suppression_active()` 主要检查 bbox、分数和 trigger/candidate 字段，没有完整检查：

- `media_candidate_allowed=False`；
- `suppressed_reason`；
- `suppression.suppressed`；
- result freshness。

已复现的输入中：

```text
media_candidate_allowed = false
a3b_state = suppressed
suppressed_reason = low_display_target_plane_prefers_A1_A2_A3
```

但 PPE suppression 仍返回 true。

因此 A3b policy 已把候选交还 A1/A2/A3 时，PPE 层仍可能隐藏 person/head/helmet 框。

### P2：suppression hold 可能远长于 result lease

A3b result lease 为 5 秒，但 pipeline suppression hold 为固定 180 processed frames：

| detector FPS | 180 帧物理时长 |
|---:|---:|
| 60 | 3 秒 |
| 30 | 6 秒 |
| 6 | 30 秒 |

pipeline 当前有意让 hold 跨越短暂后台结果间隔，因此 hold 不要求与 lease 相等；仅凭低 FPS 下折算为 30 秒，还不能确认它一定是高风险缺陷。可确认的是不同 profile 的物理持续时间差异很大，是否造成旧框长期隐藏真实 PPE，需在低 FPS profile 和攻击结束段做实验。

### 已关闭与未关闭必须区分

当前已关闭的是：

- worker 代际污染；
- timeout/lease/原子发布；
- 公共字段和错误状态。

当前仍未关闭的是：

- 缓存重复计票；
- source keyword 标签泄漏；
- policy/suppression 语义冲突；
- fixed-frame hold 与 lease 的物理时间差异及低 FPS 风险；
- 独立 A3b heldout 正例缺失。

### 结论

A3b 工程生命周期已较稳，但触发证据独立性和泛化设计仍不自洽。当前目标视频结果不能外推到普通文件、摄像头或 RTSP。

## 4.5 A4：classifier 与联合特征

### P1：资产可用不等于可交付、可复现

当前机器上的 A4 classifier 和 RAFT 资产可读取，但：

- A4 config 使用本机绝对路径；
- `rebuilt_demo/` 被 `.gitignore` 忽略；
- `src/defense/module_a/rebuilt/data/` 不存在正式 bundled 资产；
- RAFT ONNX/engine 也不受当前 Git 跟踪；
- 干净 checkout 中能力可能静默变化。

当前资产示例：

```text
A4 classifier: rebuilt_demo/data/a4_classifier.pkl
feature dimension: 20
RAFT: rebuilt_demo/data/raft_small_256.onnx
RAFT TensorRT: rebuilt_demo/data/raft_small_fp16_256.engine
```

### P1：训练 detector 与生产 detector 分布不一致

A4 批量特征采集 wrapper `collect_all.py/.bat` 默认指向过旧的三类 detector；单视频采集工具本身支持显式 `--model`，训练脚本只消费 feature CSV。当前生产则使用：

```text
classmate_maskbd_v4
classes = [helmet, head]
```

A4、A1/A2/A3/A3b 多处依赖：

- ROI 标签；
- person/target context；
- bbox 置信度；
- target disappear；
- recent target presence；
- 目标相对位置。

但当前 pkl/feature CSV 没有绑定或记录足够的采集谱系：

- detector hash；
- class map；
- feature schema hash；
- flow backend；
- 数据 manifest hash；
- 代码 commit。

更换 detector 后仍加载同一个 pkl，会形成输入特征分布漂移。

### P1/P2：错误与 schema 不匹配采用 fail-open

- `_load_classifier()` 失败时静默返回 `None`；
- `predict_proba()` 异常时静默回退规则；
- status 不明确暴露 classifier configured/loaded/error/fallback；
- feature 维度不同时会自动截断或补零继续运行；
- 历史上已经出现过 25 维模型与 runtime 20 维输入不一致。

当前虽然恰好是 20 维，但兼容策略会掩盖未来错误资产。

另有字段偏差：

```text
_compute_a4 读取 p_media_scores["display"]
A3b 实际写入 display_frame
```

当前模型最后一维 importance 为 0，现机影响有限，但名义 20 维中存在无信息输入。

### P2：TargetAnchoredAnalyzer 为死接线

rebuilt 每帧执行 analyzer，但 `_joint_decision()` 中：

```text
ta_suspicious
ta_classifier_bonus
```

只被赋值，之后没有消费者。AST 探针确认两变量均为 `stores=1, loads=0`。

同时：

- 多个输入为固定零值或空集合；
- `reset()` 未调用 `self._ta.reset()`；
- analyzer 历史和 glare EMA 可跨 source reset 保留。

当前 TA 不影响结果，因此主要是无效复杂度和潜在未来接线风险。不能直接把它接入最终 gate，否则会构成未经验证的行为变化。

### 结论

A4 当前机器“能加载”，但训练谱系、生产 detector 分布、错误可见性和 schema 约束不自洽，是 Module A 中第二个高过拟合/分布漂移风险区。

## 4.6 Joint decision、阈值与时间窗口

### 配置文本与实际逻辑漂移

基础配置含 69 个 Module A 键。静态使用点统计中，rebuilt/runtime 直接引用约 35 个，约 34 个未进入生产 rebuilt 有效逻辑或只对 legacy 生效。

典型无效/误导键包括：

```text
p_adv_threshold
alert_window
alert_trigger_count
device
require_gpu
light_flow_enabled
a3b_display_alpha
p_adv_display_alpha
temporal_threshold
temporal_grid_size
temporal_persistence_frames
temporal_adaptive_baseline
lbp_radius
natural_exposure_suppression_enabled
strong_static_glare_ratio_threshold
...
```

同时 rebuilt 的实际行为大量依赖只存在于 Python 默认值的键：

```text
rebuilt_theta_adv
rebuilt_theta_blind
rebuilt_alert_hold_frames
rebuilt_blind_sustained_floor
rebuilt_scene_baseline_window
rebuilt_sustained_adv_seconds
rebuilt_sustained_adv_run_mult
...
```

只读探针确认：

```text
configured p_adv_threshold = 0.55
effective theta_adv = 0.65

configured alert_window/count = 5/3
effective max_history = 8
```

`FrameProcessor._module_a_effective_config()` 当前主要暴露 A3b 子集，仍没有完整暴露：

- theta_adv/theta_blind；
- blind/scene baseline；
- classifier loaded/fallback；
- flow backend/device；
- fixed-frame hold 折算秒数。

### profile detector FPS 失效

runtime 优先读取 `detector_process_fps_cap`。若 profile 只覆盖 `process_fps_cap`，继承的 detector 值仍为 60。

实际探针：

| profile | process_fps_cap | effective detector cap |
|---|---:|---:|
| default | 60 | 60 |
| balanced | 30 | 30 |
| cpu_smoke | 8 | 60 |
| desktop_debug_onnx | 12 | 60 |
| edge_fast | 8 | 60 |
| empty_smoke | 5 | 60 |
| low_power | 6 | 6 |

这会使 profile 性能声明和固定帧阈值的物理时间同时失真。

### Web feature options 覆盖 profile

即使调用方没有显式 feature options，`MonitorEngine.start()` 也会生成默认：

```text
static_image_enabled = true
a3b_sensitivity = balanced
```

balanced preset 又覆盖 static image interval。实测：

- `cpu_smoke`/`empty_smoke` profile 声称关闭 A3b；
- Web/engine 默认 options 后又被重新开启，interval 变为 4。

这需要明确为产品策略，不能让 profile 文本看起来已生效。

### 配置校验不足

当前 schema 可接受：

- 拼错的 `detector_impl`；
- `edge_min > edge_max`；
- `min_window_hits > window_size`；
- 概率阈值超出 0—1；
- trigger threshold 小于 observed threshold；
- alert count 大于 window。

尤其 `detector_impl` 只有精确 `legacy` 才选择 legacy，任何拼写错误都会静默进入 rebuilt。

### 结论

joint decision 的最大工程问题不是单个阈值高低，而是“配置文件、实际默认值、profile 覆盖、UI 覆盖和 status 可观测性”没有形成唯一可信的 effective config。

## 4.7 PPE 与 Module A 抑制关系

### P1：极低置信 held helmet 可压掉当前 bare head

生产 PPE 时序 mutex 当前检查几何重叠，但没有完整检查：

- helmet 最低置信度；
- track 是否 current/held；
- misses；
- 当前 confidence 是否仍可信。

最小复现：

```text
held helmet confidence = 0.01
misses = 1
current head = 高置信
```

输出：

```text
candidate = false
head_count = 0
helmet_count = 1
temporal_helmet_mutex_count = 1
```

代码行为已确认；真实漏报程度需多人员、摘帽、遮挡和交叉运动视频确认。

### P2：其他人的 helmet 会关闭 bare-head 快速通道

PPE fast candidate 要求：

```python
helmet_count == 0
```

没有判断 helmet 与 bare head 是否属于同一人员。混合场景中，一个人戴帽可能关闭另一个裸头人员的快速确认通道；普通 3/6 确认仍可能触发，因此主要是延迟风险。

### P2：类别语义存在三套映射

- PPE summary 使用子字符串 hints；
- ROI provider 使用精确 alias；
- tracker 使用另一套精确 alias。

例如：

```text
without_helmet
```

可被 summary 计为 head，但 tracker 不识别，无法进入时序提升。

当前默认模型只有 `[helmet, head]`，现机不触发该别名问题；自定义模型存在兼容风险。

### 结论

PPE 基本业务语义可运行，但与 A3b suppression、held track、人员归属和 alias 的边界不够严格。涉及 PPE 安全语义的修复必须单独验证，不能夹带在普通 runtime 修复中。

## 五、Runtime / integration 自洽性

## 5.1 P1：stop timeout 后仍可 restart 并复用同一 pipeline

`MonitorEngine.stop()` 即使 join 超时，也会：

- 清空线程引用；
- 可保留 pipeline cache；
- 返回。

随后 `start()`：

- 再次调用 stop；
- 清除同一个 `stop_event`；
- 从 cache 获取同一个 pipeline；
- cache hit 时直接调用 `pipeline.reset()`。

只读阻塞线程探针确认：

```text
stop_pending = ['stubborn-process']
reset_while_old_process_active = true
```

即旧 process 尚未退出时，同一 pipeline 已被 reset。run ID 只能阻止旧结果发布，不能阻止新旧线程并发访问 detector/backend。

风险包括：

- Module A/PPE 时序状态被跨线程清空；
- TensorRT/CUDA backend 推理中被 reset/close；
- 旧/new source 并发访问同一 detector；
- native backend 随机异常。

真实 TensorRT crash 概率待故障注入确认，但竞态本身已确认。

## 5.2 P1：runtime error 不协调停止 peer threads

`_set_error()` 当前主要：

- 写 status error；
- `running=false`；
- 清 JPEG。

但不统一：

- 设置 stop event；
- 关闭 PreviewBus；
- 关闭 DetectionBus；
- 协调其他线程退出。

探针结果：

```text
running = false
stop_event = false
preview_bus_closed = false
detection_bus_closed = false
```

因此检测/evidence/preview 任一线程失败后，状态可能显示停止，但其他线程继续占用摄像头、RTSP、CPU/GPU。

## 5.3 P1：camera/RTSP reconnect 没有新 source epoch

重连当前只 release/open capture，并清 capture 层 previous frame；没有：

- 增加 source epoch；
- 清 DetectionBus；
- reset processor/PPE/A3b；
- finalize/reset evidence。

重连后的新画面可能继承旧连接的：

- Module A temporal window；
- scene baseline；
- PPE hold；
- A3b soft state/cache；
- detection reuse；
- 活跃 evidence event。

`StreamSource/FrameEnvelope` 已定义 stream recovery/geometry change 语义，但当前 Web 默认抓流路径未使用该 adapter，形成 ownership 漂移。

## 5.4 P1：seek 后 PreviewBus 可重新发布旧 epoch

seek 会清 DetectionBus、overlay 和 latest JPEG，但不清 PreviewBus；preview loop 发布时没有传 packet epoch。

探针确认：

```text
published_stale_epoch = true
preview_seq = 1
status_epoch = 2
```

检测旧结果有 epoch 拒收，但预览可能短暂显示旧画面。

## 5.5 P1：evidence 可错误声明保存成功

### session 目录碰撞

EvidenceSession 目录只有秒级时间。同一 source/profile 在一秒内 restart 可复用同一目录，event sequence 又从 0 开始，可能覆盖文件并把 `events.jsonl` 混入同一 session。

探针：

```text
same_session_dir = true
```

### JPEG 写失败仍报告 saved

`cv2.imwrite()` 返回 false 时，当前代码仍增加 frame count，最终写：

```text
evidence_saved = true
```

探针：

```text
imwrite_false_reported_saved = true
frame_count = 1
file_exists = false
```

因此 evidence 的“已保存”不是可靠事实。

## 5.6 P2：evidence pre-frame 和 config snapshot 不完整

- `evidence_pre_frames` 被读取但没有真正的 pre-buffer/deque；
- 配置快照只写 runtime 子配置，不是 full resolved config；
- seek/epoch 变化不会 finalize 当前 evidence event；
- 旧/new 时间段可能混入同一事件。

## 5.7 P2：配置文件变化不会使 PipelineCache 失效

cache key 只含 config path 字符串，不含文件 mtime/hash/content。服务运行期间修改同一路径配置后，stop/start 可能继续 reset 并复用旧配置 pipeline。

## 6.3 高风险混杂

### 编码器/生成器与标签混杂

heldout：

| 标签 | 数量 | 编码 |
|---|---:|---|
| attack | 21 | 全部 FMP4 |
| clean | 27 | 25 个 H.264 + 2 个 HEVC |

这形成：

```text
attack ≈ synthetic + FMP4
clean ≈ real/stock + H.264/HEVC
```

可确认的是混杂存在；模型是否实际利用编码伪特征，需要同内容配对重编码实验确认。

### 攻击 onset 模板固定

21/21 heldout attack 均使用：

```text
attack_start_frame = 4
attack_ramp_frames = 6
```

没有：

- attack-from-start；
- 随机 onset；
- 不同 ramp；
- 不同持续时长。

场景 baseline 和持续窗口可能适应固定模板。

### 来源谱系不可靠

部分 manifest 标记为 `source=real` 的攻击实际由同一个 clean 视频程序生成，却被赋予不同 scene ID；其 clean 源还出现在 heldout。

这会破坏 Group CV 的 source lineage 独立性。修正分组后的指标下降幅度待重跑确认。

### final acceptance 正例不独立

既有 v3 正例中：

- 8/12 攻击源来自 train；
- 多条正例共用同一 clean prefix/tail；
- 10/12 攻击段与 clean context 不是同一场景。

攻击 onset 同时伴随场景切换。命中是否由攻击本体还是场景切换贡献，需同源连续对照确认。

### clean 评测被 240 帧截断

27 个 clean 中 20 个超过 240 帧，最长约 75.9 秒；攻击片基本全长，clean 多数只测前 4—10 秒。

这可能低估：

- 长流基线漂移；
- blind 长记忆；
- target 离场后误报；
- 状态污染；
- 缓存累积；
- 240 帧之后的持续升级。

## 七、legacy 与默认 rebuilt 的边界

legacy 不是当前默认生产 detector，但仍公开导出且配置声称可用，因此不能完全忽略。

### 确认的 legacy 问题

1. classifier 分支首帧会因缺少 `build_classifier_features` 导入而 `NameError`；
2. `fusion_backend=classifier/rule_or_classifier` 没有真正控制最终 `p_adv`；
3. classifier=0.99 在无 ROI anomaly 时也不能按注释独立触发；
4. A3a 受到 `no_target_fallback_max_exposure_ratio=0.0005` 的异常严格门控，该门也作用于有目标路径；
5. `p_adv` 与 target-anchored 最终证据不是同一组语义；
6. motion/light-flow/temporal 可能把同一帧对产生的相关信号当多个证据轴；
7. legacy 无法满足 rebuilt 当前的严格前驱注入契约；
8. 大量 tests 直接实例化 legacy，不能证明默认 rebuilt。

### 风险级别

- 对默认 rebuilt：主要是测试误导、维护和备用路径风险，列为 P2；
- 若产品仍宣称 legacy classifier 是可用生产配置，则 `NameError` 应按该可选生产路径的 P1 处理；
- 若不再支持，应移除或拒绝这些配置，而不是保留假生效入口。

该结论已在当前工作树再次核验：`process_pipeline.py` 未导入两个 classifier feature builder，Ruff 报 `F821`，且 `process_module_a.__globals__` 中两个符号均不存在；`detector.py` 自身导入同名符号不能改变另一个模块函数的全局命名空间。

- 新真实攻击召回；
- A3b 跨路径/摄像头/RTSP 召回；
- 长流 clean 误报；
- 跨设备、编码、FPS、分辨率泛化；
- 当前 `classmate_maskbd_v4` 的最终 Module A 效果。

4. `light_flow_enabled/device` 对 rebuilt 无效；
5. strict predecessor 只同步 gray/LBP，brightness 前驱不一致；
6. detection reuse 不检查真实 source frame/time gap；
7. stop timeout 后可复用仍在旧线程使用的同一 pipeline；
8. runtime error 不统一停止 peer threads；
9. camera/RTSP reconnect 不建立新 source epoch；
10. seek 后 PreviewBus 可发布旧 epoch；
11. evidence write 失败仍可报告保存成功，且同秒 restart 可复用 session 目录；
12. A4/RAFT 依赖 ignored/absolute assets，干净部署存在静默降级风险；
13. A4 classifier 与采集 detector、feature schema、flow backend 未绑定版本；
14. 极低置信 held helmet 可压制当前高置信 bare head；
15. 当前生产 detector 与仓库最终验收报告不一致；
16. 当前 heldout 因反复调参、重复媒体和无真实攻击正例，不能支持独立泛化结论。

## 9.3 P2：设计脆弱性或待实验确认风险

1. TargetAnchoredAnalyzer 死接线且 reset 漏清状态；
2. blind target-established 长记忆；
3. 固定帧与秒级窗口混用；
4. suppression fixed-frame hold 与 result lease 的物理时间差异及低 FPS 风险；
5. square resize 导致宽高比依赖；
6. `prev_timestamp` 应表示 source cadence 还是 process cadence 尚未统一；
7. 配置 schema 不保护策略不变量；
8. PipelineCache 不按配置内容失效；
9. Web 默认 feature options 覆盖 profile；
10. evidence pre-frame 未实现、配置快照不完整；
11. PPE 多人员 fast candidate 使用全局 helmet count；
12. PPE summary/ROI/tracker alias 不一致；
13. legacy classifier 崩溃/假接线；
14. legacy A2 LBP 距离、noise suppression 和 fusion 门不一致；
15. legacy A3a 过曝硬门和 `p_adv` 证据语义不一致；
16. legacy natural exposure/strong static glare 契约冲突；
17. 编码器混杂、固定 onset、scene lineage 错误的实际指标贡献待配对实验确认。

## 9.4 P3

1. 部分 status 只在首个检测结果后才具有完整 effective values；
2. `attack_state_remaining` 仍固定为 0；
3. 一项 Web A3b 测试依赖工作目录；
4. manifest/data-spec 和训练报告信息过期或不完整；
5. 诊断工具存在复刻生产判据、可能随实现漂移的情况；
6. 多个配置键和状态字段只存在文本或初始值，没有真实运行消费者。

## 十一、最小独立验证矩阵

### 11.1 重新定义数据用途

- 当前 train：保留 train；
- 当前 heldout：正式降级为 dev/calibration；
- 现有两个 A3b 视频和其裁剪/转码：只作 train/dev；
- 新建一次性锁定的 `test_locked_v1`；
- 按原始 capture/source lineage 分组；
- split 前执行 SHA-256 和来源谱系去重；
- 任何依据 locked test 继续调参的行为，都会令该集合立即降级为 dev。

### 11.2 最小正例

以下数量仅作为工程启动下限，不是具有既定统计功效或置信区间的充分样本量。建议至少 20 个独立 capture lineage：

| 类别 | 最少数量 | 要求 |
|---|---:|---|
| 对抗贴片/高纹理扰动 | 4 | 至少 2 个真实物理、2 个受控代理 |
| 闪烁/强光 | 4 | 局部/全局、不同自然光 |
| 动态遮挡/退化 | 4 | 局部、移动、目标短时消失 |
| A3b replay | 4 | 至少 2 个显示设备 × 2 个拍摄条件 |
| A3b interference | 4 | 不同屏幕、角度、距离、光照 |

### 11.3 最小负例

建议至少 16 个完整视频：

- 近景高纹理、反光背心、格纹、门框；
- 焊接、自动曝光、进出明暗区；
- 多人交叉、正常遮挡、摘戴安全帽、离场；
- 手持运动、RTSP 丢帧、seek、断流重连。

全部运行完整时长，不使用固定 240 帧截断。

### 11.4 配对鲁棒性

从 locked 集中选择 anchor clips，做控制变量配对：

- 路径：攻击关键词目录/中性目录/随机目录；
- 编码：H.264/HEVC/FMP4；
- FPS：24/30/60；
- 分辨率：720p/1080p；
- 链路：file/camera/RTSP latest-only；
- 方向：横屏/竖屏或保持宽高比的 letterbox 对照。

同一视频字节的路径变化必须得到相同业务结论。

### 11.5 最终门禁

至少检查：

- 正例 event hit；
- 完整负例 event FP；
- 首次确认不超过预先冻结的时延；
- 攻击结束后的 linger 上限；
- source path 不变性；
- A3b 独立 result 去重；
- backend/classifier/flow 有效值、fallback 和资产哈希；
- restart/reconnect/seek 无旧状态和旧画面；
- evidence 文件真实存在且可读取；
- 按 capture lineage 聚合，不把同源派生片当独立样本。

## 十二、建议处理顺序

### 第一阶段：纯 runtime/交付安全修复，不做算法调参

1. stop pending 时拒绝 restart，禁止 reset/close/reuse 活线程持有的 pipeline；
2. 建立统一 error shutdown 协议；
3. reconnect 增加 source epoch，seek 清/重建 PreviewBus；
4. evidence session 唯一化并检查真实写入结果；
5. 配置 cache 使用内容 digest/mtime；
6. 资产加载和 fallback 进入 status，干净部署能力必须显式；
7. 补充完整 effective config。

### 第二阶段：时序接线修复

1. 建立显式 temporal context：
   - previous gray/LBP/brightness；
   - source frame index/time；
   - current frame index/time；
2. detection reuse 同时约束 processed/source/time gap；
3. TargetAnchored 先只做 reset 和 diagnostics，不直接接入判定；
4. 相关改动完成后跑当前 dev 回归和目标视频，确认没有非预期行为变化。

### 第三阶段：独立 behavior-tuning

必须单独立项并冻结数据门：

1. A3b result sequence 去重；
2. 移除或重定义 source keyword 检测门；
3. 统一 suppression policy、freshness、lease 和 hold；
4. 固定帧窗口改为物理时间或明确 profile 不变量；
5. 裁定 natural exposure/glare；
6. 裁定 blind 长记忆；
7. 裁定 TargetAnchored 是否参与 joint decision；
8. PPE held helmet/person attribution 语义。

### 第四阶段：重新建立可信验收

1. 当前 heldout 改名为 dev/calibration；
2. 清除媒体重复和 source lineage 泄漏；
3. 为当前生产 detector、A4 classifier、RAFT、代码和配置建立 hash 绑定；
4. 采集独立真实 A3b/物理攻击正例；
5. 只在 dev 上调参；
6. locked test 一次性验收；
7. 将 delay、linger、完整 clean 时长、路径不变性和 runtime lifecycle 纳入自动 fail gate。

## 十三、最终判断

Module A 目前不是“整体设计错误”，但也没有达到“各模块逻辑闭环且已证明泛化”的状态。

可保留的核心设计是：

- latest-only 实时管线；
- preview/detection 解耦；
- A1/A2/A3 多物理证据方向；
- A3b 后台 worker 的 generation/publication 隔离和资源上限；
- 统一公共结果契约；
- source epoch 对检测结果的拒收思路。

需要明确整改的核心问题是：

- A3b 缓存结果不具备独立投票语义；
- runtime soft-trigger 的 source path 直接进入补充触发决策；
- A3b policy、freshness 和 PPE suppression 不一致；
- flow/device/profile/effective config 不一致；
- restart/reconnect/seek/evidence 生命周期未完全闭环；
- A4 资产、训练 detector 和 runtime feature schema未形成可复现绑定；
- 当前数据和 heldout 不能支持独立泛化声明。

因此本轮最终结论为：

> **默认生产架构可运行，A3b 工程生命周期已有明显硬化；但 Module A 仍存在多项 P1/P2，且过拟合、标签泄漏和评测失效风险为高。当前不能宣称“功能设计没有问题”或“已证明真实场景泛化”。**
