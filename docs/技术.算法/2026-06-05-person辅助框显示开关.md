# person 辅助框显示开关

## 问题背景

用户希望增加一个取消绘制 person 的选项，使 person 继续作为 PPE 检测的辅助上下文使用，但不再强制显示为画面中的人员框。

## 当前判断

person 不应从检测、时序融合、误报抑制或状态计数中移除。它在当前三类 PPE 链路中承担人员上下文、head/helmet 空间约束和弱证据抑制作用。此次需求属于显示层控制，不属于类别语义或模型阈值调参。

## 代码链路依据

- 前端画面叠加配置从 `web/static/index.html` 的 `drawingOptions()` 发往 `/api/display-options`。
- Web API 将配置传给 `MonitorEngine.update_display_options()`。
- 后端预览帧通过 `defense.visualization.render_preview()` 绘制 PPE boxes。
- `FrameProcessor` 和 `build_overlay_record()` 仍保留完整 `ppe_tracks` 与 person/head/helmet 计数字段。

## 影响范围

新增 `show_person_boxes` 只影响预览渲染是否绘制 person 框，不影响：

- detector 输出类别；
- PPE 后处理、时序跟踪与误报抑制；
- `ppe_tracks` 状态字段；
- person/head/helmet 统计；
- A3b 或模型安全链路。

## 结论

采用显示选项过滤 person 绘制是低风险方案：person 继续作为辅助信号参与工程逻辑，画面可按用户需要只突出 head/helmet 等业务框。

## 后续建议

如果后续前端恢复 canvas 端直接绘制 overlay，应复用同一 `show_person_boxes` 语义，避免 MJPEG 后端渲染和浏览器端 overlay 出现不一致。

## 2026-06-05 暂停帧观察

用户在 Web 端暂停帧 406 反馈“为什么能丢失”。当前状态接口显示：

- `display_options.show_person_boxes=false`
- `ppe_person_count=9`
- `ppe_raw_person_count=9`
- `ppe_inferred_person_count=9`
- `ppe_helmet_count=1`
- `ppe_head_count=5`
- `ppe_tracks=15`
- `raw_boxes_count=17`

结论：该帧不是 person 检测链路丢失，而是显示层按配置隐藏了 person 框；person 仍在状态、时序轨迹和 PPE 业务判断中保留。用户肉眼看到的剩余框主要是 head/helmet，因此容易误以为人员检测丢失。后续 UI 可以考虑在关闭 person 绘制时显示“人员辅助框已隐藏”的轻量提示，或在安全帽 HUD 中区分“辅助 person 已参与但未绘制”。

待实验确认：个别人员没有 head/helmet 框时，需要进一步对比原始 detector 输出与 PPE 后处理抑制结果，确认是模型未输出、小目标低置信度、时序保持窗口结束，还是后处理过滤导致。

## 断框问题与参数边界

断框问题不能简单归因为单个参数。当前链路至少有三层会影响画面连续性：

- 原始检测层：`confidence`、`candidate_confidence` 和 NMS IoU 决定这一帧是否有原始 head/helmet/person 候选。
- PPE 显示跟踪层：`PPEDisplayTracker` 的 `hold_frames`、`small_hold_frames`、`iou_match_threshold`、`max_missed_ms`、`smooth_alpha`、`weak_promotion_hits` 决定短时漏检后是否继续保持、是否平滑、是否把弱证据晋升为可显示轨迹。
- 预览 overlay 层：文件实时预览为了避免旧框拖尾，会把 `overlay_match_window_ms`、`overlay_hold_ms`、`overlay_interpolate_ms`、`overlay_max_age_ms` 压到较短桥接窗口，并且对未匹配轨迹更保守。

当前判断：如果只是偶发 1-2 帧断框，优先属于时序保持/overlay 桥接策略问题，可以通过参数缓解；如果某个目标连续没有 head/helmet，则更可能是原始检测证据不足或 PPE 后处理抑制，不能只靠加保持时间解决。

参数调大存在副作用：保持窗口过长会减少断框，但会引入拖框、旧框滞留和错人绑定；匹配阈值过宽会降低断框，但在多人重叠场景更容易 2 人共用一个框或身份串轨。因此后续调参应按“原始输出诊断 -> 短时保持调整 -> 多人重叠回归”的顺序进行。
