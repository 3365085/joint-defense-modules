# 2026-05-31 B模块净化与三类PPE模型类别边界

## 问题背景

当前运行时检测链路已经按 PPE 场景适配 `person`、`helmet`、`head` 三类输出，但 B 模块模型安全净化与外部攻击评估默认只把 `helmet` 作为攻击/净化目标类。用户指出：原始模型训练包含 `person` 类，如果净化后模型只稳定输出 `head`、`helmet`，可能丢失人员上下文，导致安全帽业务判断出现严重误报。

## 代码链路依据

- 运行时配置 `model/configs/module_a_runtime.yaml` 中 `module_a.roi_target_labels` 已包含 `[person, helmet, head]`，说明 Module A ROI 链路接受三类目标。
- 同一配置的 `model_security.external_eval_target_classes` 当前是 `[helmet]`，说明 B 模块外部评估目标默认只看安全帽相关目标。
- `model/src/defense/model_security/scanner.py` 的 `_class_name_map()` 在未显式提供类别名时，对 YOLO/Ultralytics 默认映射为 `{0: helmet, 1: head, 2: person}`。
- `scanner.py` 的 `_external_target_resolution()` 会主动忽略名称为 `person` 的目标类；即使配置中写入 `person`，默认也不会把它作为 B 模块攻击目标。
- `model/src/defense/model_security/purifier.py` 的 `_autodetox_target_classes()` 同样过滤掉 `person`，AutoDetox plan 只围绕非 person 的 PPE 目标类生成。
- `model/src/defense/module_a/ppe_postprocess.py` 已识别 `PERSON_HINTS`、`HELMET_HINTS`、`BARE_HEAD_HINTS`，并在 `summarize_ppe_from_detections()` 中区分 `has_person_class`、`person_context_available`、`head_helmet_only`。

## 当前判断

当前代码不是完全“不支持三类模型”；更准确地说：

1. 运行时检测与 PPE 后处理支持三类输出，并且 person 会作为上下文减少孤立 head/helmet 误报。
2. B 模块扫描、外部攻击评估、AutoDetox 净化目标默认不把 person 当作攻击目标；这是业务风险控制设计，避免把大框 person 当作被净化攻击目标，破坏 PPE 小目标检测。
3. 如果净化候选来自只含 head/helmet 的外部 packaged purified 模型，或 clean anchor 是二类模型，则净化后实际可用类别会退化，person 上下文丢失，误报风险显著上升。
4. Weight-soup 类候选以原始 source 模型为模板保存，形状不匹配的 tensor 会保留 source 权重，因此在 source 是三类模型时更可能保留三类输出头；但这仍需要 full scan 与实际视频验收确认。

## 支持三类输出的收益

- 降低误报：person 框可作为 helmet/head 的人体上下文，抑制孤立小框、边缘噪声、背景误检。
- 保持业务语义：人员数量、缺帽人数、裸头与安全帽的上下文关系更完整。
- 提高跨模型兼容性：既可接入二类 PPE 模型，也可接入三类 person/head/helmet 模型。
- 改善证据解释：告警可说明是“人员上下文下的裸头”还是“无 person 上下文的弱证据”。

## 建议方案

短期建议：不要把三类原始模型净化成二类输出模型。保留当前 B 模块攻击目标只评估 `helmet/head`，但净化候选必须保持 source 模型的三类输出结构；对 packaged 二类净化模型加类别兼容性阻断或强提醒。

中期建议：增加“类别保真”验收，而不是把 person 也纳入攻击目标。也就是 full scan 后额外检查净化模型的 class map、检测输出是否仍包含 `person`，并在代表性视频上验证 `person_context_available` 没有退化为 `head_helmet_only`。

长期建议：支持三类模型作为运行时能力，但 B 模块 target policy 仍默认排除 `person`；如确需保护 person 类，应作为单独的源完整性/业务保真指标，而不是与 helmet/head 的 ODA/OGA 攻击目标混在一起。

## 结论

只检测 `head` 和 `helmet` 的方案改动少，但会放弃 person 上下文，适合纯二类模型；对已经训练了 `person` 的模型，不建议降级为二类输出。更稳妥的路线是：运行时保留三类输出，B 模块仍重点净化/评估 helmet/head 攻击面，同时新增类别保真和 person 上下文验收，防止净化候选破坏原模型业务能力。
