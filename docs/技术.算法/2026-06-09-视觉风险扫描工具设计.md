# 2026-06-09 视觉风险扫描工具设计

## 问题背景

round2 视觉验收暴露出一个流程问题：把大量 4K PNG 直接读入对话线程会触发 413，并且人工逐帧盲扫效率低。项目需要一个本地 OpenCV 工具，先把结果视频和 overlay 证据压缩成风险索引，再只查看关键帧。

## 当前判断

工具内部优先直接读取结果视频，而不是把导出的 PNG 当主输入。

原因：

1. 若 PNG 来自结果视频解码，它的质量上限仍然是结果视频本身。
2. OpenCV 直接读视频能得到同一份解码画面，减少一次导出和大量文件读写。
3. 大量 PNG 会污染对话历史并触发 413，不适合作为 agent 主输入。
4. 最终验收证据仍可导出无损 PNG，但应由风险索引决定导出哪些帧段。

## 代码链路依据

新增工具：`model/src/defense/diagnostics/visual_risk_scan.py`

它读取：

1. `result_overlay_*.mp4`：用 OpenCV 验证视频可解码、帧数、分辨率，并可抽样统计彩色框组件。
2. `overlay_*.json`：读取 `ppe_tracks`、`track_id`、`label`、`box`、`display_box_source`、`misses`、`fresh_detection`。

它输出：

1. `visual_risk_report.json`
2. `visual_risk_report.md`
3. `risk_segments.csv`
4. 可选 `risk_frames/` 临时 PNG。

## 风险类型

当前扫描以下风险：

1. `label_switch`：同一 track 在 `head` 和 `helmet` 之间切换。
2. `track_label_instability`：同一 track 在短时间内同时出现 `head/helmet`。
3. `held_track`：检测框进入 `held_static` 或 `held_extrapolated_one_frame` 连续区间。
4. `box_growth`：同一 track 框尺寸在短窗口内明显膨胀。
5. `large_head_box`：head 框相对画面过大。
6. `edge_touch`：head/helmet 框贴近画面边界。
7. `count_change`：相邻帧 PPE 可见数量突变。
8. `video_overlay_count_mismatch`：视频帧数与 overlay 记录数量不匹配。

## 影响范围

该工具不改变生产检测路径，不增加主检测 GPU 推理，仅用于诊断和验收前筛查。

它会改变视觉验收流程：

1. 先跑本地风险扫描。
2. 根据 `review_frames` 导出少量关键帧。
3. agent 或用户只查看风险帧段。
4. 若报告 verdict 为 `fail`，不得宣称视觉通过。

## 结论

OpenCV 直接检测结果视频更适合作为工具主流程；PNG 应作为最终验收证据和少量风险帧复核材料，而不是工具主输入。

本工具已经能覆盖 round2 中发现的核心问题类型：右侧小白帽 `head/helmet` 闪变、右侧前景大框膨胀、held 残留和数量突变。

## 后续建议

1. 后续 round3 先运行 `visual_risk_scan`，再导出人工复核帧。
2. 默认不要导出风险帧，只有需要人工复核时才加 `--export-risk-frames`。
3. 风险帧目录属于临时视觉产物，用户未要求保留时应及时清理。
