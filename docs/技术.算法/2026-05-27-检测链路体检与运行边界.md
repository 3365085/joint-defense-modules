# 检测链路体检与运行边界记录

## 背景

用户反馈检测链路在特定模型和视频下稳定出现检测中断、停止后最后一帧滞留、部分攻击视频无框等问题。体检对象覆盖 A 模块实时检测、PPE 后处理、A3b 状态、Web 启动/状态/预览接口，以及 B 模块准入后的运行模型链路。

关键复现样本：

- 视频：`model/run_model/oga_visible_patch/attack_test_videos/oga_visible_patch_attack_test.mp4`
- 模型：`model/run_model/oga_visible_patch/oga_visible_patch_poisoned.pt`
- 对照：`model/run_model/oda_sig_lowfreq/attack_test_videos/oda_sig_lowfreq_attack_test.mp4`
- 对照：`model/run_model/oga_semantic_vest/attack_test_videos/oga_semantic_vest_attack_test.mp4`

## 当前判断

1. 可见贴片样本的检测中断根因不是模型零输出，而是实时链路曾经额外触发 PPE ROI 复检，首次 crop 推理耗时约 1.4s，导致 overlay 空窗。当前 `FrameProcessor.process()` 已将 `ppe_roi_redetect_enabled` 默认关闭在实时路径，只有显式启用且非实时处理才允许复检。
2. 文件结束后最后一帧滞留属于生命周期状态问题。当前 `MonitorEngine.wait_latest_jpeg()`、`PreviewBus.wait_for_frame()`、`fastapi_app.mjpeg_generator()` 和前端 `normalizeLifecycleStatus()` 已按 `source_ended` 清理预览。
3. seek 或 stop 后仍可能出现迟到检测结果污染状态。当前 `MonitorEngine._after_processed()` 与 `_publish_preview()` 已按 `source_epoch` 丢弃旧 epoch 结果，避免状态面板或预览回跳。
4. `/api/start` 曾存在异步采集失败后仍返回 `ok=true` 的风险。当前 `defense.web.fastapi_app.start()` 会检查 `wait_ready_for_preview()` 返回的 `error` 和 `ready_for_preview`，未就绪则返回 `runtime_start_failed` 或 `runtime_start_timeout`。
5. ODA 类攻击视频全程无框属于模型攻击语义生效的可能性更高：`oda_sig_lowfreq_poisoned_attack` 在 overlay 记录中 raw/head/helmet 全为空，PPE 原因是 `no_head_or_helmet_evidence_detected`，不是显示层丢框。
6. OGA semantic 局部无框更像是模型输出弱 + PPE 抑制共同作用：部分帧 raw 有 helmet，但 `PPEDisplayTracker` 和 PPE 后处理将其归为弱证据，最终未渲染。

## 代码链路依据

- Web 启动入口：`src/defense/web/fastapi_app.py` 的 `start()`，先走 B 准入，再调用 `MonitorEngine.start()`。

- 采集/检测/预览线程：`MonitorEngine._backend_capture_loop()`、`_backend_process_loop()`、`_preview_render_loop()`。
- 旧 epoch 防护：`MonitorEngine._after_processed()` 与 `_publish_preview()`。
- PPE 主处理：`src/defense/runtime/frame_processor.py` 的 `FrameProcessor.process()`。
- PPE 业务状态：`src/defense/runtime/ppe_business.py` 的 `evaluate_ppe_business()`。
- PPE 类语义：`src/defense/module_a/ppe_postprocess.py` 的 `summarize_ppe_from_detections()`。

## 影响范围

- 正向修复：减少 seek/stop/source_ended 后旧结果污染；减少前端“启动成功但一直初始化”的误导；避免实时路径额外 GPU 推理导致卡顿。
- 不改变：模型权重、类别语义、PPE 阈值、A3b 策略口径。
- 已知边界：攻击视频的验收不能只用“是否有框”判断。ODA 应看 helmet disappearance，OGA 应看 head-only 被误报 helmet，不同攻击族需要不同指标。

## 后续建议

1. 为每个攻击族建立四组对照：`poisoned + attack`、`poisoned + clean/source`、`purified + attack`、`purified + source`。
2. Overlay 记录建议增加 PPE suppression 明细，例如 `helmet_fp_suppression` 和 `class_counts`，便于区分模型零输出、后处理抑制和显示层问题。
3. 如果 `model/run_model/.../attack_test_videos` 也要用于 A3b 攻击验收，需要重新审视 A3b 的路径关键字门控，当前它更偏向真实异常视频目录，不会对这些 run_model 路径自动确认报警。
4. OGA invisible SIG 等频域攻击经过 MP4 编码可能衰减，最终判断应以 canonical image-set 或无损帧序列作为补充依据。
