# 证据预览与事件索引优化记录

日期：2026-05-28

## 问题背景

Web 证据预览页能展示关键帧，但视频控件显示黑屏或 0:00。用户同时提出证据不应只靠文件夹散落管理，后续需要更像数据库一样支持检索、复现和追溯。

## 代码链路依据

- 证据生成入口：`model/src/defense/runtime/evidence.py` 的 `EvidenceSession.update()`、`EvidenceSession._finalize()`。
- 事件帧写入：`EvidenceSession._write_event_frame()`。
- 视频片段写入：`EvidenceSession._write_event_clip()`。
- 证据查询接口：`model/src/defense/web/fastapi_app.py` 的 `/api/evidence/events`、`/api/evidence/events/{event_key}`、`/api/evidence/file`。
- 证据页面：`model/src/defense/web/static/evidence.html`。

## 当前判断

关键帧已经正确保存，问题集中在事件视频片段编码。原实现使用 OpenCV `mp4v` 写 `clip.mp4`，该编码在浏览器端不稳定，容易出现视频标签加载但无法播放。优化后改为使用 `imageio_ffmpeg` 提供的 ffmpeg 生成 H.264、`yuv420p`、`faststart` MP4，保证浏览器优先可播放。

同时新增轻量 SQLite 索引 `evidence_index.sqlite3`，每个事件在保留 `event.json` 的同时写入事件索引。索引记录事件 key、通道、输入源、运行配置、帧范围、分数、原因、代表帧和视频路径，便于后续做事件管理、筛选、回放和复现实验。

## 影响范围

- 不改变检测算法和告警判定。
- 不改变关键帧落盘路径。
- 新增 SQLite 只是事件管理索引，`event.json` 仍是单事件可读源。
- 打开旧事件时，如果发现旧视频不是浏览器可播 H.264，会基于已有关键帧尝试自动重建 `clip.mp4`。

## 后续建议

1. 将证据中心扩展为完整事件管理页，支持按通道、模型、视频源、时间、告警类型筛选。
2. 给每次运行增加 run 级索引表，记录模型 hash、输入视频 hash、配置快照和运行参数。
3. 复现实验入口应基于数据库索引重放：模型路径、输入源、事件帧范围、关键配置一起传给启动接口。
4. 证据库损坏时应保留文件夹扫描降级能力，避免单点索引损坏导致历史证据不可见。
