# 模块A技术路线与实现说明

更新时间：2026-05-14

## 1. 模块定位

模块A是面向视频流的物理扰动与翻拍/假目标检测模块，目标是在本机摄像头、MP4文件、RTSP/HTTP视频流等输入上实时判断画面是否存在物理对抗扰动、视频源异常、翻拍屏幕、纸面照片或其他假目标媒介。它不是单一分类器，而是一个多分支、多证据、带状态记忆的联合防御管线。

模块A当前主要输出三类状态：

- 物理对抗扰动 `p_adv`：面向当前画面被强光、模糊、纹理、光流、轨迹异常等物理扰动攻击的检测。
- 翻拍/假目标检测 `A3b / p_media`：面向屏幕翻拍、视频中出现屏幕、纸面假目标、近距离遮边后的假目标延续。
- 视频源真实性 `p_synth`：面向视频源伪造/合成风险的独立通道，当前在主告警中会受物理扰动状态抑制，避免重复报警。

## 2. 输入与工程形态

模块A以 `VideoDefensePipeline` 作为视频处理入口，负责帧读取、检测器后端调用、ROI规范化、模块A特征计算、告警状态输出与Web展示。Web控制台位于 `tools/module_a_monitor_app.py`，支持：

- MP4路径输入。
- RTSP/HTTP视频流。
- 本机摄像头编号。
- TensorRT/GPU后端。
- YOLOv5/YOLOv8类名归一化。
- 自定义模型路径记忆。
- 模块A分支状态、帧号、置信度、原因、证据文件展示。

## 3. 总体检测流程

模块A处理一帧视频时，大体经过以下阶段：

1. 视频帧进入 `VideoDefensePipeline`。
2. 检测器后端产生目标框，例如 person、helmet、head 等。
3. ROI Provider 将 YOLOv5/YOLOv8 等不同模型输出的类别名映射为统一类别。
4. `ModuleADetector` 计算多组时序和空间特征。
5. A1/A2/A3/A3b/A4等子特征进入规则融合。
6. `AlertState` 将单帧可疑结果转为连续帧确认告警。
7. Web端展示 `p_adv`、A3b、`p_synth` 三张分支卡片。

整体可以理解为：

```text
视频帧 → 目标检测/ROI → 特征提取 → A3b媒介检测 → 规则融合 → 连续帧确认 → Web告警/证据
```

## 4. 物理扰动检测技术路线

物理扰动检测关注的是“真实摄像头画面是否被物理方式干扰”。它主要不是判断画面里有没有屏幕，而是判断当前画面的成像、纹理、运动和目标轨迹是否异常。

核心特征包括：

- 过曝/强光：检测大面积亮度异常、局部高亮区域、强光导致的画面失真。
- 时序纹理变化：检测相邻帧纹理统计突然变化。
- 局部时序纹理变化：避免全局轻微变化被误判，关注局部区域突变。
- 运动伪影：检测摄像头或画面中异常运动导致的伪影。
- 轻量光流：用于捕获局部运动方向和幅度异常。
- 清晰度退化：检测模糊、遮挡、失焦等。
- 目标轨迹一致性：跟踪目标框位置、尺度、匹配关系，发现突然消失、跳变或不合理漂移。

这些特征进入规则融合后形成 `p_adv`，再由连续帧状态机确认告警。这样做的原因是单帧异常可能来自正常手抖、压缩噪声、自动曝光或目标快速移动，不能直接作为确认攻击。

## 5. A3b翻拍/假目标检测技术路线

A3b关注的是“画面中是否出现被摄像头二次拍摄的媒介”，例如手机屏幕、电脑屏幕、纸面照片、视频中播放另一段视频等。它与 `p_adv` 的核心区别是：

- `p_adv` 判断画面是否被物理扰动。
- A3b 判断画面中是否存在屏幕/纸面/假目标媒介。

A3b当前采用多层级级联方案：

### 5.1 L0边缘候选提取

A3+路径会在灰度帧上进行边缘检测、轮廓提取、多边形近似和几何过滤，找到疑似矩形媒介区域。候选会记录：

- `bbox`：候选框。
- `area_ratio`：面积比例。
- `aspect_ratio`：长宽比。
- `rectangularity`：矩形程度。
- `edge_score`：边缘强度。
- `bg_suppressed`：是否被背景/纯色留边抑制。

### 5.2 纯色留边/比例填充抑制

用户指出的视频存在四周黑边，且未来可能出现白色、灰色或其他纯色留边。针对这种情况，候选提取阶段新增了“纯色留边抑制”：

- 不只检测黑色，而是检测任意低纹理、颜色一致的留边。
- 要求上/下或左/右形成成对纯色边，并且至少一侧贴近画面边界。
- 符合条件的候选会被标记为 `bg_suppressed`，避免把视频比例填充边界当成真实屏幕边框。
- 单侧纯色区域只给弱分，不直接抑制，避免误伤真实屏幕、墙面或地面边缘。

该逻辑位于 `defense/module_a/features/a3/static_media/candidate_extraction.py`。

### 5.3 L1候选轨迹管理

候选不是单帧独立判断，而是进入候选轨迹管理器。轨迹管理会维护：

- 候选是否连续出现。
- 候选框IoU匹配。
- 命中次数与丢失次数。
- 背景边缘抑制状态。
- 与YOLO目标的关系。

这样可以降低一帧噪声、建筑边缘、栏杆、窗框带来的误报。

### 5.4 L2平面/重放验证

A3b会进一步计算平面一致性、单应性残差、区域内外运动差异等指标：

- `plane_score`：候选是否像一个稳定平面。
- `warp_residual`：平面配准残差。
- `flow_gap`：候选内部运动与外部背景运动差异。
- `target_iou / target_proximity / target_area_ratio`：候选与人/头/安全帽等目标的关系。

屏幕翻拍或播放视频时，常见现象是外层平面稳定，但内部内容运动和外部真实世界不一致，因此 `flow_gap` 和 `warp_residual` 是关键证据。

### 5.5 L3 p_media评分

A3b将候选轨迹、平面、边缘、流差、YOLO上下文综合为 `p_media`。注意：

- `p_media` 是观察值，不等于确认告警概率。
- Web上主分显示确认/候选报警概率，辅助行显示 `p_media`。
- 正常视频里 `p_media` 可以有值，因为建筑、栏杆、柱子、门框也可能像矩形边缘，但没有进入候选/确认链路就不报警。

## 6. A3b状态机革新

早期A3b更接近“一轮轮滑窗投票”，一次确认后如果后续边框消失或变形，状态容易断掉。针对 `VID20260512200916.mp4` 这类视频，模块新增了遮边延续状态。

当前A3b状态链路为：

```text
正常 → 候选观察 → 快速确认/持续确认 → 遮边延续 → 解除
```

### 6.1 快速确认

快速确认用于边框明显、重放证据强、短窗口稳定的情况。它要求：

- `p_media` 足够高。
- 候选框面积合理。
- 候选轨迹短时稳定。
- 存在水平边缘接触或目标相关证据。
- 满足短窗口票数。

### 6.2 持续确认

持续确认用于更保守的长窗口判断。它要求：

- `p_media` 达到阈值。
- 候选不是明显目标本体。
- 候选面积不超过上限。
- 有局部时序/模糊/重放证据。
- 候选框中心和面积变化在可接受范围内。

### 6.3 遮边延续

用户提供的视频先广角拍到屏幕边框，随后镜头靠近，边框被遮住或变得不规则。如果仍然要求“当前帧必须看到完整边框”，后半段会失效。

因此新增 `p_media_occlusion_state`：

- 一旦A3b已经确认，就锁定媒介状态。
- 后续即使边框短暂消失、贴边、变形、抖动，也保持一段长时风险。
- 如果后续重新出现不规则边框或贴边大框，会刷新保持时间。
- 如果长时间没有任何媒介迹象，才解除。

这个设计的核心原则是：只有“已经确认过A3b”之后才允许遮边延续，正常视频不会因为纯贴边、柱子、门框、人靠近而直接进入遮边保持。

## 7. Web端展示逻辑

Web端将A3b从“单一置信度”改为“主分+观察值”：

- 正常：主分显示 `0.000`，辅助显示 `p_media`、快速票数、持续票数。
- 候选：主分显示候选报警概率，说明正在等待连续确认。
- 确认：主分显示确认报警概率，标注快速确认/持续确认/遮边延续来源。
- 最近原因使用中文标签，例如“目标轨迹异常”“遮边延续”“持续确认”。

这样可以解释为什么正常视频中 `p_media` 有值但不报警，也能让用户看到A3b确实在观察。

## 8. 当前验证结果

最近一次回归包括：

- 正常手机拍摄视频3条：A3b警报为0，确认警报为0。
- 干扰视频3条：均可检出。
- `VID20260512200916.mp4`：加入纯色留边抑制后，首次确认约第49帧，后续通过遮边延续保持到结尾。
- 自动测试：`61 passed`，跳过的是已知陈旧样本报告回归测试。

## 9. 性能现状与后续优化

当前A3b加入了更多状态和诊断字段，准确性提升明显，但性能后续仍需要打磨。重点方向：

- 降低L0候选提取频率或做动态调度。
- 将更多OpenCV CPU逻辑迁移到Torch/NPU友好路径。
- 对L2单应性验证做Top-K缓存和跳帧调度。
- Web端降低MJPEG与状态轮询压力。
- 对A3b事件态做轻量序列缓存，避免重复计算。
