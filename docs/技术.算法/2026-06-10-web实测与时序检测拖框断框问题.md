# Web 实测与时序检测拖框断框问题

## 问题背景

用户反馈：人物做成时序检测后，最大问题可能是拖框；实际 Web 测试效果不如生成视频，断框问题仍未修复，并追问两个视频是否来自同一份代码。

## 当前判断

人物、头部、安全帽这类目标在视频中运动时，时序检测的核心矛盾是“防断框”和“防拖框”相互牵制。检测偶发漏帧时，tracking / overlay hold 可以短时间沿用上一帧框，视觉上会减少闪烁和断框；但如果目标、相机或播放时钟继续移动，旧框没有新的检测结果校正，就会表现为拖框、滞后或框贴错位置。

因此，拖框不是时序检测的偶然副作用，而是当前策略需要明确控制的主要风险。只增加持框时长不能真正修复断框，只会把一部分断框转换成旧框滞留。

## 代码链路依据

- `model/configs/module_a_runtime.yaml` 中运行时启用了 `detector_queue_policy: latest_only`，并配置了 `overlay_match_window_ms: 180`、`overlay_hold_ms: 550`、`overlay_interpolate_ms: 400`、`overlay_max_age_ms: 950`。
- 同一配置中 PPE tracking 启用了 `hold_last_box: true`、`show_held_boxes: true`、`smooth_alpha: 0.82`、`max_missed_frames: 10`、`max_missed_ms: 700`。
- `model/src/defense/runtime/runner.py` 的 `_select_preview_overlay()` 会按视频时间选择 nearest / interpolated / held overlay；文件实时预览下会把 hold 和 match 限制到 bridge 窗口，但仍存在 held overlay。
- `model/src/defense/module_a/postprocess/ppe_tracking.py` 中渲染 gate 对 missed track 有 held box、hold eligibility、render cap 等判断，说明检测结果和显示结果之间存在时序补偿层。

## 为什么 Web 实测可能不如生成视频

生成视频和 Web 实测不一定是同一条显示链路。

如果生成的是 YOLO reference 视频，它只用于判断模型本体能否检出目标，默认不经过项目实时检测链路、latest-only backpressure、Web preview 时间匹配、浏览器 JPEG 预览和 overlay hold/interpolate。因此 reference 画面更稳定，并不代表 Web 显示也已经稳定。

如果生成的是项目离线 overlay/result 视频，它可能复用了检测与 PPE 后处理代码，但仍未完全等价于 Web 实测：离线渲染通常按视频帧顺序渲染；Web 预览则同时受实时播放时钟、检测线程处理频率、latest-only 丢帧、overlay timeline 匹配窗口、预览缩放和浏览器拉流节奏影响。断框如果发生在异步匹配或显示层，离线视频和 Web 画面就会不同。

## 影响范围

该问题影响 PPE 检测框显示质量，尤其是近景人物移动、多人交错、小目标、检测线程低于播放帧率、浏览器实时预览和用户指出的关键问题区间。它不应先归因于模型本体失效；应先用 YOLO reference 判断模型本体，再对比项目 overlay JSON、项目结果视频和 Web 实测表现。

## 结论

断框问题不能只靠“更长时间保留旧框”修复。当前更合理的诊断顺序是：

1. 用 YOLO reference 确认目标帧模型本体是否能检出。
2. 用项目实际结果视频和 overlay JSON 判断检测后处理是否产生连续 track。
3. 如果项目结果视频连续但 Web 断框，重点排查 Web preview 的 overlay 时间匹配、latest-only、held overlay、预览缩放和浏览器拉流节奏。
4. 如果项目结果视频也断框，重点排查 PPE tracking、弱检测提升、miss cap、render gate 和类别互斥策略。

## 后续建议

下一轮不要只看生成视频，应拿同一段视频、同一区间、同一权重同时产出：YOLO reference、项目结果视频、overlay JSON / CSV / summary，以及 Web 实测对应的状态或截图证据。只有这些证据对齐后，才能判断是模型本体、后处理 tracking，还是 Web 显示链路的问题。

该判断仍需在用户指出的问题区间上做实际视频和 Web 对照实验确认。

## 2026-06-10 补充定位

本轮在 `230-411` 区间补齐证据后，确认了一个 Web 专属差异点：后端 MJPEG 预览在 `runtime/runner.py` 的 `_render_backend_preview()` 中已经调用 `render_preview()` 把 PPE 框画进 JPEG；前端 `web/static/index.html` 又通过 `drawOverlay()` 从 `/api/overlay` 拉取记录，并在 `overlayCanvas` 上按前端 `OverlayTimeline` 再画一次框。

这会造成两个显示层并存：

- 后端预览层：跟随实际 preview packet 和 `_select_preview_overlay()`；
- 前端 canvas 层：跟随状态轮询、`/api/overlay` 轮询和浏览器 `requestAnimationFrame()`。

两者时钟不同步时，即使模型和项目离线结果视频没有完全断框，Web 也可能出现框变粗、拖影、旧框滞留感或位置不一致。

已做修复：后端状态增加 `preview_overlay_baked=true`，前端在该状态下跳过 canvas 框绘制，只保留 overlay 轮询用于状态面板和诊断。轻量审计显示，在同一区间内，修复前前端 canvas 会额外绘制 `79` 帧、`436` 个非 `person` 框；修复后默认后端预览模式额外绘制为 `0`。

剩余风险：这只消除了 Web 的二次叠框问题。如果用户在项目结果视频本身仍看到拖框或旧框滞留，下一步仍要调 PPE tracking / render gate 的 held track 策略；如果只在 Web 仍异常，则继续排查后端 `_select_preview_overlay()`、latest-only 调度、预览缩放和浏览器 MJPEG 拉流节奏。
