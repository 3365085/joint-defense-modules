# 语义背心 Web 投毒显示互斥修正

日期：2026-06-05

## 背景

用户反馈同一目标视频 `素材/橙色背心head测试视频/zczxc.mp4` 与同一语义投毒模型 `purification_lab/seven_experiment_archive/oga_semantic_vest/poisoned_best.pt` 下，实验对比视频的 attack 列能够稳定显示 `helmet/person`，但 Web 检测仍会出现 `head` 重复框或 `head/helmet` 混显。

## 当前判断

实验脚本在绘制前执行 display-only 的 head/helmet mutex：当 `head` 与 `helmet` IoU 达到 0.20 时，保留 `helmet`、隐藏 `head`。Web 侧原先的 PPE 后处理更偏向“抑制 helmet 误报”：当同位置 `head` 置信度高于 `helmet` 时，会把 `helmet` 放入 suppressed helmet，导致投毒场景里 raw 已经有同位 helmet，但 Web 显示仍保留 head。

## 代码链路依据

- `purification_lab/scripts/make_clean_attack_purif_comparison.py`：`_display_rows()` 固定调用 `_suppress_head_helmet_overlap(rows, threshold=0.20)`，该逻辑只作用于显示结果，raw 统计仍保留。
- `src/defense/module_a/ppe_postprocess.py`：`suppress_helmet_false_positives()` 原逻辑会在 `head/helmet` 高重叠且 `helmet` 置信度不够领先时压掉 helmet。
- `src/defense/module_a/postprocess/ppe_tracking.py`：显示 tracker 需要显式读取 `suppressed_head_indices` / `covered_head_indices`，否则已被业务后处理压掉的 head 仍可能形成或保留显示轨迹。
- Web API 复核显示：修正前目标视频 overlay 为 `records_with_head=15`、`mixed_head_helmet_records=15`；修正后为 `records_with_head=0`、`mixed_head_helmet_records=0`。

## 影响范围

本次修正不改模型权重、类别顺序、Web API 字段或 `person` 保留语义；只改变同一目标区域 `head/helmet` 冲突时的 Web 显示与 PPE 业务仲裁，使其对齐实验视频口径。raw `head` 仍可通过 `raw_head_count` 等字段保留为诊断证据。

## 结论

对于语义背心投毒场景，Web 侧现在优先保留同位 `helmet` 并隐藏重叠 `head`，避免 `head/helmet` 重复框和投毒效果被 Web 后处理削弱。后续若要进一步泛化到所有 PPE 场景，应补充非投毒裸头误报回归，确认低置信 helmet 不会过度掩盖真实未戴帽告警。

## 追加复核：早期帧业务计数与显示互斥不一致

用户随后反馈仍有重复/拖框残留，并询问 A3b 时序检测反应约 3 秒是否还能更快。用 Web API 重新跑同一视频与同一 `poisoned_best.pt` 后，实测稳定运行阶段 `source_time_s - overlay_time_s` 最大约 0.125 秒，`processing_ms` 约 27.6ms，`dropped_detection_frames=0`，说明本轮目标视频的主要问题不是 latest-only 队列积压或推理慢到 3 秒。启动冷加载或安全准入可能造成体感等待，但播放中检测结果并未落后三秒。

新的残留证据是早期和少量中后段 overlay record 中 `ppe_head_count=1`，但 `ppe_tracks` 已无 `head`，只剩 `helmet/person`。这说明显示 tracker 已按同目标区保留 helmet，而业务 summary 仍只按 IoU 判断 head/helmet mutex；当二者中心接近但 IoU 略低于 0.20 时，显示层认为是同一个 PPE 目标，业务层仍把 raw head 计为有效 head。

修正方向：在 Web 启用的 `prefer_helmet_on_head_overlap` 互斥中补充中心距离条件 `head_helmet_mutex_center_distance=0.055`，使 `ppe_postprocess` 与 `ppe_tracking` 对同目标 head/helmet 的判定口径一致。该修改不改变 raw 诊断字段，`raw_head_count` 仍可保留模型原始输出；面向 Web 展示和 PPE 业务判断的 `head_count/missing_helmet_count/candidate` 则按互斥后的有效结果输出。该判断基于当前目标视频与代码链路，仍建议后续用真实未戴帽样例做反向回归，确认不会压掉真实裸头告警。

## 追加复核：时序 helmet 轨迹与重复启动体感延迟

进一步复核发现，少量残留帧并不是同帧 raw `head/helmet` IoU 漏判，而是 raw 当前帧只输出了中心橙色背心人的 `head`，但 `PPEDisplayTracker` 已基于历史稳定轨迹把同一区域显示为 `helmet`。因此画面框层没有 `head`，右侧业务状态仍短暂显示 `ppe_head_count=1`。修正为：在 `apply_temporal_evidence()` 之后、PPE 状态机投票之前，若最终稳定显示轨迹中的 `helmet` 覆盖有效 raw `head`，则把该 raw head 作为 `temporal_helmet_mutex` 诊断保留，同时从 Web 业务 `head_count/missing_helmet_count/candidate` 中移除，并补回有效 helmet 证据。

性能方面，本轮目标视频稳定运行阶段仍不是 3 秒检测滞后：Web API 实测 overlay 与源时钟最大差约 0.083-0.125 秒，`processing_ms` 约 32-35ms，`dropped_detection_frames=0`。体感等待主要来自同模型重复启动时的冷加载；文件播放自然结束后旧逻辑会释放 pipeline cache。现已改为默认保留同模型/同配置 pipeline cache，除非显式配置 `release_pipeline_cache_on_file_end: true`。同一目标视频连续两次 Web API 启动实测：第一轮冷加载 `/api/start` 约 2230.6ms、`pipeline_cache_hit=false`；第二轮缓存命中 `/api/start` 约 156.9ms、`pipeline_cache_hit=true`。

最终目标视频 Web API 复核：两轮 overlay 均 `bad_records=0`，tracks 标签只包含 `helmet/person`，最终 `ppe_head_count=0`、`ppe_helmet_count=4`、`ppe_person_count=5`。这证明 Web 展示与右侧业务状态已对齐实验视频 attack 列的投毒显示口径。

## 追加复核：person 重复框与可见计数口径

后续目标模式复验发现，`head/helmet` 互斥已经生效后，剩余“重复框”主要不是 `head/helmet` 混显，而是同一人员目标出现两个高度重叠的 `person/person` track。典型模式是一大一小两个 person 框中心接近，较小框大部分被较大框包含，IoU 约 0.47-0.67，低于旧的同类 shadow-track 合并阈值，因此会在 Web overlay 中同时保留。

修正方向：在 `src/defense/module_a/postprocess/ppe_tracking.py` 中增加 `bbox_min_overlap_ratio()` 与 `_same_person_target()`，对 `person/person` 同目标判断同时使用 IoU、中心距离和较小框包含比例；进入 track 前的 `_merge_same_target_items()` 与已有 track 的 `_remove_shadow_tracks()` 使用同一判断。这样只合并“同一人大小框套叠”的显示重复，不合并相邻人员目标。回归测试增加了“包含式重复 person 只显示一个”和“相邻 person 不合并”两类样例。

同时修正 `src/defense/runtime/frame_processor.py` 的 Web 状态口径：`ppe_person_count`、`ppe_helmet_count`、`ppe_head_count`、`ppe_class_counts` 现在按最终可见 `ppe_tracks` 统计；`ppe_raw_person_count`、`ppe_raw_head_count`、`ppe_raw_class_counts` 继续保留模型原始诊断。这样页面/HUD 与画面框一致，不再出现“框已去重但人数按原始重复检测抖动”的误导。

最新同目标视频复验：冷启动 `/api/start` 约 2126.1ms、`pipeline_cache_hit=false`、`init_ms≈1894.8ms`；同模型第二次启动 `/api/start` 约 205.3ms、`pipeline_cache_hit=true`、`init_ms≈0.178ms`。缓存启动后的 120 条 overlay 记录中，`display_head_records=0`、`business_head_records=0`、`duplicate_records=0`、`count_mismatch_records=0`，最终状态为 `ppe_head_count=0`、`ppe_helmet_count=4`、`ppe_person_count=5`。`raw_head_records=49` 仍作为原始模型诊断存在，但不进入 Web 可见框和业务告警口径。
