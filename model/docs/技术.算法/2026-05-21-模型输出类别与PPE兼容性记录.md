# 模型输出类别与 PPE 兼容性记录

## 问题背景

用户提出需要兼容部分模型不输出 `person` 类的情况，并询问模型输出类别由什么决定。当前 PPE 业务应避免把 `person` 类作为必需输入，否则 `head/helmet` 模型会出现语义错误或状态误判。

## 当前判断

模型输出类别主要由训练数据类别表、模型检测头、权重训练结果和导出产物元数据决定。运行时后处理只能解释或过滤已有输出，不能让模型凭空输出未训练或未导出的类别。

## 代码链路依据

- `src/defense/module_a/backends/detector_backend.py`
  - `DetectionFrameResult.names` 保存模型类别表。
  - `UltralyticsDetectorBackend.predict()` 从 Ultralytics `result.boxes` 读取 `cls/conf/xyxy`。
  - `YoloV5DetectorBackend.names` 当前固定为 `{0: "helmet", 1: "head", 2: "person"}`，并在 `_non_max_suppression()` 后按类别 id 输出。
- `src/defense/module_a/ppe_postprocess.py`
  - `is_person_label()`、`is_helmet_label()`、`is_bare_head_label()` 只负责把输出标签归一化为业务语义。
  - `summarize_ppe_from_detections()` 当前消费 `person_count`、`helmet_count`、`head_count`。
- `src/defense/runtime/config.py`
  - `infer_model_family_from_model_path()` 与 `apply_custom_model()` 决定自定义模型走 YOLOv5 还是 Ultralytics 解码链路。

## 结论

PPE 算法应将 `person` 视为可选上下文，而不是必需类别。兼容策略应先根据 `DetectionFrameResult.names` 判断模型是否声明 `person` 类；无 `person` 类时，使用 `head/helmet` 主导判定；有 `person` 类时，把它用于鲁棒性兜底、ROI 复检和不确定状态。

## 后续建议

- 增加模型能力识别字段，例如 `has_person_class`、`has_head_class`、`has_helmet_class`。
- PPE 状态增加 `evidence_mode`，区分 `head_helmet_only` 与 `person_context_available`。
- 避免在无 `person` 类模型下输出“未检测到人员”，应输出“未检测到 person 类；当前基于 head/helmet 判定”。
