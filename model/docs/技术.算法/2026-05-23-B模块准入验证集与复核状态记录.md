# B 模块准入验证集与 full scan 状态记录

## 背景

用户在 Web 端点击开始检测后，页面曾长期显示“正在初始化检测”。实际链路中，A 模块没有启动，原因是 `/api/start` 先经过 B 模块模型安全准入，未通过时会直接阻断 `MonitorEngine.start()`。

## 当前判断

1. 初始阻断原因曾是 B full scan 验证集不可用。
   - 配置文件：`model/configs/model_security/heldout_sets.yaml`
   - 已改为当前机器存在的验证集：
     - `D:\security_project_d\model_b_old\datasets\mask_bd_external_eval`
     - `D:\security_project_d\model_b_old\datasets\poison_benchmark_cuda_tuned_remap_v2`
   - `ModelSecurityService._validation_assets()` 当前返回 `usable=true`。

2. 原先 `defense.model_security.scanner.full_scan()` 仍复用 `quick_scan()` 的伪激活探针，不能代表 B 模块完整验证链路。
   - 已增加 `defense.model_security.runtime_adapter.ModuleADetectorAdapter`。
   - B 模块 `model_security_gate.detox.external_hard_suite.run_external_hard_suite()` 现在通过 A 模块的 `create_detector_backend()` 调用实际运行后端。
   - 这样可兼容默认 YOLOv5 TensorRT engine，避免 B 模块旧的 Ultralytics YOLOv8 加载器无法读取 YOLOv5 PT。

3. 默认模型已完成一次 B full scan，但结果为 `suspicious`，因此不会写入白名单，也不会启动 A 模块。
   - 运行模型：`D:\联合防御模块\model\baseline_training\runs\baseline_yolov5\weights\best_fp16_640.engine`
   - 源 PT：`D:\联合防御模块\model\baseline_training\runs\baseline_yolov5\weights\best.pt`
   - 主报告：`D:\联合防御模块\model\runtime\model_security\reports\sha256_86de55260fc432d818b3f9d423652fac5956e9f178ee61a1988f5a0aba5cd866_full.json`
   - hard-suite JSON：`D:\联合防御模块\model\runtime\model_security\reports\sha256_86de55260fc432d818b3f9d423652fac5956e9f178ee61a1988f5a0aba5cd866_external_hard_suite\external_hard_suite_asr.json`
   - hard-suite CSV：`D:\联合防御模块\model\runtime\model_security\reports\sha256_86de55260fc432d818b3f9d423652fac5956e9f178ee61a1988f5a0aba5cd866_external_hard_suite\external_hard_suite_rows.csv`
   - 结果：`status=suspicious`，`risk_score=1.0`，`rows=112`，`max_asr=1.0`，`mean_asr=0.4464`。

## 代码链路依据

- `/api/start`：`defense.web.fastapi_app.create_app()` 中先调用 `ModelSecurityService.ensure_admitted()`；未通过时返回 `model_security_blocked`，不调用 `MonitorEngine.start()`。
- 准入状态：`ModelSecurityService.admission_status()` 根据白名单、扫描线程、最新 full scan 报告生成 `admission_status`。
- 白名单：`ModelTrustRegistry` 写入 `model/runtime/model_security/trusted_registry.json`；当前默认模型未写入，白名单文件仍不存在。
- full scan：`defense.model_security.scanner.full_scan()` 要求源 PT、可用验证集、runtime config 和 project root；真实结果由 B hard-suite external ASR 决定。
- Web 展示：`model/src/defense/web/static/index.html` 中 `modelSecurityBlockState` 和 `showModelSecurityBlock()` 会把主画面显示为“B模块准入阻断”，避免误显示为 A 模块初始化。

## 影响范围

- 当前生产策略仍是 `review/suspicious/unverifiable` 一律不放行。
- 默认模型在 B hard-suite 下被判定为 `suspicious`，所以 `/api/start` 返回 409，A 模块不启动，这是符合当前策略的结果。
- 这说明“B 模块全链路接入且正常运行”已成立，但“默认模型可直接进入白名单”不成立。

## 后续建议

1. 需要确认默认模型是否应该被 B 模块视为干净。如果应该干净，则要继续校准 B 的 target class、验证集选择、ASR 成功定义和阈值。
2. 如果当前 B 判定可信，则下一步应进入净化流程，而不是人工把 `suspicious` 写入 trusted registry。
3. 不建议为了启动 A 模块而静默绕过 B 准入；开发联调可另加显式调试开关，但生产白名单仍应只接受 `clean/trusted` 报告。
