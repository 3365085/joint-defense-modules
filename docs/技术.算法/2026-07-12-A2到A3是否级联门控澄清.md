# A2→A3 是否"时序纹理不一致才触发 A3"——流程与帧数影响澄清

日期: 2026-07-12
类型: 检测流程 / 性能取舍澄清
状态: 已从代码确认

## 问题背景

用户读基础文档(`doc/前置文档/基础文档.txt`)时注意到 A2 的描述:
"A2 时序纹理不一致性分析……若 Change_t 突然剧烈且无对应运动目标，则疑似动态扰动"，
进而担心是否存在"A2 判异常后才进入 A3(光流)"的级联门控——若成立会影响每帧实际执行的算子数量，
即影响帧率/性能。

## 当前判断(已从代码确认)

**不成立。当前默认启用的 rebuilt 内核里，A1→A2→A3→A4 是无条件顺序执行，不存在"A2 异常才触发 A3"的门控。**

基础文档中 A1/A2/A3/A4 的编号是**功能分层的描述顺序**，不是运行时的级联触发条件。
它们是并列的特征提取器，最后在 A4(融合) + `_joint_decision`(联合判定)汇总。

## 代码链路依据

`model/src/defense/module_a/rebuilt/detector.py` 的 `process()`(约 676-710 行)：

```
689  lbp  = self._compute_lbp(gray)          # 纹理基础(每帧)
690  flow = self._compute_flow(prev, gray)   # 光流基础(每帧)
693  a1 = self._compute_a1(lbp, rois, ...)                # 无条件
694  a2 = self._compute_a2(lbp, rois, ..., flow)          # 无条件
695  a3 = self._compute_a3(flow, rois, ...)               # 无条件
699-709  a3b = 后台线程 + _a3b_interval 节流(唯一的条件执行)
710  a4 = self._compute_a4(a1, a2, a3, a3b)               # 融合
```

`光流(_compute_flow)` 在 690 行**每个关键帧都算**，A3 直接消费该光流，A2 并不作为 A3 的开关。

legacy 路径(`module_a/process_pipeline.py` 的 `process_module_a`)同样是 A1→A2→A3→A3b→A4 顺序无条件执行，
无级联门控。两套实现口径一致。

## 真正影响帧数/性能的机制(与 A2→A3 无关)

1. **关键帧抽取(隔帧检测)**: 主项目外壳每 3 帧取 1 跑检测，静止帧复用上次结果。
   这是项目要保留的轻量化外壳(见北极星记忆)，已验证不打断光流连续性。
2. **A3b 后台线程 + `_a3b_interval` 节流**(detector.py 699-709): A3b 静态媒体检测放后台线程，
   主路径永不等待、用上一次结果；不节流会让后台线程通过 GIL 拖慢主路径约 5ms/帧。
   这是**唯一的条件执行**，且是性能优化，不是检测级联门控。

## 影响范围

- 对检测流程与帧率无需改动。A3(光流)在每个关键帧都在跑，没有被 A2 卡住。
- demo 内核本就是"实验需要光流连续"而逐帧跑(见 `model/HANDOFF_FROM_DEMO.md`)，
  移植到主项目后由外壳的关键帧抽取降频，光流连续性由复用逻辑保证。

## 结论与后续建议

- 基础文档那句话是攻击语义的**描述**(动态扰动的特征长相)，不是要求实现成级联门控；
  当前实现选择"全算子并行提取 + A4 融合"，更稳健(避免 A2 漏判导致 A3 根本不跑)。
- 无需据此修改。若未来为极致边端算力(如 Jetson Nano)要进一步降耗，
  可评估"A2 低置信时跳过 A3 光流"的条件执行，但须先量化 A3 单帧耗时占比与漏检代价，
  属**待实验确认**，当前不做。
