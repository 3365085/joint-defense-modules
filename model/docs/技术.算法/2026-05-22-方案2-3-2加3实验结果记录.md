# 方案 2 / 3 / 2+3 实验结果记录

## 实验背景

用户要求先提交当前多人弱头部显示修复，再实验三个方案：

- 方案 2：提高检测输入像素，按原视频 1/4 面积并保持宽高比例。
- 方案 3：`weakHead` 通过时序稳定性升级为有效 `head`。
- 方案 2+3：高像素输入与时序升级同时启用。

## 实验边界

- 视频：`D:\联合防御模块\素材\手机随意录制的视频\固定镜头室外视频.mp4`
- 视频属性：3840x2160，约 60fps。
- 实验帧数：前 150 帧。
- 当前生产 TensorRT engine 输入固定为 `(1, 3, 640, 640)`：
  - `baseline_training/runs/baseline_yolov5/weights/best_fp16_640.engine`
  - `baseline_training/runs/baseline_yolov5/weights/best.engine`
- 因此方案 2 使用同权重 `best.pt` 做 PyTorch 可变分辨率实验，输入为 1920x1080 内容并 pad 到 1920x1088。该结果只能说明趋势，不能等价于生产 TensorRT 性能。

## 观察结论

1. 方案 2 单独不理想。
   高像素输入确实让 `max_raw_head` 从 5 到 6，并在 150 帧中出现 1 帧 person，但有效 `head` 帧数没有增加，反而从 76 降到 70。主要原因是后处理仍会把远处小头部判为 `weakHead`，像素提高没有自动解决“弱证据不能告警”的业务语义。

2. 方案 3 提升最直接。
   在不改模型、不改主检测分辨率的情况下，稳定出现的 `weakHead` 被升级，`valid_head_frames` 从 76 到 147。视觉上远处行人从“只显示 weakHead”变成连续有效 head，但风险是：稳定误检也可能被升级。

3. 方案 2+3 召回最高，但最激进。
   `valid_head_frames` 达到 149，`promoted_frames` 达到 88。该方案对远处目标最敏感，但由于方案 2 目前依赖 PyTorch 可变分辨率实验，并且 2+3 会放大稳定弱证据，误报风险最高。

4. person 类不能作为可靠前提。
   当前模型类别表中有 person，但实验前 150 帧 baseline 的 `person_frames=0`，方案 2 也只有 1 帧输出 person。这说明“模型有 person 类”和“当前帧检测到 person”是两回事，PPE 主流程不能依赖 person 框。

## 建议

- 不建议直接上方案 2。当前 TensorRT engine 固定 640，生产化必须先导出 1280 或 1920 动态/固定 engine，并重新测速度、显存和拖框。
- 建议优先实现方案 3 的受限版本：
  - weakHead 连续命中至少 3 帧；
  - 平均置信度达到阈值；
  - 位置连续，不能跳动太大；
  - 只升级当前帧有效 head，不长时间保持旧框；
  - 对边缘区域或极小目标使用更高阈值。
- 方案 2+3 可以作为后续增强，但必须在新 engine 和更长视频集上做误报评估后再考虑。
