# 实时视频流输入链路修复(帧号/网络摄像头/超时)

日期: 2026-07-13
类型: 输入源链路诊断 / 修复
状态: 已改+已审查+本地摄像头已验证; 网络摄像头待用户在线验收

## 背景

用户截图暴露: 非本地视频输入(实时流)时"只检测第一帧"——画面 HUD 帧号恒为 00000。
且网络摄像头 `192.168.1.149:8081` 无法作为输入源。要求本地直连摄像头优先可用,
网络摄像头能正常识别才算过,并清理代码。

## 根因(均已取证确认)

### 1. 帧号卡 00000(真因,上轮漏查)
`runner.py` 采集循环统一用 `cap.get(CAP_PROP_POS_FRAMES)` 做帧号。该属性对本地文件可靠,
但对实时流(摄像头/网络流)常年返回 0 → `current_frame_idx = max(0, frame_idx-1)` 恒为 0。
帧号不递增导致 overlay/时序判定都停在第一帧语义。

### 2. 网络摄像头(HTTP MJPEG)无法输入
`192.168.1.149:8081` 实测是 `multipart/x-mixed-replace`(HTTP MJPEG),不是 rtsp。
前端"RTSP / HTTP 视频流"选项统一传 `source_type=rtsp`,而 `open_capture` 的 rtsp 分支
硬用 `cv2.CAP_FFMPEG`。FFMPEG 后端对 HTTP multipart 流会阻塞卡死(实测在线时默认后端
CAP_ANY 0.4s 秒开,FFMPEG 卡住)。

### 3. 超时不生效导致长阻塞
- 不可达 rtsp: `CAP_PROP_OPEN_TIMEOUT_MSEC=3000` 对 FFMPEG 后端实测不生效, `open` 阻塞 ~30s。
- 离线 http: 裸 CAP_ANY 构造走 FFMPEG tcp 重试卡 ~14s。
- http 断流(TCP 保持但停推帧/设备假死): read() 在 OpenCV 内部阻塞, 采集循环已有的
  "5s 无帧重连"位于 `if not ok:` 之内, 只能兜 read 快速返回失败, 无法抢占正在阻塞的 read()。

## 修复(runner.py, 全部低风险)

1. **帧号**: 采集循环按 source_type 分流。file 保留 POS_MSEC/POS_FRAMES(支持 seek);
   实时流改用单调递增的 `packet_seq` 做 `current_frame_idx`, `source_time_s = packet_seq/fps`。
2. **HTTP MJPEG**: `open_capture` rtsp 分支按 `urlparse(url).scheme` 分流。http/https 用
   `cv2.VideoCapture(); cap.open(url, CAP_ANY)`; 真 rtsp/rtmp 才用 CAP_FFMPEG。
3. **快速失败**: 新增 `_probe_tcp_reachable(url, timeout_ms)`——socket 短超时 TCP 探活,
   不可达立即抛 RuntimeError。http/rtsp 打开前都先探活。不可达 rtsp 30s→3s, 离线 http 14s→2s。
4. **超时兜底**: http 分支补 READ_TIMEOUT(让 read 有上界返回, 5s 重连能真正兜底);
   真 rtsp 用环境变量 `OPENCV_FFMPEG_CAPTURE_OPTIONS=stimeout;<us>|rtsp_transport;tcp` 注入
   FFMPEG 超时(因 CAP_PROP OPEN_TIMEOUT 对 FFMPEG 不生效)。
5. **死代码清理**: camera 分支删除永不触发的 `if cap is None` 兜底, 重构为顺序尝试
   DSHOW→MSMF→CAP_ANY。

## 审查 agent 交叉核验结论
- (a)帧号+1时序连续性: 无问题。实时流连续采集帧严格满足 current_idx==prev+1, predecessor
  是采集连续帧(非检测帧), detect_interval 丢帧不打断; 推到检测的帧对 gap 恒=1。
- (b)reconnect 后 packet_seq 不重置: 无问题。帧号持续单调, 重连首帧前驱置 None 不误判 +1,
  下游(ppe_tracking 差值/gap_frames/scene_cut)只需单调, 安全。
- (d)http 断流 read 阻塞: 已按建议补 READ_TIMEOUT。
- (e)除零/NaN: 无问题。`_read_capture_meta` 已把 fps 钳到 [1,60]。
- (c)真 rtsp OPEN_TIMEOUT: 实测确认不生效(30s), 已改 TCP 探活+stimeout 兜底(3s)。

## 已知限制(诚实记录)
- `OPENCV_FFMPEG_CAPTURE_OPTIONS` 是进程级全局环境变量, 真 rtsp 打开时设置后不清理;
  对本地文件/http 分支无实际副作用(stimeout 不触发, rtsp_transport 对非 rtsp 忽略), 可接受。
- 网络流用 packet_seq/fps 是合成时钟, 与真实到达速率会漂移, 但显示与检测记录源自同一
  packet 同一时钟, overlay 匹配在同一时间轴自洽, 不错位。
