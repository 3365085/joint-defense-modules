# 检测输出模型族状态修复记录

## 问题背景

当前工程从原 A 模块 `D:\security_project_d\Model_A` 合并后，业务层仍通过
`src/defense/runtime/ppe_business.py` 的 `PPEBusinessEvaluator.evaluate` 影响
`VideoDefensePipeline.process_frame` 输出的 `ppe_*` 检测信息。对比原项目发现，
当前检测输出异常更可能来自运行配置和 Web 状态链路，而不是 PPE 业务规则本身。

## 代码链路依据

- 自定义模型参数入口在 `src/defense/runtime/config.py` 的
  `normalize_custom_model_options` 与 `apply_custom_model`。
- 检测流水线构建在 `src/defense/runtime/pipeline_factory.py` 的
  `PipelineCache.get`，输出 `PipelineBundle`。
- 运行状态输出在 `src/defense/runtime/runner.py` 的
  `MonitorEngine._empty_status`、`MonitorEngine.start` 和
  `MonitorEngine._process_loop`。
- 前端模型选择和状态展示在 `src/defense/web/static/index.html` 的
  `inferCustomModelSelection`、`applyRuntimeCustomModelStatus`、
  `backendLabel`、`modelFamilyLabel`。
- PPE 业务输出由 `src/defense/runtime/frame_processor.py` 的
  `FrameProcessor.process` 调用 `PPEBusinessEvaluator.evaluate` 后写回
  `ppe_warning`、`ppe_tracks`、`ppe_reason` 等字段。

## 当前判断

原 A 模块会根据自定义模型文件自动推断 backend 和 model family，并把解析后的
`custom_model`、`backend`、`model_family` 写入运行状态。当前合并版本曾把
`.pt/.onnx/.engine` 自定义模型强制偏向 `yolov5`，且状态输出缺少
`model_family`，会造成前端显示与后端实际检测适配器不一致，并可能让
Ultralytics 模型进入错误的 YOLOv5 路径。

本次修复恢复以下行为：

- `apply_custom_model` 支持按文件后缀推断 backend，并校验显式 backend 与后缀是否匹配。
- `infer_model_family_from_model_path` 支持对 `.pt` 与 `.onnx` 模型做模型族推断，无法确认时默认使用 `ultralytics`。
- `PipelineBundle` 增加 `model_family`，`MonitorEngine` 状态输出恢复 `model_family` 和解析后的 `custom_model`。
- `FrameProcessor._build_status` 每帧状态改为使用 `PipelineBundle` 中的解析结果，避免首帧后回退为启动请求草稿值。
- 前端恢复“自动识别模型类型”，不再仅因文件后缀把模型族强制设置为 `yolov5`。

## 影响范围

该修复只影响自定义模型选择、运行状态字段、前端状态展示和模型适配器选择链路。
未修改模型权重、类别语义、检测阈值、PPE 业务规则、A3b 策略或主检测算法。

## 结论

检测输出异常的高风险点已定位为运行配置和 Web 状态链路的模型族信息丢失。
PPE 业务层存在且会影响检测输出，但本次对比未发现 `PPEBusinessEvaluator` 与
`FrameProcessor` 相比原 A 模块存在差异。

## 后续建议

- 用实际 `.pt/.onnx/.engine` 自定义模型分别做一次端到端启动与检测输出验收。
- 若仍有检测框、类别或 PPE 告警异常，应再进入 `src/defense/module_a` 和
  `src/defense/runtime/ppe_business.py` 做行为级实验确认。
- 模型族推断对未知模型结构仍可能回退到 `ultralytics`，该分支属于“待实验确认”。
