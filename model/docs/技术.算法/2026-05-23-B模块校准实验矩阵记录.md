# B 模块校准实验矩阵记录

## 背景

默认模型接入 B 模块准入后，完整扫描结果为 `suspicious`，导致 A 模块启动被阻断。为判断这是默认模型真实可疑，还是 B 的目标类、验证集和 ASR 判定策略不适配，执行只读校准实验。实验不写入白名单，不启动 A 模块，不修改模型权重。

## 实验入口

- 工具：`defense.model_security.calibration.run_calibration_matrix`
- CLI：`python -m defense.model_security.calibration`
- 运行环境：`D:\联合防御模块\.pixi`
- 运行模型：`D:\联合防御模块\model\baseline_training\runs\baseline_yolov5\weights\best_fp16_640.engine`
- 模型类别：`0=helmet`，`1=head`，`2=person`
- 准入阈值：`external_eval_allowed_max_asr=0.10`，`external_eval_suspicious_asr=0.50`

## 第二轮放大样本

只重跑第一轮中最接近通过的组合：

- `poison + helmet + class_presence`
- `poison + helmet+head + class_presence`
- 每攻击集抽样：`64`

## 第三轮置信度敏感性

对 `poison + helmet+head + class_presence` 执行 `external_conf` 扫描：`0.25`、`0.26`、`0.28`、`0.30`、`0.35`。

结果均为：

- `status=review`
- `max_asr=0.109375`
- `mean_asr=0.0625`
- `rows=128`
- top attack：`semantic_green_cleanlabel=0.109375`

结论：

- 该边界问题不是检测置信度阈值导致。
- 不建议通过提高 `external_eval_conf` 来让默认模型进入白名单。

## 当前判断

1. 继续把 `mask_bd_external_eval` 合入默认准入会稳定把默认模型判成 `suspicious`，应暂停作为生产准入依据。
2. `poison_benchmark_cuda_tuned_remap_v2 + helmet+head + class_presence` 是目前最接近合理的准入配置，但仍只是 `review`，不能自动入白名单。
3. 若坚持 `allowed_max_asr=0.10`，默认模型不能 trusted；下一步应检查 `semantic_green_cleanlabel` 的 7 个失败样本，判断是数据语义、标签映射、模型误检还是确有后门风险。
4. 若业务上接受 `allowed_max_asr` 从 `0.10` 放宽到 `0.11` 或 `0.12`，默认模型可以接近通过，但这属于安全策略放宽，必须由用户明确批准，不能由代码默认执行。

## 建议下一步

- 读取 `poison__helmet_head__class_presence\external_hard_suite_rows.csv`

若失败样本审计确认是验证策略误杀，再调整 B 准入默认配置；若确认是模型风险，则进入净化流程。
