# 三类 PPE 模型检测净化一体化改造记录

日期：2026-06-04

## 背景

当前系统需要从原本偏 `head / helmet` 的投毒模型检测与净化口径，升级为完整支持 `person / head / helmet` 三类模型。用户指出原检测系统中 `person` 主要作为辅助检查，投毒模型检测算法也存在二类输出模型假设；同时原净化一体化效果较差，需要接入七类投毒/净化实验成果。

## 代码链路依据

- 模型加载：`model/src/defense/module_a/backends/detector_backend.py` 中 `create_detector_backend` 创建 Ultralytics / YOLOv5 backend，检测输出统一为 `DetectionFrameResult.names/classes/boxes/confidences`。
- 运行状态：`model/src/defense/runtime/frame_processor.py` 将 PPE 结果展开为 `ppe_person_count`、`ppe_head_count`、`ppe_helmet_count`、`ppe_class_counts` 与 overlay/status 字段。
- PPE 后处理：`model/src/defense/module_a/ppe_postprocess.py` 已按 label 识别 person/head/helmet，且已有 person ROI 抑制 `helmet_without_person_context`、`oversized_helmet_vs_person` 等误报逻辑。
- 投毒检测：`model/src/defense/model_security/scanner.py` 的 `_external_target_resolution` 决定外部 hard suite 目标类；原逻辑默认偏 helmet，且显式过滤 person。
- 净化候选：`model/src/defense/model_security/purifier.py` 原先优先使用随包 B 模块 strict audit 与 packaged poisoned/purified 模型。

## 当前判断

1. 三类支持的关键风险不是 NMS 类别数量，而是 class id 到 label 的映射。YOLOv5/导出模型若缺少显式 class map，可能把 `person/head/helmet` 顺序解释错误。
2. `person` 不应在未做行为调参前改成 PPE 违规告警触发条件；当前最小安全改造是让 `person` 在定位、状态、可视化和误报抑制中拥有同等级字段，但保持 `head` 驱动未戴帽告警。
3. B 模块投毒目标默认应覆盖 `helmet/head`，同时把 `person` 作为 context/preserve class 记录；只有显式配置 `external_eval_allow_person_targets` 时才把 `person` 作为投毒目标，避免误改 PPE 安全语义。
4. 七实验归档适合作为 hash-bound catalog 接入：投毒模型 SHA 命中即阻断并给出配对净化候选；净化模型 SHA 命中时可展示攻击算法、净化算法、对比视频和复现实验路径。

## 影响范围

- 检测输出解释：`inference.names/class_names/labels` 将成为可配置 class map。
- PPE 状态字段：新增 `effective_person_count`、`weak_person_count`、`promoted_person_count` 等兼容扩展字段。
- B 模块报告：外部验证诊断新增 target/context/preserve class 信息。
- 净化一体化：可通过 `purification_lab/seven_experiment_archive/manifest.json` 识别七实验投毒/净化模型。
- 视觉面：overlay/HUD 会显示三类弱证据与时序增强计数，提交前必须按 AGENTS.md 做视频视觉验收。

## 结论

当前改造方向是兼容扩展而非重置 PPE 策略：保持旧 API/status 字段，新增三类 class map、person 定位字段、三类目标解析和七实验归档 catalog。涉及视觉显示的修改在提交前仍需跑目标视频并本地静默抽帧逐帧检查。

## 后续建议

- 对 `person` 是否能参与告警语义，应单独做行为调参和误报评估后再决定。
- 七实验归档若要作为生产白名单依据，建议后续补充严格 ASR/Wilson/mAP 审计字段；当前归档命中结论基于 SHA、用户验收视频和结构化 summary。
- 对 YOLOv5/ONNX/TensorRT 等缺失 metadata 的模型，运行配置或 UI 应显式提供 class map。

## 本轮复核与修正

日期：2026-06-04

### 背景

并行只读复核指出两个影响闭环的问题：一是当前实际运行配置仍可能把 B 模块外部评估目标限制为单类 `helmet`；二是自定义模型入口不能随模型携带 `class_names`，导致 `person/head/helmet` 顺序模型可能被按旧顺序解释。同时，PPE 摘要、显示跟踪和 backend 对 `human/pedestrian/worker` 这类人员别名的识别范围不完全一致。

### 代码链路依据

- `model/src/defense/runtime/config.py`：`normalize_custom_model_options()` 和 `apply_custom_model()` 现在接收 `class_names`，并写入 `inference.class_names`，使检测 backend、B 模块 fingerprint 和 scanner 共用同一类别映射。
- `model/src/defense/module_a/backends/detector_backend.py`：`configured_class_names()` 统一解析 `names/class_names/labels`；backend 构造时优先使用显式类别映射。
- `model/src/defense/model_security/scanner.py`：`_external_target_resolution()` 默认覆盖 `helmet/head`，`person` 默认进入 `context_classes/preserve_classes`；七实验 archive 命中时 full scan 记录 `seven_experiment_known_poisoned_archive` 或 `seven_experiment_purified_archive`。
- `model/src/defense/module_a/ppe_postprocess.py`、`model/src/defense/module_a/postprocess/ppe_tracking.py`、`model/src/defense/module_a/backends/detector_backend.py`：人员别名对齐为 `person/worker/human/pedestrian`，避免摘要、跟踪、候选阈值策略不一致。
- `model/src/defense/web/static/model_security.html` 和 `model/src/defense/web/static/index.html`：安全中心可选择类别顺序，主检测页沿用同一份本地配置并提交给后端。

### 当前判断

1. 三类模型支持的关键不是新增一个模型分支，而是保证 class id 到 `person/head/helmet` 的映射在 detector、PPE 后处理、B 模块扫描、白名单 fingerprint 和前端展示中一致。
2. `person` 当前仍不作为未戴帽违规触发类；它用于人员定位、上下文计数、显示框和抑制离体 `head/helmet` 误报。这是兼容性扩展，不是 PPE 行为调参。
3. B 模块默认扫描目标应为 `helmet/head`，`person` 作为上下文保留类；若后续要把 `person` 也作为投毒攻击目标，需要显式开启 `external_eval_allow_person_targets` 并单独评估语义影响。
4. 七实验归档接入属于 hash-bound catalog：毒模型 hash 命中即阻断并给出配对净化候选；净化模型 hash 命中可作为已验证归档模型通过扫描。严格生产审计指标仍标注为待实验确认。

### 验证

- `pixi run python -m pytest model/tests/test_model_security_bypass_and_metrics.py -q`：13 passed。
- `pixi run python -m pytest model/tests/test_yolov5_model_base_contract.py model/tests/test_detector_backend_candidate_confidence.py model/tests/test_ppe_postprocess.py model/tests/test_ppe_tracking_aliases.py model/tests/test_model_security_runtime.py -q`：67 passed。本组包含本地 ignored 测试资产，作为开发期补充验证；可提交回归已同步到已跟踪测试文件。

### 影响范围

- 运行时自定义模型：新增可选 `class_names` 字段；未提供时仍使用原配置或模型 metadata。
- B 模块准入：报告 diagnostics 会显示 `target_classes/context_classes/preserve_classes`，旧 `ppe_helmet_target_v2` 报告仍兼容。
- PPE 展示：人员别名识别更一致，`person` 框仍不触发违规告警。
- Web：安全中心新增类别顺序选择；该选择只影响运行时类别解释，不改模型权重或阈值。

### 待实验确认

- 真实目标视频中三类框是否稳定、person 上下文是否足以抑制离体 `head/helmet` 误报，仍需按 AGENTS.md 抽取连续 3 秒本地静默目检。
- 七实验归档净化模型若作为生产强白名单依据，仍建议补 Wilson/mAP/ASR 严格审计字段；当前结论依赖 SHA、归档 summary 和已验收三栏视频。

## 橙色马甲 person 不稳定复核与修正记录

日期：2026-06-04

### 背景

用户指出 `runtime/visual_acceptance/three_class_overlay_20260604/orange_vest_three_class_overlay.mp4` 中，中间橙色马甲人的 `head` 和 `helmet` 看起来稳定，但 `person` 输出不稳定。该问题属于检测效果与类别语义解释问题，需要记录到技术/算法问题档案。

### 当前判断

本次复核确认，旧验收视频的主要问题不是 Web overlay 吞掉 `person` 框，也不是该目标在模型中完全没有人框，而是验收时使用了错误的类别顺序覆盖：旧记录写入 `0=person, 1=head, 2=helmet`，但当前三类实验模型和实验区说明均为 `0=helmet, 1=head, 2=person`。因此旧视频中前景橙色马甲人的稳定大框被显示成“安全帽/helmet”，用户看到的 `person` 不稳定实际上来自被错映射后的 `helmet` 类计数。

### 代码链路依据

- `purification_lab/README.md` 明确三类实验顺序为 `0=helmet`、`1=head`、`2=person`。
- `model/baseline_training/runs/baseline_yolov8_three_put/best.pt` 的 checkpoint metadata 读取结果为 `{0: "helmet", 1: "head", 2: "person"}`。
- `model/src/defense/module_a/backends/detector_backend.py` 会优先使用配置或自定义模型传入的 `class_names` 解释 class id。
- `model/src/defense/module_a/ppe_postprocess.py` 和 `model/src/defense/module_a/postprocess/ppe_tracking.py` 均按 detector result 的 `names` 字段生成 PPE 计数与展示 track。
- `model/src/defense/visualization/overlay.py` 只按 track label 画框和 HUD，不重新推断类别语义。

### 复核结果

- 旧验收记录：`runtime/visual_acceptance/three_class_overlay_20260604/visual_acceptance_record.json`，类别覆盖为 `0=person, 1=head, 2=helmet`。
- 旧视频单帧目检：frame 24、37、60 中前景橙色马甲人大框被标成“安全帽”，HUD 的 `person` 计数随错误映射波动。
- 数值复核：按错误映射全帧 `person>=0.25` 为 106/120；但按模型真实顺序重跑后 `person_missing_frames=[]`。
- 修正版验收视频：`runtime/visual_acceptance/three_class_overlay_20260604_corrected/orange_vest_three_class_overlay_corrected.mp4`。
- 修正版抽帧区间：frame 24-95，对应 24 FPS 下约 1.00s-3.96s。
- 修正版抽帧目录：`runtime/visual_acceptance/three_class_overlay_20260604_corrected/frames_024_095`。
- 修正版检查拼图目录：`runtime/visual_acceptance/three_class_overlay_20260604_corrected/inspection_sheets`。
- 肉眼检查结论：frame 24-95 连续区间内前景橙色马甲人 `person` 大框连续存在，`head` 框稳定；未见旧视频中“前景 person 消失但 helmet 大框稳定”的现象。后段多人入画后背景框较密集，属于画面目标数量增加，不是本问题的 class map 错配。

### 影响范围

- 旧验收视频不能作为 `person/head/helmet` 稳定性结论依据，因为 `person` 与 `helmet` 语义被对调。
- Web/API 主路径在使用默认运行配置 `[helmet, head, person]` 时语义正确；风险来自用户在自定义模型类别顺序中手动选择了不匹配的 class order。
- B 模块扫描目标、PPE 计数和 overlay 标签都依赖同一 class map；类别顺序错配会同时影响显示、状态字段和扫描目标解释。

### 处理与建议

- 已将模型安全页的类别顺序选项调整为优先显示 `helmet / head / person（三类实验）`，降低误选概率。
- 对 `baseline_yolov8_three_put/best.pt` 及七实验三类模型，默认使用 checkpoint 自带 names 或显式选择 `helmet/head/person`；不要选择 `person/head/helmet`。
- 后续建议在自定义模型加载时展示 checkpoint embedded names 与用户覆盖 names 的差异提示；是否强制拦截错配属于产品交互策略，待实验确认。

### 后续防错补强

- `model/src/defense/model_security/service.py` 已新增 `class_names_diagnostics`：仅当自定义模型启用时，通过 torch zip 的 `data.pkl` pickle opcode 安全扫描读取 `.pt/.pth` checkpoint embedded names，并与运行配置中的 `class_names` 对比。
- 该诊断不参与主检测路径，不做额外 GPU 推理，不改变模型权重、检测阈值、PPE 语义或 A3b/Module A 策略。
- 当用户覆盖顺序与 checkpoint embedded names 不一致时，B 模块 status 返回 `class_names_mismatch=true` 和 `class_names_warning`；模型安全页同步在“类别映射”字段显示 warning。
- 回归测试新增默认 runtime 不读取 checkpoint names、错误覆盖时报告 mismatch、安全解析 torch zip embedded names、FastAPI status 透出 class map warning、模型安全页展示 warning 五个场景。
- 验证：`pixi run python -m pytest -q tests/test_model_security_bypass_and_metrics.py` 为 20 passed；`pixi run smoke` 为 300 passed，3 skipped。跳过项仍为 samples smoke report 缺失，与本次 class map 防错补强无直接关系。
