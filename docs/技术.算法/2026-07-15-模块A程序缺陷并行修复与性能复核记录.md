## 一、结论摘要

本轮修复程序、接线、生命周期、诊断、明确授权的性能调度，以及用户要求的 A3b 过拟合/误报行为问题；未修改模型权重、类别语义或 PPE 语义。

当前结论：

1. 已关闭本轮能够由程序修复的主要问题：
   - A3b bbox suppression 的固定帧时长和低 FPS 放大问题；
   - Web 缺省 feature options 覆盖 profile；
   - runtime 在真实 source FPS 到达前创建 evidence 和计算预算；
   - disabled evidence 仍创建目录/manifest；
   - evidence 缺 run/source lineage；
   - PipelineCache 不能识别最终模型文件内容替换；
   - PipelineCache 不能识别 A4 classifier 辅助资产内容替换；
   - A4 第 20 维特征键错误且没有稳定 schema；
   - rebuilt 分阶段 timing 被 pipeline 覆盖；
   - rebuilt A3b timing 没有进入旧公共 timing alias；
   - Web 无法观察实际 effective A3b sensitivity；
   - seek 期间 in-flight 旧帧可在 epoch 切换后重新发布；
   - A3b suppression 的 freshness/policy/candidate 续租语义不一致；
   - runtime soft-trigger 绕过 rebuilt tighten gate，导致 clean 场景大面积误报。
2. 默认 `desktop_rtx` 检测上限由 60 调整为 30 FPS；`high_quality` 仍保留 60 FPS。该调整用于限制 60 FPS source 下的持续满载，不改变 30 FPS 目标视频的有效检测上限。
3. 不能通过把 `light_flow_interval` 简单改大来节流。当前 interval skip 会直接返回空 flow，并删除该帧 A2/A3 证据，而不是对更长时间跨度进行等价积分；因此生产 profile 保持 `light_flow_interval=1`，成本通过 detector FPS cap、latest-only 和保守 detection reuse 控制。
4. 当前补丁后的目标视频仍能在生产口径进入 A3b：
   - 确定性 `FrameProcessor/A3BSoftTriggerState` 全帧口径：首次触发 source frame 30、source time 1.0 秒，280 帧中触发 42 帧；
   - 真实 `MonitorEngine/latest-only` 预览口径：首次公开 overlay trigger 为 source frame 33、source time 1.1 秒；
   - trigger source 为 `confirmed_track`；
   - backend error、worker timeout、worker rejection 均为 0；
   - 当前 runtime 预览结果写入 `target_runtime_preview_postfix.mp4` 及配套 overlay/summary JSON。
   - 早于 soft gate 修复的 frame 17 / 0.567 秒结果已被当前补丁取代，不能作为最终触发时延结论。
5. 性能在同机桌面负载下存在明显运行间波动：
   - 稳定实时复跑：198 个处理状态样本，processing p95 约 27.62 ms，pipeline p95 约 21.97 ms，最大 dropped 约 1，满足 30 FPS 的 33.33 ms 预算；
   - 另一轮桌面争用复跑：pipeline p95 约 58.60 ms、最大 dropped 约 19，状态正确显示 `processing_budget_ok=false`。
   - 因此可确认正常负载下满足目标视频 30 FPS 预算，但不能宣称在任意桌面/GPU 争用下都稳定满 30 FPS。该项属于部署负载容量风险，需用独占 GPU 或固定负载压测进一步确认。
6. 已补齐当前生产口径的 48 段 heldout CLI，并以当前工作树重跑：
   - 27 个 clean：A3b FP videos/frames = `0/0`；
   - 21 个非 A3b attack：wrong-channel videos/frames = `0/0`；
   - backend error、worker timeout/rejection、schedule blocked、stale/expired result 均为 0；
   - 48 段严格前帧缺口为 0。
7. 48 段集合已反复参与策略选择，且没有独立真实 A3b 正例，因此上述结果只能证明当前回归门禁，不等价于“无过拟合”或真实 replay 召回率证明。

### 3.1 A3b suppression 改为 source-time 租约

文件：

- `model/src/defense/pipelines/video_defense_pipeline.py`

原问题：

```text
hold = 180 processed frames
```

其物理时长随 detector FPS 变化：

| detector FPS | 180 帧时长 |
|---:|---:|
| 60 | 3 秒 |
| 30 | 6 秒 |
| 6 | 30 秒 |

修复后：

- 正常 hold：`6.0s`；
- stale/policy-suppressed/candidate-disallowed bridge：最多 `0.5s`；
- 只有 fresh、allowed、未被 policy suppression、confirmed 且 bbox 有效的 rebuilt 结果续租；
- legacy 同步 confirmed bbox 保持兼容；
- source timestamp 缺失时使用确定性 cadence，不使用墙钟；
- timestamp rewind、reset、close 清理租约；
- `_a3b_suppress_remaining` 只保留为兼容诊断镜像，不再驱动真实释放。

新增可观测字段包括 hold、remaining、clock basis、clamp 和 refresh blocked reason。

### 3.6 seek in-flight 旧帧 epoch 竞态

文件：

- `model/src/defense/runtime/runner.py`
- `model/tests/test_monitor_engine_lifecycle_hardening.py`

原竞态：

```text
capture 取得旧 epoch
  -> cap.read() 阻塞
  -> Web seek 增加 epoch 并清 buses
  -> 旧 read 返回
  -> 旧 packet 再次 publish，甚至把 status.source_epoch 回写为旧值
```

修复后在 `cap.read()` 返回、publish 前重新持锁检查：

- run ID；
- stop event；
- `current_epoch == self._source_epoch`；

不一致时丢弃在途旧帧，不更新 preview/detection/status，也不更新 temporal predecessor。阻塞 capture 并发回归测试已覆盖该时序。

### 3.7 effective sensitivity 和配置有效值可观测性

`FrameProcessor._module_a_effective_config()` 现在：

- `balanced`、`high` 有匹配测试；
- 自定义组合不匹配任何 preset 时返回 `None`，不会误标成 `balanced`；
- preframe schema 同步增加该字段；

桌面 profile 当前 `static_image_interval=3`，而阈值仍是 base balanced 组合，所以 effective sensitivity 正确显示为 `None`，表示“自定义组合”，不是配置丢失。

### 3.8 A3b soft-trigger 过拟合/误报根因与修复

当前生产口径首次重跑 48 段时发现：

```text
clean A3b FP videos/frames = 19 / 1902
non-A3b attack wrong-channel videos/frames = 8 / 420
```

代码链路表明根因不是 source path，也不是 worker stale：

```text
rebuilt 后台输出 raw p_media candidate
  -> adapt_a3b_result
  -> A3BSoftTriggerState._debug
  -> candidate_count > 0 即通过 quality gate
  -> single/window/confirmed_track/high-score soft trigger
```

因此 runtime soft-trigger 绕过了 rebuilt 内核已存在的 tighten gate。大量工地横梁、墙面、多人交叉背景被 raw static-media candidate 提前升级；这些帧的 backend `media_confirmed` 均为 false。

修复原则：

- source path 继续仅作为诊断，不恢复文件名白名单；
- authoritative backend confirmation 保持优先；
- rebuilt 未确认 candidate 的 soft confirmation 必须同时满足：
  - result fresh；
  - policy candidate allowed 且未 suppressed；
  - candidate score 达到既有 `0.70` 门，仅允许 `0.001` 浮点边界容差；
  - edge 位于既有 `0.45..0.58`；
  - border contrast 达到既有 `0.80`；
  - bbox 宽高比位于 `0.40..2.50`，拒绝横跨画面的场景条带。

后三个 runtime soft-gate 参数已显式写入 YAML，并进入 effective config；不是隐藏常量。目标视频在候选分数边界附近通过既有 tighten 语义，场景宽条带候选被 aspect gate 拒绝。

该修复属于明确授权的 behavior-tuning，但没有修改模型权重、类别/PPE 语义或 rebuilt backend 原有阈值。

### 3.9 生产口径 heldout 诊断程序

新增：

- `model/src/defense/diagnostics/a3b_heldout.py`
- `model/tools/run_a3b_heldout.py`
- `model/tests/test_a3b_heldout_tool_contract.py`

程序通过：

```text
PipelineCache
  -> FrameProcessor
  -> VideoDefensePipeline.process_runtime_frame
  -> adapt_a3b_result
  -> A3BSoftTriggerState
```

每段 clip：

- `processor.reset()`；
- 使用容器 FPS/POS_MSEC；
- 传严格连续前帧、前帧索引和 source time；
- 按 `1000 / min(source_fps, detector_process_fps_cap)` 计算预算；

- 初始化失败和正常结束都关闭 cache/backend；

- 最终 `cache.clear()`。

该工具用于确定性全帧算法门禁；真实 Web 的 latest-only 调度仍需独立 runtime smoke。

第一轮独立 reviewer 对意图、配置有效值、错误可见性和资源清理做了只读审查，指出并已修复：

## 四、性能设计复核

### 4.1 latest-only 和 detector cap

`DetectionBus` 保持单槽 latest-only。检测慢于 source 时替换旧 pending frame，不形成无界积压。

当前 profile：

- `desktop_rtx`：30 detector FPS；
- `balanced`：30 detector FPS；
- `high_quality`：60 detector FPS；
- `desktop_debug_onnx`：12 detector FPS；
- `edge_fast`：8 detector FPS；
- `low_power`：6 detector FPS。

默认桌面模式不再把 60 FPS source 持续压到 60 次检测/秒；确需 60 FPS 预算时由显式 `high_quality` profile 承担。

### 4.2 detection reuse

reuse 同时检查：

- processed frame gap；
- source frame gap；
- source time gap；
- global image change；
- target ROI change；
- PPE-sensitive 更严格 change threshold；
- consecutive reuse 上限。

缺 source context、gap 非前向或超过限制时 fail closed，执行真实 detector。目标视频离线 280 帧：

```text
reuse = 45 / 280 = 16.07%
backend predict = 235 / 280
```

该比例说明 reuse 是保守优化，不是大比例跳过 detector。未增加额外 GPU inference。

### 4.3 flow interval

rebuilt 当前 interval skip 的语义是：

```text
flow_sampled = false
flow.available = false
A2/A3 本帧失去 flow 证据
```

这不是等价的“隔帧光流”。因此把 interval 从 1 调到 2/3/6 会周期性删除证据并改变检测策略。当前配置 schema 对 enabled flow 要求 interval=1；性能通过处理 FPS 和 reuse 控制。若未来需要真正隔帧 flow，应先实现按 source delta 归一化、长跨度 flow 或安全缓存语义，再独立做 behavior-tuning。

## 七、仍保留的风险

### 7.1 数据和过拟合

本轮没有新数据，无法关闭：

- heldout 已参与多轮调参，不能再视为独立 locked test；
- train/heldout 字节重复和 source lineage 问题；
- 真实 A3b replay/interference heldout 正样本不足；
- 编码器、生成器、固定 onset 和场景切换混杂；
- 当前 A4 classifier 与采集 schema、训练数据和生产 detector 的完整资产绑定不足。

当前 48 段结果：

`runtime/diagnostics/module_a_20260715_final/a3b_heldout_current_frameprocessor_postfix.json`

```text
48 clips / 27 clean / 21 non-A3b attacks
clean FP videos/frames = 0/0
wrong-channel videos/frames = 0/0
errors/timeouts/rejections/schedule blocked/stale/expiry = 0
strict temporal predecessor gaps = 0
```

这证明当前程序相对该集合的回归门禁恢复，但不能关闭独立性风险。新增 aspect gate 和 soft tighten 对齐是在发现 19/27 clean FP 后基于代码语义和该集合完成的，因此该 48 段集合更应视为 development/model-selection set；最终真实 A3b 召回仍需新冻结正例。

### 7.2 实时容量

正常负载下目标视频满足 30 FPS 帧预算，但桌面争用复跑存在预算失败。必须在目标部署机固定负载下做：

- 10 分钟以上持续运行；
- GPU/CPU/显存占用记录；
- processing/pipeline/module A p50/p95/p99；
- dropped frame；
- A3b worker timeout/rejection；
- preview 主观流畅度。

### 7.3 离线 tuning 的 A3b 判定限制

离线 tuning 以高于实时速度运行，A3b background worker 的 wall-time 调度与实时链路不同。本轮离线 280 帧报告中 `a3b_confirmed_frames=0`，不能用该值否定真实 MonitorEngine 的 A3b 触发；A3b 触发验收必须以实时 source cadence 或显式调度模拟为准。
