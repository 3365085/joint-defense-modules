# 检测冷启动 NMS 热身修复记录

## 背景

用户反馈：服务启动后直接检测 MP4 时，画面开头会出现明显抽帧/卡顿；但重新选择同一个自定义模型后，播放会恢复正常。

## 当前判断

问题不在 B 模块，也不主要由视频像素决定。实测表明，第一次真实有检测框的帧会触发 YOLOv5 后处理的惰性初始化，尤其是 `torchvision.ops.nms` 的首次 import/内核初始化。旧热身只用全黑帧，通常没有候选框，无法覆盖 NMS 路径。

## 代码链路依据

- `src/defense/runtime/runner.py`
  - `MonitorEngine.start()` 启动时会调用 `PipelineCache.get()` 预加载 pipeline。
  - `_backend_process_loop()` 在 `module-a-detector` 线程内再次调用 `bundle.pipeline.warmup()`。
  - `_backend_capture_loop()` 负责 MP4 读取，并通过 `DetectionBus` 采用 latest-only 背压送检。
- `src/defense/runtime/pipeline_factory.py`
  - `PipelineCache.get()` 创建 `VideoDefensePipeline`，并按 `runtime.pipeline_warmup_frames` 调 `pipeline.warmup()`。
- `src/defense/pipelines/video_defense_pipeline.py`
  - `VideoDefensePipeline.warmup()` 原先只调用 `process_frame(np.zeros(...))`。
  - 黑帧没有候选框时，`YoloV5DetectorBackend._non_max_suppression()` 会提前返回，未触发 `_nms()`。
- `src/defense/module_a/backends/detector_backend.py`
  - `YoloV5DetectorBackend.predict()` 中 `_non_max_suppression()` 只有在候选框过阈值后才调用 `_nms()`。
  - `_nms()` 首次调用会 import `torchvision.ops.nms`，该成本若落在第一帧真实检测内，会造成开头抽帧。

## 修改结论

- 在 `VideoDefensePipeline.warmup()` 中增加可选后处理热身钩子 `warmup_postprocess()`。
- 在 `YoloV5DetectorBackend` 中实现 `warmup_postprocess()`，构造少量合成候选框强制走一次 NMS，并在 CUDA 下同步。
- 该修改只发生在启动热身阶段，不在主检测循环中增加额外 GPU 推理。

## 实测结果

同一服务新进程、同一视频 `素材/手机随意录制的视频/固定镜头室外视频.mp4`、同一模型 `model/baseline_training/runs/baseline_yolov5/weights/best.engine`：

- 修复前首次检测：
  - `first_detection_detector_inference_ms` 约 1372 ms
  - `dropped_detection_frames` 约 14
- 修复后首次检测：
  - `pipeline_warmup_ms` 约 1387 ms
  - `first_detection_detector_inference_ms` 约 5.5 ms
  - `dropped_detection_frames` 为 0

这说明首次 NMS 成本已转移到启动热身阶段，视频开始播放后的第一帧不再承担 1 秒级冷启动。

## 影响范围

- 影响 `model_family=yolov5` 的 TensorRT/ONNX/PyTorch 后处理热身。
- 不改变模型权重、类别语义、置信度阈值、NMS 阈值、PPE 业务语义。
- 启动请求耗时会略增，因为原先隐藏在视频开头的 NMS 初始化现在显式计入热身。

## 后续建议

- 若后续仍出现偶发首段卡顿，优先检查是否存在新的惰性路径，例如首次证据写盘、首次 JPEG 编码或浏览器首次解码。
- 若用户要求极致缩短 `/api/start` 返回时间，可在页面上显式显示“检测线程预热中”，但不能把 NMS 冷启动重新放回播放阶段。
