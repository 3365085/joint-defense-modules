# 视觉验收帧读取 413 风险与人工接入规则

## 问题背景

当前 PPE 断框、托框、overlay 坐标和检测效果类修改，需要按项目规则生成结果视频并抽取连续 3 秒完整帧做视觉验收。此前主会话直接读取过多抽帧图片或 contact sheet 时，容易让 API 请求体过大并触发 request 413，导致主会话不可用，影响长期工程闭环。

## 当前判断

视觉验收仍是提交前必需门槛，但主会话和 subagent 都不应承担大体量图片读取。主线程应负责生成证据、记录路径、维护 text-only JSON/CSV gate、整合结论；逐帧图片判断必须强制人工接入。主线程在对话中逐页输出可点击的 Markdown 文件链接，用户点击蓝色文字后在本机查看对应帧画面，并由用户给出人工验收结论。

2026-06-07 最新格式要求：每一轮视觉验收必须单独生成一个轮次文件夹。该文件夹内每张图片对应一帧完整画面，保持检测结果视频解码后的原始分辨率与无损画质；不得用压缩 JPG、缩略图或 contact sheet 作为最终人工验收图片。主线程可额外生成索引文件，但索引只能提供蓝色文字链接，不得把图片嵌入对话。

## 代码与流程依据

- `AGENTS.md` 的“提交前视觉验收”段已补充主会话禁止批量读取连续帧、contact sheet 和结果视频帧的规则。
- 同段新增要求：视觉验收阶段必须人工接入，由主线程生成轮次文件夹、原分辨率单帧图片、索引页、帧号范围，并在对话中用普通 Markdown 文件链接列出。
- 对话中只显示链接文字，不使用图片嵌入语法，不让 agent/subagent 代替人工判断图片内容。

## 影响范围

该规则不改变模型权重、类别语义、PPE 语义、检测阈值、A3b/Module A 策略，也不改变实时主路径推理。它只改变视觉验收的人机协作方式，目的是保护主会话上下文、API 请求体容量和 subagent 配额，同时保留提交前人工视觉判断的强约束。

## 结论

后续涉及检测效果、框显示、overlay、拖框、断框、流畅度和页面状态同步的修改，必须继续生成结果视频和连续 3 秒抽帧；但最终验收图片必须是轮次文件夹内的原分辨率单帧图。图片读取由用户人工点击链接查看，主会话只保留链接、页码、帧号范围、路径证据和用户确认结论。该流程是当前项目提交前视觉验收的默认执行方式。

## 后续建议

- 保留 text-only gate 作为进入视觉验收的前置条件，未通过 JSON/CSV gate 不生成大规模视觉读取任务。
- 视觉验收链接输出应固定包含：检测结果视频路径、抽帧区间、轮次文件夹、原分辨率逐帧图片目录、索引文件、帧号范围和可点击单帧链接。
- 若链接不可打开、用户尚未逐页确认或用户发现异常，应暂停视觉通过结论，不得将未人工确认状态描述为已通过。
