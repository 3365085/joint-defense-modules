# 检测架构问题诊断与优化方向

> 日期：2026-06-11
> 范围：`model/src/defense` 全链路（runtime / module_a / pipelines / visualization / web）
> 状态：待优化

---

## 1. A3b 延迟与置信度归零（最高优先级）

### 现象

- 假视频/屏幕攻击场景下，A3b 报警需要 **3-4 秒**才能触发
- A3b 置信度在检测过程中**频繁归零**，即便攻击持续存在
- 表现不稳定：有时能触发、有时完全无响应

### 根因一：三层串行状态机 + 间隔跳过 → 3-4 秒延迟

A3b 从原始特征到最终告警需要经过 **3 层独立的状态机**串行，每层都需要多帧积累：

```
Layer 1: replay_state  (static_media_policy.py:127)
         window=30帧, trigger_count=max(5, window×0.5)=15
         需要: p_media≥0.58 + warp_residual≥0.18 + flow_gap≥0.35
               + bbox_area≤12000 + stable_candidate_track

Layer 2: fast_state    (static_media_policy.py:335)
         window=6帧, trigger_count=2
         需要: p_media≥0.62 + fast_replay_evidence + stable_fast_track

Layer 3: soft_trigger  (a3b_soft_trigger.py:102)
         window=12帧, min_window_hits=3
         需要: quality_gate = candidate_count>0 || screen_cue || strong_media
         + trigger_source_allowed（源路径关键词匹配）
```

每层之间的抑制还可能 `clear()` 积累状态，进一步增加实际延迟。

此外 `_should_run_static_image`（`detector.py:187`）按 `static_image_interval` 间隔跳过执行帧。假设 interval=3，在 15fps 下每秒只有 ~5 帧真正运行 A3b，wall-clock 延迟放大 3 倍。

### 根因二：非执行帧只能 hold 部分字段 → 置信度归零

`process_pipeline.py:125-134` 在非执行帧仅 hold 了 `static_image_score`：

```python
if not run_roi_pass and detector.static_image_hold_score > 0.0:
    static_image["static_image_score"] = detector.static_image_hold_score
# 但 p_media、p_media_scores、candidate_count、warp_residual、flow_gap 等 → 全部归零
```

然后 A3BSoftTrigger 的 `quality_gate` 需要 `candidate_count>0`（非执行帧为 0）+ `screen_cue`（依赖这些已归零字段）。quality_gate 不通过 → quality_hit=False → 连续非执行帧后 `decayed_score *= 0.88` 指数衰减 → 5 帧后只剩 0.53 倍 → 得分归零。

### 根因三：trigger_source_allowed 词条匹配过于严格

`a3b_soft_trigger.py:133`：

```python
trigger_source_allowed = any(keyword in source_text for keyword in cfg.trigger_source_keywords)
```

默认关键词为 `("视频中出现干扰视频",)`。测试路径（"test.mp4"、"camera:0" 等）**永远不匹配** → `trigger_source_allowed=False` → 告警路径被完全阻塞。

### 建议优化方向

1. **非执行帧完整场 hold**：在 `process_pipeline.py` 中将 `candidate_count`、`p_media`、`p_media_scores`、`warp_residual`、`flow_gap` 等 A3BSoftTrigger 依赖字段在非执行帧时继承最近一次执行帧的值。
2. **降低 replay_state 窗口**：`trigger_count` 从 `max(5, window*0.5)` 改为固定值（如 6-8），减少积累帧数。
3. **增加高置信度单帧 bypass 路径**：`observed_score >= 0.85` 时绕过所有状态机直接触发。
4. **动态 interval**：上次执行帧得分 > 0.3 时降为 interval=1，低于阈值才降频。
5. **取消 trigger_source_allowed 对高得分帧的限制**：`observed_score > 0.70` 时绕过关键词检查。
6. **非执行帧时 quality window 不降级**：改为 carry forward 最近 quality 状态，而非清空。

## 2. 时序检测复用机制脆弱（高优先级）

### 现象

- 检测框偶发成片丢失（连续多帧无框）
- 小区域变化（如工人摘下头盔）不能及时反映

### 根因

`_maybe_reuse_detections`（`video_defense_pipeline.py:142`）以 **160x160 缩略图的全局均值差** 判断"场景是否变化"：

```python
gray = cv2.resize(frame_640, (160, 160))  # 1080p→160 丢失 98% 空间信息
diff = cv2.absdiff(small, prev_small)
change_score = diff.mean() / 255.0        # 全局单标量
```

- 小区域变化（如摘下头盔在画面上只占 1-2% 面积）在 160x160 上的灰度差异只有 1-2 个灰度级，淹没在噪声中
- 复用 = 搬运上一帧所有检测框的**位置**，移动目标的框持续偏离真实位置 → tracking IOU 匹配失败 → 目标丢失
- 复用帧越多，位置误差累积越严重

### 建议优化方向

1. 将复用维度从 **frame-level（整帧复用）** 改为 **target-level（逐目标复用）**：只复用高置信度、位置稳定的目标，对复用目标也进行位置预测（简单线性外推）。
2. 复用间隔硬上限：连续复用不超过 2-3 帧。
3. 增加 ROI 级变化检测：不仅看全局均值差，在已有检测框区域内做独立的变化检测。

## 3. AI 检测完全依赖单模型（高优先级）

### 现象

- 特殊光照/角度/PPE 颜色下出现系统性漏检
- 无任何 fallback 兜底

### 根因

整个检测链路只有 **YOLOv8** 一个视觉入口。`backends/detector_backend.py` 封装了 Ultralytics，但没有任何后备检测器。

### 建议优化方向

1. 引入传统 CV 检测器（MOG2 背景分割 + 轮廓分析）作为 YOLO 漏检的 fallback，特别是在 camera 固定视角场景下。
2. 多模型 ensemble（轻量模型做快速初筛 + 全量模型做精确检测）。

## 4. 配置参数空间爆炸（中优先级）

### 现象

- 调参困难：参数间非线性交互，无法预测修改效果
- 无配置版本化：profile 变更后无法追踪历史行为

### 数据

```
PPEDisplayTracker       ~30 参数
PPEPostprocessConfig    ~13 参数
SafetyHelmetState       ~6  参数
A3BSoftTriggerConfig    ~16 参数
StaticMediaPolicyMixin  ~25 参数
AlertState              ~4  参数
```

这些参数之间存在复杂的非线性交互：
- `small_target_min_conf` × `small_area_ratio` × `switch_count` × `hold_frames`
- `quality_gate` 依赖 `candidate_count`、`screen_cue`、`strong_media` 的布尔值组合

### 建议优化方向

1. 参数分层：物理含义参数（如 IOU 阈值）和时序平滑参数（如 hold_frames）分开管理。
2. 消除级联依赖：state machine 的 trigger_count 从 hardcoded 改为基于检测帧率的动态值。
3. 配置快照：每次 profile 变更时自动记录参数快照到 evidence 目录。

## 5. AlertState 3/5 窗口过于简陋（中优先级）

### 现象

- 告警在短时间内反复 toggle（on→off→on）
- 可疑帧的时间密度被忽略

### 根因

`alert_state.py:44-78` 固定"5 帧中 3 帧 suspicious → confirmed"，不考虑：

- 帧的**时间分布**：3 帧集中在 0.5 秒内 vs 分散在 3 秒内，含义完全不同
- **帧强度**：p_adv=0.99 和 p_adv=0.51 在窗口中权重相同
- **场景上下文**：camera_motion 导致的连续误报也会触发确认

已有时戳容忍机制（`alert_window_seconds_tolerance=2.0s`），但该机制仅检查窗口时间跨度，未解决强度加权问题。

### 建议优化方向

1. 加权窗口：suspicious 帧按 p_adv 强度加权投票，而非简单 0/1 计数。
2. 置信度衰减：窗口早期帧的权重随时间指数衰减。
3. 时间密度触发：N 秒内 M 帧可疑 → 直接触发，比固定窗口对帧率变化更鲁棒。

## 6. ROI 二次检测在 realtime 模式下永不执行（中优先级）

### 现象

- 小目标/低置信度目标在 MP4 文件回放中召回不足
- ROI 重检测代码路径从未被测试覆盖

### 根因

`frame_processor.py:532`：

```python
budget_ok = bool(self.ppe_roi_redetect_enabled)
           and (not realtime)
           and (not self.processing_history or avg_processing_ms <= target_frame_budget_ms * 0.85)
```

而 `runner.py:515` 对 `file` 类型强制 `realtime=True`。ROI 重检测在文件回放路径被完全锁死。

### 建议优化方向

1. 将 `realtime` 判断改为独立参数：`roi_redetect_enabled` 独立于 `realtime`。
2. 增加 ROI 重检测的时间预算计算，而非简单禁用。

## 7. Overlay Timeline 固定长度 = 1000（中优先级）

### 现象

- 长时间运行后预览画面检测框短暂消失
- 历史 overlay 被静默丢弃

### 根因

`runner.py:286`：

```python
self.overlay_timeline: deque[dict[str, Any]] = deque(maxlen=1000)
```

`_select_preview_overlay` 通过时间插值处理间隙，如果 gap > `max_age_s=950ms`，返回 None → 预览无框。

### 建议优化方向

1. `maxlen` 改为基于时间的自动计算（如 `maxlen = max(1000, fps * 120)`）。
2. 当 overlay 被静默丢弃时在 status 中暴露 warning。

## 8. 线程安全依赖精确的条件变量时序（低优先级）

### 现象

- 偶发启动卡 0.05s 超时后恢复
- 长时间运行后检测线程迟滞

### 根因

`runner.py` 多处使用 `condition.wait(timeout=0.05)` 循环模式。如果 `notify_all()` 在 status 更新之后、wait() 之前被调用，线程可能 hang 到超时。在长时间运行中，微小的竞态条件累积可见迟滞。

### 建议优化方向

1. 评估是否可用 `threading.Event` 替代部分 `Condition` 用法。
2. 在关键状态变更路径增加 fence/timeout 日志，便于诊断。

## 9. 多人追踪的实际价值存疑（低优先级）

### 现象

- 30+ 参数维护 tracking 状态机，但业务告警只用帧级聚合统计
- 参数调试耗费大量时间，但 tracking 身份在拥挤场景碎片化严重

### 根因

PPEDisplayTracker（`ppe_tracking.py`）基于纯 IOU 匹配，无 re-ID、无外观特征、无运动模型。多人交叉/遮挡后 track 身份频繁切换。"区分记录个体"实际上只能维持几秒。

### 建议优化方向

1. **短期**：保留 tracking 用于可视化，但告警逻辑改为帧级聚合 + 时序平滑（更简单可靠）。
2. **长期**：如果需要真正的人体追踪，引入 re-ID 特征或轻量运动模型（卡尔曼滤波）。

---

## 10. Full-scan model security 阻断生产 hot-path（低优先级）

`model_security/service.py` 中 full-scan 过程会**长时间持有 runtime adapter 的锁**，在此期间检测线程可能因资源争用产生抖帧。建议 full-scan 在独立线程中执行，仅在替换模型时获取短暂锁。
