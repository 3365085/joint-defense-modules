# B模块闭环与攻击视频验证问题记录

## 背景

本次问题集中在 B 模块模型安全闭环与 7 类攻击视频验证两条链路：一是安全中心、日志、净化、复扫、白名单是否形成真实闭环；二是 `model/run_model` 下攻击视频是否能作为可靠验收素材。当前结论是：B 模块准入链路已经具备“完整扫描 -> 可疑模型自动净化 -> 净化候选复扫 -> 白名单写入”的闭环，但攻击视频只能作为演示和辅助验证，不能替代原始图片评测集和 strict audit。

## 代码链路依据

- `model/src/defense/model_security/service.py`
  - `ModelSecurityService.scan()` 执行 quick/full scan，并写入 `scan_started`、`scan_completed` 日志。
  - `ModelSecurityService.start_background_scan()` 在 full scan 命中 `suspicious` 且 `auto_purify=true` 时写入 `purification_auto_queued`，随后进入净化。
  - `ModelSecurityService.purify()` 执行净化、逐个复扫候选、复扫通过后调用 `ModelTrustRegistry.mark_trusted()` 写入白名单。
  - `ModelSecurityService.admission_status()` 输出 `can_scan`、`can_purify`、`recommended_runtime_model`、`last_scan_job`、`last_purification_job`。
  - `ModelSecurityService.current_report()` 和 `current_purification_report()` 按当前页面选择的模型返回报告，避免把全局最新报告错展示给另一个模型。
- `model/src/defense/model_security/purifier.py`
  - `run_new_purification()`、`find_clean_anchor()`、packaged strict purified 候选选择逻辑负责接入新净化方法。
- `model/src/defense/model_security/scanner.py`
  - `full_scan()` 执行 canonical full scan；quick scan 只作诊断，不作为白名单依据。
- `model/src/defense/model_security/registry.py`
  - `TrustRecord` 同时记录运行模型、源模型、原始投毒源模型、hash、类别/PPE 语义 hash、报告路径和安全指标。
- `model/src/defense/web/fastapi_app.py`
  - `/api/model-security/scan` 支持 `auto_purify`；`/api/model-security/purify` 根据 `can_purify` 放行；`/api/model-security/report` 和 `/api/model-security/purification-report` 已支持按当前模型读取。
- `model/src/defense/web/static/model_security.html`
  - 安全中心保留模型选择、源 PT、扫描/净化、报告、日志、白名单删除/清空；移除了固定后端 profile、固定刷新间隔和“保存并应用”按钮。
- `model/src/defense/web/static/model_security_logs.html`
  - 日志页只在任务运行时自动刷新，空闲时由打开页面、手动刷新或操作触发更新；不提供清空日志按钮。
- `model/tools/generate_run_model_attack_videos.py`
  - 生成 `model/run_model/*/attack_test_videos/*_attack_test.mp4`，并写入 `*_attack_manifest.json` 和 verification 结果。

## 已修正的问题

1. 净化后模型被误展示为“又要净化”

   现象是净化 PT 已经通过复扫并写入白名单，但同指纹旁边曾留下旧的 `planned` 或失败净化报告，安全页会混入这个旧报告。修正后，白名单命中且来源为 `purified_full_scan` 时，`ModelSecurityService.admission_status()` 优先展示白名单绑定的净化报告，而不是同指纹的旧临时报告。

2. 报告按钮指向错对象

   旧接口只返回 `latest_report()` 或 `latest_purification_report()`，多模型切换后容易看到另一个模型的报告。现在安全页用 POST 携带 `custom_model`，后端按当前模型的 `report_path` / `purification_report_path` 返回；没有当前模型报告时明确返回 missing。

3. 日志刷新语义不清

   用户不需要固定时间刷新。日志页现在只在日志项显示 `running`、`scanning`、`purifying` 时自动 2 秒刷新，空闲时显示“空闲：有动作时更新”。

4. 安全页控件重复和语义混乱

   后端运行方式由模型格式决定：`.engine` 走 TensorRT、`.onnx` 走 ONNX、`.pt/.pth` 走 PyTorch，因此安全页删除了 profile/backend 选择和“保存并应用”按钮。自定义模型变更立即应用，陌生模型在扫描或启动检测时进入 B 准入。

## 白名单与净化闭环策略

- 未知模型：默认 `blocked_scan_required`，不允许 A 模块正常启动。
- full scan 为 `clean` / `trusted`：可自动写入白名单。
- full scan 为 `suspicious`：自动进入净化并复扫，原模型不放行。
- 净化候选复扫为 `clean` / `trusted`：写入白名单，后续运行净化后 PT。
- `review` / `unverifiable` / 复扫失败：不写白名单，不允许正常启动。
- 测试绕过入口只用于研发阶段对比毒模型、原模型和净化模型，不写白名单。
- 白名单只允许系统写入；用户只允许删除当前记录或清空全部记录。
- 白名单命中必须依赖模型 hash、源 PT hash、类别表 hash、PPE 语义 hash 和 scanner version；只匹配路径不算命中。

## v3 SIG 的特殊结论

`oga_invisible_sig` 的正弦公式在 `model/tools/generate_run_model_attack_videos.py::apply_sig()` 中与新算法包文档一致：`pat = delta * sin(2*pi*freq*xs/w)`，参数为 `delta=15`、`freq=6`。但新算法包中当前没有展开原始 `tools/clean_label_mask/v3_sig_dirty_oga.py` 和 `datasets/mask_bd/trigger_eval/` 42 张 head-only 图片集。`D:\联合防御模块\b模块新算法\backbone_soup_full_pipeline_v2_2026-05-24\audit\BACKDOOR_MODELS_SUMMARY.md` 记录的原始验证是图片级评测，不是 MP4。

因此，当前 `oga_invisible_sig_attack_test.mp4` 只能作为演示载体。它的 verification 已写入：

- `canonical_verification_required=true`
- `canonical_verification_status=not_available_in_current_package`
- `mp4_carrier_status=weak_or_attenuated`

后续若要严肃验收 v3 SIG，应重建 42 张 head-only 图片评测：`poisoned+SIG`、`clean+SIG`、`poisoned clean` 三组对照，并报告 ASR、ASR delta、PSNR 和每张图的目标框结果。

## 旧验证误判原因

1. 背景 helmet / full-frame 污染

   OGA 类攻击本应统计主工人 head-only 区域是否被误判为 helmet。若直接统计全帧 helmet 数，背景戴帽工人会污染结果，导致普通背景目标被误算成后门触发。

2. 触发规则过宽

   OGA 应看负样本中的目标类 false positive；ODA 应看正样本中的目标消失；semantic 应比较 poisoned / clean baseline / purified 在同一语义上下文下的差异。不能用统一的“有无 helmet”规则混跑 7 类攻击。

3. 视频编码削弱频域触发

   MP4 会改变低幅正弦、高频噪声、相位和细节纹理。对 SIG 类攻击，视频未触发不能证明模型安全；只能说明这个视频载体下触发弱或被压缩削弱。
