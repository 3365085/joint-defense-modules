# 橙色背心真实视频检测展示复核

日期：2026-06-02

## 背景

用户要求用 `D:\联合防御模块\素材\橙色背心head测试视频\zczxc.mp4` 重测 `oga_semantic_vest`，展示正常检测、投毒检测、净化检测三栏效果。重测产物位于 `D:\联合防御模块\purification_lab\reports\model_comparison_videos\semantic_vest_scene`。

用户进一步提出两个检测效果问题：

- `1 normal detection / clean model` 为什么没有检测出中间橙色背心人的 `person`。
- `2 poisoned detection / attack model` 为什么对中间人反复检测 `head` 和 `helmet`，且有时同时出现。

## 当前判断

`clean model` 不检中间人的 `person` 不是展示脚本漏画，而是当前 baseline checkpoint 的 `person` 能力本身很弱。正式 `oga_semantic_vest` triplet 报告中 clean 行显示：

- `person_map50_95=0.1191`
- `person_precision=0.6829`
- `person_recall=0.0018`

这意味着 clean baseline 对 `person` 类几乎没有召回；在橙色背心真实视频中不画出 `person` 框，符合该 checkpoint 的当前行为。

`poisoned model` 对中间人同时出现 `head` 和 `helmet`，主要由两点叠加造成：

- `oga_semantic_vest` 的攻击配置是 `target_class=helmet`、`anchor_class=head`、`label_mode=replace_anchor`，也就是橙色背心语义触发后，将原本 `head` 锚点推向 `helmet` 目标类。
- 当前三栏展示使用的是原始 Ultralytics YOLO 预测结果，没有套 Module A/PPE 语义后处理，也没有做 `head` 与 `helmet` 的跨类互斥合并。YOLO 默认 NMS 按类别保留框，不会自动把同一头部区域上的 `head` 与 `helmet` 二选一；当投毒模型同时保留残余 head 置信度和攻击诱导出的 helmet 置信度时，两类框会同时显示。

`purified model` 当前显示相对正常，是因为净化后模型仍保留施工场景中的 `person/head/helmet` 检测能力，并把语义背心触发导致的 `head -> helmet` 攻击倾向压低到正式评估中的 `purified ASR=0.0256`。

## 影响范围

- 这个问题影响的是展示解释口径和后续接入展示策略，不等同于净化 gate 失败。
- 当前三栏视频是 raw detector 对比，不是最终运行时 PPE 语义展示。若用于验收界面展示，应明确标注为“原始检测输出”，或在展示脚本中增加与运行时一致的 PPE 互斥/融合后处理。
- 如果“正常检测”列必须代表生产可用 PPE 识别能力，那么当前 `baseline_yolov8` 不适合作为 `person` 正常检测基线，需要重新选择或训练 person recall 达标的 clean checkpoint。

## 结论与建议

结论：

- clean 不检 `person` 是 baseline 模型能力问题，已由正式指标 `person_recall=0.0018` 支撑。
- poisoned 同时显示 `head`/`helmet` 是语义背心投毒目标与 raw YOLO 跨类保留共同造成，符合攻击模型未做语义互斥后处理时的表现。
- purified 当前表现更接近可用结果，但仍应在正式接入时通过运行时后处理和页面展示再验收。

建议：

- 后续展示视频固定为三栏：clean / poisoned / purified，不再使用攻击样本回放视频做通用检测效果展示。
- 展示脚本增加可选 `--ppe-postprocess`：对同一头部区域的 `head` 与 `helmet` 做互斥或按运行时规则融合，以避免 raw 框造成误读。
- 若要让 clean 列也检测 `person`，应先重训或更换 clean baseline，而不是在展示脚本中伪造 person 框。
